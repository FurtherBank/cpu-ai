# 多投影信息体系：探索草稿

> 本文从《WIP 信息架构演进-探索草稿》中拆出。
>
> 拆出原因：这些问题讨论的是「围绕同一对象，设计文档、源码产物、使用文档等多种信息载体如何相互依赖、相互校验、相互投影」。它不属于主问题中「在不改变既有设计和内容的前提下，降低当前信息体系总信息熵」的范围。

## 问题边界

这里讨论的不是单纯整理一堆既有信息，而是一个产品信息体系如何在多个产物之间保持一致、可追溯、可演进。

以 SDK 为例，至少存在这些信息状态：

- 设计文档：表达 SDK 为什么这样设计，承载设计意图、取舍、约束和未来规划。
- 源码产物：表达 SDK 实际如何运行，承载 API 签名、行为语义、实现边界和版本事实。
- 使用文档：表达 SDK 如何被用户正确接入和使用，承载任务路径、示例、公共承诺和使用约束。
- 信息基座：不是额外的一份物理文档，而是这些信息要素及其关系构成的逻辑全集。

## 信息基座不是物理文档

信息基座不应被理解为项目里必须额外维护的一份全集文档。

更准确的定义是：

> 信息基座是某个范围内，所有可作为判断依据的信息要素及其关系所构成的逻辑全集。它可以分散存在于源码、配置、测试、文档、Issue、提交记录、运行行为、团队约定中。

实际信息是载体，信息基座是逻辑结构。

例如 SDK 项目中，实际载体可能包括：

- 源码文件
- 类型声明
- 测试用例
- README
- API Reference
- 示例代码
- CHANGELOG
- 设计文档或 ADR
- Issue / PR 讨论
- 发布包和线上行为

信息基座不是复制这些载体，而是建立这些载体中信息要素的结构化关系：

- 能力有哪些
- API 是什么
- 行为语义是什么
- 哪些约束成立
- 哪些用法被支持
- 哪些边界不支持
- 哪些设计取舍存在
- 哪些事实由源码保证
- 哪些事实由文档承诺
- 哪些事实由测试验证

## 唯一事实源按信息要素划分

唯一事实源不应按产物整体划分，比如「源码永远是唯一事实源」或「文档永远是唯一事实源」。

更合理的是按信息要素划分：

- API 真实签名：源码 / 类型声明是唯一事实源。
- 用户可见行为：源码 + 测试是事实源。
- 用户公开承诺：发布后的使用文档是事实源。
- 内部设计理由：设计文档 / ADR 是事实源。
- 版本变更：CHANGELOG / release tag 是事实源。
- 安装入口：package 配置是事实源。
- 示例可运行性：示例代码 + CI 是事实源。

冲突时，判断也要按信息要素处理：

- 文档说支持，但源码不支持：对用户是文档契约违约，对实现是缺陷。
- 源码支持，但文档没写：可能是未公开能力，不一定能对外承诺。
- 设计文档说要做，但源码没做：这是未实现设计或实现缺口。
- 测试覆盖的行为和设计文档冲突：要判断测试是在固化历史行为，还是设计已过期。

所以信息基座的关键不是集中存储，而是建立：

```text
信息要素 → 所在载体 → 事实源优先级 → 依赖/校验关系 → 更新传播路径
```

## 四类要素的成立依赖

反馈关系可以很多，但如果目标是跑通「SDK 设计 → 编码实现 → 编写使用文档 → 用户正确使用」，成立依赖不能被抹平。

最小主链路是：

```text
需求 / 场景 / 约束 / 现状
→ SDK 信息基座
→ 设计决策 / 设计文档
→ 源码产物 / SDK 包
→ 使用文档
→ 用户正确使用
```

不可移除的依赖有三类：

- 设计必须依赖问题、场景、约束和对象模型。
- 实现必须依赖设计决策。
- 使用文档必须依赖真实可用的公共契约。

可以没有正式的设计文档文件，但不能没有设计决策。设计文档的价值是把设计决策显性化，使其可评审、可追溯、可传播。

可以从源码直接生成部分文档，但使用文档不能只依赖设计意图，它必须依赖最终对用户可见的事实：真实 API、真实安装方式、真实调用行为、真实错误表现、真实版本限制、真实示例可运行性。

## 四类要素的反馈关系

成立依赖是主链路，反馈关系是修正机制。

```text
                 ┌────────────┐
                 │  设计文档   │
                 │ 规划性写入  │
                 └─────▲───┬──┘
                       │   │
        读取已有基座   │   │ 写入设计决策
                       │   ▼
┌────────────┐     ┌────────────┐     ┌────────────┐
│  使用文档   │◄────│  信息基座   │────►│  源码产物   │
│ 消费性投影  │     │ 结构化状态  │     │ 事实性实现  │
└─────┬──────┘     └────▲───▲────┘     └─────┬──────┘
      │                 │   │                │
      │ 契约/校验写入    │   │                │ 事实写入
      └─────────────────┘   └────────────────┘
```

设计文档向信息基座写入「应该如何设计」。

源码产物向信息基座写入「实际如何运行」。

使用文档向信息基座写入「外部如何理解和依赖」。

信息基座负责把设计意图、实现事实、对外契约统一成一个可持续校验、可演进的结构化状态。

## 设计轴与使用轴

同一套 SDK 信息，至少同时存在两条结构轴：

```text
设计成立轴：核心原理 → 能力体系 → API / 参数 / 行为
使用理解轴：能力总览 → 接入任务 → 具体能力 → 入参确定 → 调用验证
```

这两条轴都合理，但它们不是同一条顺序。

设计侧由核心原理向下推导能力和契约：

```text
原理层 → 能力层 → 契约层
```

使用侧由用户任务组织能力和契约：

```text
任务层 → 呈现层
能力层 + 契约层 → 任务层
```

同一个「入参」在两侧的位置不同：

- 设计侧：入参是核心原理与能力边界推导出的契约结果。
- 使用侧：入参是用户完成某一步接入时必须填写的操作对象。

因此，设计原理不一定等于用户理解路径，但用户理解路径不能违背设计原理。

## 同源多轴组织

可以把这个模式命名为「同源多轴组织」：

> 同一信息基座中，设计轴按成立关系组织，使用轴按任务路径组织。两者不要求顺序一致，但要求事实一致、概念一致、边界一致，并且用户文档中的简化表达不能破坏原理层约束。

这个问题的核心不是「如何给既有信息降熵」，而是「同一对象在多个信息投影中如何保持一致、可追溯、可校验」。

## 完全可索引的实现模式

如果目标是在某个收束范围内，让信息基座全集完全可索引，不一定要把全集写成一份文档，而是要建立索引系统。

基本模式是：

```text
范围定义
→ 信息要素分类
→ 事实源登记
→ 关系索引
→ 查询入口
→ 一致性校验
```

以 SDK 的公开 API 使用面为例，需要做到：

- 每个公开 API 有唯一标识。
- 每个 API 能索引到源码定义。
- 每个 API 能索引到类型声明。
- 每个 API 能索引到测试用例。
- 每个 API 能索引到使用文档段落。
- 每个 API 能索引到示例代码。
- 每个 API 能索引到版本变更记录。
- 每个 API 能索引到设计理由或 ADR，若存在。

现实实现形态可以是：

```text
docs/
  guides/
  api/
  adr/

src/
  public-api.ts
  ...

tests/
  public-api/*.test.ts

examples/
  basic-usage/

metadata/
  api-manifest.json
```

其中 manifest 不复制全部内容，只记录索引关系：

```json
{
  "api": "createClient",
  "source": "src/createClient.ts",
  "types": "src/createClient.ts",
  "tests": ["tests/public-api/createClient.test.ts"],
  "docs": ["docs/api/create-client.md", "docs/guides/quick-start.md"],
  "examples": ["examples/basic-usage/create-client.ts"],
  "introducedIn": "1.2.0",
  "status": "stable"
}
```

这是一种「信息基座可索引化」，不是「信息基座全文档化」。

## 与信息降熵主问题的边界

本议题关注的是多种信息产物围绕同一对象如何协同成立和相互校验。

而信息降熵主问题关注的是：

> 在不改变既有设计和内容的前提下，如何通过分块、索引、归属、关系显式化、噪声压缩等方式，降低当前信息体系的总信息熵，使信息更容易被注意力机制正确处理。

因此，本议题应作为相邻问题保留，不应直接混入信息降熵主 SOP。
