# 优化维度清单

本清单用于后续设计完整信息优化 SOP 时参考。

## 通用优化维度

这些维度适用于大多数信息类型。

| 维度 | 检查问题 | 典型动作 |
| --- | --- | --- |
| 文档身份 | 这份信息是什么，不是什么？ | 改标题、补边界、移除越界内容 |
| 事实一致性 | 是否存在自相矛盾的事实承诺？ | 建事实承诺表、修正冲突 |
| 概念闭合 | 概念能否支持下一步判断？ | 补定义、边界、使用口径、消歧 |
| 信息归属 | 信息是否放在正确层级和位置？ | 上提全局约束、下沉局部细节 |
| 权威来源 | 规则由谁维护？运行时参考谁？ | 建权威来源表 |
| 关系显式化 | 来源、生产、消费、维护、校验关系是否清楚？ | 补关系说明 |
| 过时概念 | 当前设计中不存在的概念是否残留？ | 删除旧概念，不解释幽灵概念 |
| 来源保真 | 是否混入未标注推断？ | 标注来源、假设、待确认项 |
| 噪声压缩 | 是否有重复、旁支、无关解释？ | 合并、删除、折叠 |
| 表达清晰 | 是否有含混指代、空泛动词、跳跃表达？ | 改写、拆句、统一术语 |
| 可校验性 | 优化结果能否被检查？ | 补结构校验清单 |

## 信息架构四层的使用方式

信息架构四层的定义以上游探索草稿为唯一事实源。本文不重新定义四层，只记录在本专项中的检查用途：

- 用上游「索引层」检查入口、标题树和全局约束位置。
- 用上游「内容层」检查信息块纯度和粒度。
- 用上游「关系层」检查来源、消费、维护、校验等关系是否显式。
- 用上游「校验层」检查是否存在结构性验收信号。

## 领域特定优化

### 流程 / SOP 文档

重点检查：

- 阶段顺序。
- 触发条件。
- 跳过条件。
- 输入与输出。
- 完成判定。
- 异常路径。
- 执行主体。

典型问题：

- 只写「要做什么」，不写「何时做」。
- 只写正向路径，不写跳过和分支。
- 完成判定不可核对。

### 架构设计文档

重点检查：

- 模块边界。
- 权责分层。
- 单一事实源。
- 依赖方向。
- 状态流转。
- 兼容关系。

典型问题：

- 每个模块单独看合理，整体关系断裂。
- 局部设计覆盖全局约束。
- 一个概念在多个模块中有不同含义。

### 规则 / 规范文档

重点检查：

- 适用范围。
- 权威来源。
- 执行主体。
- 例外情况。
- 违规后的处理。

典型问题：

- 规则看似明确，但不知道谁执行。
- 提到外部规范但没有索引。
- 把建议写成强制，把强制写成建议。

### Schema / 契约文档

重点检查：

- 字段定义。
- 必填 / 可选。
- 生产方。
- 消费方。
- 兼容性。
- 示例是否完整。

典型问题：

- 字段有名字但没有语义。
- 示例和定义不一致。
- 没有说明谁消费这个字段。

### AI 工作流文档

重点检查：

- 主控还是 Subagent。
- 上下文如何传递。
- 哪些资料运行时可参考。
- 哪些资料只是维护设计资料。
- 何时执行、跳过、分支。
- 输出如何被下一步消费。

典型问题：

- 把维护资料当运行时依据。
- 只说明任务，不说明上下文边界。
- Subagent prompt 缺少硬依赖。

### 历史演进文档

重点检查：

- 当前概念和历史概念是否隔离。
- 旧概念是否仍有效。
- 是否需要解释历史，还是应直接删除。
- 迁移关系是否会污染当前规则。

典型问题：

- 为过时概念写解释，导致读者误以为它仍存在。
- 历史设计和当前设计混在同一规则层。

## 优化动作关系

推荐顺序：

```text
文档身份 -> 来源保真 -> 粗切块 -> 事实一致性 -> 概念闭合 -> 权威来源 -> 结构重排 -> 领域补齐 -> 表达整理 -> 结构验收
```

不建议顺序：

```text
表达润色 -> 总结压缩 -> 结构重排 -> 事实检查
```

原因：

- 润色会掩盖事实矛盾。
- 过早压缩会丢失来源。
- 结构重排如果早于概念闭合，可能把信息放到错误位置。
