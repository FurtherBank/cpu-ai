# 实践复盘

## 案例一：`task-step-schema.md`

### 初始问题

文档原题为「任务步骤四要素推导方法论」。

主要问题不是内容错误，而是文档身份不清：

- 看起来像实际推导流程。
- 实际需要表达的是每个工作流环节规格的四要素是什么。
- 文档和某个 skill 的关系不应在文中出现，因为它是客观定义型参考文档。

### 优化动作

1. **校准文档身份**
   - 标题改为「工作流环节规格四要素说明」。
   - 将「推导步骤」等过程化表述改为「应包含」「确认项」「判定口径」。

2. **统一术语**
   - 将「步骤」统一为「环节」。
   - 避免标题与正文概念打架。

3. **重构 `Execution Mode` 概念**
   - 从「主控执行或 Subagent 委派」升级为「执行模式控制」。
   - 补充触发条件、跳过条件、分支条件。
   - 再次确认执行主体仍是主轴：先判断主控还是 Subagent，再判断何时执行、何时跳过、如何分支。

4. **去除越界信息**
   - 不提具体 skill 关系。
   - 不写实际生成工作流时的推导过程。

### 抽取经验

- 标题会强烈影响读者对文档身份的判断。
- 定义型文档不应混入流程型指令。
- 概念扩展时要保留主轴，否则新增维度会稀释核心判断。
- 示例顺序也会塑造认知，不能只改定义，不改示例。

## 案例二：`development-mindset.md`

### 初始问题

文档承载了很多框架事实，但信息组织混杂：

- 公共约束分散在后文。
- dima / todo 的关系存在事实歧义。
- 仓库协议的消费方被泛化到框架层。
- 过时概念 `i-spec-execute` 被解释，反而制造新框架中的悬空概念。
- 很多细节规则其实属于对应 skill 或工具，不应由本文档定义。

### 优化动作

1. **公共约束上提**
   - 路径解析根。
   - `iteration-context.json`。
   - reqId / roundId。
   - 阶段完成判定。
   - `.iteration/` 目录含义。

2. **事实矛盾检查**
   - 发现「所有需求统一转 todo」与「进入统一迭代后的元数据落在 dima 或 todo」存在冲突。
   - 修正为：dima 和直接需求是两种来源的同类输入，进入迭代后都经 `i-todo` 形成统一 todo；dima 目录保留 resolve 状态与来源关联。

3. **权威来源归位**
   - `i-todo` 负责统一 todo 与需求类型路由。
   - `i-verify` 消费仓库协议。
   - stage-bound skill 的产物契约由其设计资料决定。
   - skill 运行时行为由可执行正文决定。
   - `iteration-ctl` 维护阶段状态与 CLI 行为。

4. **过时概念清理**
   - `i-spec-execute` 在当前框架中不存在，不需要解释。
   - 从框架文档中完全移除该概念。

5. **校验层补充**
   - 增加结构校验清单。
   - 将易退化的关键约束转为检查问题。

### 抽取经验

- 很多「文档不完备」不是因为缺少细节，而是缺少权威来源索引。
- 框架文档应定义全局事实源和流转关系，不应替具体 skill 定义运行细节。
- 历史概念如果在当前设计中不存在，应删除而不是解释。
- 「不讨论什么」也是重要边界，例如不定义哪些必须人工、哪些必须自动。

## 本轮实践暴露出的优化类型

| 优化类型 | 说明 | 案例 |
| --- | --- | --- |
| 文档身份校准 | 确认文档是什么，不是什么 | `task-step-schema.md` 从方法论文档校准为定义文档 |
| 信息组织降熵 | 调整顺序、分块、归属、索引 | `development-mindset.md` 上提全局约定 |
| 事实一致性修正 | 消除事实承诺之间的冲突 | dima / todo 关系修正 |
| 概念闭合 | 让概念能支持下一步判断 | `Execution Mode` 升级 |
| 权威来源归位 | 指明规则由谁维护 | `i-verify` 消费仓库协议 |
| 过时概念清理 | 移除当前设计中不存在的概念 | 删除 `i-spec-execute` |
| 校验层补齐 | 把关键约束转为检查项 | 结构校验清单 |
