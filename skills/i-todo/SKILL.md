# i-todo：直接需求入轨

将用户直接提出的需求登记为本地 todo 项，并按类型路由到对应执行技能。与 dima 工单轨道共享后续设计 → 规划 → 编码 → 验收的完整流程，差异仅在于：dima 工单需要经过 `i-dima-resolve` 研判后入轨，直接需求在创建时即完成类型确认。

> **路径根**：`{workspaceRoot}` 指 **Cursor 工作区根目录**（studio 多仓顶层，与根级 `AGENTS.md`、`.cursor/` 同级），**非**各子项目 Git 根。本文档中凡 `.iteration/` 均相对于 `{workspaceRoot}`。

> **任务阶段规划**：使用 TodoWrite 按以下四个步骤创建待办，每步一项，顺序不得跳过。

> **调度兼容说明**：本 skill 支持由 `iteration-ctl tick` / `dispatch` 在 `INTAKE` 阶段派发；也支持直接 `@i-todo` 调用。若没有迭代上下文注入，则按原线性流程执行。

## 阶段断点路由（Stage Hooks）

> 被调度执行时，仅执行与当前 `currentStage` 匹配的 `When stage == ...` 节。完成本节 Yield 条件后立即结束当前 agent 调用。

### When stage == INTAKE

**前置输入**：

- 用户原始需求描述
- 当前 `{workspaceRoot}/.iteration/iteration-context.json`

**核心动作**：

1. 执行下方步骤零判断是否需要拆分；若需拆分，与用户确认拆分方案。
2. 对每个独立子需求依次执行步骤一至步骤三（确认入轨信息 → 创建 todo 文件 → 路由确认）。
3. 每个子需求各自调用 `iteration-ctl allocate-req-id --source todo` 获取独立 3 位 `reqId`。
4. 每个子需求写入独立 `.iteration/todo/[<reqId>]<emoji> <needName>.json`，JSON 中包含 `reqId`、`iterationId`、`source: "todo"`、`skillName`、`needName`、`workItemId`、`status: "in-progress"`，以及（拆分场景）`splitGroup`、`dependsOn`。
5. 不在 INTAKE 阶段直接执行出口技能；后续由 `iteration-ctl tick` 在对应阶段派发。

**产出（完成契约）**：

- 制品路径：每个子需求一个文件 `{workspaceRoot}/.iteration/todo/[<reqId>]<emoji> <needName>.json`
- 文件内容含完整路由元数据，可被 tick 扫描派发。

**Yield 条件**：
完成所有子需求产出后立即结束当前 agent 调用；最终响应末尾输出 `[stage-yield] stage=INTAKE artifact=<路径列表>`。

## 输入

- 需求描述（用户原文）
- 类型（可选，用户显式提供）：`bug` | `feature` | `rfc`；若未提供，在步骤一中推断并确认

---

## 步骤零：需求划分判断

**执行模式**：主控执行

**动作**：

1. 通读用户原始需求描述，依据末尾 [需求划分基准（与 i-dima-resolve 共用）](#需求划分基准与-i-dima-resolve-共用) 节判断是否需要拆分。
2. 判断结果分两种：
   - **不拆分（默认）**：当前需求作为单个子需求处理，跳到步骤一（共生成 1 份 todo JSON）。
   - **需拆分**：列出所有子需求清单（每行：编号 + 简名 + 类型 + 该子需求负责的范围 + 与其他子需求的依赖关系），**向用户确认后**再继续。
3. 若用户对拆分方案有异议（如认为"应合并"或"应再细分"），按用户意见调整后再次确认。
4. 拆分确认通过后，生成同批拆分的 `splitGroup` 标识（建议格式：`split-<YYYYMMDD-HHmmss>`），后续每个子需求 JSON 中均写入此字段以便追溯。

**完成判定**：拆分清单已确定（含 1~N 项子需求）；如有拆分，已与用户确认；`splitGroup` 已生成（拆分场景）。

> **典型场景**：用户描述「希望系统支持开放平台租户体系，包括引入租户领域概念、租户代行鉴权机制、API 文档自动生成」——若三部分变更范围互不重叠（不同表/不同核心服务/不同代码区块）、可独立设计与验收，则拆分为 3 个子需求；否则视为一个 RFC 由 i-spec-design 整体设计。

---

## 步骤一：确认入轨信息（对每个子需求执行）

**执行模式**：主控执行

**动作**（对步骤零产出的清单中**每个**子需求执行）：

1. 确认子需求描述已获取（拆分场景下为该子需求对应的范围说明 + 原始需求中的相关原文）。
2. **确认类型**：
   - 若用户已显式提供类型 → 直接使用
   - 若未提供 → 依据末尾 [路由判断基准](#路由判断基准) 节中的规则推断类型，**向用户确认后**再继续（一句话说明推断依据）
3. 生成 `needName`：从子需求标题或摘要提取适合文件名的名称段；若无标题则取当前时间戳，格式 `YYYYMMDD-HHmmss`。
4. 分配 `reqId`：调用 `iteration-ctl allocate-req-id --source todo`；拆分场景下**每个子需求各调用一次**，得到不同的 `reqId`。
5. 确定文件前缀 emoji：`bug` → `🐞`，`feature` → `🪶`，`rfc` → `📋`。
6. （拆分场景）整理本子需求与其他子需求的顺序依赖，记录到 `dependsOn: [reqId]` 列表。

**完成判定**：每个子需求的类型、`needName`、`reqId`、emoji、`dependsOn` 均已就绪。

---

## 步骤二：创建 todo 文件（对每个子需求执行）

**执行模式**：主控执行

**动作**：

确保 `{workspaceRoot}/.iteration/todo/` 目录存在（不存在则创建），对每个子需求各写入一份 `[<reqId>]<emoji> <needName>.json`：

```json
{
  "workItemId": "YYYYMMDD-HHmmss",
  "reqId": "001",
  "iterationId": "sprint-2026-04-25",
  "source": "todo",
  "skillName": "i-bug | i-spec-feature | i-spec-design",
  "needName": "需求名称",
  "type": "bug | feature | rfc",
  "description": "需求原文（拆分场景下为该子需求对应的范围说明）",
  "status": "in-progress",
  "createdAt": "ISO 8601 时间",
  "routedTo": "i-bug | i-spec-feature | i-spec-design",
  "splitGroup": "split-20260427-1130（仅拆分场景）",
  "dependsOn": ["002"]
}
```

> `status` 生命周期：`in-progress`（已入轨，等待阶段调度）→ `done`（后续阶段完成后由编排器或人工更新）。
>
> `splitGroup`：仅在步骤零判定为「需拆分」时填写；不拆分时省略此字段。
>
> `dependsOn`：本子需求依赖的其他子需求 reqId 列表；无依赖时省略此字段或填空数组。

**完成判定**：所有子需求 todo 文件均已写入，文件名前缀 emoji 与 `type` 对应；可读取并解析 JSON 通过。

---

## 步骤三：路由执行

**执行模式**：主控执行

**路由表**：

| 类型      | 出口技能（本次仅确认 skillName，不直接执行） |
| --------- | -------------------------------------------- |
| `bug`     | `i-bug`                                      |
| `feature` | `i-spec-feature`                             |
| `rfc`     | `i-spec-design`                              |

> **边界约定**：本技能只负责触发入口技能，不串联后续技能。`i-spec-feature` 执行完不会自动进入下一阶段，`i-spec-design` 执行完不会自动进入 `i-spec-feature`；后续阶段各有独立的触发方式。
>
> **拆分场景边界约定**：每个子需求**独立**走完 design → plan → execute → verify 全链路，子需求之间在 EXECUTION 阶段也彼此独立（除非通过 `dependsOn` 指定顺序）。下游各 skill **不**做跨子需求合并。

**动作**：

1. 在迭代调度模式下，本步骤只确认 `skillName` 已写入每个子需求的 todo JSON，不直接执行出口技能。
2. 在直接调用兼容模式下，可对**每个子需求**依次读取出口技能的 SKILL.md（`.cursor/skills/{技能名}/SKILL.md`），以该子需求的描述作为输入，严格按其线性流程执行。

**完成判定**：迭代调度模式下，所有子需求 todo 文件可被 `iteration-ctl tick` 扫描；直接调用兼容模式下，每个子需求的出口技能已完整执行并按旧逻辑更新状态。

## 需求划分基准（与 i-dima-resolve 共用）

本节定义「一个原始需求/工单是否需要拆分为多个独立子需求」的判断规则。`i-todo` 与 `i-dima-resolve` 在确认类型/Scope **之前**共同使用同一套规则。

> **核心目标**：让单次出口技能（`i-bug` / `i-spec-feature` / `i-spec-design`）调用只处理一个独立子需求，避免下游技能在内部做切片、并行编排或跨需求合并。

### 何时拆分（满足以下**全部**条件即应拆分）

1. **变更范围互不重叠**：各子需求的物理变更不相交——
   - 不同时改同一张 DB 表 / 同一个 entity 文件
   - 不同时改同一个核心 service / 同一个接口
   - 不同时改同一个前端关键模块
2. **可独立设计、规划、验收**：每个子需求都有清晰可独立验证的产出（独立 `design.md` / `change-plan.md` / 可独立合入 PR）
3. **依赖关系简单**：子需求间最多存在**单向顺序依赖**（用 `dependsOn` 表达），不存在双向耦合或共享设计决策

### 何时**不**拆分

- 子需求是同一功能的多个层（DB + API + 前端 UI 一起完成一个特性）—— 这属于一个 feature 内的不同 Cx 条目，由 `i-spec-feature` 在 PLANNING 阶段消化
- 子需求都改同一张表的不同字段、同一个 service 的不同方法 —— 应作为一个需求，由对应 skill 内部分批
- 拆分后子需求之间存在**共享的设计决策**（如统一的鉴权策略、统一的命名约定、共享的领域模型）—— 应作为一个需求由 `i-spec-design` 整体设计后再视情拆分
- 单个子需求**独立完成会破坏系统状态**（如新表创建必须与读写逻辑同期上线，不能拆为两个独立合入）—— 应作为一个需求

### 拆分判断口诀

> **如果两个子需求可以由两个不沟通的工程师并行完成、各自独立合入主干、互不打架，那它们应该拆分；否则不拆。**

### 拆分后的处置

| 项                | 处理方式                                                                                           |
| ----------------- | -------------------------------------------------------------------------------------------------- |
| **reqId**         | 每个子需求独立分配（多次调用 `iteration-ctl allocate-req-id`）                                     |
| **状态文件**      | 每个子需求一份独立 JSON（i-todo 路径 / i-dima 路径），文件名前缀 reqId 各不相同                    |
| **类型路由**      | 每个子需求按下方 [路由判断基准](#路由判断基准) **各自识别**为 bug / feature / rfc，可异型混合      |
| **追溯字段**      | 同批拆分的子需求 JSON 中**统一**写入相同的 `splitGroup` 标识，便于跨子需求查找                     |
| **顺序依赖**      | 用 `dependsOn: [reqId]` 表达；无依赖时省略                                                         |
| **dima 工单字段** | 拆分场景下，**所有**子需求 JSON 共享同一个原始 `workItemId`（dima 工单本身不拆分）                 |
| **拆分前确认**    | i-todo 路径：与用户口头确认；i-dima-resolve 路径：在工单评论中明确说明拆分方案与各子需求的处理方向 |

### 不拆分时的处置

- 单个子需求按原流程入轨，不写入 `splitGroup` / `dependsOn` 字段（或填空）。
- 即使原始需求看起来「比较大」，只要不满足上述拆分条件，就交给下游 skill 处理（`i-spec-design` / `i-spec-feature` 自身能在内部分批分阶段，不需要 INTAKE 阶段越权切片）。

## 路由判断基准

本节定义「给定一个工作项，应走哪条路径」的判断规则，供 `i-todo` 在未显式提供类型时使用；与 `i-dima-resolve` 入轨写入的 `reqType` / `skillName` 及 `iteration-ctl` 扫描 JSON 时的 `inferSkillName` 规则对齐。

> 拆分场景下，本基准**对每个子需求各自应用一次**。

### Bug → `i-bug`

满足**以下任一**条件：

- 有明确的回归语义：「以前能用 → 现在不行」「之前正常 → 现在报错」
- 有可复现路径（页面/操作序列/接口）且有异常表现（报错/数据错乱/功能异常）
- 不涉及新能力开发，仅涉及修复已有功能与预期的偏差

### Feature → `i-spec-feature`（直接规划，无需设计文档）

满足**以下全部**条件：

- 需求边界清晰，在探索代码后即可确定实现路径，无需方案论证
- 涉及模块有限（≤ 3 个服务/模块），改动为**增量型**（新增字段/接口/组件，或修改现有逻辑），不涉及模块间关系重组
- 无重大技术风险或架构取舍需要在设计阶段与团队对齐

典型场景：新增一个 API 字段、新增页面上的功能入口、修改某个业务规则的判断条件。

### RFC → `i-spec-design`（先设计，再规划）

满足**以下任一**条件：

- 需要设计新的数据模型、服务或跨模块协作协议（涉及架构决策）
- 需求描述模糊，实现路径在未经设计论证前无法确定
- 涉及 ≥ 3 个服务/模块，或涉及对现有模块间依赖关系的重组
- 存在明显的方案取舍（方案 A vs 方案 B），需要在设计阶段与人工对齐后再执行

典型场景：引入新的认证体系、重构核心数据模型、设计跨服务事件通知机制。
