# i-spec-feature：规范差距分析 → 变更清单

给定 i-spec-design 产出的规范文档（理想终态）与当前系统现状，通过条款索引化 → 系统探索 → 概念落点映射 → 差距分析 → 充分性验证，产出**结构化、可追溯、置信度标注**的变更清单，作为 i-spec-execute 的直接输入。

**核心挑战**：变更清单的「充分性」——规范描述的是终态，系统的复杂度可能远超规范覆盖范围，遗漏任何变更项都会导致 i-spec-execute 阶段发现漏项引发返工。

**第二核心挑战**：变更清单的「粒度」——i-spec-execute 是 AI coding agent，它不做任何设计决策，只负责写代码和运行测试。规范中常包含「触发某个 agent 执行 X」类粗粒度要求，其内部的实现设计（prompt 框架、触发条件逻辑、与现有系统的集成方式）必须在 i-spec-feature 阶段完全确定，不得以任何形式转移给 i-spec-execute。

> **调度兼容说明**：本 skill 支持由 `iteration-ctl tick` / `dispatch` 在 `PLANNING` 阶段派发；也支持直接 `@i-spec-feature` 调用。若没有迭代上下文注入，则按原线性流程执行。

## 全局约束

- **不重新讨论规范**：规范是已定产物（i-spec-design 的输出），本 skill 只负责「如何从现状到达规范状态」
- **不决定实现细节**：只决定「要建哪些表/接口/配置（存在性与职责）」；具体实现（索引大小、TTL 值、分表方案）留给 i-spec-execute
- **以代码仓库为准**：现状判断以实际代码/migration 为准，不依赖假设或口头描述
- **设计决策不转移给 i-spec-execute**：变更清单中不能出现「参考现有实现」「待确认」「暂定 X」等开放描述，除非已显式标注为「粒度留白（已授权）」；所有「需设计型」缺口必须在本 skill 内完成设计（可通过专项设计 subagent）
- **仓库范围可追溯**：须继承 `design.md` 中 **「工作区与仓库范围」** 专节（RFC 路径）；**feature 直达**无 `design.md` 时，须在写出 `change-plan.md` 前建立**与之等价**的范围声明（主目标仓 + 协同/排除），并与编排给定的「目标仓库路径」一致。若差距分析发现规范**隐含跨仓**而上游未声明，须**先补全声明**再继续阶段四及以后。
- **变更可归因到仓**：每条涉及代码或仓库内配置的变更须能对应到某一「范围内仓库」；跨仓条目须在声明中已有依据并在条目中显式标注仓名或根路径前缀。

**目标**：将规范条款原子化，产出可追溯的条款索引表，使每条规范语句可被变更清单独立引用

**执行指引**：执行到本步骤时，**必须**阅读 [workflow/step1-spec-parse.md](workflow/step1-spec-parse.md) 作为指导。

## 注意事项

1. **严格遵守阶段顺序**，禁止跳过任何阶段，各阶段内的校验门禁（参考对应 workflow 文档）不可绕过
2. 技术决策（阶段三）不能推迟到 i-spec-execute：「影响要改什么」的决策必须在本 skill 内完成
3. 实现设计也不能推迟到 i-spec-execute：规范中的「agent 类」「自动化流程类」要求，其内部实现设计必须在本 skill 内通过阶段三点五完成，不能以「hardcode 一个 prompt」「参考现有实现」作为变更清单的最终状态
4. 发现任何 MUST 类条款无覆盖：立刻回到阶段四补条目，不能仅在覆盖度矩阵里标「缺失」
5. 变更清单的描述字段必须精确到「在 token 表新增 `token_type` VARCHAR(20) 字段」，而非「token 要支持多类型」
6. 粒度留白须显式声明：凡有意授权 i-spec-execute 自行决策的点，必须用「粒度留白（已授权）」格式标注，无此标注即视为变更清单已完全收敛、i-spec-execute 不需要补充任何设计
7. **置信度 ≠ 粒度**：高置信度描述「方向是否正确」，粒度描述「是否足够具体到可直接实现」；两者独立检查，不能以高置信度代替粒度检查
