# 背景

该项目使用了一套 Spec-Driven 的迭代模式。其中该技能负责的环节是**讨论可行方案**。

整个迭代工作流的设计信息参考 `reference/spec-driven.md`。

> **调度兼容说明**：本 skill 支持由迭代编排派发或用户直接 `@i-spec-design` 调用。若无工单 / 迭代上下文注入，则按本文的 scope 化设计流程执行。

## 输入与完成标准

**典型输入**：

- 若由编排注入：`iterationId`、`reqId`、`workItemId`、`currentRoundId`（若有）、`skillName` 等上下文
- `.iteration/dima/[<reqId>]*.json` 或 `.iteration/todo/[<reqId>]*.json` 中的 RFC 需求描述
- 目标仓库 `AGENTS.md`

**须完成的工作**：按下文「方案设计流程」执行——先把原始需求建模为目标、非目标、事实、约束、假设和待确认项，再冻结工作区与仓库范围，建立 scope 架构，将持续获取的上下文归位到各 scope、scope 间关系或全局约束，为每个 scope 完成方案构思与详细设计，并将全设计共用的上下文参照归档到制品根目录 `references/`。

**产出须满足**：

- 制品根目录：`{workspaceRoot}/.iteration/recordings-design/[<reqId>] <needName>/`
- 总览索引：`design.md`，必须包含「工作区与仓库范围」专节、需求目标、非目标、全局约束、scope 地图、scope 间关系、全局风险和后续规划输入。
- Scope 设计文档：`scopes/<scope-id>/design.md`，每个独立 scope 必须各有一份，包含该 scope 的边界、目标、约束、具体设计、风险、验证与实施输入。
- 设计共用上下文参照目录：`references/`（与 `design.md`、`scopes/` 同级），**全设计共用**一份归档空间；可在其下按主题使用一个或多个 `.md` 文件、或子目录存放摘录与素材。每条 reference 条目须写清支撑的设计判断，并标注**关联 scope id**（或显式标注「全局 / 多 scope」及具体 id）；不要求也不使用 `scopes/<scope-id>/references.md` 按 scope 分文件。
- 不写 `iteration-context.json` 的业务字段。

## 工作区与仓库范围（须先确认）

当前 Cursor 工作区根下通常有**多个独立 Git 仓库**（各子目录一个仓）。在进入上下文盘点与方案设计前，须先**冻结「会碰哪些仓」**：

1. **枚举**：列出与需求可能相关的子目录，使用**相对工作区根**的路径（如 `alipay-weavefox-studio-webdev/`），并区分哪些是 Git 仓库、哪些仅为文档或工具目录（按需）。
2. **写入声明（必进 `design.md`）**：在后续产出的 `design.md` 中设专节 **「工作区与仓库范围」**（或等价标题），至少包含：
   - **主目标仓库**：本需求代码/配置改动的**主承载**仓路径；须与工单、编排注入或用户明确输入对齐。
   - **协同仓库**（可选）：若设计会**修改**或强依赖第二仓及以上，逐条写路径 + 原因；若无则显式写「无」。
   - **排除说明**（可选）：易与需求混淆、但确定**不改**的邻仓，可单列以减少误扫、误改。
3. **默认边界**：无明确跨仓依据时，**仅**将可交付变更落在主目标仓库；扩大范围前须有需求/设计依据或用户确认，并写入上述专节。

若无法确定主目标仓库：**暂停**并向用户或编排方确认后再进入「建立 Scope 架构」。

# 方案设计流程

## 输入建模与边界冻结

在构思 scope 前，先把原始需求拆成可归位的信息块，避免把假设、约束、目标和实现偏好混写成最终方案。

1. **原始输入分类**：提取需求目标、非目标、事实、约束、假设、风险和待确认项。外部系统、跨团队接口、第三方平台规则等无法从代码或文档确认的信息，必须进入待确认清单。
2. **来源约束优先**：任何关键设计判断都必须能追溯到用户输入、代码事实、文档、PoC 或显式假设。确认前不得把假设写成确定结论。
3. **待确认清零**：若待确认项会影响 scope 边界、核心协议、数据模型、API 契约、发布/回滚策略，暂停并向用户或编排方一次性确认；不能带着阻断性未知继续生成最终设计。
4. **工作区与仓库范围冻结**：按上文「工作区与仓库范围」完成主目标仓、协同仓、排除说明声明后，再进入 scope 架构。

## 建立 Scope 架构

在持续读取上下文前，先建立稳定的 scope 架构。scope 是最终设计的基本组织单元：上层 `design.md` 管理 scope 划分与 scope 间关系，下层 `scopes/<scope-id>/design.md` 递归展开 scope 内部细节。

**执行指引**：候选 scope 从何而来、如何用「归属」归位信息、scope 间关系如何分层与校验（含七类关系的精确定义与持续输入下的确认顺序）、references 如何支撑设计判断、以及新上下文推翻结构时应先重构再落文档等，一律按 [`reference/scope-design-methodology.md`](reference/scope-design-methodology.md) 执行；本 skill 此处不重复展开方法论正文。

## 上下文资源盘点与归位

根据对齐后的任务改造目标，确认需要参考的上下文内容。

1. 通过对齐后的任务改造目标和 scope 架构，先确定需要具体参考哪些方面的上下文内容。如：

- 当前版本方案设计
- 梳理相关依赖关系，如：
  - 上游：我们的方案强依赖了哪些模块/服务
  - 下游：我们的方案被哪些模块/服务依赖
  - 数据：我们的方案涉及哪些数据库、缓存、消息队列
- 代码中所涉及模块的技术债务，包括：
  - 扫描代码中的 TODO、FIXME
  - 复杂的“黑魔法”实现逻辑
  - 其它具有“坏味道”可能有坑的逻辑。具体可参考：[代码风险信号](reference/代码风险信号.md)

2. 确定好获取这些上下文内容的方式，如：

- 直接阅读对应文件目录下的内容，或基于这些内容分析得到结论
- 通过特定 MCP 获取
- 访问或搜索对应网站获取
- 需要用户提供获取方式

其中，如果发现无法通过代码获取的外部依赖信息，必须暂停并询问用户，生成“待确认信息清单”，一次性问清。

3. 每条上下文信息获取后必须立即归位，并写入制品根目录 `references/` 下合适文件的草稿（可按主题拆分为多个 `.md`，避免单文件过大；与代码仓无关的摘录、下载材料可置于 `references/<子目录>/`）：

- 若信息只支撑一个 scope，写入共用 `references/`，该条目正文内标注关联 **scope id**
- 若信息描述多个 scope 之间的关系，在总览 `design.md` 的「scope 间关系」中登记，并在 `references/` 对应条目中列出涉及 scope 并互链或交叉引用
- 若信息是全局约束或全局背景，登记在总览 `design.md`，并在 `references/` 对应条目中标注「全局」或列出全部相关 scope id
- 若信息无法归属，暂停写结论，先判断是否需要新增 scope、合并 scope、调整边界或向用户确认

`references/` 内每条 reference 条目至少包含：来源类型、路径或 URL、关键符号或段落、获得方式、支撑的设计判断、**关联的 scope id 或关系说明**。不要只写“参考现有实现”，必须精确到可追溯位置。

4. 若上下文推翻了既有 scope 边界或关系，先修改 scope 架构与关系登记，再继续展开局部设计；不得在旧结构下追加相互冲突的正文。

## 方案构思

**动作**: 先提出全局技术方向，再按 scope 分别提出初步技术选型或改造路径。跨 scope 的技术决策必须写清楚依存、协议、时序、交换或影响关系；外部约束应登记为全局约束或 scope 约束，不写成 scope 间关系。

**注意事项**:

- 对于关键技术点选型（如：新旧库替换兼容性、特定 SQL 语法支持、高并发下的锁机制）的可行性，如果难以直接判断最终结论，且无更多外部信息支持判断，直接按 `workflow/poc.md` 进行实验验证。
- 方案构思只能基于已归位的来源、约束、scope 关系和明确假设；材料不足时补上下文或提待确认项，不用“合理补全”替代证据。

## 详细方案设计

**动作**: 调用 `tech-design`技能，根据以上信息，为每个 scope 构建完整可行的改造方案设计。

总览 `design.md` 负责说明：

- 需求目标与非目标
- 工作区与仓库范围
- Scope 地图：scope id、名称、职责边界、产物路径、状态
- Scope 间关系：归属、依存、协议、时序、交换、影响、协同
- 全局技术方向、全局约束、全局风险
- 后续规划输入：推荐实施顺序、并行边界、需要由 i-spec-feature 消费的关键决策

每个 `scopes/<scope-id>/design.md` 负责说明：

- Scope 边界：属于本 scope 的事项与明确不属于本 scope 的事项
- Scope 目标、约束与验收标准
- Scope 内部的业务流程、数据模型、API、时序、详细设计、监控、发布、回滚、测试用例等必要内容
- 与其他 scope 的关系：依存、协议、时序、交换、影响、协同目标
- 风险、PoC 结论、待确认事项清零状态
- 面向后续规划与实现的输入

`references/` 目录（全设计共用）整体负责说明：

- 支撑各 scope 与全局结论的上下文来源清单（条目中须能还原「这条证据服务哪些 scope」）
- 每个 reference 条目对应的设计判断与关联 scope
- 跨 scope 共用、冲突或互补的 reference 如何在条目中标注与交叉引用
- 外部依赖与用户确认记录
- PoC 报告条目：报告路径、实验分支或目录、支撑或否定的设计判断、关联 scope id

## 全局验收

定稿前按 [`reference/scope-design-methodology.md`](reference/scope-design-methodology.md) 的「全局验收清单」逐项自检，尤其关注：

- 目标、非目标、硬约束是否被覆盖或显式标注不适用
- 每个关键设计判断是否能追溯到 reference、用户确认、PoC 或显式约束
- 所有待确认项是否清零；未清零且影响关键设计时必须暂停
- Scope 间关系是否使用归属、依存、协议、时序、交换、影响、协同分类，未把外部约束误写成关系
- DB、API、触发条件、数据格式、错误处理、发布/回滚策略是否达到后续规划可消费粒度
