# 项目 Spec 迭代模式工作流

该项目使用了一套 Spec-Driven 的迭代模式。其中分为三个核心流程，分别是：

- 核心目标对齐
- 讨论可行方案
- 方案具体实施

## 核心目标对齐

这部分用户会根据团队成员和客户反馈的情况去收集一轮，并最终对齐形成**讨论可行方案**流程的核心输入。这其中包括：

- 改造现在要直接解决的问题/核心痛点具体是什么
- 能力诉求，比如支持 xxx 特性，提升 xxx 性能指标到 xxx
- 对这块接下来应当改成什么样的建议
- 可以考虑尝试和论证的修改方向，如通过 xxx 框架或方案解决某问题
- 改造的关键约束，如负面清单(哪些内容不改)、兼容性约束等

## 讨论可行方案

- 尝试提出解决问题的基本方向和思路(若需要)
- 盘点当前上下文资源情况。内容包括：当前版本方案设计、相关依赖梳理、历史债务识别等
- 再通过`tech-design`技能构建完整可行的改造方案设计。
- 顺带看有没有新的机会，即在这个改造过程中可以顺道做的 ROI 高且有价值的事情。

方案确定初稿后，用户会进行人工评审过程。评审完毕后，会给出若干建议，你需要根据建议再去调整方案。

## 具体实施

- 将拆解方案成具体执行的 tasks。
- 通过`feature`技能执行具体的 task，并对执行情况进行反馈
