# Scope 化设计方法论

本文是 `i-spec-design` 的正式执行依据，用于把持续输入的需求、代码事实、约束和设计判断组织成可被 `i-spec-feature` 继续消费的 scope 化设计文档。

## 核心原则

1. **来源约束优先于表达完整**：没有来源支撑的判断只能标为假设或待确认，不能写成确定设计。
2. **先稳定上层结构，再展开局部细节**：先确定目标、非目标、全局约束和 scope 边界，再写 scope 内部方案。
3. **新信息先归位，再生成结论**：任何新上下文都必须先归到 scope、scope 间关系、全局约束或待确认清单。
4. **关系显式化**：跨 scope 的依赖、协议、时序、交换和影响必须写清楚，不能只写并列模块列表。
5. **面向后续规划收敛**：最终设计必须让 `i-spec-feature` 能继续做差距分析，不能把字段名、接口契约、触发条件、关键策略留给后续猜测。

## 设计状态机

`i-spec-design` 应按以下状态迁移推进；若后续发现上游状态不成立，先回退修正上游结构，再继续写文档。

| 阶段 | 当前状态 | 允许动作 | 禁止动作 | 验收信号 |
| --- | --- | --- | --- | --- |
| 输入建模 | 原始需求未结构化 | 提取目标、非目标、事实、约束、假设、待确认项 | 直接写最终方案 | 输入类型清晰，悬空问题可见 |
| 范围冻结 | 工作区与仓库范围未确定 | 确认主目标仓、协同仓、排除仓 | 跨仓扩大设计范围 | `design.md` 有范围声明 |
| Scope 架构 | 问题空间未拆分 | 定义 scope、边界、职责、候选产物 | 写 scope 内部细节 | 每个事项可归属或进入待确认 |
| 关系建模 | scope 已存在 | 标注依存、协议、时序、交换、影响、协同 | 只写模块并列清单 | 跨 scope 关系可追溯 |
| 上下文归位 | 代码和材料持续输入 | 建立 references 条目并关联 scope / 关系 / 全局约束 | 把材料直接塞进正文 | 证据能支撑设计判断 |
| 方案生成 | 来源、约束、关系已稳定 | 生成全局方向与 scope 详细设计 | 绕过约束生成理想方案 | 每个关键设计点有依据 |
| 全局验收 | 局部设计已形成 | 检查覆盖、冲突、遗漏、规划可消费性 | 只润色表达 | 输出可被后续阶段直接消费 |

## 输入类型

在输入建模时，至少区分以下类型：

| 类型 | 定义 | 归档位置 |
| --- | --- | --- |
| 目标 | 本次设计要达成的结果 | `design.md` 需求目标 |
| 非目标 | 本次明确不解决的事项 | `design.md` 需求目标或范围说明 |
| 事实 | 代码、文档、接口、数据表等可验证现状 | `references/` 条目 |
| 约束 | 外部输入对设计施加的限制 | `design.md` 全局约束或 scope 约束 |
| 假设 | 尚未证实但用于临时推理的判断 | 待确认清单，确认前不得定稿 |
| 决策 | 已选择的设计方向或取舍 | `design.md` / scope `design.md`，并引用依据 |
| 风险 | 可能阻断或放大成本的问题 | 全局风险或 scope 风险 |
| 待确认 | 必须由用户、外部系统或实验补足的信息 | 待确认清单，清零后再定稿 |

## Scope 划分准则

一个 scope 应该是后续可以独立规划、独立实现或独立验证的设计单元。候选 scope 通常来自：

- 独立业务能力或用户流程。
- 独立数据模型或存储边界。
- 独立 API / 协议 / 集成契约。
- 独立发布、灰度、回滚或迁移边界。
- 独立风险面，或难以直接判断最终结论且缺少外部信息支撑、需要 PoC 的技术选型。

划分时使用以下检查：

- **边界清晰**：能写出属于本 scope 的事项，也能写出明确不属于本 scope 的事项。
- **职责单一**：scope 内部围绕一个主要目标组织，不混入只是相邻但目标不同的内容。
- **可交付**：后续 `i-spec-feature` 能围绕该 scope 继续产生变更清单。
- **可验证**：scope 有自己的验收标准、风险和验证方式。
- **关系外显**：如果两个 scope 需要协同，优先写关系，不要强行合并。

若某信息无法归属到任何 scope，先判断是缺 scope、scope 边界错误、关系内容，还是待确认问题；禁止为了赶进度把它塞进最近的 scope。

## Scope 间关系

Scope 间关系用于描述多个设计单元如何共同成立、交互和演化。关系类型如下：

| 类型 | 定义 | 关键问题 |
| --- | --- | --- |
| 归属 | 某事项是否属于某 scope | 这件事归谁管？ |
| 依存 | A 的成立需要 B 提供存在性条件 | 没有 B，A 能成立吗？ |
| 协议 | A 与 B 交互时共同遵守的稳定契约 | 双方遵守什么接口、数据、状态或异常规则？ |
| 时序 | A 与 B 的事件或阶段存在先后、并发或互斥 | 它们以什么顺序发生？ |
| 交换 | A 与 B 在运行时传递 payload、消息、控制信号或能力调用 | A 给了 B 什么？ |
| 影响 | A 的变化会传播并改变 B 的状态或行为 | 动 A 会动哪里？ |
| 协同 | 多个 scope 通过上述关系共同完成上层目标 | 它们合起来完成什么目标？ |

强提醒：**约束不是 scope 间关系**。约束是外部输入对 scope、关系、流程、产出或质量的限制。协议可以被约束修订，但协议属于关系层，约束属于输入层。

### 关系扫描顺序

对任意一对相关 scope，按以下顺序扫描：

```text
[ ] 归属：两者边界各包含/排除什么？
[ ] 依存：A 依存 B 吗？B 依存 A 吗？
[ ] 协议：双方是否有接口、数据、状态机、错误码、重试、降级等契约？
[ ] 时序：双方事件是否有先后、并发或互斥要求？
[ ] 交换：双方传递什么 payload 或控制信号？
[ ] 影响：任一方变更会影响另一方吗？
[ ] 协同：它们共同支撑哪个上层目标？
```

不是每对 scope 都有全部关系，但跳过扫描会留下盲点。

## References 归档契约

制品根目录 `references/` 是全设计共用的证据空间，不按 scope 分散到各 scope 目录。每条 reference 至少包含：

- 来源类型：代码 / 文档 / 用户确认 / 外部链接 / PoC / 运行观测。
- 来源位置：文件路径、URL、命令、关键符号或段落。
- 获取方式：阅读、搜索、实验、用户提供等。
- 支撑的设计判断：这条证据证明或限制了什么。
- 关联对象：scope id、scope 间关系，或「全局 / 多 scope」并列出具体 id。

归档时遵守：

- 同一证据支撑多个 scope 时，在同一条目中列出全部 scope，不复制多份。
- 证据与设计判断之间必须能追溯，不能只写“参考现有实现”。
- 若证据之间冲突，不要抹平；在条目中记录冲突，并进入待确认或设计取舍。

## 全局验收清单

定稿前逐项检查：

- [ ] 原始需求中的目标、非目标、硬约束均已覆盖或明确不适用。
- [ ] 工作区与仓库范围已冻结，主目标仓、协同仓、排除说明清晰。
- [ ] 每个 scope 有边界、目标、约束、验收标准、风险和后续实施输入。
- [ ] 每个关键设计判断能追溯到 reference、用户确认、PoC 或显式约束。
- [ ] Scope 间关系使用归属、依存、协议、时序、交换、影响、协同分类，且没有把外部约束误写成关系。
- [ ] 所有待确认项已清零，或明确阻断并暂停，未被伪装成确定结论。
- [ ] 涉及 DB、API、触发条件、数据格式、错误处理的设计已精确到后续规划可消费粒度。
- [ ] 难以直接判断最终结论且无更多外部信息支持判断的关键技术点，已完成 PoC、引用 PoC 报告，或明确写出不执行 PoC 的理由。
- [ ] 总览 `design.md` 与各 scope 文档互相一致，没有同一概念多种定义。
