# 确定完成标准 执行指导

> 本文件是主控 Agent 在阶段一中的详细执行指导。

## 步骤 1：理解用户真正意图

**重要**：对于正常复杂度的功能特性实现，必须执行深度思考过程，不能直接开始编写完成标准。

**深度思考过程**：

1. **需求分析**：
   - 仔细分析用户的需求描述，逐字逐句理解
   - 识别显式需求和隐式需求
   - 理解需求的业务背景和目的

2. **理解项目基本情况上下文**：
   - 探索项目代码的组织模式，了解对应部分业务场景和技术背景（如需要）
   - 理解功能在整体系统中的位置，分析与其他模块的关联关系和依赖（如需要）

3. **隐含需求挖掘**：
   - 基于已探索的代码上下文，识别隐含需求和约束条件
   - 考虑用户未明确提及但实际需要的功能点
   - 分析可能的边界情况和异常场景
   - 考虑性能、安全、用户体验等非功能性需求

**输出**：

- 用户意图的清晰表述
- 功能在业务场景中的定位
- 与其他模块的关联关系
- 识别出的隐含需求和约束条件

## 步骤 2：制定完成标准

**重要**：在制定完成标准时，必须进行深度思考，确保标准完整、准确、可执行。

**深度思考要点**：

- 基于对用户意图的深度理解，系统性地思考主流程的每个环节
- 从多个维度思考边界情况：数据边界、状态边界、权限边界、时间边界等
- 考虑异常场景和错误处理，确保系统的健壮性
- 思考完成标准是否真正覆盖了所有必要要素，避免遗漏关键点
- 验证完成标准的可执行性和可验证性

完成标准应具备以下性质：

**a. 主流程完整性**

- 按照标准执行，主流程应能一口气走通
- 标准中找齐了跑通主流程依赖的所有必要要素，不需要后续补充修复
- 明确每个步骤的输入、输出和验证点

**b. 边界情况覆盖**

- 覆盖绝大多数边界情况的处理
- 识别关键边界：空值、异常、并发、权限、数据格式等
- 按照你的完成标准指导执行后，需要刻意修复的边界 case 等 bug 问题应当非常少

**c. 交付自测用例定义**

- 参考项目中已有的主流程用例文件（一般命名格式：`主流程测试用例.md`、`xxx 主流程用例.md`）
- 确定本次修改的自测用例
- 标记重要路径：
  - ⭐ 标记：用户使用 10% 以上执行情况会确实走到的路径
  - ⭐ 标记：一旦未成功会有重大影响的路径（影响完整核心流程或用户数据资产完整性）
