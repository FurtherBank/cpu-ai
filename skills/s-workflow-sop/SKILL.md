# S-Workflow-SOP：沙盘推演 + Workflow Skill 全链路

👤 作为一个顶级的任务执行策略专家。你拥有极其强大的"沙盘推演"与"前置想象"能力。
你的主管（Manager）已经明确表示，不仅仅是接下来的任务会交给你，如果你的表现能够让主管满意，以后所有此类任务，就都会交给你来做。

在正式执行主管分配的第一个实操任务之前，为了确保你未来能毫无瑕疵地独立 hold 住这类任务，你需要进行深度的**"沙盘推演" (Workflow Simulation)**，在脑海中完整模拟出这类任务从接收到完美交付的全过程，并输出一套极其严密的标准化作业程序 (SOP)。

**阶段权威**：下文**阶段一至阶段七**为唯一顺序与命名；不得省略、重命名或调换。

## 工作流程（主控 Agent 调度模式）

> **第一步**：使用待办列表工具（TodoWrite）按下方阶段创建待办——**每个阶段一项**，阶段名称与顺序须与本文档 **「阶段一」至「阶段七」** 完全一致，不得自行拆分或改名。

### 阶段一：场景发散与需求拆解（主控 Agent 负责）

**Input**：原始用户输入（主管对任务类型的文字描述）。无前置阶段产物依赖。

**Goal**：把模糊任务描述抽象为「一种任务类型」，构造一组**可派单、彼此覆盖不同关键维度**的具体场景实例，作为阶段二每次派单 Prompt 中「场景与交付标准」的直接填充内容，每个实例须信息自洽、可独立成段。  
完成判据——任意两个实例之间可指出至少一处**实质差异**（路径、风险或交付形态）；每个实例的 DoD 足以让阶段二判断推演是否「跑完且跑对」。  
**注意力边界**：只做场景实例化——定义覆盖不同执行路径的场景参数与 DoD；不预判执行结果，不做推演。

**Guidance**：
- 先列出影响执行路径或风险形态的**覆盖维度**（如信息完整度、工具/环境、外部依赖、交付物类型、是否不可逆操作等），用于检查实例是否**同质化**。
- 实例数量一般为 3–7 个，以覆盖关键差异为准，数量服从质量；每个实例须在至少一个维度上与其他实例**显著不同**，避免仅换名词的重复。
- 每个实例须包含以下要素（将原样进入阶段二 `${阶段一对应场景的需求描述与交付标准}`）：
  - **背景与目标**：触发情境、核心诉求、最终服务的业务或工程目的。
  - **约束**：硬性边界（技术底线、资源、时限等）及合理推断的主管质量期望（非凭空加码）。
  - **信息盲区**：执行前必须澄清或确认的未知点（优先写清「尚不知道什么」）。
  - **成败关键维度**（供推演加压与阶段三提炼共用）：执行卡点（可能根本推不动的环节）、隐患风险（若前提不成立会导致返工的关键假设）、交付标准（DoD，可验证地界定「做完」与「做好」）。

**Execution Mode**：主控执行（不委派 Subagent）；建议按以下步骤顺序推导（可内化不必逐条输出）：

1. **理解意图**：用一两段话概括「这类任务在组织里解决什么问题、不算这类任务的范围边界」。
2. **定义覆盖维度**：列出影响执行路径或风险形态的维度，供同质化检查用。
3. **实例化**：按维度组合出典型实例，逐项填入背景与目标、约束、信息盲区、成败关键维度。

### SOP 步骤序列（仅正向路径）

- 以执行顺序列出步骤；**通常 5–10 步**；过少说明未拆透，过多可合并子步骤。
- 每步须包含：
  - **步骤名**（简短、稳定，便于下游引用）
  - **触发时机**：进入本步的前置状态或条件（含分支入口）

  - **完成判定标准**：可客观核对（工具输出、文件存在、检查结果、明确 Yes/No）
  - **条件分支（如有）**：条件 → 走向哪条替代路径（仅限输入类型差异触发的正常分支，不含异常/失败分支）

**注意力边界**：同时完成结构决策（节点划分、依赖关系、产物契约、分支点、异常路由入口）与质量标准定义；不关注节点内部如何实现（留阶段五），不关注具体检测操作（留阶段六）。

**Guidance**：
- 参照 [references/workflow-arch-methodology.md](references/workflow-arch-methodology.md) 按任务自然断点（认知切换、中间过程对后续无直接价值等）拆分子工作流；下一阶段只消费产出物，不依赖全量对话历史。
- 质量标准写法须达到"可比对"粒度，常见形态：
  - 结构完整性：「输出须包含 X、Y、Z 字段/章节，无任何字段缺失」
  - 内容正确性：「步骤描述须使用动词+对象格式，不含模糊动词（分析/处理/优化）」
  - 接口对齐：「输出的 ${字段} 可被下游节点直接作为 ${输入} 消费，无需转换」
  - 覆盖度：「须覆盖 N 个场景，每个场景有对应的 DoD 验证点」

**Execution Mode**：主控执行（高污染、高风险原子操作交 Subagent）；将本 Skill 规定的**工作流阶段**与 `TodoWrite` 待办项**一一对应**，执行中不得跳过阶段或擅自合并步骤。输出子工作流骨架（含各节点契约与质量标准），并**为每个节点同步填写四要素种子模板**：Input 硬依赖骨架（上游产出直接对应的必要输入）、Goal 骨架（Delta + 完成判定框架，与质量标准直接对应）、Execution Mode 架构决策（主控/Subagent），作为**阶段五**的唯一上游输入。

### 阶段五：工作流各环节详细设计

**Input**：阶段四产出的工作流骨架，含各节点的**四要素种子**（Input 硬依赖骨架 / Goal 骨架含质量标准 / Execution Mode 架构决策）；[references/task-step-schema.md](references/task-step-schema.md) 作为推导方法论索引（软上下文）。

**Goal**：在阶段四四要素种子的基础上，对每个业务执行环节逐一完成**四要素精炼落地**，把「节点名称」写成「可直接执行的规格」。  
完成判据——每个环节均有完整的 Input / Goal / Guidance / Execution Mode 四要素；每个环节的 Goal 与阶段四对应节点的质量标准逐条可对照，无遗漏、无自行发明的额外标准；Guidance 须包含可区分新手与专家执行差异的方法论（不得为空或仅转述 SOP 步骤）；若某条 Input 或 Guidance 被删掉就会导致幻觉或无法判对错，则该条必须保留。  
**注意力边界**：聚焦「如何执行」——工具选择、命令级动作、产出形态；**不重新定义质量标准**（已在阶段四完成），**不重新划分节点边界**（已在阶段四完成）。

**Guidance**：本阶段认知分工分为三层，按以下顺序推进（方法论细节参照 [references/task-step-schema.md](references/task-step-schema.md)）：

**层一（填充）— Input 软上下文**：在阶段四已给出硬依赖骨架的基础上，识别执行中"按需查阅"的背景信息（规范文档、方法论引用、目录结构等），以文件路径/章节引用形式提供，禁止全量注入。实践自问："删掉这条，Agent 会用幻觉填补 → 硬依赖（已在阶段四）；按需翻阅 → 补入软上下文"。

**层二（精炼）— Goal 与 Execution Mode**：在阶段四骨架基础上做细化，**禁止重写质量标准**：
- Goal：补充**交付物物理形态**（"输出含 X 字段的 JSON"、"在终端执行带 4 个参数的命令"）与 **Negative Prompts**（"不准在本步骤生成代码"），不得另起炉灶重新定义质量标准。
- Execution Mode：补充工具/命令级动作与产出，确认阶段四架构决策；若环节存在嵌套（即为**子工作流**），内层环节同样递归完成四要素。

**层三（创作）— Guidance 专家萃取**（本阶段唯一从零出发的工作）：
- **正常路径**：以专家视角克隆执行直觉——"顺利时第一反应是什么？"，萃取出可区分新手与专家的操作要诀。
- **异常路径**：穷举该环节最易踩的坑，注入"遇到 X 时应当 Y 而非立即放弃"的机灵特质（参照阶段三排雷指南对应条目）；可指向规范/模板/方法论索引，但不得以"参见文档"替代本条应有的专家直觉内容。

**Execution Mode**：主控执行；按子工作流顺序逐环节推导四要素，完成后做完成自检（上述 Goal 完成判据逐项核对）。

### 阶段六：校验/测试（纯检测层）

**Input**：各环节已生成的产物；阶段四定义的质量标准（作为检测基准，逐条引用）。

**Goal**：对实际产物进行比对检测，找出差距、确定问题、给出建议。  
完成判据——检测报告包含：逐条对照阶段四质量标准的检测结果（达标/有差距/无法判断）、差距的具体表现（可引用产物中的具体位置或内容）、改进建议（方向性即可）；「达标」结论须带可核对证据，无证据视为未检测。  
**注意力边界**：只负责检测执行与报告反馈；不定义质量标准（标准已在阶段四确立），不修改产物，不补充实现。

**Guidance**：先做类型判定，再选检测手段——
- **校验（Validation）**：面向**形式与静态事实**（结构完整性、字段规范符合性、接口对齐、阶段四质量标准中的静态可查项），由主控自行执行。
- **测试（Testing）**：面向**行为与功能验证**（功能/目标符合性、端到端贯通、场景覆盖度），**必须**通过 Subagent 执行；主控在长流程后易产生**证实偏差**，禁止用自读代码或主观确认替代 Subagent。
- 常用维度按节点选用，勿堆砌。

**Execution Mode**：校验由主控执行；测试须委派 Subagent，派发 Prompt 须嵌入 `pua-testing.md` 要求的明确验证标准（直接引用阶段四质量标准条目）、强制证据、PUA 激励、报告格式，禁止模糊指令。**硬约束**：主控不得自行执行归类为「测试」的验证。
