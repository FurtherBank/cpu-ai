# S-Workflow-SOP：沙盘推演 + Workflow Skill 全链路

本 Skill 用于先沙盘推演一类任务，再沉淀 SOP，最后落地为可复用 Workflow Skill。

当我要把一类复杂任务固化成 Workflow Skill 时，我不会一上来就写流程。直接写出来的流程往往只覆盖理想路径，缺少被信息缺失、工具边界、外部依赖和异常分支压过的证据。

所以我先把用户的模糊需求抽象成一种任务类型，再构造多个差异足够大的场景；我让每个场景独立沙盘推演，只记录执行痕迹，不急着总结规律。等材料足够后，我才跨场景提炼 SOP，把正向路径、异常路径、节点契约、质量标准和测试闸门逐层落到可复用 Skill 里。

这个 Skill 的核心不是“生成一份看起来完整的流程”，而是防止凭空编排：先发现，再抽象，再架构化，再内建质量关卡，最后才落盘。

**阶段权威**：下文**阶段一至阶段七**为唯一顺序与命名；不得省略、重命名或调换。

## 工作流程（主控 Agent 调度模式）

第一步必须使用 `TodoWrite` 创建待办：**每个阶段一项**，阶段名称与顺序须与本文档 **「阶段一」至「阶段七」** 完全一致，不得拆分或改名。

---

### 阶段一：场景发散与需求拆解（主控 Agent 负责）

**我为什么先做这一步**：我需要先知道这类任务会在哪些条件下变形。若场景同质化，后续 SOP 只会覆盖一个顺利样本，无法暴露真正的风险边界。

**Input**：原始用户输入。无前置阶段产物依赖。

**Goal**：把模糊任务描述抽象为「一种任务类型」，构造 3–7 个可派单、信息自洽、覆盖不同关键维度的场景实例。实例将原样填入阶段二派发 Prompt 的「场景与交付标准」。  
完成判据：任意两个实例至少有一处**实质差异**（路径、风险或交付形态）；每个实例的 DoD 足以让阶段二判断推演是否「跑完且跑对」。  
**注意力边界**：只定义场景参数与 DoD；不预判执行结果，不做推演。

**Guidance**：
- 先列影响执行路径或风险形态的**覆盖维度**，如信息完整度、工具/环境、外部依赖、交付物类型、是否不可逆操作等，用于避免实例同质化。
- 实例数量服从质量；每个实例须至少在一个维度上与其他实例**显著不同**，避免只替换名词。
- 每个实例必须包含：
  - **背景与目标**：触发情境、核心诉求、最终服务的业务或工程目的。
  - **约束**：硬性边界（技术底线、资源、时限等）及合理推断的质量期望；不得凭空加码。
  - **信息盲区**：执行前必须澄清或确认的未知点，优先写清「尚不知道什么」。
  - **成败关键维度**：执行卡点、隐患风险、交付标准（DoD）；供阶段二推演加压与阶段三提炼共用。

**Execution Mode**：主控执行（不委派 Subagent）。建议按顺序内化推导：
1. 概括这类任务解决什么问题，以及不属于此类任务的边界。
2. 列出覆盖维度，作为同质化检查依据。
3. 组合典型实例，并填入背景与目标、约束、信息盲区、成败关键维度。

---

### 阶段二：独立沙盘推演（委派 Imagine Subagent + 主控汇总）

**我为什么委派独立推演**：我需要得到多个互不污染的执行痕迹，而不是主控提前总结出的理想路径。此阶段只生产事实材料，不生产 SOP。

**Input**（每次 Subagent 派发消费）：单个阶段一实例的完整要素；原始用户输入作为身份定位背景。**禁止**将多个场景合并传入同一 Subagent。

**Goal**：对阶段一的**每个**场景分别生成完整沙盘推演报告，至少覆盖：按时间顺序的关键操作（命令/改动/交互等具体层级）、每步后的观测与判断、异常与阻碍台账、侧翼突围或替代路径。  
完成判据：全部场景报告已回传主控；每份报告足以支撑阶段三汇总，无关键决策点遗漏；多场景推演彼此独立、互不干扰。

**Guidance**：Subagent 须先阅读 `./workflow/imagine.md`，并严格按四条原则执行：端到端实操、绝对客观、实时观测反馈、异常捕获与侧翼突围。只做发现和记录，不提炼 SOP 结构，不总结规律；不得只做宏观规划而缺少操作级细节。

**Execution Mode**：使用 `Task`（`subagent_type="generalPurpose"`），**一场景一次派发**；多个场景可并行或串行，但**禁止**合并进同一次 Task。主控收齐报告后进入阶段三。

**派发 Prompt**（将 `${...}` 替换为该场景在阶段一中的完整要素）：

```markdown
## 原始用户输入

${原始用户输入}

## 任务身份定位

你是任务执行策略专家，需要对这类任务做完整沙盘推演，并产出可支撑 SOP 提炼的事实材料。

---

## 本阶段任务：单场景沙盘推演（发现层）

你必须先阅读本 skill 内的 `./workflow/imagine.md`，并严格按其四条原则（端到端实操、绝对客观、实时观测反馈、异常捕获与侧翼突围）执行；不得只做宏观规划而缺少可落地的操作级细节。

**注意力边界**：只做**发现和记录**，忠实呈现执行过程；不提炼 SOP 结构，不总结规律。

以第一人称执行者身份，仅针对下列**单一**场景做端到端沙盘推演。

推演结束后，将**完整推演过程**回传给主控 Agent，至少覆盖：按时间顺序的关键操作（命令/改动/交互等具体层级）、每步后的观测与判断、异常与阻碍台账、侧翼突围或替代路径。

## 场景与交付标准

${阶段一场景实例：背景与目标；约束；信息盲区；核心制约与成功标准（含执行卡点、隐患风险、交付标准 DoD）}
```

**主控**：必须收齐各场景推演材料后，再进入阶段三。

### 阶段三：跨场景汇总 + SOP 提炼（主控执行）

**我为什么分成 3A/3B/3C**：我不把发现、正向路径和异常处理混在同一次抽象里。先压缩事实材料，再提炼 Happy Path，最后把异常挂回具体步骤，能避免 SOP 被流水账拖散，也避免异常吞掉主流程。

**Goal**：通过 3A、3B、3C 产出完整 SOP 文档（步骤序列 + 排雷指南），作为阶段四唯一上游输入。  
完成判据：3A 推演汇总表四类归纳齐全；3B 产出仅含正向路径的 SOP 步骤序列；3C 整合排雷指南并完成逐场景覆盖度核查；合并后的 SOP 文档可直接移交阶段四。  
**注意力边界**：3A 做跨场景压缩（发现 → 素材），3B 只提炼正向路径（素材 → 步骤序列），3C 整合异常层（素材 + 步骤序列 → 排雷指南）；**每个子步骤只做一种抽象，严禁越界**。

---

#### 子步骤 A：跨场景分析（主控 Agent）

**Input**：阶段二回传的全部场景推演报告（各场景独立完整，主控可稳定引用场景名称或 ID）。

**Goal**：将 N 份场景推演报告压缩为结构化**「推演汇总表」**，作为子步骤 B 和 C 的主输入，避免后续处理原始流水账。  
完成判据：四类归纳信息齐全、结论可追溯至具体场景、异常已去重；**不含任何 SOP 条文**，只提供素材与分支锚点。

**Guidance**：按顺序完成四类归纳：
- **共性路径识别**：多数或全部场景重复出现的关键步骤，标注出现场景及所处流程阶段。
- **差异点与条件提取**：各场景在何处分歧，由哪些**可判定条件**触发，作为后续 SOP 的 **If/Else 分支锚点**。
- **异常全集（去重）**：汇总所有坑位、报错与意外；每条含触发条件、涉及场景数、严重程度（阻断失败/可恢复/非阻断）、应对方向。
- **制胜要素抽象**：成功路径上可复用的关键动作（前置检查、绕过方式、工具用法或顺序），区别于「分歧条件」本身。

**Execution Mode**：主控执行（不委派 Subagent）；按四类归纳整理并输出推演汇总表。

---

#### 子步骤 B：SOP 步骤序列提炼（主控执行，仅正向路径）

**Input**：子步骤 A 产出的推演汇总表；原始用户输入作为身份定位背景。

**Goal**：提炼正向执行路径（Happy Path），使另一名 AI Agent 仅依据该 SOP 即可按序独立执行，无需再读原始推演全文。  
完成判据：步骤通常 5–10 步；每步包含步骤名、触发时机、核心动作（动词+对象）、预期产出、完成判定标准、条件分支（如有）；**不含**排雷指南或覆盖度报告。  
**注意力边界**：只处理步骤序列、触发条件、核心动作、预期产出、完成判定标准、正常条件分支；**不处理**异常/失败场景；**不做**覆盖度自验。

**Guidance**：
- 仅基于汇总材料中的事实；材料不足时写明假设，不得编造；未覆盖场景须显式标注，不得假装已覆盖。
- 步骤按执行顺序列出；通常 5–10 步，过少说明未拆透，过多则合并子步骤。
- 每步须包含：**步骤名**（简短稳定）、**触发时机**（前置状态或条件）、**核心动作**（动词+对象，禁用「分析、处理、优化、完善」等空动词）、**预期产出**（文件/命令/结构化记录等物理形态）、**完成判定标准**（可客观核对）、**条件分支（如有）**（仅限正常输入差异，不含异常/失败分支）。
- 粒度须达到**单步执行者无需猜测即可照做**。
- 输出用 Markdown，**直接以正文开始**，**只输出 SOP 步骤序列**。

**Execution Mode**：主控执行（不委派 Subagent）；按 Guidance 推导并输出 SOP 步骤序列。

**产物**：SOP 步骤序列（正向路径，无异常处理内容）。

---

#### 子步骤 C：异常指南整合（主控 Agent）

**Input**：3A 产出的异常全集（去重）；3B 产出的 SOP 步骤序列。

**Goal**：将每条异常挂载到具体步骤，整理为**「排雷指南」**；并对 3B 步骤序列做逐场景覆盖度核查，缺口直接补入步骤序列。  
完成判据：每条异常都有所属步骤，无孤立条目；覆盖度核查已对照 3A 全部场景完成；**若有覆盖缺口须直接补入步骤序列**，禁止只用「本 SOP 未覆盖 X」收尾。  
**注意力边界**：
- 关注：异常发生步骤、触发条件、涉及场景数、严重程度、应对方向（只需明确方向，具体执行细节留阶段六）。
- 关注：逐场景核查 3B 步骤序列能否覆盖 3A 的共性路径。
- 不关注：重新构造步骤流程、重复描述正向路径、节点间结构关系（留阶段四）。

**Guidance**：排雷指南须按类型归纳（**信息缺失** / **工具能力边界** / **外部依赖不可控** / **逻辑死锁** / 其他）。每条包含：所在步骤（引用 3B 步骤名）、识别信号、预防措施（对应哪一步、如何提前规避）、处理路径（含替代方案或「侧翼突围」）。

**Execution Mode**：主控执行（不委派 Subagent）；先挂载异常，再做逐场景覆盖度核查，最后输出排雷指南与覆盖度核查说明（逐场景列出「对应步骤序列 → 偏差或材料不足说明」）。

---

**阶段三 → 阶段四移交说明**

阶段三产出完整 **SOP 文档**，由 3B **步骤序列**（正向路径层）与 3C **排雷指南 + 覆盖度核查**（异常层）拼合而成。

阶段四以该 SOP 文档为**唯一上游输入**，将 SOP 步骤映射为子工作流节点，定义节点契约（输入/输出）与质量标准，并同步输出四要素种子模板（Input 硬依赖骨架 / Goal 骨架 / Execution Mode 架构决策）。阶段五只在阶段四工作流骨架（含质量标准与四要素种子）上细化，不从零推导。

---

### 阶段四：工作流架构设计

**我为什么在这里定架构**：SOP 说明“怎么做”，Workflow 还要说明“每个节点消费什么、产出什么、如何判定质量”。我在这一阶段固定节点边界和质量标准，避免阶段五重新发明流程。

**Input**：阶段三完整 SOP 文档（3B 步骤序列 + 3C 排雷指南与覆盖度核查）。

**Goal**：把 SOP 转化为结构清晰、上下文可控的「子工作流骨架」；为每个节点显式定义质量标准，并同步输出四要素种子模板，作为阶段五唯一上游输入。  
完成判据：每个子工作流均有硬依赖输入、精炼输出、质量标准三项契约；质量标准达到「可比对」粒度，阶段六检测 Subagent 拿到后无需猜测即可逐条核对；每个节点已输出四要素种子（**Input 硬依赖骨架** / **Goal 骨架含 Delta+完成判定框架** / **Execution Mode 架构决策**）；节点边界、契约与质量标准无待定项。  
**注意力边界**：完成结构决策（节点划分、依赖关系、产物契约、分支点、异常路由入口）与质量标准定义；不设计节点内部实现（留阶段五），不设计具体检测操作（留阶段六）。

**Guidance**：
- 参照 [references/workflow-arch-methodology.md](references/workflow-arch-methodology.md)，按任务自然断点（认知切换、中间过程对后续无直接价值等）拆分子工作流；下一阶段只消费产出物，不依赖全量对话历史。
- 质量标准须达到「可比对」粒度，常见形态：
  - 结构完整性：「输出须包含 X、Y、Z 字段/章节，无任何字段缺失」
  - 内容正确性：「步骤描述须使用动词+对象格式，不含模糊动词（分析/处理/优化）」
  - 接口对齐：「输出的 ${字段} 可被下游节点直接作为 ${输入} 消费，无需转换」
  - 覆盖度：「须覆盖 N 个场景，每个场景有对应的 DoD 验证点」
  - 可审计性：「该节点完成后行为日志须更新章节 X、Y；Subagent 返回须含 behaviorLogPath（若适用）」——参照 `$behavior-log-audit` 与 [references/behavior-log-audit-bridge.md](references/behavior-log-audit-bridge.md)

**Execution Mode**：主控执行（高污染、高风险原子操作交 Subagent）。本 Skill 的**工作流阶段**须与 `TodoWrite` 待办项**一一对应**，不得跳过阶段或擅自合并步骤。输出子工作流骨架（含各节点契约与质量标准），并为每个节点同步填写四要素种子：Input 硬依赖骨架（上游产出的必要输入）、Goal 骨架（Delta + 完成判定框架，与质量标准对应）、Execution Mode 架构决策（主控/Subagent）。**每个节点的质量标准须含可审计性维度**（Audit Hook 落点）。

---

### 阶段五：工作流各环节详细设计

**我为什么只做细化**：阶段四已经决定节点边界和质量标准。此时我只把每个节点写成可执行规格，补齐 Input、Goal、Guidance 和 Execution Mode，不重新裁决工作流结构。

**Input**：阶段四产出的工作流骨架，含各节点**四要素种子**（Input 硬依赖骨架 / Goal 骨架含质量标准 / Execution Mode 架构决策）；[references/task-step-schema.md](references/task-step-schema.md) 作为推导方法论索引（软上下文）。

**Goal**：基于阶段四四要素种子，逐一完成各业务环节的**四要素精炼落地**，把「节点名称」写成「可直接执行的规格」。  
完成判据：每个环节均有完整 Input / Goal / Guidance / Execution Mode；每个环节的 Goal 与阶段四对应质量标准逐条可对照，无遗漏、无自行发明的额外标准；Guidance 须包含可区分新手与专家执行差异的方法论，不得为空或仅转述 SOP 步骤；删掉会导致幻觉或无法判对错的 Input / Guidance 必须保留。  
**注意力边界**：聚焦「如何执行」：工具选择、命令级动作、产出形态；**不重新定义质量标准**，**不重新划分节点边界**。

**Guidance**：按三层推进（方法论细节参照 [references/task-step-schema.md](references/task-step-schema.md)）：

行文模式参照 [references/workflow-writing-styles.md](references/workflow-writing-styles.md)：需要建立阶段判断力、风险感知和承接关系的位置，按执行者心智叙事写；需要固定输入、目标、完成判定、返回格式和执行边界的位置，按规范化契约表述写。

**层一（填充）— Input 软上下文**：在阶段四硬依赖骨架基础上，识别执行中「按需查阅」的背景信息（规范文档、方法论引用、目录结构等），以文件路径/章节引用形式提供，禁止全量注入。判断口径：删掉会让 Agent 用幻觉填补的是硬依赖（阶段四已给出）；按需翻阅的是软上下文。

**层二（精炼）— Goal 与 Execution Mode**：在阶段四骨架基础上细化，**禁止重写质量标准**：
- Goal：补充**交付物物理形态**（如「输出含 X 字段的 JSON」「在终端执行带 4 个参数的命令」）与 **Negative Prompts**（如「不准在本步骤生成代码」），不得另起炉灶定义质量标准；**工作流型 Skill 须含 Audit Hook**：本阶段完成后须更新行为日志哪些章节。
- Execution Mode：补充工具/命令级动作与产出，确认阶段四架构决策；若环节存在嵌套（即为**子工作流**），内层环节同样递归完成四要素；Subagent 委派须在返回格式含 `behaviorLogPath`（显式 Skill 调用）或父日志一行摘要（纯 Task）。

**层三（创作）— Guidance 专家萃取**（本阶段唯一从零出发的工作）：
- **正常路径**：提炼专家顺利执行时的第一反应，沉淀可区分新手与专家的操作要诀。
- **异常路径**：穷举该环节最易踩的坑，注入「遇到 X 时应当 Y 而非立即放弃」的机灵特质（参照阶段三排雷指南对应条目）；可指向规范/模板/方法论索引，但不得用「参见文档」替代应有的专家直觉内容。

**Execution Mode**：主控执行；按子工作流顺序逐环节推导四要素，完成后逐项核对 Goal 完成判据。

### 阶段六：校验/测试环节设计（工作流嵌入层）

**我为什么把质量关卡嵌入流程**：测试不是最后补一句“认真检查”。我需要在关键产物被下游消费前放置校验或测试节点，让错误在放大前被拦住。

**Input**：阶段五输出的子工作流环节规格（含各环节 Input / Goal / Guidance / Execution Mode）；阶段四定义的质量标准与关键产物清单。

**Goal**：审视已设计的工作流，在**必要位置**插入校验/测试步骤，形成内建质量闸门。  
完成判据：对每个产出关键产物的环节，明确是否需要在其后插入校验/测试节点；若需要，写清节点类型（校验 or 测试）、检测目标（对照哪条质量标准）、执行主体（主控 or Subagent）；若不需要，说明理由（如产物已在 Goal 完成判据中内建自检，或由后续环节天然覆盖）。不得笼统跳过，每个关键产物节点须有明确结论。  
**注意力边界**：只设计并嵌入校验/测试节点；不执行检测，不修改其他环节的业务规则。

**Guidance**：
- **校验（Validation）**：面向**形式与静态事实**（结构完整性、字段规范符合性、产物契约对齐），设计为**主控自执行节点**，直接在对应环节的 Goal 完成判据或其后插入自检清单。
- **行为日志完整性（必检项）**：对照 `$behavior-log-audit` 的 [validate-log.md](../behavior-log-audit/workflow/validate-log.md) 与目标 Skill 的 `references/audit-quality-standards.md`，在 Skill 封口前校验日志结构、阶段 Audit Hook 覆盖、子 Skill 链（若适用）。
- **测试（Testing）**：面向**行为与功能验证**（功能/目标符合性、端到端贯通、场景覆盖度），设计为**Subagent 委派节点**；须预先写好派发 Prompt 模板，包含明确验证标准、强制证据要求、报告格式，禁止以「要认真测试」类劝诫替代具体指令。
- 嵌入位置：置于**产物生成后、下游环节依赖前**；避免在每个微步骤后堆砌，优先聚焦高风险/不可逆产物节点。

**Execution Mode**：主控执行；逐环节审查后，将校验/测试节点作为独立步骤（具备完整四要素）**补写进工作流规格**，或以「子步骤」形式嵌入对应环节；完成后做整体一致性自检（节点覆盖是否与质量标准对应、是否遗漏高风险产物）。

### 阶段七：Skill 完整编写

**我为什么最后才写 Skill 文件**：只有当前六阶段已经产出 SOP、节点契约、质量标准和测试设计后，落盘才是映射与拆分工作。此阶段不再重写业务规则，避免把前面的推导结论冲掉。

**Input**：阶段一至六全部定稿产物：SOP 文档、工作流架构与质量标准、环节四要素规格、检测报告。

**Goal**：按产物契约落盘为可立即加载的 Cursor Skill，完成「调度层（主文件）— 细节层（workflow）— 依据层（references）」分工；不重写业务规则，只做映射、拆分与引用校验。  
完成判据：生成后逐项自检全部通过：
- **结构与体量**：目录符合模板；`SKILL.md` 主文件小于 500 行；过长时把步骤细节与长文依据分别下沉至 `workflow/`、`references/`，主文件保留索引式指针而非重复正文。
- **内容与契约**：每个需执行的步骤具备四要素（Input / Goal / Guidance / Execution Mode），并与 [references/task-step-schema.md](references/task-step-schema.md) 一致；阶段名称与顺序与 `TodoWrite` 规划一致，无跳步或歧义指代。
- **行文模式**：主体内容须符合 [references/workflow-writing-styles.md](references/workflow-writing-styles.md)；阶段心智、风险信号和承接关系使用执行者心智叙事，输入输出、完成判定、委派 Prompt、校验测试使用规范化契约表述。
- **主控与上下文**：强上下文依赖步骤标为**主控执行**，且通过「指导文件链接 + 摘要」做索引式加载，细节在对应 `workflow/*.md`。
- **测试与质量闸门**：测试类环节须通过 Subagent 执行；测试派发 Prompt 须符合 [references/pua-testing.md](references/pua-testing.md) 定义的**质量关卡与证据契约**，避免仅写「要认真测试」类劝诫。

- **可审计性（必达）**：调用 `$behavior-log-audit` 完成注入——`SKILL.md` 含「行为日志执行协议」节；`references/audit-quality-standards.md` 存在；各阶段含 Audit Hook；Subagent 返回格式含 `behaviorLogPath`（若适用）。详见 [workflow/inject-new-skill.md](../behavior-log-audit/workflow/inject-new-skill.md)。
- **术语**：全文术语与前置阶段及本 Skill 前文一致，无同一概念多种叫法。

**Guidance**：以 [templates/workflow-skill-template.md](templates/workflow-skill-template.md) 为目录与 `SKILL.md` 形态的**唯一参照**；不在本阶段重写业务规则，前置阶段定稿内容只做映射和拆分。

**Execution Mode**：主控执行；实例化 `SKILL.md`、`workflow/`、`references/`、`templates/`（按需），生成后逐项执行完成判据自检，未通过则补全后再结束。
