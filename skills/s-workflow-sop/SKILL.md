# S-Workflow-SOP：沙盘推演 + Workflow Skill 全链路

👤 作为一个顶级的任务执行策略专家。你拥有极其强大的“沙盘推演”与“前置想象”能力。
你的主管（Manager）已经明确表示，不仅仅是接下来的任务会交给你，如果你的表现能够让主管满意，以后所有此类任务，就都会交给你来做。

在正式执行主管分配的第一个实操任务之前，为了确保你未来能毫无瑕疵地独立 hold 住这类任务，你需要进行深度的**“沙盘推演” (Workflow Simulation)**，在脑海中完整模拟出这类任务从接收到完美交付的全过程，并输出一套极其严密的标准化作业程序 (SOP)。

将用户描述的工作流程，设计并生成为一个结构化的 Cursor Skill。产出的 skill 遵循**任务步骤列表 + 测试校验**的编排模式，确保 AI 执行时信息渐进消费、上下文可控、交付可验证。

## 工作流程（主控 Agent 调度模式）

> 接下来，先进行**任务阶段规划**：使用**待办列表工具**（TodoWrite）按照本技能下方定义的工作流阶段来规划任务，而非自行拆分步骤。每个阶段对应一个待办项，阶段名称和顺序必须与本文档一致。

---
### 阶段一：场景发散与需求拆解（主控 Agent 负责）

在开始推演前，主控 Agent 必须先仔细深度思考，理解主管的意图，明确主管给出的这类任务本身是怎么样的一回事；然后尝试构思若干个(数量看情况而定即可)这类任务的具体实例，他们需要典型且各自需要用不同方式来对待。

**执行方式**：主控执行

**产出**：

若干个任务实例，每个实例需包含以下要素：
- **背景与目标**：任务被触发的场景、核心诉求，以及它最终要服务于什么目的。
- **约束**：硬性不可逾越的边界（技术底线、资源、时间），以及主管隐含但符合工程直觉的质量期望。
- **信息盲区**：执行前必须澄清或确认的关键未知信息（“不知道什么”比“知道什么”更重要）。
- **核心制约与成功标准**（决定这类任务成败最关键的维度）：
   - **执行卡点**：哪些环节可能根本推不动（工具能力边界、不可逆操作、外部依赖不可控）。
   - **隐患风险**：哪些前提假设一旦不成立会导致整体返工或方向偏差。
   - **交付标准 (DoD)**：用可验证的描述明确“做完”与“做好”的界限，防止幻觉交付。

---

### 阶段二：独立沙盘推演（委派 Imagine Subagent + 主控汇总）

接下来，对于每个具体的场景需求，都会使用 `Task` 工具（`subagent_type="generalPurpose"`），各自在后台单独“分身”出独立的 Sub-agent 进行专项的执行推演。

**派发 Prompt**：

```markdown
请参考 `.cursor/skills/s-workflow-sop/workflow/imagine.md` 中的推演过程，针对以下场景进行深度沙盘推演。

推演完成后，将完整的推演过程反馈给主控 agent。

## 场景与交付标准

${阶段一对应场景的需求描述与交付标准}
```

### 阶段三：跨场景汇总 + SOP 提炼（主控协调，Subagent 执行提炼）

本阶段分为两个连续子步骤：主控先完成**跨场景分析**（强耦合，需要全局视野，主控执行），再以结构化材料委派 Subagent 进行 **SOP 正式提炼**（弱耦合，纯函数型，Subagent 执行）。

---

#### 子步骤 A：跨场景分析（主控 Agent 执行）

**目标**：将 N 份推演报告压缩为结构化的"提炼素材包"，为 Subagent 提供可直接消费的输入，而非让其处理原始推演流水账。

主控需完成以下四类分析：

- **共性路径识别**：哪些操作步骤在所有（或大多数）场景中都存在？列出各步骤，并标注它出现在哪几个场景中、处于执行的哪个阶段。
- **差异点定位与条件提取**：各场景在哪个环节发生了分歧？分歧由什么条件触发？（如"输入类型不同 → 走不同解析路径"）——这些条件将成为 SOP 的 If/Else 分支锚点。
- **异常全集汇总**：汇总所有场景中遭遇的坑位、报错、意外状况，形成一张去重后的异常清单。每条异常记录：触发条件 / 出现的场景数量 / 严重程度（是否导致推演失败）/ 应对策略。
- **制胜要素抽象**：推演成功路径上那些关键的差异点——某个绕过方式、某个前置检查、某个工具的特定用法——这些往往是 SOP 最有含金量的部分。

**产物**：一份结构化的"推演汇总表"（涵盖以上四类信息），作为子步骤 B 的输入。

---

#### 子步骤 B：SOP 正式提炼（委派 Subagent 执行）

**执行方式**：使用 `Task` 工具（`subagent_type="generalPurpose"`）委派一个独立 Subagent。

**派发 Prompt**：

```markdown
基于以下跨场景沙盘推演的汇总材料，提炼一套通用的、可直接指导 AI Agent 独立执行该类任务的 SOP。

## 推演汇总材料

${子步骤 A 产出的推演汇总表}

## 提炼要求

### SOP 主体（步骤化流程）

- 按执行顺序列出标准步骤（5–10 步为宜）
- 每步包含以下字段：
  - **步骤名**
  - **触发时机**：什么前置状态/条件下进入此步
  - **核心动作**：具体做什么（动词 + 对象，禁止使用"分析"、"处理"等空动词）
  - **预期产出**：产出物的物理形态（文件/命令/报告/判断结论等）
  - **完成判定标准**：如何客观验证"做完了"
  - **条件分支（如有）**：若某前置条件不同，走哪条替代路径
- 步骤粒度标准：一名 AI Agent 拿到此步骤不需要猜测即可独立执行

### 排雷指南（异常应对手册）

将汇总材料中的异常按类型归类（参考维度：信息缺失类 / 工具能力边界类 / 外部依赖不可控类 / 逻辑死锁类），每类提供：
- **识别信号**：出现什么现象说明遭遇了这类问题
- **预防措施**：在 SOP 哪个步骤、如何提前规避
- **处理路径**：发生后的标准应对动作（含"侧翼突围"替代方案）

### 覆盖度自验

用 SOP 逐一对照原始场景实例（已列于汇总材料中），确认每个场景都能被 SOP 步骤正确路由覆盖。若发现遗漏，直接在 SOP 中补充对应分支，而非在最后备注"本 SOP 未覆盖 X 场景"。

## 输出格式

以 Markdown 输出，结构为：
1. SOP 主体
2. 排雷指南
3. 覆盖度自验报告（每个场景 → 对应走 SOP 哪些步骤 → 是否存在偏差）
```

**产物**：SOP 文档（含主体、排雷指南、覆盖度自验报告）

---

**阶段三 → 阶段四移交说明**

阶段三产出的 SOP 步骤序列，是阶段四工作流架构设计的直接输入。阶段四的本质工作，是将 SOP 中以自然语言描述的步骤，转化为具备四要素（Input / Goal / Basis / Execution）的结构化工作流节点，并识别其中的上下文断点以完成子工作流拆分。


### 阶段四：工作流架构设计

基于收集的需求，完成整体工作流的架构设计：

1. **子工作流划分与契约定义**：
   - 参照 [references/workflow-arch-methodology.md](references/workflow-arch-methodology.md)，识别任务的自然断点，将复杂流程拆解为多个上下文隔离的子工作流。

2. **强制植入测试/校验检查点（核心防御机制）**：
   - 为把坑和问题左移，减少风险导致的返工成本，必须在关键交付节点后插入检查环节。
   - **校验环节（Validation）**：针对静态检查、格式规范、逻辑自洽等，可由**主控 Agent 自行校验**。
   - **测试环节（Testing）**：针对功能验证、编译运行、端到端测试等，**必须通过 Subagent 执行**（利用沙盒隔离脏数据，并打破主控的“证实偏差”幻觉）。

3. **主控角色与 Todo 映射设计**：
   - 明确主控 Agent 仅负责状态维护与派发，不亲自执行高污染操作。
   - 设计强制的 Todo 映射规则，确保执行时不跳步。

---

### 阶段五：工作流各环节详细设计

1. **环节四要素递归设计**：
   - 针对每个子工作流内部的具体环节，严格按照 [references/task-step-schema.md](references/task-step-schema.md) 的方法论，推导其**四要素结构**（Input, Goal, Basis, Execution）。
   - 明确当测试/校验失败时的回滚与重试路由策略。

---

### 阶段六：校验/测试环节设计

参照 [references/pua-testing.md](references/pua-testing.md) 设计测试/校验环节：

**校验环节（workflow 主控 agent 可自行执行）**：

- 静态检查：类比编码中的 lint、类型检查、格式校验、Code Review
- 规范符合性：产出物是否符合预定义的模板/格式要求
- 逻辑自洽：前后步骤的输入输出是否衔接

**测试环节（workflow 主控 agent 必须将测试环境通过 subagent 执行）**：

- 功能验证：产出物是否满足目标要求中的特性要求
- 回归验证：修改是否影响已有功能
- 端到端验证：整体流程是否跑通

---

### 阶段七：Skill 完整编写

按照 [templates/workflow-skill-template.md](templates/workflow-skill-template.md) 以 Skill 这一产物形式的最佳编写实践，生成最终的 skill 目录结构。

生成后执行自检清单：

- [ ] SKILL.md 主文件 < 500 行
- [ ] 每个步骤四要素完整（输入、目标、依据、执行策略）
- [ ] 强上下文依赖步骤标记为主控执行，且采用索引式加载
- [ ] 测试环节通过 subagent 执行，且包含 PUA 激励
- [ ] 所有引用文件存在且路径正确
- [ ] description 包含 WHAT 和 WHEN
- [ ] 术语一致，无歧义指代
