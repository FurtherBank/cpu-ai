# Workflow Skill 行文模式：执行者心智叙事与规范化契约表述

Workflow Skill 的主体内容应同时完成两类任务：

1. 让执行者读起来像在进入一个自然、连续、可自我校验的工作过程。
2. 让 Agent 执行起来有明确输入、目标、边界、分支、校验和交付格式。

因此，本文把 Workflow Skill 的主体内容归纳为两种基础**行文模式**：

- **执行者心智叙事**：负责建立理解、判断力、风险感知和阶段承接。
- **规范化契约表述**：负责固定输入输出、行为边界、完成判定、协作交接和验证要求。

二者不是按“是否第一人称”“是否表格”区分，而是按文本的主要作用区分：它是在塑造执行者的内部判断，还是在外化可执行、可复核的契约。

## 推荐主标题

当需要在 Skill 或模板中显式说明这组行文模式时，推荐使用：

```markdown
## 行文模式：执行者心智叙事 + 规范化契约表述
```

若上下文更偏方法论，也可使用：

```markdown
## Workflow Skill 行文模式
```

这里讨论的是贯穿整个 Skill 主体的行文模式，不是仅用于开头说明的局部段落。

## 执行者心智叙事

**定义**：执行者心智叙事是一种 Workflow Skill 行文模式。它用第一人称或近第一人称的连续叙事，呈现执行者在每个阶段的判断位置、注意力边界、风险感知、回退信号和阶段间承接关系。

它主要回答：

- 我现在面对什么输入。
- 我为什么不能直接进入下一步。
- 我先保护、发现、验证或隔离什么。
- 我如何判断当前阶段是否足够扎实。
- 如果读起来或执行起来不顺，可能意味着哪个信息缺口、流程断点或质量标准不足。

**适用位置**：

- Skill 开头的整体心智模型。
- 每个阶段或子步骤开始前的“我为什么做这一步”。
- 阶段间移交说明，尤其是为什么上游产物是下游的唯一输入。
- Guidance 中的专家判断、风险信号、回退信号。
- 复杂方法论文件的开头或关键章节开头。
- 容易被误解为“可省略”“可随便合并”“只是形式要求”的环节。

**质量判断**：

- 读者是否知道为什么要做这一步。
- 读者是否知道当前阶段承接什么、产出给谁。
- 读者是否能把“不顺畅感”识别为回查输入、阶段边界、上游产物或完成标准的信号。
- 读者是否知道何时继续、澄清、回退、委派或升级。

## 规范化契约表述

**定义**：规范化契约表述是一种 Workflow Skill 行文模式。它把执行要求外化为可执行、可复核、可委派、可测试、可维护的结构化约束。

它主要回答：

- 必须消费哪些输入。
- 要把系统从什么状态变到什么状态。
- 完成时必须满足哪些客观条件。
- 哪些事情明确禁止做。
- 由主控执行还是委派 Subagent。
- 失败、跳过、分支和回退如何处理。
- 产物用什么格式交付，如何被下游消费。

规范化契约表述包含命令式表达，但不等于命令。它可以表现为命令、清单、表格、状态机、决策矩阵、Subagent Prompt、输出模板、测试报告格式、证据要求等形态。具体采用哪种形态，取决于当前内容要稳定的是动作、判断、协作、状态转移还是产物结构。

**适用位置**：

- Input / Goal / Guidance / Execution Mode 四要素。
- 完成判据、DoD、质量标准。
- Subagent 派发 Prompt。
- 输出模板、返回格式、表格字段。
- 校验清单、测试用例、证据要求。
- 禁止事项、阶段权威、执行顺序、命名约束。

**质量判断**：

- 是否可执行：执行者能否照着做。
- 是否可复核：外部能否判断完成/未完成。
- 是否可委派：Subagent 是否能仅凭输入包完成任务。
- 是否可测试：结论是否能绑定证据。
- 是否可维护：长期修改时是否能保留同一语义。

## 区分维度

两种行文模式的核心区分维度是：文本主要作用于执行者的内部判断，还是作用于外部可验证的执行契约。

| 维度 | 执行者心智叙事 | 规范化契约表述 |
| --- | --- | --- |
| 作用对象 | 执行者的理解、注意力、风险感、阶段感 | 行为、输入输出、边界、格式、完成标准 |
| 主要问题 | 我为什么这样做？哪里不顺说明什么？ | 必须做什么？做到什么算完成？ |
| 校验方式 | 读起来是否顺、是否能感知异常、是否知道何时回退 | 是否可执行、可复核、可委派、可测试 |
| 典型语言 | 我现在面对……我先……如果卡住说明…… | 必须……不得……输入为……输出为……满足 P1/P2 |
| 失败表现 | 机械、跳步、读者不知道为什么 | 自由发挥、不可验、交付不稳定 |
| 价值 | 提高不确定场景下的判断力 | 提高稳定场景下的执行正确率 |

## 是否需要第三种行文模式

在 Workflow Skill 主体内容编写中，暂不需要引入与这两种并列的第三种基础行文模式。

很多看似独立的表达形态，如协作剧本、证据化审查报告、状态机转移说明、决策矩阵、产物模板，本质上都是**规范化契约表述**在不同内容对象上的形态变化。它们不是第三种同级模式，而是契约表述用于稳定协作、证据、状态、分流或产物结构时的具体形式。

复杂性通常不来自第三种行文模式，而来自两种模式在不同抽象层级上的组合：

| 抽象层级 | 执行者心智叙事 | 规范化契约表述 |
| --- | --- | --- |
| 全局层 | 我为什么不直接执行或生成 | 阶段权威、全局禁止事项 |
| 阶段层 | 我为什么做这一阶段 | 阶段 Input / Goal / Execution Mode |
| 协作层 | 我为什么委派或回收 | Subagent Prompt、回传格式 |
| 产物层 | 我为什么这样组织结果 | 模板、字段、报告结构 |
| 校验层 | 我为什么在这里拦截风险 | 测试用例、证据要求、通过标准 |

## 选择规则

| 情况 | 优先行文模式 | 判断口径 |
| --- | --- | --- |
| 需要理解 Workflow 为什么存在 | 执行者心智叙事 | 先解释我面对什么问题、为什么不能直接做、这套流程防什么风险 |
| 阶段开始前需要建立注意力边界 | 执行者心智叙事 | 写清我为什么做这一步，以及什么现象说明上游不够扎实 |
| 需要定义输入、输出、完成标准 | 规范化契约表述 | 使用 Input / Goal / Guidance / Execution Mode 或清单/表格 |
| 需要委派 Subagent | 规范化契约表述 | Prompt 必须结构化、硬依赖完整、返回格式明确 |
| 需要描述专家经验或异常信号 | 先心智叙事，后契约表述 | 先说明风险判断，再给可执行处理路径 |
| 需要定义测试、校验、证据要求 | 规范化契约表述 | 必须可核对、可重复、可阻断 |
| 读起来突然跳步、机械、卡顿 | 补执行者心智叙事 | 把卡顿当作信号，检查阶段意图、前后承接或缺失输入 |
| 执行起来自由发挥过多、不可验证 | 补规范化契约表述 | 增加完成判定、禁止事项、返回格式或校验项 |

## 推荐组合形态

一个阶段的理想行文结构通常是：

```markdown
### 阶段 X：阶段名称

**我为什么做这一步**：用执行者心智叙事说明阶段存在的原因、它防止的风险、它承接什么上游产物、给下游提供什么稳定输入。

**Input**：用规范化契约表述列出必须入参和软上下文索引。

**Goal**：用规范化契约表述定义状态变化和完成判定。

**Guidance**：先写必要的执行者心智判断，再写具体操作规则、异常处理和禁止事项。

**Execution Mode**：用规范化契约表述说明主控或 Subagent 如何执行、何时跳过、如何分支、返回什么。
```

这个组合的重点是：叙事负责让流程有心智连续性，契约负责让流程可执行和可验证。

## 最终判断

写 Workflow Skill 时，默认采用两种行文模式协作：

- 用**执行者心智叙事**建立理解、判断力和异常感知。
- 用**规范化契约表述**固定执行、交接和验证。

如果只能保留一种，应先保留规范化契约表述，保证最低可执行性；但对于复杂、认知型、编排型 Workflow，缺少执行者心智叙事会显著降低理解顺畅度和不确定场景下的执行正确率。
