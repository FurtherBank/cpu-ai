# 任务步骤四要素推导方法论 (Task Step Schema & Methodology)

**核心设计哲学：把 Agent 当作"高智商但无常识的外星实习生"**
确定要素内容的最大陷阱是"知识诅咒"（人类觉得理所应当的前提，Agent 完全不知道）。推导要素内容的过程，本质上是**"隐性知识显性化"**的过程。

我们将工作流中每个环节步骤，视为一个**"状态机（State Machine）"**。  
每个步骤的具体内容实现，都需要明确给出以下要素内容：

---

## 一、 输入信息 (Input)：当前状态的切片

**目标**：确定执行该步骤所需的最小必要信息集合，并为每条信息选择合适的提供方式——使 Agent 既不因信息缺失产生幻觉，也不因信息冗余导致注意力涣散。

**方法论：最小必要信息法（Minimum Necessary Input）**

推导输入信息时，最容易犯的错是"给得太多"（导致注意力涣散）或"给得太少"（导致幻觉）。正确做法分两步：**先判断哪些信息是必要的，再决定以何种形式提供**。

### 推导步骤

1. **盲盒测试——确定哪些信息是必要的**：触发一轮新的`深度思考`，进行专门探讨：  
假设你把一位顶尖工程师关在小黑屋里，断掉互联网，只允许他看屏幕上的字来完成"这个特定步骤"。他必须要哪些信息？逐条列出。

2. **依赖分级——区分硬依赖与软上下文**：对步骤一列出的信息逐条分类：
   - **硬依赖（必须入参）**：缺失则无法启动的变量，通常是上一步的直接输出或用户的明确指令；必须显式写入 Input。
   - **软上下文（按需索引）**：执行过程中可按需查阅的"字典"型背景信息；以文件路径或章节引用形式提供，让 Agent 自行读取，不做全量注入。

3. **颗粒度控制——软上下文只传索引，不传全文**：不要直接传入整个代码库或长文档，而是传入文件路径大纲、目录结构，让 Agent 按需读取。避免"把字典当课文背"式的 token 浪费。

> **实践自问（两问）**：  
> - **第一问**："如果去掉这条信息，Agent 会用幻觉来填补吗？" → 若是，则为硬依赖，必须入参。  
> - **第二问**："这条信息是一次性消费，还是执行中按需查阅的？" → 若是后者，只传路径/引用，不传全文。

---

## 二、 目标职责 (Goal)：环节预期做到的状态变化

**目标**：准确定义该环节/子工作流的目标职责，并在执行结果产物层面，刻意阐述其内容形式/物理形态等具体要求，以及若干细节性的"特性要求"，来作为交付成果的质量标准，实现问题的"左移"拦截。

### 推导步骤

触发一轮新的`深度思考`，专门探讨该环节/子工作流目标职责的准确定义:

这里的 Goal 是一个**可判定的状态转移契约**，精确定义为：

`Goal = 状态变更 (Delta) + 完成判定 (Done Predicate)`

- **状态变更（Delta）**：这一步执行后，相比输入状态，系统里"新增/改变/消除"了什么事实。
- **完成判定（Done Predicate）**：在不看主观解释的前提下，外部审查者（脚本、subagent 或人）可用同一标准判断"已完成/未完成"。

准确定义应当可直接落地成以下模板：
```
在输入 I 条件下，把系统从 S0 变到 S1；当且仅当满足 P1..Pn 时判定完成。
```

其中：
- `S1` 是"下一个状态"的事实描述（目标本体），不是交付物清单。
- 完成判定条件中，需要按照事实情况明确提到：
  - **定义"交付物（What）"的物理形态**：触发一轮新的`深度思考`，进行专门探讨：目标必须是具体的名词（如"输出一份包含 3 个数组的 JSON"或"在终端执行一条带有 4 个参数的 bash 命令"）。
  - **定义"特性要求（How good）"的检验函数（左移思维）**：触发一轮新的`深度思考`，进行专门探讨：去复盘人类过去在这个环节踩过的坑。例如人类常忘写异常捕获，特性要求就必须加上："必须包含 try-catch 兜底，并记录 error log。"
  - **划定"不做什么（Negative Prompts）"的边界**：Agent 极容易"越俎代庖"。如果当前步骤只是"确定修改路径"，目标中必须明确特性："不要生成具体代码，只输出文件路径。"

**one-shot 示例（步骤：定位 Bug 根因）**

✅ 正例：
```
在输入「报错日志 + 相关代码文件」条件下，
把系统从「只知道报错现象」变到「已锁定 Bug 的精确成因」；
当且仅当同时满足以下条件时，判定完成：
  P1. 指出具体文件路径、函数名及触发该错误的代码行号
  P2. 用"输入 → 中间状态 → 报错"的完整逻辑链条解释为何出错
  P3. 结论可被日志或代码片段中至少一处原文交叉验证
  P4. 未在本步骤生成任何修复代码（越权）
```

❌ 反例：
```
目标：定位 Bug，理解根本原因，为下一步修复做准备。
```
> "理解"无法判定，P1–P4 全部缺失；Agent 猜一个可能原因就会宣布完成。

> **实践自问**："当 Agent 说它完成了这一步时，我能写一段脚本（或用另一个 subagent）自动化地判断它及格了吗？如果不能，说明目标定得不够具体。"

## 三、 工作依据 (Guidance)：执行的知识武装

**目标**：为 Agent 提供执行该步骤时的参考材料，内容可以包含两类：**流程规范依据**（可选）和**方法论依据**。确保执行过程的专业性与完备性。

其中：
- 流程规范依据对偏执行类工作流大概率存在，对创造性工作则可能完全没有；
- 方法论依据几乎总是需要的，负责传递专家直觉、例外处理与"机灵特质"

**方法论：专家经验萃取与例外管理（Expert Heuristics Extraction）**

这是四要素中**最难也是含金量最高**的部分。流程规范（SOP）不一定存在，"方法论依据"则往往藏在资深员工的直觉里，需要主动萃取。

### 推导步骤

1. **确认流程规范依据(如有)**：先判断该步骤是否有现成的 Wiki、编码规范或标准操作流程。  
若有，抽取其核心路径，制作成 `.md` 索引供 Agent 读取，而不是全塞进 prompt；  
若无（如创造性任务），直接跳过，进入下一步。
2. **正常路径的思维链克隆**：触发一轮新的`深度思考`，进行专门探讨：以专家视角深度自问："执行这一步时，脑子里的第一反应是什么？顺利时的操作直觉是什么？"（例如排查 Bug 优先从后往前看 error trace）。萃取出的直觉即为方法论依据的核心。
3. **异常路径的陷阱与"机灵特质"注入**：触发一轮新的`深度思考`，进行专门探讨：穷举出该步骤最容易出现的异常与边界情况。写入依据："遇到 `command not found` 时，不要立即放弃报错，你应当主动使用 `ls` 和 `find` 寻找正确路径，尝试不少于 3 次修复后再向主控报告失败。"

> **实践自问**："一个新手拿到该步骤的输入，照着网上的常规做法一定会搞砸的地方在哪里？把避免搞砸的诀窍写进方法论。"

## 四、 执行模式 (Execution Mode)：
**目标**：确认这个步骤在 Agent 流程层面的实际执行模式，是应当由主控 Agent 执行，还是应当在一个单独的上下文中由 subagent 执行。


**方法论：上下文耦合度量测（Context Coupling Measurement）**

决定是"主控执行"还是"subagent 委派"，核心依据是**上下文的"污染率"和"折旧率"**。

### 推导规则
1. **评估上下文耦合度（依赖过去的记忆吗？）**
   - **强耦合**：Agent 需要默认理解上下文中**难以结构化提供的信息**进行调整思路等后续操作。例如：上一步设计的架构意图，需要看全局中细节性的状态，或根据刚才几轮的修改报错等。
     👉 **结论：主控执行（渐进式加载）**。
     Skill 内容编写方式：将指导文件写在 `workflow/step_X.md` 中，并提示让主控读取。  
     指导文件中需要提示需要着重关注主控上下文哪些方面的内容
   - **弱耦合（纯函数型）**：不关心之前发生了什么，只看当前输入（如"跑一遍单元测试提取报错行数"）。
     👉 **结论：Subagent 委派**。一次性打包上下文，通过 Task 派发，做完就销毁。防干扰且省 token。
2. **评估执行风险度（需要隔离沙盒吗？）**
   - 涉及执行未知代码、危险 Shell 操作（如 `rm -rf`、`curl | bash`）。
     👉 **结论：Subagent（`shell` 类型）**。主控绝不能自己去执行可能导致系统崩溃的操作。
3. **评估任务的可拆分性与并行度（是单一线索还是多分支？）**
   - **多独立分支**：上一阶段给本阶段的输入，可以明确按照边界（scope/task、不同文件、不同模块）划分为多条性质相同、相互独立且具备一定规模的任务。
     👉 **结论：Subagent 并发委派**。主控将任务结构化拆解后，通过 Task 工具同时拉起多个 Subagent 并行处理，最后由主控统一汇总结果。既能极大提升执行效率，又能避免主控在长线重复劳动中丢失上下文。

### Subagent 委派：入参与提示词构造规范

当判定为 Subagent 委派时，**提示词（prompt）是唯一的信息通道**——subagent 没有主控的对话历史，没有隐性上下文，一旦漏传关键信息就会产生幻觉或跑偏。因此，提示词内容必须显式、完整地传入所有必要信息。

#### 提示词必须包含的内容层级（强制）

| 层级 | 内容 | 说明 |
|------|------|------|
| **原始需求（必传·原样）** | 用户最初的完整需求描述 | **禁止摘要或改写**，必须原文照搬。subagent 需要"知道自己最终为谁服务、为什么做这件事"，摘要会丢失边界条件和隐性约束。 |
| **本步骤入参（必传）** | 该步骤的硬依赖数据，即上一步的直接输出或主控执行过程中产生的关键变量 | 对照本要素"一、输入信息"中确定的硬依赖列表逐条传入，不可省略。 |
| **本步骤目标与完成判定（必传）** | 该步骤的 Goal，含交付物形态和完成判定条件 | subagent 必须知道"做到什么程度算完成"，否则无法自我判断是否达标。 |
| **方法论依据（必传）** | 专家直觉、异常处理诀窍等 Guidance 内容 | 必须传入。若存在对应的方法论文件（如 `./workflow/step_X.md`），**必须以文件路径索引形式传入**（如"以 `./workflow/step_X.md` 为指导依据"），不得将文件全文内联进提示词；若无独立文件，则将核心方法论要点以文字形式直接写入提示词。 |
| **软上下文索引（按需传）** | 文件路径、目录结构等可供 subagent 自行读取的背景资料索引 | 传路径/引用，不传全文，防止 token 浪费。 |

#### 推导步骤：如何构造 Subagent 提示词

1. **锚定原始需求**：提示词开头必须以一个明确的块（如 `## 原始需求` 段落）原样嵌入用户最初的需求全文。这是 subagent 理解"任务语境"的根基，任何缩写或转述都会引入偏差。

2. **逐条检验硬依赖是否已在提示词中出现**：对照步骤一（输入信息）中确定的硬依赖清单，逐项确认其已被显式写入提示词正文（或以确定性文件路径引用），不依赖 subagent 自行推断。

3. **写明完成判定条件**：将 Goal 的 Done Predicate（P1..Pn）完整复制进提示词，让 subagent 在完成时能自我对照校验，而不是自己判断"差不多了"。

4. **设定返回格式约定**：明确 subagent 应以何种格式/结构将结果返回给主控（如 JSON、纯文本行、特定 Markdown 结构），避免主控解析失败。

> **实践自问**："如果把这段提示词发给一个与本任务毫无关联的全新 Agent，它能在不追问任何问题的情况下完成这个步骤并返回正确格式的结果吗？" → 若不能，说明有信息遗漏，必须补入。

## 综合实战演练：自动修复代码 Bug 工作流的"步骤 2：定位根因"

**1. 思考 Input（盲盒测试）：**
- 必须有步骤1的"报错日志"，和报错对应的"代码文件内容"。
- *推导结果*：入参=`报错日志`；上下文=`发生报错的相关代码文件（通过 explore 搜索读取）`。

**2. 思考 Goal（判定驱动）：**
- 怎样算"定位成功"？必须指出具体位置和修改方向。
- *推导结果*：目标=`锁定导致Bug的具体文件、函数及代码行号，并阐述错误发生的逻辑链条`。特性要求=`不准在这里直接写出修复代码，必须先解释机理`（左移检验）。

**3. 思考 Guidance（专家萃取）：**
- 专家是怎么定位这种 Bug 的？
- *推导结果*：方法论=`1. 优先阅读业务代码堆栈，忽略第三方库报错；2. 重点排查异步状态更新；3. 遇到前端状态不一致，先假设是竞态条件。`

**4. 思考 Execution Mode（耦合度量测）：**
- 找 Bug 需要不断根据线索去翻阅不同文件，高度依赖中间过程的发现。
- *推导结果*：`主控执行`。

> **终极检验**：不要把写 Prompt 当成写文章，要把写要素当成**"编写具备自愈能力的防御性代码"**。每写下一条输入、目标或依据，都要问自己："这行约束能封死哪一种 AI 偷懒、跑偏或产生幻觉的可能？"这样设计出的工作流，才是工业级可用的。
