# 工作流环节规格四要素说明

每个工作流环节都应被写成可执行、可核对的状态转移规格，而不是泛泛的说明。环节规格由四要素组成：Input / Goal / Guidance / Execution Mode。

核心原则：把隐性知识显性化。不要假设 Agent 自动知道背景、边界、完成标准或异常处理方式。

我写四要素时，会把自己脑子里的“当然应该知道”全部外化。Input 防止执行者缺上下文，Goal 防止完成标准漂移，Guidance 防止新手按表面动作踩坑，Execution Mode 防止主控和 Subagent 的职责混在一起。

---

## Input：当前状态切片

**定义**：Input 描述执行当前环节所需的最小必要信息。它既要避免信息缺失导致幻觉，也要避免信息过载导致注意力涣散。

**应包含**：

- **硬依赖**：缺失就无法启动或无法判对错的信息，通常是上一环节直接输出或用户明确指令；必须写入 Input。
- **软上下文**：执行中按需查阅的背景资料，如规范、目录结构、方法论文档；提供路径或章节引用，不全文注入。
- **粒度边界**：软上下文只给索引，不把整份代码库或长文档塞进环节规格。

**确认项**：

- 去掉这条信息，Agent 是否会用幻觉填补？若是，它是硬依赖。
- 这条信息是一次性消费，还是执行中按需查阅？若是后者，只给路径或引用。
- Input 中是否混入了本环节不会消费的信息？若是，应移除或下沉为软上下文索引。

## Goal：可判定的状态变化

**定义**：Goal 描述当前环节要把系统从什么状态变到什么状态，以及外部如何判定它完成。

Goal 的形式：

```text
Goal = 状态变更（Delta） + 完成判定（Done Predicate）
```

推荐表达形式：

```text
在输入 I 条件下，把系统从 S0 变到 S1；当且仅当满足 P1..Pn 时判定完成。
```

**应包含**：

- **S1 是事实状态**，不是单纯的交付物清单。
- **完成判定必须可核对**，外部审查者、脚本或 Subagent 能按同一标准判断通过/未通过。
- **交付物物理形态要明确**，如“输出包含 3 个数组的 JSON”“执行带 4 个参数的命令”。
- **完成质量要可判定**，与本环节完成状态直接相关的质量条件应写入完成判定。
- **边界要写清**，如“本环节不生成代码，只输出修改路径”。

示例：定位 Bug 根因。

```text
在输入「报错日志 + 相关代码文件」条件下，
把系统从「只知道报错现象」变到「已锁定 Bug 的精确成因」；
当且仅当同时满足以下条件时，判定完成：
P1. 指出具体文件路径、函数名及触发错误的代码位置
P2. 用“输入 → 中间状态 → 报错”的链条解释为何出错
P3. 结论可被日志或代码片段交叉验证
P4. 未在本环节生成修复代码
```

反例：

```text
目标：定位 Bug，理解根本原因，为下一步修复做准备。
```

问题：“理解”不可判定，缺少位置、逻辑链、证据和越权边界。

**确认项**：

- 当 Agent 说完成时，能否写一段脚本或派另一个 Subagent 判断是否及格？若不能，Goal 还不够具体。
- Goal 是否只描述本环节的状态变化，而没有越界要求下游环节的产出？
- 完成判定是否能逐条核对，而不是依赖“理解”“优化”“完善”等主观词？

## Guidance：执行依据与专家经验

**定义**：Guidance 描述执行当前环节时应遵守的依据、应采用的经验判断，以及常见异常的处理方向。

Guidance 可包含三类内容：

- **规范依据**：必须遵守的外部规则、模板、Wiki 或编码规范；以索引形式引用，避免全文注入。
- **专家经验**：帮助执行者做出更好判断的优先级、操作顺序或识别信号。
- **异常提示**：常见失败模式、容易踩坑的位置，以及推荐处理方向。

**应包含**：

- 有现成规范时，给出文件路径或章节引用。
- 有经验性判断时，说明它适用的场景和避免的问题。
- 有异常提示时，写清识别信号和处理方向。
- 当环节包含多个高承重点，且后续需要统一判断、总方案或框架整合时，引用 [attention-budget-integration-guidance.md](attention-budget-integration-guidance.md)。此类环节必须先让每个承重点完成单点封口，再进入关系判断和整合；不得为了降低整合上下文难度而削弱上游单点探索。

示例：

```text
遇到 command not found 时，不要直接判定失败。
先确认当前目录和 package.json scripts，再检查项目文档中的命令约定；
若仍无法定位，向主控报告缺失的命令依据和已检查位置。
```

**确认项**：

- Guidance 中哪些内容是必须遵守的规范，哪些只是经验建议？二者是否容易混淆？
- 新手照常规做法最容易搞砸哪里？避免搞砸的诀窍是否已经写入 Guidance？
- Guidance 是否只写“参考某文档”，却没有提取本环节必须知道的关键判断？

## Execution Mode：执行模式控制

**定义**：Execution Mode 描述当前环节由谁执行，以及在什么条件下执行、跳过或进入不同分支。执行主体是 Execution Mode 的主轴；触发、跳过和分支条件用于控制该执行主体在实际工作流中的运行方式。

**应包含**：

- **执行主体**：本环节由主控 Agent 执行，还是由 Subagent 委派执行；如需委派，说明委派类型和信息传递要求。
- **触发条件**：什么前置状态、输入条件或上游结果出现时，本环节应执行。
- **跳过条件**：什么情况下本环节不应执行，或应直接交给后续环节处理。
- **分支条件**：存在不同执行路径时，写清每条路径的判定条件和对应动作。

**执行主体判定口径**：

- **主控执行**：环节强依赖过程上下文、架构意图、连续发现或前后状态维护。
- **Subagent 委派**：环节弱依赖历史上下文，只看当前输入即可执行，或需要隔离执行噪音/风险。
- **高风险操作**：涉及未知命令、大型测试、浏览器长流程、高污染日志等，主控不得直接执行，应委派合适类型的 Subagent。

### 当 Execution Mode 为 Subagent 委派时，Prompt 应包含

Subagent 没有主控的对话历史，Prompt 是唯一信息通道。必须显式传入：

| 层级 | 内容 | 要求 |
|------|------|------|
| 原始需求 | 用户最初的完整需求描述 | **必传且原样传入**；禁止摘要或改写，避免丢失边界条件和隐性约束 |
| 本环节入参 | 硬依赖数据 | 对照 Input 清单逐项传入 |
| 目标与完成判定 | Goal 和 Done Predicate | 让 Subagent 能自检是否达标 |
| 方法论依据 | Guidance 或指导文件路径 | **必传**；有文件时传路径索引，不内联长文；无独立文件时写入核心方法论要点 |
| 软上下文索引 | 可查阅的文件路径、目录结构 | 按需传，不传全文 |
| 返回格式 | JSON、Markdown 或特定结构 | 避免主控无法解析 |

**Prompt 信息完整性要求**：

1. 是否原样保留原始需求。
2. 硬依赖是否全部出现。
3. 完成判定是否完整复制。
4. 方法论依据或指导文件路径是否明确。
5. 返回格式是否可被主控消费。

**确认项**：

- 把这段 Prompt 发给一个全新 Agent，它能否不追问就完成任务并按格式返回？若不能，补齐缺失信息。
- Execution Mode 是否先明确了本环节由主控执行还是 Subagent 委派执行？
- Execution Mode 是否说明了本环节何时执行、何时跳过、何时进入不同分支？
- Execution Mode 是否说明了选择主控或委派的原因？
- 委派任务的输入、完成判定和返回格式是否足以被主控继续消费？

## 综合示例：定位 Bug 根因

**Input**：

- 硬依赖：上一环节产出的报错日志。
- 软上下文：报错相关代码文件路径，由主控按需搜索读取。

**Goal**：

- 状态变化：从“只知道报错现象”变为“已锁定具体成因”。
- 完成判定：指出文件/函数/位置；解释错误链条；结论可被日志或代码验证；不生成修复代码。

**Guidance**：

- 优先阅读业务代码堆栈，先忽略第三方库噪音。
- 重点排查异步状态、边界输入和最近改动。
- 遇到线索不足时，先补充日志或缩小复现路径，不直接猜根因。

**Execution Mode**：

- 执行主体：默认主控执行。原因是定位过程依赖连续线索、上下文判断和多文件阅读。
- 触发条件：已获得报错日志，且日志足以指向至少一个可排查的代码区域。
- 跳过条件：没有报错日志或复现现象时，不进入根因定位，先回到信息补充环节。
- 分支条件：若日志指向业务代码，优先主控连续阅读；若日志主要来自大型测试或高污染命令输出，可委派 Subagent 做隔离排查。

---

## 四要素完整性检查

- Input 是否足以让执行者知道本环节依赖什么。
- Goal 是否足以让外部判断本环节是否完成。
- Guidance 是否足以降低常见误判、越界或失败风险。
- Execution Mode 是否先说明由谁执行，再说明何时执行、何时不执行、如何分支，以及在委派时需要传递哪些信息。
- 每条 Input、Goal、Guidance 和 Execution Mode 约束，是否都能减少一种幻觉、跑偏或误判风险。
