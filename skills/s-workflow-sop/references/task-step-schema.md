# 任务步骤四要素推导方法论 (Task Step Schema & Methodology)

**核心设计哲学：把 Agent 当作"高智商但无常识的外星实习生"**
确定要素内容的最大陷阱是"知识诅咒"（人类觉得理所应当的前提，Agent 完全不知道）。推导要素内容的过程，本质上是**"隐性知识显性化"**的过程。

我们将工作流中每个环节步骤，视为一个**"状态机（State Machine）"**。  
每个步骤的具体内容实现，都需要明确给出以下要素内容：
- **Input (输入信息)**：当前状态的切片。
- **Goal (目标要求)**：环节预期做到的状态变化。
- **Basis (工作依据)**：状态转移的规则与判断准则。
- **Execution (执行策略)**：状态转移的计算引擎。

---

## 一、 输入信息 (Input)：当前状态的切片

**目标**：定义执行任务的"硬依赖"（必须入参）与"软上下文"（背景补充），并遵循"渐进式消费"原则，优先以索引形式提供信息，避免 Agent 注意力涣散。

**方法论：最小依赖与盲盒测试法（Information Dependency Graph）**

推导输入信息时，最容易犯的错是"给得太多"（导致注意力涣散）或"给得太少"（导致幻觉）。

### 推导步骤
**输入溯源（盲盒测试）**：触发一轮新的`深度思考`，进行专门探讨：  
假设你把一位顶尖工程师关在小黑屋里，断掉互联网，只允许他看屏幕上的字来完成"这个特定步骤"。他必须要哪些信息？

**注意事项**
- **信息内容按照是"硬依赖"与"软上下文"区分提供策略**：
  - 硬依赖（必须入参）是没有就无法启动的变量，通常是上一步的直接输出或用户的明确指令；
  - 软上下文（索引补充）是执行过程中按需查阅的"字典"，一般以对应所在范围的引用/路径形式提供即可。
- **控制颗粒度（Token 经济学）**：不要直接传入整个代码库，而是传入文件路径大纲或 AST 结构作为索引，让 Agent 按需读取，避免注意力涣散。

> **实践自问**："如果去掉这条信息，Agent 会用幻觉来填补吗？如果会，这就是必须入参。"

## 二、 目标职责 (Goal)：环节预期做到的状态变化

**目标**：准确定义该环节/子工作流的目标职责，并在执行结果产物层面，刻意阐述其内容形式/物理形态等具体要求，以及若干细节性的"特性要求"，来作为交付成果的质量标准，实现问题的"左移"拦截。

### 推导步骤

触发一轮新的`深度思考`，专门探讨该环节/子工作流目标职责的准确定义:

这里的 Goal 是一个**可判定的状态转移契约**，精确定义为：

`Goal = 状态变更 (Delta) + 完成判定 (Done Predicate)`

- **状态变更（Delta）**：这一步执行后，相比输入状态，系统里"新增/改变/消除"了什么事实。
- **完成判定（Done Predicate）**：在不看主观解释的前提下，外部审查者（脚本、subagent 或人）可用同一标准判断"已完成/未完成"。

准确定义应当可直接落地成以下模板：
```
在输入 I 条件下，把系统从 S0 变到 S1；当且仅当满足 P1..Pn 时判定完成。
```

其中：
- `S1` 是"下一个状态"的事实描述（目标本体），不是交付物清单。
- 完成判定条件中，需要按照事实情况明确提到：
  - **定义"交付物（What）"的物理形态**：触发一轮新的`深度思考`，进行专门探讨：目标必须是具体的名词（如"输出一份包含 3 个数组的 JSON"或"在终端执行一条带有 4 个参数的 bash 命令"）。
  - **定义"特性要求（How good）"的检验函数（左移思维）**：触发一轮新的`深度思考`，进行专门探讨：去复盘人类过去在这个环节踩过的坑。例如人类常忘写异常捕获，特性要求就必须加上："必须包含 try-catch 兜底，并记录 error log。"
  - **划定"不做什么（Negative Prompts）"的边界**：Agent 极容易"越俎代庖"。如果当前步骤只是"确定修改路径"，目标中必须明确特性："不要生成具体代码，只输出文件路径。"

**one-shot 示例（步骤：定位 Bug 根因）**

✅ 正例：
```
在输入「报错日志 + 相关代码文件」条件下，
把系统从「只知道报错现象」变到「已锁定 Bug 的精确成因」；
当且仅当同时满足以下条件时，判定完成：
  P1. 指出具体文件路径、函数名及触发该错误的代码行号
  P2. 用"输入 → 中间状态 → 报错"的完整逻辑链条解释为何出错
  P3. 结论可被日志或代码片段中至少一处原文交叉验证
  P4. 未在本步骤生成任何修复代码（越权）
```

❌ 反例：
```
目标：定位 Bug，理解根本原因，为下一步修复做准备。
```
> "理解"无法判定，P1–P4 全部缺失；Agent 猜一个可能原因就会宣布完成。

> **实践自问**："当 Agent 说它完成了这一步时，我能写一段脚本（或用另一个 subagent）自动化地判断它及格了吗？如果不能，说明目标定得不够具体。"

## 三、 工作依据 (Basis)：状态转移的规则

**目标**：提供完成任务的 SOP 规范或方法论经验，包括对特定情况的处理指导和激发 Agent "机灵"特质的激励性提示，确保执行过程的专业性与完备性。

**方法论：专家经验萃取与例外管理（Expert Heuristics Extraction）**

这是四要素中**最难也是含金量最高**的部分。流程规范（SOP）容易找，但"方法论依据"往往藏在资深员工的直觉里。

### 推导步骤
1. **SOP 的锚定（寻找客观依据）**：抽取现成 Wiki、编码规范的核心路径，制作成 `.md` 索引供它读取，而不是全塞进 prompt。
2. **思维链的克隆（推导方法论）**：触发一轮新的`深度思考`，进行专门探讨：问人类专家："你在做这一步时，脑子里的第一反应是什么？如果卡住了你会怎么办？"（例如排查 Bug 优先从后往前看 error trace）。
3. **陷阱与"机灵特质"注入**：触发一轮新的`深度思考`，进行专门探讨：穷举出该步骤最容易出现的异常。写入依据："遇到 `command not found` 时，不要立即放弃报错，你应当主动使用 `ls` 和 `find` 寻找正确路径，尝试不少于 3 次修复后再向主控报告失败。"

> **实践自问**："一个新手拿到这几个入参，照着网上的常规做法一定会搞砸的地方在哪里？把避免搞砸的诀窍写进方法论。"

## 四、 执行策略 (Execution)：
**目标**：确认这个步骤在 Agent 流程中，是应当由主控 Agent 执行，还是应当在一个单独的上下文中由 subagent 执行。

**方法论：上下文耦合度量测（Context Coupling Measurement）**

决定是"主控执行"还是"subagent 委派"，核心依据是**上下文的"污染率"和"折旧率"**。

### 推导规则
1. **评估上下文耦合度（依赖过去的记忆吗？）**
   - **强耦合**：Agent 需要知道上一步设计的架构意图，需要看全局状态，或根据刚才几轮的修改报错来调整思路。
     👉 **结论：主控执行（渐进式加载）**。将指导文件写在 `workflow/step_X.md` 中让主控读取。
   - **弱耦合（纯函数型）**：不关心之前发生了什么，只看当前输入（如"跑一遍单元测试提取报错行数"）。
     👉 **结论：Subagent 委派**。一次性打包上下文，通过 Task 派发，做完就销毁。防干扰且省 token。
2. **评估执行风险度（需要隔离沙盒吗？）**
   - 涉及执行未知代码、危险 Shell 操作（如 `rm -rf`、`curl | bash`）。
     👉 **结论：Subagent（`shell` 类型）**。主控绝不能自己去执行可能导致系统崩溃的操作。

## 综合实战演练：自动修复代码 Bug 工作流的"步骤 2：定位根因"

**1. 思考 Input（盲盒测试）：**
- 必须有步骤1的"报错日志"，和报错对应的"代码文件内容"。
- *推导结果*：入参=`报错日志`；上下文=`发生报错的相关代码文件（通过 explore 搜索读取）`。

**2. 思考 Goal（判定驱动）：**
- 怎样算"定位成功"？必须指出具体位置和修改方向。
- *推导结果*：目标=`锁定导致Bug的具体文件、函数及代码行号，并阐述错误发生的逻辑链条`。特性要求=`不准在这里直接写出修复代码，必须先解释机理`（左移检验）。

**3. 思考 Basis（专家萃取）：**
- 专家是怎么定位这种 Bug 的？
- *推导结果*：方法论=`1. 优先阅读业务代码堆栈，忽略第三方库报错；2. 重点排查异步状态更新；3. 遇到前端状态不一致，先假设是竞态条件。`

**4. 思考 Execution（耦合度量测）：**
- 找 Bug 需要不断根据线索去翻阅不同文件，高度依赖中间过程的发现。
- *推导结果*：`主控执行`。

> **终极检验**：不要把写 Prompt 当成写文章，要把写要素当成**"编写具备自愈能力的防御性代码"**。每写下一条输入、目标或依据，都要问自己："这行约束能封死哪一种 AI 偷懒、跑偏或产生幻觉的可能？"这样设计出的工作流，才是工业级可用的。
