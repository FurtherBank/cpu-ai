# 注意力预算型多环节整合 Guidance

当工作流环节包含多个高承重点，且下游还要做统一判断、总方案或框架整合时，引用本 Guidance。它的目的不是让流程更复杂，而是防止为了整合便利而压缩上游单点质量。

## 适用信号

- 一个环节中有多个方向、多个判断点、多个争议命题或多个候选分支。
- 每个点单独确认都可能需要约 1000 字以上的信息量，包括反例、备选、约束推演、证据梳理或多轮讨论。
- 分支之间互相依赖，但任一分支的结论都会改变后续整合方式。
- 主控已经出现“先概括一遍”“每点写一段”“最后统一总结”的倾向，而用户实际要求深挖。

## 核心原则

1. **先单点封口，再整合**：每个承重点都达到可被下游依赖的局部状态后，才允许做综合结论。
2. **整合难度不能反向削弱上游投入**：不能因为上游充分展开会增加后续整合成本，就把单点探索压薄。
3. **关系判断晚于单点判断**：先确认每个分支自身，再判断分支之间是互斥、互补、层级、条件分流，还是共同指向新上层问题。
4. **上下文用交接摘要降载**：上游材料很长时，生成可追溯的短摘要给整合层消费，而不是减少上游探索。

## 单点封口标准

每个承重点至少输出以下局部状态，才算有资格进入整合：

- 暂定判断或当前最强命题。
- 支撑理由、证据或现实约束。
- 强反例、失败方式或近敌。
- 适用边界与不可用场景。
- 仍未知的问题。
- 可追溯到完整材料的交接摘要。

如果任一项缺失，后续环节只能输出“待整合缺口”，不得强行总结合并。

## Execution Mode

- 聚焦形式只有两种：主控刻意聚焦，或单独 Subagent 聚焦。
- 单个承重点预计需要千字级探索时，必须使用单独 Subagent 聚焦；不要写成“可能使用 Subagent”“考虑 Subagent”或“Subagent/新鲜上下文均可”。
- 一个 Subagent 只处理一个精确承重点；不要把多个高承重点合并成“综合讨论”。
- 主控职责是定义承重点、派发单点任务、检查封口质量、维护依赖图、最后整合关系。
- 整合层如果本身也需要千字级推演，必须作为单独环节处理；达到 Subagent 条件时，同样必须使用单独 Subagent，输入只包含各单点的封口摘要和必要原文索引。

## 交接摘要格式

```markdown
### 承重点

### 暂定判断

### 支撑依据

### 反例/失败方式

### 边界

### 未知

### 对整合层的影响
```

## 完成判定

一个多环节整合步骤完成，当且仅当：

- 所有必要上游承重点都有封口摘要。
- 未封口承重点被明确标为阻塞项或排除项。
- 整合结论逐项说明依赖了哪些封口结果。
- 冲突没有被措辞抹平，而是被标注为互斥、条件分流或待研究。
- 整合没有新增未验证的上游命题；如新增，必须回到单点封口流程。
