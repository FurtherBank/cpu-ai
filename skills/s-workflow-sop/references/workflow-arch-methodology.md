# 工作流架构设计方法论

工作流架构的目标是控制全局复杂度、降低上下文衰减，并保证最终交付不偏离目标。复杂工作流应被设计成带检查点的状态机流水线：每个阶段只消费上游精炼产物，不依赖全量对话历史。

## 阶段划分

**目标**：通过上下文隔离降低信息过载和幻觉风险。

划分准则：

- **寻找自然断点**：识别认知负荷切换点，如从方案设计转入编码、从实现转入验证。
- **隔离冗余细节**：若阶段 A 的尝试过程、报错修复、废弃方案对阶段 B 没有直接价值，应拆成独立阶段。
- **定义阶段契约**：每个阶段必须有明确的硬依赖输入和阶段性交付物。下一阶段只消费交付物，不消费完整过程。

## 风险左移

**目标**：在错误放大前拦截问题，并用独立验证降低确认偏误。

关键交付节点后必须设计校验或测试：

- **校验（Validation）**：面向形式与静态事实，适合主控 Agent 执行。检查对象包括 Schema、字段完整性、格式、规范、输入输出契约、逻辑自洽。
- **测试（Testing）**：面向行为与功能验证，必须委派 Subagent 执行。测试负责验证真实场景、端到端链路、回归影响和场景覆盖度。

## 主控与 Subagent 边界

**目标**：让主控维护状态和调度，Subagent 在隔离上下文中执行具体任务。

- **主控 Agent**：维护 DoD、当前阶段状态、上游产物、派发上下文和回收结果。强上下文依赖的工作由主控执行。
- **Subagent**：执行弱上下文耦合、可独立验证、高污染或高风险的原子任务，并返回精炼结果。
- **高污染/高风险操作**：主控不得亲自执行；如大型测试、海量日志分析、未知命令验证、浏览器长流程，应委派合适类型的 Subagent。

## Todo 映射

Skill 执行时必须使用 `TodoWrite` 将预定义阶段映射为待办项。阶段名称和顺序应与 Skill 文档一致，防止执行中跳步、合并或自行改名。

## 契约化派发

唤起 Subagent 时，主控必须使用预设 Prompt 模板，并传入必要上下文：

- 原始需求或必要背景。
- 当前步骤硬依赖输入。
- 当前步骤目标与完成判定。
- 方法论依据或指导文件路径。
- 返回格式要求。

前序阶段产物应原样填入模板占位符，避免主控二次加工导致契约失真。

## 归档与可追溯

长流程的最后应设计归档环节，用于记录核心决策、DoD、修改路径、验证结论和遗留风险。归档结果既服务人工 Review，也为后续任务提供稳定软上下文。

## 设计推导 SOP

1. **定义终局**：明确最终交付物的物理形态，如文件、服务、配置、文档或报告。
2. **拆分阶段**：按上下文切换点和信息折旧情况拆分子工作流，并定义输入输出契约。
3. **植入防御**：在关键产物后插入校验（主控）或测试（Subagent）节点。
4. **细化环节**：对每个步骤应用 `task-step-schema.md` 推导 Input / Goal / Guidance / Execution Mode。
5. **定义模板**：为 Subagent 派发和关键产物输出准备模板。
6. **设计归档**：定义最终记录文件的格式和存放位置。
