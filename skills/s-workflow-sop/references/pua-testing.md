# 测试质量关卡与证据契约

> 文件名因历史原因保留为 `pua-testing.md`。本文在本 Skill 信息架构中的职责是：定义测试与校验这两个 scope 的边界、执行关系、证据标准和回传结构。

我设计测试节点时，不把它当作流程末尾的礼貌性确认。主控在长流程后最容易相信自己的产物已经正确，所以我把形式校验留给主控，把行为验证交给独立 Subagent，并要求每个结论都有可追溯证据。

## 信息架构定位

测试/校验相关信息必须先归位，再写入工作流：

- **归属**：校验属于主控自检 scope；测试属于独立验证 scope。
- **依存**：测试依赖明确的验证标准、测试范围、用例或场景、环境信息、证据要求。
- **约束**：测试结论必须由证据支撑；不得用“看起来正常”“代码推断通过”替代实际验证。
- **时序**：校验/测试节点应位于关键产物生成之后、下游依赖消费之前。
- **交换**：测试 Subagent 必须返回结构化报告，主控只消费报告中的结论、证据、失败项和遗留风险。
- **影响**：测试失败会影响上游步骤，需要回退修复并重新测试。
- **协同**：校验负责低成本拦截形式问题；测试负责隔离上下文并验证行为正确性，二者共同构成质量闸门。

## 校验 vs 测试

| 类型 | 执行者 | 目的 | 示例 |
|------|--------|------|------|
| **校验（Validation）** | 主控 Agent | 检查产物是否满足形式、结构、静态事实要求 | lint、类型检查、格式校验、Schema/契约对齐 |
| **测试（Testing）** | **必须委派 Subagent** | 验证功能、行为、端到端链路和场景覆盖 | 功能测试、回归测试、端到端验证 |

测试必须委派 Subagent：主控在长流程后容易受自身上下文影响，倾向于确认产物正确。独立 Subagent 用新的上下文执行验证，能降低确认偏误并隔离执行噪音。

## 测试 Subagent Prompt 契约

测试 Prompt 必须显式包含以下内容。

### 明确验证标准

不要写模糊指令：

```markdown
请测试一下这个功能是否正常。
```

应写成可执行、可判定的标准：

```markdown
请逐项验证以下 ${N} 个用例。每个用例必须返回：实际执行的操作、预期结果、实际结果、是否通过。不允许跳过任何用例。
```

### 强制证据输出

每个测试项都必须附带可验证证据：

- 命令行操作：记录完整命令和关键输出。
- 浏览器操作：描述操作路径和观察到的页面结果；必要时附截图说明。
- 代码检查：引用具体文件路径、符号或相关片段。

声称“已通过”但没有证据的测试项，视为未执行。

### 质量关卡说明

在测试 Prompt 中明确 Subagent 的职责边界：

```markdown
## 执行要求

你是本次交付的独立质量关卡。请按验证标准实际执行测试，并用证据支撑结论。

要求：
- 每个测试用例必须实际执行，不允许只基于代码阅读推断结果。
- 发现问题时，必须给出复现步骤、实际现象和可能涉及的位置。
- 如果所有用例都通过，请额外检查一个最可能出问题的边界场景。
- 输出先列失败项和风险，再列通过项；没有失败项时，也必须说明证据为何足以支撑通过结论。
```

### 回归测试补充

涉及修改已有功能时，Prompt 还必须要求验证影响面：

```markdown
本次修改涉及 ${N} 个文件。请验证：
1. 修改功能本身是否正确。
2. 修改是否影响已有功能。

请主动识别受影响的上下游模块并验证。不要只测试改动点本身。
```

### 场景化 Prompt 片段

根据测试类型选择或组合以下片段。片段只定义行为要求，不使用威胁式激励。

**通用测试片段**：

```markdown
## 执行要求

你是本次交付的独立质量关卡。你的结论必须来自实际执行和可验证证据。

要求：
- 每个测试用例必须实际执行，不允许基于代码阅读推断结果。
- 发现问题时，必须给出复现步骤、实际现象、影响范围和可能涉及的位置。
- 如果所有用例都通过，请额外检查一个最可能出问题的边界场景。
- 输出先列失败项和风险，再列通过项；没有失败项时，也必须说明证据为何足以支撑通过结论。
```

**功能测试片段**：

```markdown
你是独立测试执行者，职责是验证功能是否真实满足目标，而不是确认实现“看起来没问题”。

要求：
- 对每个功能点，至少覆盖正常路径和 1 个异常路径。
- 测试前先阅读相关实现，识别最薄弱或最容易遗漏的路径并优先验证。
- 没有发现问题时，必须用执行证据说明为何当前功能点可以判定通过。
```

**回归测试片段**：

```markdown
本次修改涉及 ${N} 个文件。请同时验证：
1. 修改功能本身是否正确。
2. 修改是否影响已有功能。

对于第 2 点，请主动识别受影响的上下游模块并验证。不要只测试改动的部分。
```

## 测试报告标准格式

测试 Subagent 必须按以下结构返回：

```markdown
# 测试报告

## 测试环境
- 执行时间：${timestamp}
- 测试范围：${scope}

## 测试结果汇总
- 总用例数：${total}
- 通过：${passed}
- 失败：${failed}
- 跳过：${skipped}（必须说明原因）

## 详细结果

### 用例 1：${用例描述}
- **执行操作**：${具体操作}
- **预期结果**：${expected}
- **实际结果**：${actual}
- **结论**：${通过/失败/跳过}
- **证据**：${命令输出/截图说明/代码引用}

## 额外发现
${测试过程中发现的非用例覆盖问题}

## 遗留风险
${未覆盖范围、环境限制或需要人工确认的事项}
```

## 校验环节设计指南

校验由主控 Agent 执行，设计时遵守：

1. **校验项可自动化**：优先使用命令、脚本、结构化检查或明确清单。
2. **校验失败即阻断**：校验未通过时不得进入下一步骤。
3. **校验结果可追溯**：记录每项校验的命令、检查对象和结果。

```markdown
**校验清单**（示例）：
- [ ] `npm run lint` 通过，无新增 error
- [ ] `npm run type-check` 通过，无类型错误
- [ ] 产出文件符合模板格式，关键字段非空
- [ ] 前后步骤的输入输出契约匹配
```
