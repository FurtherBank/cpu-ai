# 测试校验环节 PUA 激励策略

测试和校验是把坑和问题左移、减少返工成本的关键环节。本文件定义了如何在工作流的测试环节中融入 PUA 激励策略，确保 agent 尽全力执行而非敷衍了事。

## 校验 vs 测试

| 类型 | 执行者 | 目的 | 示例 |
|------|--------|------|------|
| **校验** | 主控 agent 自行执行 | 快速检查产出物是否符合形式要求 | lint、类型检查、格式校验、规范符合性 |
| **测试** | **必须通过 subagent** | 验证产出物的功能正确性和完整性 | 功能测试、回归测试、端到端验证 |

测试必须通过 subagent 执行的原因：主控 agent 在长流程后容易产生"确认偏误"——倾向于认为自己的产出没问题。独立 subagent 无此偏误。

## 测试 Subagent Prompt 构造规范

### 核心原则：不给偷懒空间

测试 subagent 的 prompt 必须包含以下要素：

#### 1. 明确的验证标准（而非模糊指令）

```markdown
❌ "请测试一下这个功能是否正常"
✅ "请逐项验证以下 ${N} 个用例，每个用例必须给出：实际执行的操作、预期结果、实际结果、是否通过。不允许跳过任何用例。"
```

#### 2. 强制证据输出

```markdown
在你的测试报告中，每个测试项必须附带可验证的证据：
- 命令行操作：贴出完整的命令和输出
- 浏览器操作：描述具体的操作路径和观察到的结果
- 代码检查：引用具体的文件路径和行号

声称"已通过"但没有证据的测试项视为未执行。
```

#### 3. PUA 激励嵌入

在测试 subagent 的 prompt 末尾追加激励段落，从以下模板中选取适合当前场景的内容：

**通用测试激励**：
```markdown
## 执行要求

你是本次交付的最后一道质量关卡。你的测试结论直接决定这个功能能否上线。

请注意：
- 每个测试用例必须实际执行，不允许基于代码阅读"推断"结果
- 发现问题时，必须精确定位到代码位置和复现步骤
- 如果所有用例都通过，请再额外检查一个你认为最容易出问题的边界场景
- 你的测试报告会被人工审查，敷衍的报告会被打回重做

记住：漏掉一个 bug 到线上 = 生产事故。你就是防线。
```

**功能测试激励**：
```markdown
你是独立的测试工程师，不是开发者的助手。你的职责是找出问题，而不是确认"没问题"。
- 对每个功能点，至少测试正常路径 + 1 个异常路径
- 测试前先看代码，找到最薄弱的地方优先测试
- 输出时先报问题，后报通过项。没有问题比有问题更需要充分证据证明
```

**回归测试激励**：
```markdown
本次修改涉及 ${N} 个文件。你需要验证：
1. 修改的功能本身是否正确（新功能测试）
2. 修改是否影响了已有功能（回归测试）

对于第 2 点，请主动识别受影响的上下游模块并验证。不要只测改动的部分。
改了 A 不验证 B 和 C，等线上出问题就是你的责任。
```

## 测试报告标准格式

测试 subagent 必须按此格式返回：

```markdown
# 测试报告

## 测试环境
- 执行时间：${timestamp}
- 测试范围：${scope}

## 测试结果汇总
- 总用例数：${total}
- 通过：${passed}
- 失败：${failed}
- 跳过：${skipped}（必须说明跳过原因）

## 详细结果

### ✅ / ❌ 用例 1：${用例描述}
- **执行操作**：${具体操作}
- **预期结果**：${expected}
- **实际结果**：${actual}
- **证据**：${命令输出/截图描述/代码引用}

### ...

## 额外发现
${测试过程中发现的非用例覆盖的问题}
```

## 校验环节设计指南

校验由主控 agent 自行完成，设计时注意：

1. **校验项必须可自动化**：能用命令/脚本完成，不依赖主观判断
2. **校验失败即阻断**：校验不通过时禁止进入下一步骤
3. **校验结果可追溯**：记录每项校验的执行命令和输出

```markdown
**校验清单**（示例）：
- [ ] `npm run lint` 通过，无新增 error
- [ ] `npm run type-check` 通过，无类型错误
- [ ] 产出文件符合模板格式（关键字段非空）
- [ ] 前后步骤的输入输出接口匹配
```
