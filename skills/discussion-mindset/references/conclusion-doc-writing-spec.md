# 研讨结论说明文档：写作规格

## 这份规格管什么

研讨（`discussion-mindset`）跑完、多个承重内容单元已单点封口之后，如果要把这一阶段的结论沉淀成一份**完整、可长期复用的说明文档**，按本规格写。

它只管一件事：怎么把**已经封口的研讨结论**写成给人读的文档。研讨本身怎么做（方向校准、灵魂点搜寻、隔离聚焦、单点封口）不归本规格管，那是 `discussion-mindset` 本体加 `f-attention-budget-constraint` 的事。

## 何时用（触发条件）

仅在以下任一条件成立时才产出这类文档，它**不是研讨的默认收尾**：

- 用户明确要求把结论写成文档、说明、报告。
- 研讨结论需要长期复用、跨语境引用，或要交给第三方，口头小结已经承载不住。

如果只是想让用户看见研讨过程、或留一个接力接口，用 `discussion-mindset` 的「对外呈现」问题包就够了（即该技能 SKILL.md「对外呈现」一节给出的那套轻量问题包结构：当前锋利判断、为什么重要、备选灵魂点、反例与现实约束、搜索质量、下一轮问题），不必上这份规格。

## 前置硬条件：结论必须已单点封口

这类文档是**整合呈现，不是新推导**。动笔前先确认：文档要写的每个核心结论，都已经在研讨阶段达到单点封口——有暂定判断、支撑理由、强反例、适用边界、未知项。

如果还有承重单元没封口，先回 `discussion-mindset` / `f-attention-budget-constraint` 把它封口，**不要在写文档的过程中顺手补推导**。写作主线里补出来的判断没经过隔离确认，混进正式产物就是把未确认内容伪装成已确认结论。

## 产出目标和定位

我们通过研讨得出的结论，一般是**高信息密度**，**高度理论化**的语言文字风格来表达的。当我们真正将这份结论输出为一篇定稿后，实测这种风格的表达会让其他主体很难做到能够直接的深度理解。  
因此，我们编写结论说明文档，除了写明结论内容本身外，还需要通过合适的方式举例子——通过叙事的形态，以**情境中具体发生了什么，走向了什么情况的方式**，让读者可以很容易的理解，这一点本身是在讲什么，对应的不同情况会意味着什么。  
- **档位**：默认按 `writing-expression` 的第二档（人类读的持久文档）处理；若要落进知识库长期复用或对外发布，按第三档处理，走该规则的落盘关口。
- **先判读者的阅读任务**：即读者要用这份文档完成什么。研讨结论文档的读者任务通常是两条——「理解这个话题到底在讲什么」加「能在自己的情境里认出它、知道不同情况各意味着什么」。全篇组织都服务这两条，而不是罗列研讨产出了多少东西。

## 写作规格

### 按读者动线组织，不照搬推导单元的输出

研讨阶段常把话题拆成多个承重单元分别封口，可能由多个 Subagent 各自产出一份结论。写文档时**不要**把这些单元的原始输出按产出顺序拼起来。要按读者的理解路径重排，典型顺序是：**这是什么 → 为什么它反直觉 / 为什么重要 → 怎么识别或怎么做 → 边界和例外**。

子单元的结论是原料，读者动线才是产物。同一个封口结论，在文档里出现的位置由「读者此刻需要它来理解什么」决定，而不是由它在研讨里被推导出来的先后决定。

### 叙事例子是承重内容，不是装饰（本规格的重点）

每个承重概念都要配至少一个**情境化叙事例子**，让抽象结论在具体场景里被读者认出来。一个合格的叙事例子满足三条：

- **是具体情境，不是概念复述**：有主体、有动作、有走向，读者能在脑里想象出画面。「某人在某处做了某事，然后发生了什么」，不是把结论换个说法再讲一遍。
- **显形分叉**：同一个起点，因为某个关键处理方式不同，走向不同结局；把「不同情况各意味着什么」显式摆出来，而不是只讲一条正确走法。分叉本身就是让读者看懂「这一点在讲什么」的机制——读者是通过看到两条路的差别，才理解这个点在管什么。
- **锚回抽象点**：例子末尾用一句话点明它对应文档里的哪个结论（例如统一用「这个例子在讲什么」收尾），不要让读者自己猜例子想说什么。

叙事例子承重，所以在 `f-attention-budget-constraint` 的闸门下，每个关键例子都应作为承重内容单元对待：先确认它真的显形了目标分叉，再确认它没有为了生动而扭曲原结论。为了画面好看而牺牲结论准确，是这一体裁最典型的失败方式。

### 自造术语落白话注解

沿用研讨中产生的自造术语时，遵守 `writing-expression` 的造词契约：首次出现给定义，紧跟一行白话注解（「说白了就是……」）或一个微型示例；一个概念自始至终只用一个名字。

研讨内部为了高密度沟通造的命名，不要直接搬进面向读者的文档而不解释——研讨参与者脑里的语境，裸读者没有。

### 边界、置信与反例：不许为了收束而删

研讨结论几乎总带着反例、适用边界、未知项、残留硬核。这些**恰恰是文档最容易被误当「万能结论」而用错的地方**，必须保留并显式成节，通常包括：适用边界（哪些场景不管用）、置信分层（实证 / 推断 / 未知）、已知反例或残留问题。

不要为了让文档显得干净统一而压掉单点的复杂度。这与 `f-attention-budget-constraint` 整合层「不得为漂亮统一感回头压低任何单点的复杂度」同源：整合是关系判断的落笔，不是把粗糙抹平。

### 格式

遵循 `markdown-writing`：标题默认不编号，流程 / 时序 / 状态 / 关系用 Mermaid，只有对外分享或明确要求图文的文档才加图片占位。

## 生产流程

满足「何时用」、且研讨上下文较重（封口结论都在当前对话历史里）时，按以下方式产出：

1. **fork 一个 Subagent 承担写作**：用 `f-attention-budget-constraint` 的 fork 方式（cursor：`Task` 且 `resume: "self"`；codex：`fork_context: true`），让写作线程继承整段研讨历史，从而直接拿到全部单点封口结论，避免把大量结论重新塞进 prompt 时失真或遗漏。派发时在 prompt 里额外写清：产出目标和定位（一份完整说明文档）、落盘路径、读者与档位、以及「按本规格 `references/conclusion-doc-writing-spec.md` 写作」。
2. **写作线程走 `f-attention-budget-constraint`**：把文档的承重部分——核心定义、每个叙事例子、边界与置信声明、一句话总括——当承重内容单元，按其闸门与动作分档处理，而不是一路顺写到底。
3. **落盘关口**：第三档产物或对外发布物写完后，按 `writing-expression` 的落盘关口做冷读裸测；第二档做落笔自查。

如果研讨上下文很轻，或用户只要一小节说明，可以不 fork，直接在主线按本规格写。**fork 与否是成本权衡，写作规格本身不变。**

## 完成自检

- 每个核心结论都能在文档里找到对应的叙事例子，且例子显形了分叉、锚回了抽象点。
- 自造术语首次出现都有白话注解，且一概念一名。
- 适用边界、置信分层、反例 / 未知项成节保留，没有为收束被压掉。
- 组织是读者动线，不是推导单元的顺序拼接。
- 所有结论都来自已封口的研讨判断，没有在写文档时新混入未确认的推导。
