# 「粒度」概念升维重构报告

> 基于 CCR-SOP（认知概念升维重构标准操作程序）产出

---

## 前置条件：什么样的信息才值得讨论粒度

粒度只对**具备执行意图的信息**有意义。需同时满足以下三个条件，粒度讨论才有价值：

| 条件 | 含义 | 缺失时的情形 |
|------|------|------------|
| **有执行意图** | 信息携带了「让某人/某系统做某件事」的方向，而非纯粹描述事实 | 纯描述/记录/存档类信息，粒度无意义 |
| **有执行主体** | 能够确定由谁来消费并执行这段信息——可以是显式指定的，也可以从场景、文档归属、组织惯例中隐式推断 | 无法从任何上下文中推断出执行方是谁（信息真正悬空、无人认领），粒度无意义 |
| **有结果期望** | 作者希望执行方做完后产生某种结果——不论这个期望有多模糊 | 作者完全不在意执行方做完后是什么状态（纯发散讨论、头脑风暴），粒度无意义 |

> **注意**：「结果期望是否具体/收敛」不是前置条件的判断标准，而是粒度本身在度量的内容。只要存在「希望发生某件事」的方向，前置条件即满足——期望越模糊，说明粒度越粗，而非条件不满足。

**适用范围的一句话判断：**

> 「如果执行方没有按照这段信息去做任何事，我会感到有什么地方出了问题吗？」
> 若是 → 这段信息具有执行意图，粒度值得讨论。
> 若否 → 这是描述性信息，粒度不是这里的核心问题。

## 阶段一：解剖与参数化

### 函数签名

$$Granularity(A, S, D, T)$$

| 参数 | 含义 |
|------|------|
| A (Artifact) | 被描述的内容单元本身 |
| S (Subject) | 执行方（人、AI、系统、团队） |
| D (Domain) | 内容所属问题域（技术方案、产品设计、操作手册……） |
| T (Task) | 内容服务的目标动作（评审、编码、部署、讨论……） |

### IPO 模型

**Input（触发粒度感的因素）**

- 执行方读取内容后，尚需补充的**背景知识量**
- 内容所描述步骤的**执行分支数**（可能路径的空间大小）
- **信息缺口**：执行方必须自行引入的外部决策数量与复杂度

**Process（粒度的本质结构）**

粒度是对**「执行方必须自行生成的决策的总搜索空间」**的度量：

$$G(A, S) = \sum_{i=1}^{n} \log_2 |\Omega_i|$$

- $n$：执行方面临的决策缺口数量
- $|\Omega_i|$：第 $i$ 个决策的可选答案空间大小（越开放，值越大）
- 决策被显式确定 → 对应 $|\Omega_i| = 1$ → 该项贡献归零
- 所有决策归零 → $G = 0$ → 极细粒度，可直接执行

**Output（可观测现象）**

| 粒度 | 执行方行为 | 结果 |
|------|-----------|------|
| $G \gg 0$（粗） | 自行补充决策，各人补法不同 | 产出分歧、返工、「没想到」 |
| $G \approx 0$（恰当） | 无需补充，直接推进 | 精确落地，无意外 |
| $G < 0$（过细） | 被不必要的约束限制合理实现 | 过度规约，僵化 |

## 阶段二：现实压力测试

### 常态切片（食谱场景）

| 指令 | 决策缺口 | $G$ |
|------|---------|-----|
| 「调味」 | 调料种类 + 用量 + 时机 + 方式 ≥4个 | 高 |
| 「加入适量盐」 | 「适量」= 多少？+ 时机 | 中 |
| 「翻炒30秒后加入3克盐」 | 0个 | ≈0 |

→ 三层指令被精确区分，定义有效 ✅

### 极化切片（军事命令）

- **粗**：「攻占高地203」→ 表面只有1个决策，但 $|\Omega|$ 趋于无限（路线、时机、兵力全部开放）
- **细**：「明日0600，第一连主攻，沿东侧山路，0800前完成」→ $G \approx 0$

→ 暴露了定义的关键修正点：**决策数量不等于粒度**，决策的搜索空间大小同样关键。纯粹用"数量"度量会失准，必须引入 $|\Omega_i|$。

### 突变切片（执行方背景切换）

「由柯西不等式，可得……」

- 对博士生：$G \approx 0$，直接推进
- 对高中生：需查定义、验证条件 → $G \gg 0$

→ **同一内容，执行方背景不同，粒度感完全不同**，验证了粒度的相对性。定义中 $S$（执行方）作为参数是必要的 ✅

## 阶段三：划定防守边界

### 近敌辨析

| 维度 | **粒度** | **详细程度（Verbosity）** | **完整性（Completeness）** |
|------|---------|------------------------|--------------------------|
| 关注焦点 | 执行方需要自补的**决策搜索空间** | 内容的**篇幅与字数** | 是否覆盖**所有场景/边界** |
| 是否随人变化 | 是（与执行方背景强相关） | 否（内容固有属性） | 否（内容固有属性） |
| 核心问题 | 「还需要想什么才能做？」 | 「写了多少字？」 | 「有没有漏掉情况？」 |
| 反例共存 | — | 高详细但高粒度：废话连篇但结论模糊 ✅ | 完整但高粒度：列出所有case但每个都没说怎么做 ✅ |

### 必要的二元区分：粒度不足 vs 粒度留白

这两者在结构上形态相同（都存在开放决策缺口），但性质根本不同：

| | **粒度不足（缺陷型）** | **粒度留白（授权型）** |
|--|--|--|
| 决策缺口的识别 | 作者未意识到此处存在决策 | 作者明确识别并有意保留 |
| 执行方的感知 | 不知道「这里需要我来决策」 | 清楚地知道「这里授权给我」 |
| 典型场景 | 设计文档遗漏 agent 提示词设计 | 法律条文保留法官自由裁量空间 |
| 结果 | 产出分歧、意外返工 | 产出合理但有差异（可接受） |
| 修复方式 | 必须补充设计 | 不需要修复 |

**一句话结论：**

> 粒度 = 执行方在理解或执行某内容单元时，必须自行生成的所有决策的「搜索空间之和」；与内容长短无关，与执行方背景强相关；粗粒度本身中性，「粒度不足」与「粒度留白」的区别在于作者是否有意识地做出了这个选择。

## 阶段四：杠杆定律

### 客观定理（描述性）

**定理一：粒度的相对性定律**
粒度没有绝对值。当执行方的背景知识 $B$ 增加时，同一内容的等效粒度变细。「写到什么层次」的本质是对执行方背景的校准问题，而非对内容本身的属性判断。

**定理二：粒度临界点定律**
存在一个临界粒度 $G^*$（由执行方背景决定），在该点以下执行产生收敛（结果可预测）；高于该点执行产生发散（结果依赖执行方个体）。对执行准确性有要求的场景，必须将内容粒度降至 $G^*$ 以下。

**定理三：信息缺口的来源二分法**
执行方需补充的决策来自两种来源，成本差异巨大：

| 类型 | 来源 | 补全成本 |
|------|------|---------|
| 可寻找型 | 信息存在于代码库、文档、领域惯例中，但未被引用进来 | 低（查阅即可） |
| 需设计型 | 信息在任何地方都不存在，必须凭空决策 | 高（等同于把设计任务转移给执行方） |

粗粒度内容中，若缺口是「需设计型」，则该内容实质上是在把**设计决策**悄无声息地压给了执行方。

**定理四：过细粒度的副作用**
当粒度细到「规约了本不需要规约的细节」时，会剥夺执行方的合理实现空间，产生不必要的刚性约束。最优粒度是 $G \approx G^*$，而非 $G = 0$。

### 行动准则（规定性）

**判断粒度是否足够的标准问题：**
> 「如果让两个背景相似但互不沟通的人各自独立执行这段内容，他们的产出会一致吗？」
> 若否 → 粒度不足，存在未被显式化的决策。

**永远不要**用「描述意图」代替「定义行为」来结束一个需要执行的设计结论——这只是把决策转移给了执行方，并制造了一个隐形的分歧点。

**降低粒度的操作路径（按成本从低到高）：**

- **引用已有信息**：把散落在代码、文档、惯例中的「可寻找型」信息直接写进来（如字段名、接口路径格式）
- **枚举决策分支**：把「执行方可能面临的选择」显式列出并给出选择结论
- **设计并固化**：对「需设计型」信息缺口，在内容本身中完成设计（如写出 agent prompt 框架），而不是留白

**识别「伪细粒度」的方法：**
内容看起来很详细（字数多、有例子），但关键决策点仍然是开放的——这是详细程度高但粒度仍粗的典型陷阱。测试方法同上：两人独立执行，结果是否收敛。

## 阶段五：粒度演进体系

粗粒度要求不能直接执行，需要通过补充信息演进为细粒度要求，才能保证执行不出问题。这个过程形成一个**要求树**，理解演进机制是管理复杂任务的核心能力。

### 演进的基本结构

一个粗粒度要求 $R$ 演进为细粒度的过程：

$$R_{粗} + \Delta I \rightarrow \{R_1, R_2, \ldots, R_n\}_{细}$$

$\Delta I$ 是补充进来的信息。**演进有效的充分条件是：$\Delta I$ 正确且完整。**

- **正确**：$\Delta I$ 与实际约束一致，推导链可被他人独立复现
- **完整**：没有遗漏决策缺口（用「两人独立执行，结果是否收敛」验证）

### $\Delta I$ 的三种来源与识别方法

三种类型通过级联排除判断，不是并列的特征匹配：

```
面对一个决策缺口：
  ↓
Q1：给定当前已知的约束集合，能否用纯逻辑推导出唯一答案？
  → 是：可推断型（写出推导链，验证逻辑可被他人独立复现）
  → 否：↓

Q2：信息是否存在于某个外部来源？能否给出具体查找位置（文件路径/文档章节/代码位置）？
  → 是：可寻找型（给出引用位置，验证适用性）
  → 否：需设计型（显式标记，记录选择理由和备选方案）
```

**关键约束：** 无法给出具体推导链的就不是可推断型；无法给出具体引用位置的就不是可寻找型。这是排除法，不允许模糊归类。

### 演进失误的两种模式

**忽视型**：执行方没有意识到某个维度存在决策缺口，直接执行时遗漏了该维度的考量。

发生机制有三种：

| 机制 | 原因 | 示例 |
|------|------|------|
| 领域惯例掩盖决策性 | 该维度在执行方经验中有"默认答案"，看起来不像需要决策 | 「生成文档」→ 默认同步生成，未意识到同步/异步是决策点 |
| 反馈延迟掩盖决策性 | 决策后果不在当下体现，注意力聚焦在立即可验证的部分 | 权限粒度设计——初期功能正常，数月后才暴露越权问题 |
| 注意力焦点距离远 | 注意力集中在要求主干，边缘配套决策点超出当前注意力范围 | 实现主逻辑时，「失败错误处理」在注意力外围被遗漏 |

**错误判断型**：意识到了需要决策，但选错了方向。通常因参照来源有误或推断前提不成立。

参照来源可信度信号：

| 信号 | 判断 |
|------|------|
| 能给出具体位置（文件/行号/章节） | 可信度高 |
| 来源是「我之前做过类似的」 | 需警惕：上下文可能不同，须验证适用性 |
| 来源与当前场景存在显著差异 | 须显式验证适用性，否则视为无效引用 |

### 最隐蔽的失误：来源类型误判

**把「需设计型」误当成「可推断型」或「可寻找型」**，是代价最大、最难察觉的失误——执行方以为有依据，实际上是在无意识中即兴发明了一个设计决策。

预警信号：

**误当可推断型时：**
- 推导链中出现「通常」「一般来说」「应该」「合理的做法是」等模糊词
- 被追问「为什么」时，答案诉诸直觉或经验，而非具体的前提条件
- 推导链无法被另一个人独立复现

**误当可寻找型时：**
- 找不到具体的引用位置（只能说「参考了业界惯例」）
- 引用来源与当前场景有显著差异，未做适用性验证
- 引用来源是自己过去的决策（循环引用，没有独立依据）

### 要求树的完整结构

完整演进后，一个粗粒度要求会生长为一棵**要求树**，形成事情的完整执行计划。

**节点：** 每个要求 $R_i$，携带：
- 当前粒度评估 $G(R_i, S)$
- 若已演进：$\Delta I$ 的来源类型标注
- 验证记录（推导链 / 引用位置 / 设计决策理由）

**边：** $R_{粗} \xrightarrow{\Delta I} \{R_1 \ldots R_n\}$，每条边的 $\Delta I$ 须标注来源类型和验证状态

**叶节点终止条件：** $G(R_i, S) \approx 0$，执行方无需补充任何决策即可产出确定性结果。终止条件依赖执行方 $S$——换一个背景的执行方，原叶节点可能需要继续展开。

**树的健康度判断：**

```
对每条演进边的 ΔI：
  可推断型 → 推导链可被他人独立复现             → 健康
  可寻找型 → 有具体引用位置，且适用性已验证      → 健康
  需设计型 → 决策已显式标记，理由和备选项已记录  → 健康
  以上任何一条不满足                            → 该节点存在隐蔽风险
```

**核心警告：** 一棵看起来很细的树也可能是不健康的。只要某个中间节点的 $\Delta I$ 是未被识别的「需设计型」决策，其所有下游节点即便粒度极细，也是**精确地建立在错误假设上**。

细粒度保证执行的确定性，但不保证执行方向的正确性。

### 演进体系的核心结论

要求树的健康度由三个独立条件共同决定：

- **来源识别的准确性**：每个 $\Delta I$ 通过级联判断正确归类，无类型误判
- **验证动作的完成**：可推断型有推导链，可寻找型有具体引用且适用性已验证，需设计型有显式决策记录
- **终止条件的相对性**：叶节点的判定要对齐具体的执行方，换执行方可能需要重新展开
