# 阶段四：差距分析 → 变更清单 执行指导

## 输入信息

**必须入参**：

- 阶段三概念→落点映射表（含技术决策结论）— 提供「在哪里」
- 阶段二系统状态摘要 — 提供「当前状态是什么」
- 阶段三点五设计文档（若有）— 提供「实现设计类」条款的具体实现规格，**必须在写对应变更条目前阅读**

**上下文参考**：

- 四态差距类型定义（见下方）
- 置信度定义（见下方）

## 目标要求

**任务**：对每个规范落点，对比当前状态，用四态标注产出结构化变更清单

**目标**：变更清单按五层组织（DB/API/服务/配置/数据迁移），每条精确到表/字段/接口粒度

**特性要求**：

- 每条变更有 ID、四态类型、具体描述、规范条款 ID、置信度
- 描述字段禁用模糊词（「相关逻辑」「需要改」「鉴权模块」等），必须到具体落点（表名/字段名/接口路径）
- 五层均有记录，即使某层为空也显式说明「无此层变更」
- 低置信条目有具体验证建议（非「需要再确认」）
- **不产出依赖顺序或分批计划**（留阶段六）
- **「实现设计类」条款对应的变更条目，必须将阶段三点五设计文档的具体规格内联到描述中**（不能写「参考设计文档」，必须将关键规格直接写进条目）

## 工作依据

### 四态差距类型定义

**新增**：规范要求该落点存在，但当前系统完全没有

- 示例：规范要求「开发者 App 表」，系统无此表 → 新增 `developer_app` 表

**修改**：当前有对应物，但字段/行为需要变更（语义一致，但不完整或不符合约束）

- 示例：`auth_token` 表有，但缺少 `token_type` 字段 → 修改，新增 `token_type` 字段

**保留+扩展**：当前有对应物，语义与规范一致，只需在此基础上添加（不破坏现有）

- 示例：`user` 表语义与规范的「产品用户」一致，只需新增 `personal_scope` 关联表 → 保留+扩展

**冲突重写**：当前有对应物，但语义与规范冲突，不能简单修改（需要区分新旧语义）

- 示例：`tenant_id` 当前是组织 ID 语义，规范要求开放平台租户语义 → 冲突重写（必须新建字段，不能复用旧字段）
- **触发条件**：术语映射表中「语义是否一致=否」的条目，或同名字段但含义不同

**「冲突重写」的处理规则**：

- 绝不复用旧列的语义（即使字段名相同），必须新建字段并在命名上区分
- 旧字段/接口保留（不删除），等待调用方迁移完成后再规划下线
- 必须触发「调用方影响评估」

---

### 五层结构

**按此顺序逐层完成，不能跳过层**：

**DB 层**：表的新增/修改/删除；字段的新增/修改；索引（若影响「要不要」而非「怎么建」）

- 特别注意：若有字段新增且旧数据需要回填，在「数据迁移层」也要有对应条目

**API 层**：接口路径的新增/修改/废弃；请求/响应字段变更；行为变更（默认值/错误码）

**服务层**：核心业务逻辑的新增/分拆；服务边界变化；依赖关系变化

**配置层**：特性开关、权限配置、允许列表、环境变量；若引入新配置源（Redis/DB/文件）也在此层记录

**数据迁移层**：旧数据回填、数据格式转换、数据清洗；NOT NULL 约束收紧前的回填；语义变更后的数据更新

---

### 置信度规则

**高**：migration + 路由/handler 双重证据，无矛盾

**中**：单一证据（只有 migration 或只有 handler 引用），或存在继承/全局 filter 未完全展开

**低**：动态注册/动态路由；无 migration 文件；基于推断而非直接证据

**低置信条目的验证建议格式**：

- 要具体可执行（`rg "字段名" --type ts src/` 这样的粒度）
- 不能写「需要进一步确认」「建议复查」等无操作性的内容
- 示例：「`rg "tenant_id" --type ts app/module/weavefox_openapi/` 确认是否有其他使用点」

---

### 「冲突重写」类变更的调用方影响评估

若变更清单中出现「冲突重写」类条目，必须额外完成：

1. 全量检索：搜索所有引用受影响字段/接口的代码位置
2. 每个调用方评估：
   - 调用路径（文件路径 + 行号范围）
   - 影响类型（读 / 写 / 既读又写）
   - 兼容策略（双读期 / 版本切换 / 同期改造）

## 粒度检查（变更清单初稿完成后必须执行）

变更清单写完后，对每条变更条目逐一执行以下扫描。**发现任何一项，立即修复后再进行下一步，不得带着问题进入阶段五**：

**信号 1：「参考现有实现」类间接引用**

- 识别模式：「参考同目录下已有 X」「遵循 X 的实现模式」「类似 Y 的方式」
- 处理：回到阶段二实现范式摘要，找到具体文件路径 + 类名/函数签名，替换间接引用

**信号 2：未定义的函数/类调用**

- 识别模式：变更条目描述中出现了一个函数名/类名，但它在：①本变更清单中没有对应新增条目，②在现有代码库中也不存在
- 处理：为该函数/类补充定义变更条目，或将调用改为现有实现

**信号 3：「待确认」设计决策**

- 识别模式：「待与产品确认 X」「根据实际情况调整」「暂定 X，后续可改」
- 处理：回到该条款，识别这是「粒度不足（须补设计）」还是「粒度留白（已授权）」。若是前者，回到阶段三点五补充设计；若是后者，改写为显式的粒度留白声明（见下方格式）

**粒度留白声明格式**（用于有意授权 spec-execute 自行决策的场景）：

```markdown
> **粒度留白（已授权）**：[具体的开放点描述]
> **授权边界**：[spec-execute 可以自行决策的范围限制，如「函数命名风格由实现者决定，但函数须接受 (context: AgentContext) 参数」]
> **不在授权范围内**：[明确禁止 spec-execute 自行决定的内容]
```

**判断标准（置信度 vs 粒度的区分）**：

「置信度」描述「这个变更条目的方向是否正确」，不描述「这个变更条目是否足够细」。一个高置信度条目仍然可能粒度不足。对每条标注「高置信度」的条目，额外问：「如果两个不沟通的工程师各自执行这条变更，他们的实现会有哪些实质差异？」若答案非空，粒度仍不足，须修复。
