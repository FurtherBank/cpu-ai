# 阶段三：概念落点映射 + 技术决策 执行指导

## 输入信息

**必须入参**：

- 阶段一条款索引表（领域概念列表 + 技术决策依赖标记）
- 阶段二系统状态摘要（当前落点信息）
- 阶段二术语映射表（若存在）

**上下文参考**：

- 技术决策边界定义：「影响要改哪些表/接口/配置（存在性）= plan 层必须决定；影响怎么实现（索引大小/TTL/分表）= execute 层决定」

## 目标要求

**任务**：为每个规范领域概念找到在当前系统中的落点；完成所有「待技术决策」项的决策

**目标**：概念→落点映射表无「待定」项；每个落点已唯一确定（不含互斥候选方案）；所有「实现设计类」条款的需设计型缺口已被识别并列出

**特性要求**：

- 每个规范条款的领域概念有至少一个具体落点或明确「无对应物」标注
- 「待技术决策=是」的概念，在本阶段出口时必须已有技术决策结论
- 技术决策记录覆盖「存在性与职责」（要不要这张表/这个接口），不要求覆盖实现细节
- **不产出变更清单**：本阶段只判断「在哪里」，不判断「要不要改」
- **「实现设计类=是」的条款，必须产出对应的「需设计型缺口描述」**：缺什么设计、缺口的决策空间有多大（影响哪些变更条目的具体内容）

## 工作依据

### 概念落点映射方法

**对每个领域概念（从条款索引表中的「领域概念」类条款提取）**：

1. 在系统摘要的「数据模型」章节查找：是否有对应表或字段
2. 在系统摘要的「接口全景」章节查找：是否有对应接口或服务方法
3. 在系统摘要的「鉴权矩阵」章节查找：是否有相关鉴权逻辑

**当前状态的三种情况**：

- **有对应物，语义一致**：记录具体路径（`auth_token` 表 / `AuthService.verifyToken()` 方法）
- **有对应物，但语义不符**（需参照术语映射表）：记录具体路径 + 标注「语义不符」
- **无对应物**：直接写「无对应物」

**集合运算类概念的落点（关键）**：

- 「代理权限集 = A ∩ B ∩ C ∩ D」中的每个因子（A/B/C/D）都要**单独找落点**
- 不要把「权限集整体」当一个落点处理
- 若某个因子找不到落点 → 标「无对应物（需新增）」+ 技术决策依赖=是（存储方案需要决定）

**找不到落点时的处理**：

1. 先检查「是否规范用了系统中没有的术语」→ 查术语映射表
2. 若确实无对应物 → 标「无对应物」，记「候选落点类型：DB表/Redis/接口/配置」（2-3 个候选供技术决策用）
3. **不要跳过**——每个概念必须有结论

---

### 技术决策流程（条件执行）

**触发**：概念→落点映射草稿中存在「待技术决策=是」的条目

**决策边界判断**（关键，每次决策前先问）：

- 问「如果选方案 X vs 方案 Y，变更清单里的表名/接口名/配置项会不同吗？」
  - **会不同** → 必须在本阶段决定（plan 层决策）
  - **不会不同，只影响实现细节** → 留给 spec-execute（execute 层决策）

**常见需要在 plan 层决定的技术决策类型**：

- 存储方案选型（DB/Redis/JWT payload 三者之一或组合）
- 是否引入新表（vs 在现有表新增字段）
- 是否引入新服务边界（vs 在现有服务新增方法）
- 是否引入配置化（vs 硬编码）

**技术决策记录格式**：

```markdown
### 技术决策：[概念名]

**问题**：[待决策的问题描述]

**选定方案**：[方案名称及简述]

**排除方案**：

- 方案A：[名称] — 排除原因：[具体原因，不能只说「不好」]
- 方案B：[名称] — 排除原因：[具体原因]

**对变更清单的影响**：

- 会产生的变更条目：[表名/接口名/配置项列表]
```

**典型技术决策示例（存储方案选型）**：

| 方案                     | 优势                     | 劣势                           | 典型排除原因         |
| ------------------------ | ------------------------ | ------------------------------ | -------------------- |
| MySQL（SoT）+ Redis 缓存 | 持久化、可审计、灾备友好 | 需管理缓存一致性               | 通常是最稳健选择     |
| Redis（SoT）             | 写入快、延迟低           | 无原生持久化审计               | 合规要求强审计时排除 |
| JWT payload 全量携带     | 无需回源                 | 8KB 上限、权限更新无法即时生效 | 动态修改诉求时排除   |

## 需设计型缺口清单（条件，若存在「实现设计类=是」条款）

| 缺口 ID | 对应条款 ID | 缺口描述（缺少什么设计） | 影响的变更条目类型 | 已有系统探索结论 |
|---------|------------|------------------------|-------------------|----------------|
| D-001 | S9 | openapi agent 的 prompt 框架（system role、输入文件范围、输出格式约束、质量验证方式）均未定义 | SVC-xxx（agent 实现）、CFG-xxx（prompt 配置） | 基类：`BaseVibeAgent`，调用方式：`agentRunner.run(...)` |
| D-002 | S10 | 「判断本轮是否有服务端代码变更」的具体条件表达式未确定，影响 workflow 触发条件逻辑 | SVC-xxx（pro-mode.ts 修改） | 插入点：sendDingcardMessage 后，context 结构已探明 |

**缺口识别方法**：

对每个「实现设计类=是」的条款，问：「如果让两个工程师各自实现，他们的实现会有哪些维度的实质差异？」每个可能产生差异的维度，就是一个需设计型缺口。

**缺口分级**：

- **需专项 subagent 设计**：缺口复杂（如 prompt 框架设计，包含多个相互约束的决策），需要专项研究才能产出足够细粒度的设计结论 → 进入阶段三点五
- **可在主控内决策**：缺口简单（如单一参数选择），主控可在本阶段直接决策并记录 → 记入技术决策记录，不进入阶段三点五

- [ ] 技术决策结论已更新到概念→落点映射表（「待定」已全部消除）
- [ ] 所有「实现设计类=是」条款有对应需设计型缺口描述（即使缺口为空也须显式说明「无需专项设计」及理由）
- [ ] 每个需设计型缺口已判断「需专项 subagent」vs「主控直接决策」
