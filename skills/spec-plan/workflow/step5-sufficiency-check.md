# 阶段五：充分性验证 执行指导

## 输入信息

**必须入参**：

- 阶段四变更清单初稿
- 阶段一条款索引表（完整版，含所有 MUST/SHOULD/MAY 条款）

## 目标要求

**任务**：通过双向追溯和四类遗漏源专项检索，确保变更清单对规范的「充分性」

**目标**：MUST 类条款 100% 有变更条目覆盖（或明确「不适用」说明）；无孤立变更条目

**特性要求**：

- 覆盖度矩阵完整，每个 MUST 类条款有结论
- 四类遗漏源各有专项检索记录
- 「无规范条款引用」的变更条目已处理（保留并补标，或移入可选清单，或删除）

## 工作依据

### 双向追溯验证

**正向（条款 → 变更条目）**：

对每条 MUST 类规范条款，在变更清单中查找对应变更条目

- 有对应条目 → 在矩阵中标「✅ 覆盖：[条目 ID]」
- 无对应条目 → 先确认是否真的不需要变更（极少数情况）：
  - 若不需要：在矩阵中标「不适用：[原因]」，原因必须说明清楚（如「当前系统已满足此约束，无需变更」）
  - 若需要但漏了：**立刻回到阶段四补条目**，然后重新跑正向验证

**反向（变更条目 → 规范条款）**：

对每条变更条目，检查是否有对应规范条款 ID

- 有条款 ID → 正常
- 无条款 ID → 判断来源：
  - 「实现时顺便发现的其他改进」→ 移入「可选/附加变更」清单，不作为主要变更
  - 「确实对应某条规范但漏标了」→ 补标规范条款 ID
  - 「属于技术实现细节」→ 移入 spec-execute 的范围，从本清单删除

---

### 四类遗漏源专项检索（每次必须执行）

这四类来源是最常被静默遗漏的依赖，必须逐一检索：

**缓存层**：

- 检索 Redis key 的构建位置，看是否包含受影响的字段
- 搜索词：`redis.set`、`cache.set`、受影响字段名（如 `tenant_id`、`token_type`）出现在 key 拼接中
- 若发现：在配置层或服务层补充「Redis key 命名空间变更」或「缓存失效策略调整」条目

**消息队列**：

- 检索事件生产者/消费者中是否传递了受影响字段
- 搜索词：`emit`、`publish`、`subscribe`、`consumer`、`queue`、`event`
- 若发现：在服务层补充相应条目

**内部服务调用**：

- 检索内部 HTTP/RPC 调用中是否传递了受影响字段
- 搜索词：`chromeFetch`、`axios.get`、`fetch(`、受影响接口路径片段
- 若发现：在 API 层或服务层补充相应条目

**前端/SDK 依赖**：

- 若存在前端同仓库或 npm 包，检索受影响字段在前端的使用
- 搜索词：受影响字段名、受影响接口路径
- 若发现：在变更清单中补充「对外 API 契约变更」说明

### 粒度独立检查（每次必须执行）

双向追溯完成后，对最终变更清单执行粒度扫描。此步骤独立于置信度评估——高置信度条目也须通过此检查。

**检查方法**：对每条变更条目，问「如果两个不沟通的工程师各自执行这条变更，他们的实现会有哪些实质差异？」

| 检查项 | 扫描内容 | 发现时处理 |
|-------|---------|-----------|
| 间接引用 | 描述中是否含「参考现有」「类似 X」 | 回阶段四修复，补充具体路径 |
| 未定义调用 | 描述中是否有未在清单中定义、也不在代码库存在的函数/类 | 补充对应新增条目 |
| 开放设计决策 | 描述中是否有「如何实现 X」仍未确定（且未标为粒度留白） | 回阶段三点五补设计 |
| 伪代码占位 | 代码片段中是否有注释形式的「// 实现具体逻辑」而无实际内容 | 要求给出完整代码逻辑或拆分为单独变更条目 |

**粒度检查产出**：

```markdown
## 粒度扫描结果

| 条目 ID | 扫描结论 | 问题描述（若有） | 处理动作 |
|---------|---------|---------------|--------|
| SVC-002 | ✅ 通过 | — | — |
| SVC-004 | ❌ 未通过 | `runOpenApiDocAgentIfNeeded` 未定义 | 补充 SVC-006 条目定义该函数 |
| CFG-001 | ❌ 未通过 | prompt 内容为空位 | 回设计阶段补充，或标粒度留白 |
```

若粒度扫描发现任何「❌ 未通过」条目：**必须修复后重新执行粒度扫描，直到全部通过，再进入行为验证**。

### 行为验证（条件执行）

**触发条件**：变更清单中存在「冲突重写」或「修改」类条目

执行方式：委派 Subagent（参照 SKILL.md 中的测试派发 Prompt 模板）

Subagent 需要做：

- 用 ripgrep 确认所有引用受影响字段/接口的代码路径
- 对比「调用方影响评估」中的覆盖声明与实际发现
- 返回每个条目的「覆盖完整 / 发现遗漏：具体位置」结论

收到报告后：

- 若有遗漏 → 补入最终变更清单，重跑覆盖度矩阵相关行
- 若无遗漏 → 标注「行为验证：通过」，进入阶段六
