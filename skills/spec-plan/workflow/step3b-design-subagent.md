# 阶段三点五：专项设计子任务 执行指导

> **条件阶段**：仅在阶段三产出的「需设计型缺口清单」中存在「需专项 subagent」级别缺口时执行。若所有缺口均可在主控内直接决策，跳过本阶段直接进入阶段四。

## 输入信息

**必须入参**：

- 阶段三「需设计型缺口清单」（每个缺口的 ID、描述、对应条款 ID、已有系统探索结论）
- 阶段二实现范式摘要（目录结构、基类信息、调用方式等）
- 规范文档原文（设计 subagent 需要理解业务背景）

## 目标要求

**任务**：对每个「需专项 subagent」级缺口，派发独立的设计 subagent，产出设计文档；主控汇总设计结论，整合进概念→落点映射表

**目标**：每个缺口在阶段三点五出口时，有足够细粒度的设计结论（两个独立 spec-execute 根据此结论实现，行为可预测收敛）

**特性要求**：

- 每个缺口一次独立 subagent 派发，禁止合并
- 设计文档保存至规范文档同目录，文件名格式：`design-<缺口ID>-<缺口简名>.md`
- 设计文档的内容须包含：方案选型（若有多个候选）、具体规格（可被直接引用到变更清单的内容）、与现有系统的集成约束

## Subagent 派发模板

对每个缺口，将以下模板中的 `${...}` 替换后派发：

```markdown
## 专项设计任务：${缺口描述简名}

你是一名资深系统设计工程师。你的任务是为下方的「实现设计缺口」产出完整的设计规格，使其可被另一名工程师直接引用到代码实现中，而无需任何额外的设计决策。

**严格约束**：

- 你的产出将被直接写入变更清单，由 spec-execute（AI coding agent）执行
- spec-execute 不做任何设计决策，因此你的规格必须达到「两个不沟通的工程师各自实现后，行为收敛」的精确度
- 禁止出现「根据实际情况」「可以考虑」「参考行业惯例」等开放式描述
- 禁止把「需要进一步研究的问题」留给 spec-execute

**完成判定（DoD）**：

你的设计文档必须通过以下测试：
- 任何人读完你的文档后，应能独立写出相同的代码结构（类定义、方法签名、prompt 内容、触发条件表达式等）
- 文档中不应存在任何「这里由实现者决定」的开放点，除非你已明确说明这是「粒度留白（已授权）」并给出边界约束

---

## 缺口信息

**缺口 ID**：${缺口ID}
**对应规范条款**：${条款ID} - ${条款描述}

**缺口描述**：
${缺口的详细描述，包含「缺少什么设计」}

**已有系统探索结论**（可直接复用，无需重新探索）：
${阶段二实现范式摘要中该条款对应的内容}

**业务背景**（来自规范文档）：
${规范文档中与该条款相关的原文}

---

## 设计产出要求

请产出以下内容（按顺序）：

### 1. 方案选型（若有多个候选方案）

| 方案 | 描述 | 选/排除 | 原因 |
|------|------|---------|------|
| 方案A | ... | 选定 | ... |
| 方案B | ... | 排除 | ... |

### 2. 实现规格

**对「agent 类」缺口，必须包含**：

- System prompt 完整内容（不是「应该包含 X」，而是实际文本）
- 输入数据的获取方式（从哪个 API 读取哪些字段，过滤条件是什么）
- 输出格式约束（具体到字段名、数据类型、存储路径）
- 质量验证逻辑（如何判断 agent 执行结果可用）
- 错误处理策略（失败时的行为：重试/静默跳过/上报）

**对「触发条件逻辑」类缺口，必须包含**：

- 触发条件的完整布尔表达式（具体到变量路径，如 `context.chatInfo?.app_setting?.enableOpenAPI === true`）
- 判断各子条件的数据来源（从 context 哪个字段读，字段类型和含义）
- 边界情况处理（条件为 null/undefined 时的行为）

### 3. 与现有系统的集成约束

- 依赖的现有接口/函数（文件路径 + 函数签名）
- 必须遵守的现有约定（如文件保存格式、状态更新方式）
- 不能破坏的现有行为（负向约束）

### 4. 粒度留白声明（若存在有意授权的开放决策）

| 开放点 | 授权边界约束 | 授权理由 |
|--------|------------|---------|
| ... | ... | ... |
```

## 汇总与整合

Subagent 回传设计文档后，主控执行以下整合步骤：

**步骤 1：设计文档保存**

将设计文档保存至规范文档同目录：
```
<规范文档目录>/design-<缺口ID>-<缺口简名>.md
```

**步骤 2：映射表更新**

在概念→落点映射表中，为对应条款更新：
- 「技术决策依赖」列：若设计文档已解决存在性决策，更新为「已决策」
- 备注列：添加「设计文档：design-<缺口ID>-<缺口简名>.md」

**步骤 3：确认缺口关闭**

逐个检查需设计型缺口清单，确认每个「需专项 subagent」缺口已有设计文档，且设计文档符合 DoD：
- [ ] 没有「根据实际情况」类开放描述
- [ ] agent 类缺口有完整 prompt 内容（不是占位符）
- [ ] 触发条件类缺口有完整布尔表达式
- [ ] 粒度留白均已显式声明

若设计文档不符合 DoD，退回 subagent 要求补充（说明具体缺失项），不进入阶段四。

## 完成判定

- [ ] 所有「需专项 subagent」缺口均有设计文档，保存路径已记录
- [ ] 每份设计文档通过 DoD 自检
- [ ] 概念→落点映射表已更新，无「待定」项
- [ ] 主控已将设计结论摘要内化（能在阶段四直接引用，无需再读原文档）
