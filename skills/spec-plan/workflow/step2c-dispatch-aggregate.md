# 并行切片模式：P5–P6 执行指导

> 本文件在主控 Agent 执行「并行切片模式」的派发和汇总阶段时必须完整阅读。
> 对应 SKILL.md 中的「阶段一点五」P5–P6 两个子节点。

---

## P5：并行/分阶段派发

### 输入信息

**必须入参**：
- P4 产出的每切片 prompt（完整内容，直接作为 Task prompt 传入）
- P3 切片清单（用于确认派发顺序：有前置依赖的切片不能与被依赖切片同批并行）

### 目标要求

**完成判定（必须同时满足）**：
- P1. 所有切片均已回传（无超时未回的）
- P2. 每份报告包含完整四节（节 A / 节 B / 节 C / 节 D，缺任一节均标为格式不合格）
- P3. 节 C 每个条款块含 8 字段（条款ID / 当前状态 / 落点 / 差距 / 变更方向 / 本地编号 / 置信度 / 涉及共享文件）
- P4. 节 D 每行含 4 字段（类型 / 描述 / 涉及条款 / 严重程度）

### 工作依据

**派发规则**：
- 使用 `Task` 工具，`subagent_type="generalPurpose"`，`readonly=true`
- 一切片一次 Task（**禁止将多切片合并传入同一 Task**，会导致上下文混淆）
- **无前置依赖的切片**：在同一消息中并行发出所有 Task（同时放多个 Task 调用）
- **有前置依赖（两阶段）**：
  1. 先并行发送阶段一组的所有切片
  2. 等阶段一组全部回传
  3. 从回传中提取前置信息（FK 类型、主键定义等）
  4. 再并行发送阶段二组的所有切片

**回传质量快速验证**（主控收到报告后立即检查，发现问题立即标记，不等全部汇总后再处理）：

```
✓ 检查四节是否完整（寻找「## 节 A」「## 节 B」「## 节 C」「## 节 D」标题）
✓ 检查节 C 第一个条款块是否含 8 字段（若不含，说明格式自检指令未生效）
✗ 不需要在此阶段验证内容正确性（留 P6）
```

**格式不合格时的处理**：
- 节 C 格式错误（自由文本代替结构块）：主控从报告中提取有效信息，手动重构为结构块，标注「主控重构」
- 节 D 格式错误（无四字段）：主控按节 D 标准格式重新整理，不重新派发（避免浪费）
- 四节不完整（缺节 A 或节 B）：重新派发该切片，prompt 末尾补充「**本次尤其注意**：必须在回复最开始输出「## 节 A 数据层现状」，即使该切片无 DB 层变更也要输出此节标题并写「本切片无 DB 层变更」」

**已知坑位**：
- `readonly=true` 只防止写文件，不能防止 subagent 读范围外文件；越界探索的防线在 prompt 的禁止段
- subagent 可能在节 C 产出「空条款块」（只有条款 ID，其余字段为「未探索到」）——这是正常情况（探索盲区），不是格式错误；主控在 P6 关卡一处理

---

## P6：汇总整合

### 输入信息

**必须入参**：
- P5 全部切片报告（N 份完整四节内容）
- P2 Owner Mapping 表（用于共享变更归一校验）
- P3 切片清单（用于消费方依赖计数校验）

### 目标要求

**完成判定（必须同时满足）**：
- P1. ID 唯一性：`set(all_global_ids).length == len(all_global_ids)`（机械验证，不依赖主观检查）
- P2. 共享资源条目各一：每个共享表/共享库变更只有一条全局条目，归属 owner 切片
- P3. 消费方依赖计数正确：依赖共享库的服务数量 == 对应 go.mod/package.json 升级条目数
- P4. 跨切片语义依赖：Greenfield 切片依赖的存量变更有显式「前置于: Cx」注记
- P5. 无孤立变更条目（每个条目引用的表/函数在现有代码库或同一清单的新增条目中存在）
- P6. 跨切片审查记录附于产物末尾（书面可追溯）
- P7. 产出可直接进入 spec-plan 阶段三的「系统状态摘要」和「变更清单初稿」（格式与串行路径产出等价）

### 工作依据

**三道关卡（顺序执行，不可跳过）**：

---

### 关卡一：节 D 处理

汇总所有切片的节 D，按类型逐项处理：

**「共享文件变更意向」**：
1. 汇总所有切片中同一共享文件的变更意向
2. 检查命名冲突（不同切片是否对同一 enum name 有不同值的建议）
3. 无冲突时：合并为一条「SHARED-xxx」变更条目（同一个共享文件所有变更合入一条）
4. 有冲突时：主控裁决（选规范中有更明确定义的一方，记录裁决理由）
5. 产出合并后的 SHARED 条目，归属「主控」，在跨切片审查记录中标注来源切片

**「文件散落」**：
1. 节 D 报告了「路径 X 文件不在切片探索范围，可能涉及 [条款 Sy]」
2. 主控直接探索该文件（主控不受 readonly 约束）
3. 确认相关条款是否影响该文件
4. 若影响：补充变更条目，标注「来源：主控补充探索（[切片名] 节 D 触发）」
5. 若不影响：在审查记录中注明「已探查，无需变更，理由：X」

**「意外模块依赖」**：
1. 记录跨模块依赖关系（从哪个切片到哪个模块，具体 import/调用）
2. 判断此依赖是否影响规范变更：「新字段/新接口是否改变这条依赖的签名或语义？」
3. 若影响：补充调用方影响评估（同 spec-plan 阶段四的「冲突重写调用方影响评估」格式）
4. 若不影响：在审查记录中标注「跨切片调用：签名/语义无影响，理由：X」

**「探索盲区」**：
1. 对应变更条目标注低/中置信度
2. 补充具体可执行的验证建议（`rg "字段名" src/ --type ts` 这类粒度，不接受「需要进一步确认」）

---

### 关卡二：三道去重检查

**检查一：共享表变更唯一性**

扫描所有节 C 条款块中「落点」字段，提取所有数据库表名：
```
发现重复表名时：
- 比对重复条目的「变更方向」和「差距描述」
  - 内容一致 → 保留 owner 切片的条目，删除其他切片的副本
  - 内容冲突 → 主控裁决，选 owner 切片的版本（owner 是唯一写入方，最了解该表）
  - 记录裁决过程到跨切片审查记录
```

**检查二：共享库代码变更归属**

扫描所有条目的文件路径，找路径含 `libs/`、`shared/`、`packages/`、`node_modules/` 的条目：
```
规则：
- 若该共享库在 P3 切片清单中有对应的「共享库切片」→ 此条目必须来自该共享库切片
- 若其他切片（如 identity、resource）的节 C 包含了共享库的代码变更条目 → 删除，仅保留共享库切片的版本
- 例外：各服务自己的 go.mod/package.json 升级条目（这是消费侧的本地文件，不是共享库代码）
```

**检查三：消费方依赖计数**

目标：确认所有依赖某共享库的服务，都有对应的「升级依赖版本」变更条目。

```
操作：
1. 从 P3 切片清单中，找到「共享库切片」（如 auth-middleware）
2. 统计依赖该库的服务切片数量（来自 P2 的 Owner Mapping 中「禁止产出变更的切片」列，
   那些切片就是该库的消费方）
3. 检查这些消费方切片中，是否每一个都有「go.mod / package.json 升级」的变更条目
4. 若某服务切片无此条目 → 主控补充该条目，标注「来源：主控汇总补充（消费方计数检查）」
```

坑位：消费方有时把 go.mod 升级和业务变更合并描述到一个条目中 → 检查条目描述文本是否包含 go.mod / package.json 的关键词，不只是数条目个数。

---

### 关卡三：跨切片语义依赖检查

目标：识别 Greenfield 切片的变更条目是否引用了「本次也在变更的」存量表/接口，建立顺序依赖注记。

```
操作：
1. 枚举所有 Greenfield 切片的变更条目
2. 对每条 Greenfield 条目，检查其「落点」或「差距」描述中是否引用了：
   - 本次变更清单中另一个切片（Brownfield）的新增字段
   - 本次变更清单中另一个切片（Brownfield）的新建表（作为 FK 目标）
   - 本次变更清单中另一个切片（Brownfield）的新增接口（作为调用依赖）
3. 若发现：在 Greenfield 条目后附加「前置于: Cx」注记，Cx 为该 Brownfield 条目的全局 ID
4. 这些顺序依赖将在阶段六 DAG 中自动成为 DAG 边
```

举例（以场景 E 为例）：
```
ODEV-DB-001（新建 developer_app 表，含 FK → users.id）
→ 检查：users 表是否在本次变更中？不是（users 表未被修改）→ 无需前置注记

ODEV-SVC-001（DeveloperService.createApp() 校验 scope 是否合法）
→ 检查：scope 合法性依赖 auth_token.scope 字段的枚举值 → auth_token.scope 字段在本次变更清单中（AUTH-DB-001）
→ 添加注记：「前置于: AUTH-DB-001（auth_token.scope 字段必须先存在）」
```

### 全局 ID 分配

完成三道关卡后，对所有有效变更条目分配全局 ID：

**分配策略**：
- 按层按模块顺序：DB 层先（所有切片的 DB 条目），再 API 层，再服务层，再配置层，再数据迁移层
- 同层内按切片顺序（与 P3 切片清单的顺序一致）
- 共享资源条目（SHARED 类）排在所有常规层之后，数据迁移层之前

**命名格式**：`DB-001`、`API-001`、`SVC-001`、`CFG-001`、`MIG-001`、`SHARED-001`

**唯一性机械验证**：
```python
# 伪代码，主控手动执行等价验证
all_ids = [每个条目的全局 ID]
assert len(set(all_ids)) == len(all_ids), "ID 冲突！重复项：" + str([x for x in all_ids if all_ids.count(x) > 1])
```
