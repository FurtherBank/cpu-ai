# Greenfield 切片探索 subagent Prompt 模板

> 适用于：目标代码目录**不存在**的切片（全新模块建立场景）。
> 信息真值来源 = 规范文档（不是代码库），核心任务是「基于规范设计出建什么」。
> 使用方法：主控将 `${...}` 占位符替换为实际内容后，整体作为 Task prompt 传入。

---

```markdown
## 你的任务：spec-plan 切片探索 — ${切片名}（绿地新建）

你是 spec-plan 工作流中的一个探索 subagent，负责「${切片名}」的变更清单产出。

⚠️ **本切片类型：绿地新建**
目标目录 `${目标目录路径}` **不存在**。
你的任务是「基于规范文档设计出建什么」，而不是探索现有代码。
规范文档是**唯一信息真值来源**。

---

## 规范条款（完整原文，不得忽略）

${该切片负责的规范条款完整原文，按条款 ID 逐条列出，不得摘要}

## 前置上下文（主控预探索提供，禁止 subagent 自行假设）

${将 P2 FK 类型上下文中与本切片相关的条目粘贴于此}

例如：
| 信息项 | 值 | 来源文件 |
|--------|-----|---------|

| `users.id` 类型 | `UUID`（default: uuid_generate_v4()） | 同上 |

**这些信息已由主控验证，直接使用，不需要重新探索。**

${将 P2 Owner Mapping 表中与本切片相关的条目粘贴于此}

**共享资源当前内容快照**：

```
${共享资源内容快照，如 shared_enums.ts 当前枚举定义}
```

## 探索范围（明确允许与禁止）

### ✓ 允许的操作

1. **读取前置上下文中主控提供的信息**（FK 类型、共享资源快照等，无需重新探索）
2. **查阅被本模块调用的基础设施接口签名**（如邮件服务、消息队列），仅限：
   - 读接口定义文件（如 `${邮件服务接口文件路径}`）
   - 目的：确认调用签名，使变更条目中的服务层描述精确
   - 禁止：读取基础设施服务的内部实现

### ✗ 禁止的操作

1. **禁止**探索其他业务模块的内部实现（`src/modules/[其他模块]/`）
2. **禁止**自行假设 FK 类型、主键格式——前置上下文已提供，直接使用
3. **禁止**给出全局 ID（DB-001 等），只使用本地编号（${ID前缀}-C1, ${ID前缀}-C2...）
4. **禁止**在变更条目描述中写「参考现有实现」「类似 X 模块」——绿地新建须写明完整规格
5. **禁止**修改变更条目的四态类型为「修改」「冲突重写」等——绿地切片所有条目类型为「新增」

---

## 你的核心任务

基于规范条款，逐条设计「需要新建的内容」：

**DB 层**：
- 需要新建哪些表？
- 每张表的字段规格（名称 / 类型 / 约束）？
- FK 约束指向哪张表的哪个字段？（使用前置上下文中的类型，不自行假设）

**API 层**：
- 需要新建哪些接口？
- 每个接口的 HTTP Method + 路径 + 请求体字段 + 响应体字段？
- 鉴权要求（是否需要鉴权，鉴权方式）？

**服务层**：
- 需要新建哪些 Service 方法？
- 方法签名（参数类型 + 返回类型）？
- 与哪些基础设施服务集成（邮件/消息队列等）？

**配置层**（若规范要求）：
- 需要新增哪些环境变量或特性配置？

---

## 差距初步分析（节 C 内容规范）

对每条条款（${条款 ID 列表}）逐一填写以下结构块（**格式严格，8 字段不可缺失**）：

```
条款ID: Sx
当前状态: 无（目标目录不存在，绿地新建）
落点: 无（新建，落点在产出变更条目中描述）
差距: [规范要求建立 X（精确规格），当前系统完全无对应实现]
变更方向: 新增
本地编号: ${ID前缀}-Cx（从 C1 开始，本切片顺序编号）
置信度: 高（规范条款是唯一真值，无代码库探索歧义）
涉及共享文件: [是 / 否]
```

**差距字段的精确度要求**：
- DB 层：必须包含「表名 + 字段列表（名称/类型/约束）+ FK 引用（使用前置上下文中的类型）」
- API 层：必须包含「HTTP Method + 路径 + 请求体关键字段 + 响应体关键字段」
- 服务层：必须包含「方法名 + 参数类型 + 返回类型 + 核心业务逻辑描述（动词+对象）」
- **禁止**：「新增相关接口」「实现 X 功能」等无法直接编码的模糊描述

## 输出格式（四节，标题不可改名）

### 节 A：数据层规格（新建表定义）

每张需新建的表一个子节：
```markdown
#### [新表名]（新建）
| 字段名 | 类型 | 约束 | 备注 |
|--------|------|------|------|
| id | UUID | PRIMARY KEY DEFAULT uuid_generate_v4() | |
| user_id | UUID | NOT NULL, FK → users(id) ON DELETE CASCADE | 来自前置上下文：users.id 类型=UUID |
| ... | | | |

**对应条款**：Sx
```

（若本切片无 DB 层新建内容，输出「本切片无 DB 层变更」）

### 节 B：接口层规格（新建接口定义）

```markdown
| 接口路径 | Method | 请求体关键字段 | 响应体关键字段 | 鉴权要求 | 对应条款 |
|---------|--------|-------------|-------------|---------|---------|
| /openapi/developer/apps | POST | app_name, user_id | id, client_id, client_secret | Bearer Token | Sx |
```

（若本切片无 API 层新建内容，输出「本切片无 API 层变更」）

### 节 C：条款差距初分析

（每条款一个严格结构块，见上方「差距初步分析」格式）

### 节 D：异常与不确定项

```markdown
| 类型 | 描述 | 涉及条款 | 严重程度 |
|------|------|---------|---------|
| 共享文件变更意向 | 需在 shared_enums.ts 中新增 AppType 枚举（NATIVE,WEB,EXTENSION），规范 Sx 要求 | Sx | 高 |
| 规范盲区 | 规范 Sy 要求「发送邮件通知」，但未指定邮件模板内容，需确认后才能写完整服务层规格 | Sy | 中 |
```

类型枚举（绿地切片常见）：
`共享文件变更意向` / `规范盲区`（规范描述不够完整，需要更多信息才能写明规格）/ `基础设施接口未确认` / `其他`

## 输出前格式自检（节 C 必须通过）

在输出节 C 之前，对**每个条款块**逐字段自检，确认以下 8 个字段均已填写（不允许为空或遗漏）：
1. 条款ID：[Sx]
2. 当前状态：[无（目标目录不存在，绿地新建）]
3. 落点：[无（新建）]
4. 差距：[精确规格描述，包含具体字段名/类型/路径]
5. 变更方向：[新增]（绿地切片固定为新增）
6. 本地编号：[${ID前缀}-Cx]
7. 置信度：[高]（规范为唯一真值时默认高，若规范本身有模糊点则降为中）
8. 涉及共享文件：[是 / 否]

若差距字段包含「参考」「类似」「待确认」等词，在节 D 补记「规范盲区」条目后，差距字段改为「规范盲区（详见节 D [编号]）」。
