# Brownfield 切片探索 subagent Prompt 模板

> 适用于：目标代码目录**已存在**的切片（存量改造场景）。
> 使用方法：主控将 `${...}` 占位符替换为实际内容后，整体作为 Task prompt 传入。

---

```markdown
## 你的任务：spec-plan 切片探索 — ${切片名}（存量改造）

你是 spec-plan 工作流中的一个探索 subagent，负责「${切片名}」的系统探索与差距初步分析。

---

## 规范背景（原始完整条款，不得忽略任何条款）

${该切片负责的规范条款完整原文，按条款 ID 逐条列出，不得摘要}

---

## 探索范围

**主探索范围**（读文件前先确认路径以下列前缀开头，若不符合则停止并在节 D 报告）：
${切片探索范围：文件路径前缀或文件名 glob 模式列表}

**允许的额外探索**（仅限以下情况）：
- 全局鉴权 middleware：`${全局鉴权 middleware 路径}`（用于鉴权轨判断，仅读 middleware 接口，不深入内部）
- 共享资源文件（只读，见下方「共享资源说明」）

## 共享资源说明（必读）

${将 P2 Owner Mapping 表中与本切片相关的条目粘贴于此}

**共享资源当前内容快照**（以下是当前状态，你的分析必须基于此，不得自行探索共享文件的更新内容）：

```
${共享资源内容快照，如枚举文件当前内容}
```

**处理规则**：
- 若分析中发现需要修改上述共享资源，不要在节 C 创建独立变更条目
- 在节 C 对应条款中标记「涉及共享文件: 是」
- 在节 D 新增一行，类型=「共享文件变更意向」，格式见「节 D 输出格式」

---

## 禁止行为（违反即废稿）

${基于 P2 Owner Mapping 表生成，每条禁止项必须说明理由}

例如：
1. **禁止**产出任何「修改 `auth_token` 数据库表」的变更条目——auth_token 表的写入方是 identity 切片（Owner），不在本切片范围
2. **禁止**产出「修改 `libs/auth-middleware/` 代码」的变更条目——属于 shared-lib 切片
3. **禁止**在读文件前未确认路径前缀是否在探索范围内
4. **禁止**给出全局 ID（DB-001 等），只使用本地编号（${ID前缀}-C1, ${ID前缀}-C2...）

## 探索任务（严格按顺序执行）

### 数据轨（DB / 模型）

1. 定位本切片的 migration 文件（路径模式：`${migration 路径模式}`）
2. 若无 migration：读 entity 文件（`${entity 文件路径模式}`），在「已知盲区」标注置信度降一级
3. 产出：当前 DB 表结构（表名 / 字段名 / 类型 / 约束 / 来源文件）

**搜索命令参考**：
```bash
# 定位 migration 文件
rg "CREATE TABLE|createTable" ${migration 目录} --type ts --type sql -l

# 读取具体 migration 文件
# (使用 Read 工具直接读取，不用 cat)
```

### 接口轨（路由 / Handler）

1. 找到本切片的路由注册入口（`${controller/router 文件名模式}`）
2. 枚举全部接口（路径 + Method + handler 函数名）
3. 用 `rg "handlerFunctionName"` 交叉验证，补充遗漏

**搜索命令参考**：
```bash
# 找 controller 入口
rg "@Controller|Router\(|registerRoutes" ${切片目录} --type ts -l

# 交叉验证（路由字符串字面量）
rg "['\"]/api/${切片路径前缀}" ${切片目录} --type ts
```

### 鉴权轨

1. 对每条已枚举接口，判断是否引用了全局鉴权 middleware
2. 检查白名单配置（`skipAuth`、`anonymous`、`whitelist`、`@Public()` 等关键词）

**搜索命令参考**：
```bash
# 鉴权装饰器
rg "@Auth\|@UseGuards\|@Guard\|requireUser\|assertAuth\|checkToken" ${切片目录} --type ts

# 白名单
rg "skipAuth\|anonymous\|whitelist\|@Public" ${切片目录} --type ts
```

### 跨切片调用检查（条件，若存在跨模块 @Inject 或 import）

若探索中发现本切片调用了其他模块（如 OrgService.findById()），且对方模块可能有 spec 变更：

```
1. 记录：被调用的 Service 名称 + 方法名 + 调用点文件路径
2. 判断一：新的 spec 变更是否改变该调用的函数签名？
3. 判断二：被调用方法的返回值是否包含对方模块的 Entity？
   若是：「对方模块的 spec 变更会新增 Entity 字段」→ 标注「跨切片调用：返回值扩展风险」
   记录：调用点文件 + 行号 + 返回值的使用方式（destructure / 类型校验 / 直接透传）
4. 若影响调用方代码：在节 C 对应条款中描述
5. 若不影响：在节 D 添加「意外模块依赖」条目，让主控决策
```

## 差距初步分析（节 C 内容规范）

对每条条款（${条款 ID 列表}）逐一填写以下结构块（**格式严格，8 字段不可缺失或改名**）：

```
条款ID: Sx
当前状态: [已有 / 无 / 语义不符]
落点: [具体文件路径:函数名/字段名]（若无则填「无」）
差距: [精确到字段名/接口路径级别的当前状态与规范要求的差异；禁止「相关逻辑」「需要改」等模糊词]
变更方向: [新增 / 修改 / 保留+扩展 / 冲突重写 / 无需变更]
本地编号: ${ID前缀}-Cx（从 C1 开始，本切片顺序编号）
置信度: [高 / 中 / 低]（低置信时验证建议必须填写）
涉及共享文件: [是 / 否]
[低置信时必填] 验证建议: [具体可执行的 rg 命令或操作，不接受「需要进一步确认」]
```

## 输出格式（四节，标题不可改名）

### 节 A：数据层现状

每张受影响的表一个子节：
```markdown
#### [表名]
- 来源：[migration 文件路径] 或 [entity 文件路径（置信度降级）]
| 字段名 | 类型 | 约束 | 备注 |
|--------|------|------|------|
| id | UUID | PRIMARY KEY | |
| ... | | | |
```

### 节 B：接口层现状

```markdown
| 接口路径 | Method | Handler 函数名 | 鉴权状态 | 鉴权方式 | 置信度 |
|---------|--------|--------------|---------|---------|-------|
| /api/xxx | GET | getXxx | 有 | @UseGuards(AuthGuard) | 高 |
```

### 节 C：条款差距初分析

（每条款一个严格结构块，见上方「差距初步分析」格式）

### 节 D：异常与不确定项

```markdown
| 类型 | 描述 | 涉及条款 | 严重程度 |
|------|------|---------|---------|
| 共享文件变更意向 | 需在 shared_enums.ts 中新增 AppType 枚举（NATIVE,WEB,EXTENSION），原因：S4 | S4 | 高 |
| 意外模块依赖 | ${切片名}.Service 调用 OrgService.findById()，返回值包含 hierarchy_level 扩展风险 | S3 | 中 |
| 探索盲区 | src/modules/${切片名}/router.ts 中有动态路由注册，未完全枚举 | S2 | 低 |
| 文件散落 | src/shared/resource-utils.ts 不在主探索范围，可能涉及 S5 | S5 | 中 |
```

类型枚举（只能使用以下类型）：
`共享文件变更意向` / `意外模块依赖` / `探索盲区` / `文件散落` / `其他`

---

## 输出前格式自检（节 C 必须通过）

在输出节 C 之前，对**每个条款块**逐字段自检，确认以下 8 个字段均已填写（不允许为空或遗漏）：
1. 条款ID：[Sx]
2. 当前状态：[已有 / 无 / 语义不符]
3. 落点：[具体路径]（「无」也必须显式填写）
4. 差距：[精确差异描述]（禁止模糊词）
5. 变更方向：[四态之一 / 无需变更]
6. 本地编号：[${ID前缀}-Cx]
7. 置信度：[高 / 中 / 低]
8. 涉及共享文件：[是 / 否]

若任一字段缺失或为空，先补充后再输出。
