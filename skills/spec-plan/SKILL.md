# spec-plan：规范差距分析 → 变更清单

给定 spec-design 产出的规范文档（理想终态）与当前系统现状，通过条款索引化 → 系统探索 → 概念落点映射 → 差距分析 → 充分性验证，产出**结构化、可追溯、置信度标注**的变更清单，作为 spec-execute 的直接输入。

**核心挑战**：
1. 变更清单的「充分性」——规范描述的是终态，系统的复杂度可能远超规范覆盖范围，遗漏任何变更项都会导致 spec-execute 阶段发现漏项引发返工。
2. **粗粒度条款要在这里落成可执行方案**：规范常只写「要什么」，少写「怎么做」。凡是仍属设计空白的部分——例如 prompt 结构、触发与分支逻辑、与现有系统的集成与对接方式——都必须在 spec-plan 中演化定案并写进变更清单。**spec-execute 只负责按清单编码与跑测，不承担补设计**；这类决策不得以任何形式留给后续阶段。

## 全局约束

- **不重新讨论规范**：规范是已定产物（spec-design 的输出），本 skill 只负责「如何从现状到达规范状态」
- **不决定实现细节**：只决定「要建哪些表/接口/配置（存在性与职责）」；具体实现（索引大小、TTL 值、分表方案）留给 spec-execute
- **以代码仓库为准**：现状判断以实际代码/migration 为准，不依赖假设或口头描述
- **设计决策不转移给 spec-execute**：变更清单中不能出现「参考现有实现」「待确认」「暂定 X」等开放描述，除非已显式标注为「粒度留白（已授权）」；所有「需设计型」缺口必须在本 skill 内完成设计（可通过专项设计 subagent）

## 工作流程（主控 Agent 调度模式）

> **任务阶段规划**：使用 TodoWrite 按以下阶段创建待办项，每个阶段一项，名称与顺序与本文档一致，不得跳过或合并。

### 阶段一：规范解析

**执行方式**：主控执行

**输入**：

- 规范文档（MUST 完整阅读）
- 改造目标描述（一句话表达「从哪个状态到哪个状态」）

**目标**：将规范条款原子化，产出可追溯的条款索引表，使每条规范语句可被变更清单独立引用

**产出**：条款索引表（见 [templates/change-list-template.md](templates/change-list-template.md) 中「条款索引」节）

实际执行到该步骤时，**必须**阅读 [workflow/step1-spec-parse.md](workflow/step1-spec-parse.md) 作为指导。

### 阶段二：系统探索

**执行方式**：主控执行。**[并行策略]** 若规范条款涉及多个独立的代码目录或子模块（可按 scope/task 明确划分边界），主控应将探索任务结构化拆解，通过 Task 工具同时拉起多个 Subagent 并行执行各独立分支的探索，最后由主控统一汇总整合系统状态摘要。
- **Subagent 提示词要求**：必须为每个 Subagent 提供明确的探索边界（特定目录/文件/服务）、对应的规范条款原文及关注点、以及共享资源的上下文（避免重复探索），并要求其严格产出该切片内的客观系统状态摘要。

**输入**：

- 阶段一产出的条款索引表（驱动定向搜索）
- 目标仓库路径或 AGENTS.md（确认搜索边界）

**目标**：通过数据/接口/鉴权/实现范式四轨探索，产出当前系统现状的客观摘要，达成所有闭包完成信号

**产出**：当前系统状态摘要；[条件] 术语映射表（若发现现有实现与规范命名重叠）；[条件] 实现范式摘要（若条款索引表中存在「实现设计类=是」的条款）

实际执行到该步骤时，**必须**阅读 [workflow/step2-system-explore.md](workflow/step2-system-explore.md) 作为指导。

### 阶段三：概念落点映射 + 技术决策（条件）

**执行方式**：主控执行

**输入**：

- 阶段一条款索引表
- 阶段二系统状态摘要（含术语映射表如有、含实现范式摘要如有）

**目标**：建立规范概念与代码/DB 落点之间的精确映射；完成所有影响「要改什么」的技术决策；识别「实现设计类」条款内的需设计型缺口

**产出**：概念→落点映射表；[条件] 技术决策记录（若有「待技术决策」标记的概念）；[条件] 需设计型缺口清单（若存在「实现设计类=是」条款）

实际执行到该步骤时，**必须**阅读 [workflow/step3-concept-mapping.md](workflow/step3-concept-mapping.md) 作为指导。

### 阶段三点五：专项设计子任务（条件阶段）

**触发条件**：阶段三产出的「需设计型缺口清单」中，存在「需专项 subagent」级别的缺口

**执行方式**：对每个缺口，主控派发独立 subagent；主控汇总并整合产物

**输入**：

- 阶段三需设计型缺口清单
- 阶段二实现范式摘要
- 规范文档原文（subagent 背景理解用）

**目标**：为每个「需设计型」缺口产出足够细粒度的设计文档，使对应变更清单条目可以直接引用具体规格（prompt 骨架、触发条件表达式、集成约束等），而非留下开放决策

**产出**：每个缺口一份设计文档，保存至规范文档同目录（`design-<缺口ID>-<缺口简名>.md`）；概念→落点映射表更新（标注设计文档路径）

实际执行到该步骤时，**必须**阅读 [workflow/step3b-design-subagent.md](workflow/step3b-design-subagent.md) 作为指导。

### 阶段四：差距分析 → 变更清单

**执行方式**：主控执行。**[并行策略]** 若变更涉及多个相互独立的模块或范围（scope/task），主控可将分析任务拆解，通过 Task 工具并发委派多个 Subagent 分别产出各子模块的变更清单初稿，最后由主控进行跨边界的去重、语义依赖检查与汇总整合。
- **Subagent 提示词要求**：必须向每个 Subagent 注入该切片对应的「概念→落点映射表」与「系统状态摘要」，严格要求按四态（新增/修改/保留+扩展/冲突重写）输出结构化的变更清单初稿，并明令禁止使用「参考现有实现」等模糊词。

**输入**：

- 阶段三概念→落点映射表（含技术决策结论）
- 阶段二系统状态摘要（当前状态详情）
- 阶段三点五设计文档（若有）

**目标**：对每个规范落点，对比当前状态，用四态标注（新增/修改/保留+扩展/冲突重写）产出结构化变更清单；将设计文档中的具体规格内联到对应变更条目

**产出**：结构化变更清单初稿（DB/API/服务/配置/数据迁移五层）；[条件] 调用方影响评估

**自检清单**（完成判定，完成后检查所有项）：

- [ ] 五层均有记录（即使某层为空也有显式说明）
- [ ] 每条变更有 ID、四态类型、规范条款 ID、置信度
- [ ] 无描述字段包含「相关逻辑」「需要改」等模糊词
- [ ] 低置信条目有具体验证建议（非「需要再确认」）
- [ ] **粒度检查通过**：无「参考现有实现」类间接引用；无未定义的函数/类调用；无「待确认」开放决策（除已显式标注的粒度留白外）

实际执行到该步骤时，**必须**阅读 [workflow/step4-gap-analysis.md](workflow/step4-gap-analysis.md) 作为指导。

### 阶段五：充分性验证

**执行方式**：主控执行（双向追溯）；[条件] Subagent 执行（行为验证）

**输入**：

- 阶段四变更清单初稿
- 阶段一条款索引表

**目标**：通过正向（规范条款 → 变更条目）和反向（变更条目 → 规范条款）双向追溯，确保 MUST 类条款 100% 覆盖，无范围蔓延

**产出**：覆盖度矩阵；遗漏补扫记录；最终变更清单

实际执行到该步骤时，**必须**阅读 [workflow/step5-sufficiency-check.md](workflow/step5-sufficiency-check.md) 作为指导。

**行为验证（条件）**：若变更清单中存在「冲突重写」或「修改」类条目，通过 Subagent 执行行为验证：

```markdown
## 任务：spec-plan 变更清单行为验证

你是一名严格的测试工程师。你的任务是验证下方「变更清单」中「冲突重写」和「修改」类条目对当前系统行为的影响是否已被正确识别。

**不允许的行为**：

- 对「行为影响描述」内容只做表面复述，不进行实质验证
- 认为「已有描述就等于已验证」
- 遇到困难时只说「无法确认」而不提供具体验证路径

**验证方式（按优先级选用）**：

1. 静态代码分析：用 ripgrep 确认所有引用受影响字段/接口的代码路径，列出每一处
2. 调用图追踪：从变更条目的「落点」出发，向上追溯调用链，确认影响范围是否与清单中「调用方影响评估」一致
3. 若发现遗漏调用方：列出具体文件路径和调用位置，并说明影响类型

**必须提供的证据**：

- 每个被验证的变更条目：列出检索命令 + 输出结果摘要
- 实际发现的调用方路径列表
- 「清单声明覆盖」与「实际发现」的对比（一致 / 偏差说明）

**报告格式**：每个条目一节，包含：条目 ID / 验证方法 / 证据 / 结论（覆盖完整 or 发现遗漏：具体内容）

你就是防线。漏掉一个遗漏点 = 后续阶段返工。

## 输入

[变更清单（冲突重写 + 修改类条目）]
[调用方影响评估章节]
[目标仓库路径]
```

_收到报告后，若有遗漏，补入最终变更清单，再进入阶段六。_

### 阶段六：DAG + 分批计划（条件阶段）

**触发条件**：变更项数量 ≥ 10，或变更项间存在跨批次部署依赖

**执行方式**：主控执行

**输入**：

- 阶段五最终变更清单
- 部署约束（是否支持停机窗口、是否允许功能降级）

**目标**：产出依赖 DAG 和分批计划，使 spec-execute 可按批次独立上线，每批后系统可用

**产出**：依赖 DAG（Mermaid 格式）；分批计划表（每批：变更项 + 系统状态描述 + 回滚方案）

**自检清单**（完成判定）：

- [ ] 每条 DAG 边有明确依赖类型（FK约束/行为依赖/迁移依赖）
- [ ] 存量数据迁移批次在「读逻辑上线」批次之后
- [ ] 每批有至少一种技术回滚方案

实际执行到该步骤时，**必须**阅读 [workflow/step6-dag-batch.md](workflow/step6-dag-batch.md) 作为指导。

### 阶段七：产物落地

**执行方式**：主控执行

**输入**：

- 阶段一改造目标描述
- 阶段二系统状态摘要
- 阶段五最终变更清单
- 阶段六分批计划（若已执行；否则由主控根据变更清单直接编排批次）

**目标**：将所有分析产出整合为一份可供 spec-execute 直接消费的 Markdown 产物文件

**文件路径约定**：在规范文档所在目录下创建 `change-plan.md`（同目录无歧义；若规范文档路径不明确，在工作区根目录创建）

**文件结构**（严格按以下模板生成，各节内容来源如注释所示）：

```markdown
# <改造标题>

## 改造目标

<直接引用规范文档中的目标语句原文，不做改写>

## 现状背景

<基于阶段二状态摘要，聚焦「理解为什么要改」必须了解的现状；按子系统/子模块分不同子标题，不写流水账>

## 变更清单

### C1 - <动作> <对象>

<精确描述：字段名/接口路径/文件路径/配置键名均须写明；置信度 + 四态类型 + 规范条款 ID 以括注形式附于标题后>

### C2 - ...

（其余变更项以此类推）

## 分批执行计划

### Batch 1 - <系统侧描述> 编码实现

| 变更项 | 说明                       |
| ------ | -------------------------- |
| C1     | <本批次该变更项的具体工作> |
| C2     | ...                        |

- **批次结束状态**：<该批次完成后系统可独立运行、且对下游无破坏性的状态描述>

### Batch N - 本地新链路自测验证

- **自测方案**：<具体验证方式，如 openapi-test E2E / 单测 / 手动 curl 等>

### Batch N+1 - <系统侧描述> 预发验证/发布

- **前置**：<必须先完成的批次或条件>
- **灰度方案**：<是否需要灰度及具体方式，无需灰度须说明理由>
- **回滚方案**：<回滚操作描述>
```

**自检清单**（写文件前逐项确认）：

- [ ] 改造目标为规范原文直接引用，未二次改写
- [ ] 现状背景来自阶段二客观摘要，无主观评价
- [ ] 每条变更项标题格式为「Cx - 动作 对象」，描述精确到字段/路径级别
- [ ] 分批计划中每批有明确结束状态描述
- [ ] 含预发/发布的批次均有前置、灰度方案、回滚方案三项

## 注意事项

1. **严格遵守阶段顺序**，禁止跳过任何阶段
2. 技术决策（阶段三）不能推迟到 spec-execute：「影响要改什么」的决策必须在本 skill 内完成
3. 实现设计也不能推迟到 spec-execute：规范中的「agent 类」「自动化流程类」要求，其内部实现设计（prompt、触发条件、集成方式）必须在本 skill 内通过阶段三点五完成，不能以「hardcode 一个 prompt」「参考现有实现」作为变更清单的最终状态
4. 发现任何 MUST 类条款无覆盖：立刻回到阶段四补条目，不能仅在覆盖度矩阵里标「缺失」
5. 变更清单的描述字段必须精确到「在 token 表新增 `token_type` VARCHAR(20) 字段」，而非「token 要支持多类型」
6. 粒度留白须显式声明：凡有意授权 spec-execute 自行决策的点，必须用「粒度留白（已授权）」格式标注，无此标注即视为变更清单已完全收敛、spec-execute 不需要补充任何设计
7. **置信度 ≠ 粒度**：高置信度描述「方向是否正确」，粒度描述「是否足够具体到可直接实现」；两者独立检查，不能以高置信度代替粒度检查
