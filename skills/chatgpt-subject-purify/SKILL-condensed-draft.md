# chatgpt-subject-purify（凝练重构实验稿）

> 实验稿，未启用。按「凝练表达形态」主题的推荐组合重构：产物职责句 + 唯一收录判据 + 标注推论承载一致性约束；政策约定单列；逆默认回归防线外置到 [references/integration-regression-checklist.md](references/integration-regression-checklist.md)。对照原版：[SKILL.md](SKILL.md)。

意图：把散落在 ChatGPT App、Sider 与 `cpu-matrix/public/chats/` 的对话拉取进来，将**本次新拉取的内容**净化收束进 `cpu-matrix/public/subjects/<主题>/`，形成可继续讨论、可联合其他主题使用的高质量上下文资产。终点不是大纲，也不是问答流水账搬家。

## 工作流总览

| 阶段 | 执行单元 | 输出 |
| --- | --- | --- |
| 拉取 | 独立 Subagent，自行读取 `workflow/pull-stage.md` 执行 | 拉取阶段总结（范围、命令、写入资产、认证/覆盖缺口、轻量候选清单） |
| 整合 | 主控 Agent + `$info-purify` | 主题产物更新（见产物模型） |

两阶段以「拉取阶段总结」交接：拉取只把材料拉齐、记录范围、暴露缺口，不写 subjects、不做纳入判断；主题归属、信息块拆分、判断分层和正文写作全部属于整合阶段。

整合阶段全程遵循 `f-attention-budget-constraint`。

## 产物模型

每个产物由一句职责定义 + 一条收录判据锚定；标注为「推论」的条目可从判据推出，列出仅作提醒，遇到清单外的新情形以判据为准。

### 核心上下文.md（主题主入口）

职责：目标读者继续讨论、判断、应用、维护本主题时的唯一正文入口。

收录判据：**一段内容能进正文，当且仅当目标读者消费主题时需要它**——净化后的定义、事实、观点、关系、边界、应用入口、待确认命题。

推论：维护过程材料（候选筛选表、旧新映射、审计表、逐轮问答台账）不进正文，归净化日志；主题归入边界归 `主题边界.md`；来源清单归 `索引.md`，正文只保留理解强事实所需的 `Sxx` 锚点。

结构从内容长出来，二级标题划分稳定内容范围：

```markdown
# <主题>：核心上下文
## 主题定位与使用方式
## <按内容划分的模块，1..N>
## 应用入口：后续讨论或具体问题时怎么用
## 当前可用判断与不可用判断
## 待讨论命题与研讨接口
```

写作前先判闭环成熟度，它决定结论姿态：

| 状态 | 写法 |
| --- | --- |
| 未闭环 | 事实、观点、命题、待确认分层写，不做终局裁决 |
| 阶段性闭环 | 上提当前可用判断，保留边界、关系和待确认 |
| 已定型 | 可写成操作手册或稳定方法论，过程下沉到来源和日志 |

### 主题边界.md（固定文件）

职责：回答「一份新材料属不属于本主题」——本主题处理什么、不处理什么、归入判断方法、当前使用方式。

收录判据：只写本主题自身的判断依据。推论：不借其他主题解释本主题边界；未被正文实际使用的主题不出现。

### 索引.md（导航资产）

职责：**读者质疑正文某条判断时，沿索引找到出处**。它是回溯入口，不是第二份正文，也不是材料台账。

收录判据：一条来源能进索引，当且仅当删掉它会让正文某个最终内容失去出处。写完每一行用这个判据自查。

推论：排除、暂缓、相邻、背景、仅用于筛选的材料不承担出处职责，不进索引；其他主题自己的依赖不支撑本主题正文，不级联写入；对话目录一行已足以定位出处，不逐轮展开 `001.md` 清单。

schema：

```markdown
# <主题>：索引

> 本索引只收录实际用于形成本主题最终内容的来源。使用主题内容以
> `核心上下文.md` 为入口；判断材料归属以 `主题边界.md` 为入口。

## 主题入口            # 2–3 行固定
- 行格式：[文件名](路径)：一句话入口身份

## 明确依赖来源
### 对话来源           # 0..N 行；每个对话目录至多 1 行
- 行格式：Sxx：[对话标题](../../chats/<目录>/)：来源角色
  （核心材料 | 关键事实来源 | 关键观点来源）
- 例外：某一轮是唯一出处时才允许指到具体轮次，单列并说明理由
### 直接依赖主题       # 0..N 行
- 行格式：Txx：[主题名](../<主题>/核心上下文.md)：正文依赖对方哪部分
### 底层概念           # 如存在
- 行格式：[概念](./<概念>.md)：一句话入口身份
```

### 底层概念文件

职责：当某个概念是全文评价轴（约束多个后续模块）时，单独承载其定义、边界、非范围和关系约束，并作为最高权威来源。只属于单一模块的定义留在模块内，不单独成文。

### 净化日志.md / 工作记录.md（可选过程记录）

职责：过程追溯——拉取阶段总结、候选筛选记录、结构诊断、旧新映射、事实/概念/关系核查、专项交接、升级候选、校验记录、剩余风险。

落盘分档：轻量任务不落盘，关键风险在交付说明汇报；标准任务（长期维护、材料多、需复核）可落盘；审计任务或用户要求可追溯时必须落盘。

若某项风险会影响读者使用主题，先转译成正文中的「使用限制 / 待确认命题 / 当前不可用判断」，日志只保留完整追溯。

### 历史 核心判断.md

既有 `核心判断.md` 是历史主入口和高权重输入：读取吸收其有效内容，当前主入口落在 `核心上下文.md`；旧文件默认保留，仅在用户明确要求时迁移或删除。

## 结构归属判据

对每个候选（按信息块粒度，不被对话标题绑死），顺序回答四问，取第一个成立的：

1. 它改变判断对象和长期消费路径吗？→ **新主题**（标志：独立消费路径、独立核心对象、需要自己的概念与边界、维护生命周期不同、塞进旧主题会改变旧主题总定位）
2. 它在既有主题内立起一个稳定子问题吗？→ **新内容模块**（标志：内部能形成完整信息结构，后续材料会继续追加，与其他模块有明确关系）
3. 它能说出接到既有哪个模块、补强哪条事实或关系吗？→ **归入已有模块**（说不出精确脉络接点的，归入状态不得写「纳入既有主题」；「内容也属于这个大方向」不是接点）
4. 都不是 → **补充 / 暂缓 / 排除**，只留筛选记录

配套判断：

- **对话漂移**：持续进行的同一对话可能中途改变讨论对象；按轮次和信息块重新判断归属，不整段沿用前半段的主题。
- **拆分归属**：一个对话的多个信息块可分属不同主题；原始对话目录只保留一个来源资产，不复制不改写。
- **边界升级 vs 泛化**：新材料要求既有主题升级维度时——补一句边界声明能解决的是必要边界升级；升级后该主题会承包一切问题的是泛化风险，改建派生主题；用户明确确认重定义核心对象的才按 info-purify 审计模式重构，不顺手改。
- **多因成因栈**：领域问题由多个独立因素叠加时，正文单列成因栈（因素 / 来源 / 独立性 / 作用关系 / 处理归属），不把全部困难归因到最相似的上游主题。

## 阶段一：拉取

主控只确定主题名、目标目录、数据源范围和时间窗口，然后派发独立 Subagent：只传 `workflow/pull-stage.md` 路径和用户输入，主控不打开该文件，Subagent 自行读取执行并返回拉取阶段总结。

当前环境无法启动 Subagent 时，先停止并说明；仅在用户明确允许主控代跑时执行，并在交付中标记偏离。

Subagent 返回后更新行为日志（关键行为、影响清单、子 Skill 调用）。

## 阶段二：整合

### 步骤 1：盘点与候选筛选

在 `cpu-matrix` 项目根读取既有主题文件（索引、主题边界、核心上下文、概念文件）和拉取阶段总结列出的候选。

候选来源约定：候选只来自拉取阶段总结（默认 `aichat sync` 时间窗内已变更对话，缺省窗口 2 天）；用户给出时间范围、URL、id 或本地路径时以用户为准；仅当用户明确要求全量/历史回溯时扩大 sync 范围或用 `aichat view <regex>` 分批查阅。推论：整合阶段不自行遍历 `public/chats/` 目录。

对每个候选执行结构归属四问，产出轻量筛选记录（归净化日志，不进三个主题文件；索引只从中摘取实际形成出处的来源）：

| 字段 | 说明 |
| --- | --- |
| source_id | 稳定编号 `Sxx` |
| 数据源 | ChatGPT / Sider / 既有本地文档（双源进同一份记录，主题结构不按数据源拆分） |
| 路径 / 标题 | 对话目录粗粒度链接，不逐轮展开 |
| 边界理由 | 为什么属于 / 相邻 / 不属于 / 暂不属于 |
| 归入状态 | 纳入 / 排除 / 暂缓 / 相邻主题 |
| 来源角色 | 核心材料 / 补充案例 / 边界反例 / 相邻主题材料 / 明确排除 |
| 结构动作 | 新主题 / 新模块 / 归入已有模块 / 拆分归属 / 历史案例 / 升级候选 |
| 脉络接点 | 归入既有主题时必填：接到哪个模块、补强哪条事实或关系 |
| 必要依赖来源 | 正文判断实际依赖的其他主题或文件；无实际引用则留空 |

拉取中发现目标之外的新候选时单独登记，按四问独立判断；暂不确定的只进筛选记录，不因「内容符合」并入宽泛命名的旧主题。

### 步骤 2：底层概念与架构位置

若 subject 已有概念文件，按最高权威来源处理，并检查架构位置：它是全文评价轴还是模块内定义？是否约束后续模块、改变章节顺序或权威优先级？其「非范围」能否防止近敌概念被误写成主题核心？多上游主题并存时，各概念分别提供机制、元治理、应用边界还是历史案例？

### 步骤 3：运行 info-purify，写正文

读取 [info-purify](/Users/a1111/Desktop/ai-workspace/.cursor/skills/info-purify/SKILL.md) 按其阶段执行；标准模式为默认，长期维护/材料多/概念复杂时用审计模式。两层映射：info-purify 的「产物本体」→ `核心上下文.md` 正文（必要时拆概念文件）；其结构诊断、旧新映射、事实承诺、关系表、专项交接、升级候选 → 净化日志记录。`主题边界.md` 与 `索引.md` 由本 Skill 固定维护。

每个正文模块的局部净化顺序：

1. 确认该模块的消费动作（后续讨论 / 联合判断 / 直接应用 / 维护复核）。
2. 把材料拆成信息块：事实、观点、命题、反证、边界、候选方案、待确认。
3. 只写服务模块主题的内容，按收录判据取舍；把散落问答**聚合**成结构，不照搬原始顺序。
4. 显式写出关系：`grounds / constrains / supports / depends_on / complements / resolves / modifies`。
5. 未闭环部分写成待讨论命题或使用限制，不为完整感强行收束。
6. 回查该模块的结构归属（应拆分 / 上提新主题 / 仅历史案例时回退修正）。

### 步骤 4：更新索引

按索引 schema 重写或合并 `索引.md`。默认合并重排，不删除用户原有精心维护的链接，除非明确替换。

### 步骤 5：校验

逐条执行 [references/integration-regression-checklist.md](references/integration-regression-checklist.md)（覆盖历史翻车点，必须实际打开逐组核对，不凭印象封口），另加：

- 链接检查：三个主题文件的相对链接可解析（可用只读 Node 脚本）。
- 工作区检查：`git -C .../cpu-matrix status --short` 汇报状态；前置提交与产物提交分工正确。
- 行为日志封口或声明审计缺口（对照 [references/audit-quality-standards.md](references/audit-quality-standards.md)）。

## 补丁进入协议（维护者须知）

本 Skill 后续新增约束时，先问「哪个既有职责句 / 收录判据没盖住这个失败，还是只是表达失败」，再按类型路由；**禁止在正文平铺追加禁令**：

- 揭示职责定义或判据本身有误 → 修改产物模型对应判据（罕见，需明确理由）。
- 新翻车防线（模型默认倾向导致的错误）→ 追加到 `references/integration-regression-checklist.md`，附一行触发场景。
- 字段级边界 → 写进对应 schema 的「收 / 不收」行内。
- 用户新偏好 / 外部契约 → 约定表。

同一约束语义在正文出现多于一处，视为架构失败，须收敛回唯一表述点。

## 交付口径

只汇报高信号内容：拉取阶段是否由独立 Subagent 完成及其总结要点（范围、写入资产、缺口）；主题目录更新了哪些文件；`核心上下文.md` 是否已可作主入口（旧 `核心判断.md` 的处置）；索引是否只含实际出处来源；净化日志落盘档位及理由；概念架构位置是否变化；校验结果与剩余风险；行为日志路径与终态。若拉取未按标准隔离或整合未独立复核，明确说明偏离与授权。
