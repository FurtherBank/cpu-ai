# 步骤四：工单性质语义锁定 执行指导

## 目标

在查阅代码之前，用工单描述和评论文本锁定工单性质。

**两条核心防线：**

> **防线一（认知漂移）**：若先搜索代码，再判断性质 → 搜索结果（「功能确实不存在」）会潜移默化影响受理决策（「既然没有，顺势做」）。正确顺序：先锁定性质 → 再用性质决定搜索目的 → 代码搜索结论服从性质结论。
>
> **防线二（Bug 锚定独立于性质判定）**：工单有「曾经正常 → 现在不正常」的语义，只能说明性质是 Bug，**不代表可以 accepted**。`accepted` 的前提是步骤五能通过代码搜索将问题收窄到可定位的功能模块。若信息不足，步骤五必须出 `CANNOT_ANCHOR`，先追问再接受。

---

## 性质判定规则

**以描述语义为唯一依据，不依赖工单 type 字段（客户可能填错）。**

### Bug（已有功能的非预期失效）

充要条件：描述语义中存在**「曾经正常 → 现在不正常」**的失效表述。

典型特征：
- "以前能用，现在不行了"
- "页面出错"、"接口报 500"、"功能失效"
- "升级/更新后开始出现"

### 需求（新能力扩充）

充要条件：描述语义为**「希望有但目前没有」**的正向扩充，且不含失效语义。

典型特征：
- "希望增加…"、"支持…功能"、"能否添加…"
- "我们需要一个…的功能"

⚠️ 常见混淆：客户写「系统没有 X 功能，这是个 Bug」——注意：**缺少某功能不构成 Bug**，以「非预期失效」为唯一标准，不被措辞中的「Bug」带偏。

### 咨询（使用方式/机制疑问）

充要条件：描述语义为**询问现有功能的用法、机制或限制**。

典型特征：
- "怎么配置…"、"是否支持…"、"有效期是多久"
- "如何…"（不含失效意味）

**咨询类的处理路径**：判定为咨询后，性质标签直接标记为 `CONSULTING_RECEIVED`，**不进入步骤五**，直接跳到步骤六撰写回复评论。

咨询不涉及代码变更，工单状态为 received，**不打 accepted**。

---

## 需求合理性评估（性质标签 = 需求时执行）

**功能需求不等于超范围**。当工单为需求时，需先评估是否可以接受，再决定代码搜索目的。

### 评估结论：可接受（IN_SCOPE_FEATURE）

满足以下**全部条件**时，功能需求可以 accepted：

| 条件 | 判断方式 |
|---|---|
| 技术职责在 cwd 内 | 功能实现落在当前 cwd 代码仓库范围，不需要跨系统协调或其他团队介入 |
| 无明显技术不可行 | 不涉及需要专项技术评估的方向（如集成外部支付/AI预测/硬件接入等） |
| 需求规模增量合理 | 是在现有系统上扩展某个功能点，而非系统性重构或全新平台建设 |
| 描述足够具体 | 能理解「要做什么」，不需要进一步产品设计就能评估实现路径 |

**典型可接受示例**：
- 导出 Excel 报表（支持自定义列）
- 为某列表页增加筛选条件
- 在现有通知功能中增加邮件渠道

### 评估结论：不可接受（OUT_OF_SCOPE_FEATURE）

满足以下**任意一条**时，引导客户走产品需求评审通道：

| 条件 | 示例 |
|---|---|
| 明显需要跨团队/跨系统协调 | 「与 ERP 系统打通」「接入第三方支付」 |
| 涉及产品/业务层决策，非技术执行层能独立决定 | 「修改权限体系」「调整定价策略」 |
| 规模过大，属于新产品/新平台级 | 「设计一套完整的工作流引擎」「重构整个前端架构」 |
| 描述过于模糊，无法推断实现路径 | 「优化一下用户体验」（需回到追问路径补充） |

## 代码搜索目的声明（Bug / 需求类必填；咨询类跳过）

在进入步骤五搜索代码之前，用一句话写出本次搜索的目的。目的必须与性质标签绑定：

| 性质标签 | 目的声明模板 |
|---|---|
| Bug | "定位可能的失效代码模块，确认问题在 scope 内可锚定；若无法锚定则出 CANNOT_ANCHOR" |
| 需求（IN_SCOPE_FEATURE） | "确认当前已有哪些相关能力，评估增量实现的代码接入点，辅助受理判断" |
| 需求（OUT_OF_SCOPE_FEATURE） | "确认当前已有哪些相关能力，辅助评论撰写，**不改变引导产品通道的决策**" |
| 咨询（CONSULTING_RECEIVED） | **不写目的声明，不进步骤五** |

---

## 边界场景处理

**描述语义模糊，无法区分 Bug 与需求**：
- 走 received 追问，要求客户补充「这个功能之前是否正常工作过」
- 不在信息不足时做猜测性判断

**工单描述极短（5 字以内，如「不好用」）**：
- 仅凭工单标题/描述无法完成性质判定
- → 直接进入追问路径（评论要求补充具体描述），不尝试代码搜索

**有回归语义但描述极度模糊（无接口名、无报错文案、无操作路径）**：
- 性质可判定为 Bug（有「以前正常→现在不行」），但步骤五代码搜索无法锚定到具体模块
- → 步骤五必须出 `CANNOT_ANCHOR`，评论要求补充：具体接口名/功能入口、报错信息或截图、问题首次出现时间
- → 下一轮信息充足后再重新评估是否 `IN_SCOPE_BUG`
- **不允许在无锚点的情况下 accepted，也不允许在评论中说「已纳入处理队列」**
