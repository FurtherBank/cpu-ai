# nextack-spm：PC Web 中台 SPM 埋点

**PC Web 中台**常见为 React Router、Vite/Webpack 等自管路由，通常不能依赖 `spm` 自动从框架托管配置里解析 B 位；需要在 **前端包根目录**（该前端 **`package.json` 所在目录**，见下文「执行上下文与目录」）维护 **`spm.config.json`**，并在页面或路由配置中声明 **`spmBPos` / `spmb`**。命令行工具使用 npm 包 [`@alipay/fast-spm`](https://www.npmjs.com/package/@alipay/fast-spm) 提供的 **`spm` CLI**。

SPM 四层结构：**spma（站点）→ spmb（页面）→ spmc（模块）→ spmd（点击位）**。中台落地时重点核对 **A 位来源、B 位目录约定、`ab=` 入参、Tracert 初始化、CD 命名与平台验收**。

常用参考：

- **初始化（含 `spm.config.json`、Tracert 4.x HTML 片段、CLI、A/B 位、首次申请、TS）**：`references/pc-web-spm-init.md`。
- **命名与分屏**：`references/naming-rules.md`。

## 执行上下文与目录（多包 / 多仓库）

- **根目录（本文一律指）**：项目仓库内 **前端 npm 包的根目录**，即该包 **`package.json` 所在目录**；**`spm.config.json` 与该 `package.json` 同级**。单包前端仓时多与 Git 仓库根重合；**monorepo 时多为 `apps/web` 等子包目录，不是整个 monorepo 的 Git 根**（除非仓库根本身就是该前端包）。执行 **`spm` / `spm -c`** 及依赖安装时，约定 **`cd` 到此前端包根**。
- **`spm.config.json` 位置**：放在 **前端包根目录**（与 `package.json` 同级）。`page` 等路径相对 **在该目录下执行 `spm` 时的当前工作目录** 解析，通常写 `src/pages` 等即可。
- **`index.html` 位置**：每个独立 Vite 应用的 **`root`** 下各有一份（与 `spm.config.json` 未必同目录，但仍在同一前端包或约定子路径内）；Tracert 片段按应用改，勿按错误的父目录或仓库根改。
- **依赖安装**：在 **前端包根** 下按项目约定执行（如 `npm install`、`pnpm install`、`yarn`、`tnpm install` 等），以该包 `README` 或贡献指南为准。

## 约定速查表

| 项 | 约定 |
|----|------|
| **框架默认** | PC Web 中台（React + 自管路由为主） |
| **`appType` 默认** | **`site`**（PC 站点埋点）；纯移动 H5 仓改为 **`webapp`** |
| **声明式入口** | **前端包根** **`spm.config.json`**（与 `package.json` 同级；见 `references/pc-web-spm-init.md`） |
| **Tracert 运行时 HTML** | Vite 应用根 **`index.html`** 的 **`<head>`**：启动器 → CDN → `set`（见 `references/pc-web-spm-init.md`） |
| **页面代码位置** | 默认 **`src/pages/<页>/`**（相对前端包根；monorepo 下物理路径常为 **`apps/<app>/src/pages/`**） |
| **A 位** | 组织流程申请/绑定；写入 **`spm.config.json` 的 `spma`** 与 HTML；**不**由工具「发明」A |
| **B 位** | 源码 **`spmb`/`spmBPos`**（页面中文）+ 平台已注册的数字 **`b`**；可选 **`pages-spm.json`** 台账 |
| **声明 → 点位代码** | 写好中文 B/CD 声明后执行 **`spm`**（或 `spm path=… ab=`）；成功后 diff 中应出现 **`data-aspm-click`** 等回填 |
| **申请命令** | 多应用/单页补洞可 `path=`；`path=` + **`ab=`** 时核对 **`spm --help`**：部分 `spm.config.json` 布尔项为真会走另一套解析分支，易导致 **不按 `ab=` 申请 CD** |
| **CD 命名** | 源码写 `spm="模块名-按钮名"`，详见 `references/naming-rules.md` |
| **属性键** | 含 **C+D** → `data-aspm-click`；**仅 C** → `data-aspm`；勿机械按 `spm -c` 误报改错（**先读表达式是否含 d 位**，见下文「spm -c 检测」） |

## 开发验证工作流


每次处理埋点需求时，严格按照下面动作流程推进：

### spm -c 检测

**目标**：在不上报、不跑浏览器的前提下，用静态扫描尽早发现「该埋未埋」、属性键与 C/D 语义不一致、曝光缺失等常见问题，作为申请回填后的第一道机械门禁；**不能**替代下文充分度评审与平台验证。

```bash
spm -c
# 或（在前端包根下，path 相对该目录）
spm -c path=src/pages
```

关注：缺 SPM 的 `onClick`/`onTap`、`href` 的 `<a>`、错误 key、仅 C 却写在 `data-aspm-click`、已埋点缺 `data-aspm-expo` 等（细则见下文「为元素声明 spm」与 `references/naming-rules.md`）。

### 为元素声明 spm

- 跳过已有 `data-aspm` / `data-aspm-click` / `spm` 的元素。
- 跳过带 `// ignore fast-spm` 的元素。
- **`<a href>`** 与点击区同等对待，需导航语义命名。
- CD 声明形态还包括：`const spmOk = "模块-按钮"`、`Tracert.click("模块-按钮")` 等（以 **`spm --help`** 与埋点平台当前支持为准）。

### spm 申请与 path+ab 补洞

| 变更范围 | 命令 | 说明 |
|------|------|------|
| 当前包内多页或不确定遗漏范围 | **`spm`** | 中文 B/CD 写好后执行 **`spm`**（俗称「三个字」）统一扫描、申请并回填；适合首次补齐基线后的全量申请，也适合后续多页一起补洞 |
| 仅本次变更目录且已知 **A.B** | `spm path=src/pages/某页 ab=<spma>.<spmb>` | `b` 须在平台存在且与该页一致；适合新增/调整某个已定位页面的 CD 补洞。**若接口无申请结果**，对照 **`spm --help`** 检查 `spm.config.json` 是否与 `path=`+`ab=` 用法冲突 |

成功后从控制台复制 **埋点平台迭代/需求链接**（`spm` 常会打印 `可在埋点平台查看: https://...`，常见形态如 `https://log.alipay.com/index_v4.htm#/iterationDetail?demandId=…`），写入 MR 描述或报告最后一行；**随后须先完成下文「埋点充分度（评审自检）」并输出充分度简报，再进入「埋点需求平台侧验证」**。

### 分屏回填修正

申请后搜索 `` `${index}` ``，按外层 `.map((_, idx)` 等改为实际索引变量名（分屏回填占位须与真实循环变量一致）。

### 浏览器与 DOM 抽检

无登录场景时，可用 **`vite preview`** 或已启动的 dev 地址打开页面，检查关键按钮、链接、表格行操作等可交互区域仍存在且可点击。可用 **`[spm='…']`**、`[data-aspm-click='…']` 等选择器辅助确认 DOM 中的声明或回填属性。

DOM 抽检只证明页面元素与属性存在；是否已上报到埋点平台，仍须按 **「埋点需求平台侧验证」** 做 **PC + 实时验证**。可与「埋点充分度」代码评审穿插进行，但 **不得** 用 DOM 存在属性替代充分度清单对照。

### 埋点充分度（评审自检）

**执行时机（须在平台验证之前）**：在 **`spm` / `spm path=` 已成功、`spm -c` 已达团队约定门槛、diff 已审查（含分屏）之后**，**必须先于**下文「埋点需求平台侧验证」的步骤说明或「验收提示模板」执行本节的 **代码逻辑评审**；**禁止**仅以「`spm -c` 无报错」或「已回填属性」代替充分度结论。

**执行方式**：Agent（或贡献者）须结合 **本次变更实际涉及的源码路径、页面/路由结构、事件与分支逻辑**（含异步成功/失败、条件渲染出的按钮、抽屉 footer、表格行操作等），对照下列清单 **逐项** 判断已覆盖 / 已知缺口 / 不适用（不适用须写明原因，如「本 PR 无表单提交」）。

1. **可点即可见数**：用户可点的控件（含图标按钮、行内操作、空状态主按钮、抽屉 footer）均有 CD 或等价 API 埋点。
2. **链路关键步**：创建/编辑/删除/提交/取消、导出、权限错误后的「去申请」等 **业务状态迁移** 至少一端有埋点。
3. **列表与表格**：批量操作、行操作、排序/筛选若可点，**模块名一致、列表项带 `_N` 或明确索引变量**。
4. **外链与协议**：`<a target="_blank">`、下载链接不遗漏。
5. **曝光**：默认带 `data-aspm-expo` 的点击埋点有利于点击率；模块级曝光按数据同学要求补 `data-aspm` + expo。
6. **重复名**：同一页面内不同按钮不得共用无法区分的 `spm` 中文描述。

**充分度简报（必选输出）**：在向用户给出 **「埋点需求平台侧验证」的具体操作步骤**或粘贴 **「验收提示模板」** 之前，须先输出一段 **简短充分度执行情况**（建议 **3～8 行**），至少包含：**对照了哪些页面/目录或模块**（与代码路径对应）；上列六项各用 **已覆盖 / 缺口 / 不适用** 之一概括（缺口须点出交互或路径名）；若存在缺口，说明是 **当场补埋点**、**记入 MR 待办** 还是 **与数据方确认可后补**。完成该简报后，再接平台验证说明与模板。

### 埋点需求平台侧验证（PC + 实时）

**前置**：已完成上文「埋点充分度（评审自检）」的代码对照，并已输出 **充分度简报**。

在 **`spm` / `spm path=` 已成功**、且已从控制台取得 **埋点需求/迭代详情链接**（全文复制，勿截断 query）后，按下述步骤验收（平台文案若有微调，以线上页面为准）：

1. **打开埋点需求链接**（即本次命令输出中的地址），在站点内找到并点击 **「开始验证」**。
2. 在弹出对话框中：**平台类型** 选择 **「PC」**；**验证方式** 选择 **「实时验证」**；在 **网页地址** 输入框填入待测页 URL（**支持本地** `http://localhost:<端口>/` 或 `http://127.0.0.1:<端口>/`，与当前 `pnpm dev` / `vite preview` 实际地址一致）；然后点击对话框中的 **「开始验证」**，按平台引导完成后续操作验证。

若本次申请**未成功**、控制台**未打印**可打开的迭代/需求链接，则须先排查接口报错或 B 位等问题，**勿**让用户在无效需求页上操作。

**验收提示模板**（将 `<埋点需求链接>`、`<本地页面地址>` 替换为实际值）：

```text
埋点 CLI 已执行完毕，请开始平台验证：
1. 打开埋点需求链接：<埋点需求链接>，在页面中点击「开始验证」。
2. 在弹窗中：平台类型选「PC」，验证方式选「实时验证」；网页地址填写 <本地页面地址>（支持 localhost）；再点击弹窗内的「开始验证」，按平台引导完成验证。
```

### CI 与 MR 门禁（推荐）

- **MR 描述**：至少包含 **本次 `spm` 控制台输出的埋点需求/迭代链接**（若本次为纯 `spm -c` 修复且无新申请，注明「无新迭代」及原因）、**主要变更页面或模块**。
- **CI**：将 job 的 **`working_directory`**（或等价配置）设为 **前端包根**（`package.json` 所在目录），再执行与本地一致的校验，例如 `spm -c path=src/pages`（`path` 相对该目录，按仓库调整）。失败时阻断合并；**WARN 是否阻断**由团队约定写入流水线注释。
- **与 `spm` 命令同版本**：CI 镜像或 job 中安装的 `@alipay/fast-spm` 主版本宜与贡献者本地一致，避免「本地过 CI 挂」。

### 发版与线上复核

- **构建产物抽检**：对预发/生产构建后的 `index.html`（或等价入口）确认 **CDN `tracert_{spma}` 与内联 `spmAPos`** 与当前环境约定一致；若用 **`body data-aspm`** 做页级 B，确认与平台当前页 **b** 位一致。
- **发版后**：结合「如何看数据」一节，用平台 **实时/调试** 能力做关键路径 **T+0** 抽检；正式报表结论仍以 **T+1** 为准。

### 迭代维护：新页面与路由变更

- **新增一级页面目录**：回到上文统一流程，先同步增加 **`spmBPos`/`spmb`**（及路由侧若需的 pathname 映射），再按变更范围执行 **`spm`** 或 `path=… ab=`；**勿**在未声明 B 位时批量复制其它页的 `data-aspm-click`。
- **路由 path 调整而 UI 仍是同一「分析页」**：仍按统一流程执行，重点评估是否应合并 B 位中文或拆分新 B，避免报表口径无意识合并/分裂。
- **下线功能**：删除或注释声明与回填属性，并重新跑 **`spm -c`** 与充分度评审，避免残留无效点位污染分析。

## 如何看数据：实时与 T+1

以 **埋点平台 / 行为分析产品** 的说明为准（不同租户与产品模块可能不同）。一般经验：

| 类型 | 常见时效 | 说明 |
|------|----------|------|
| **调试/抽样/实时大屏**（若平台提供） | 分钟级～准实时 | 用于联调、发布当日验证；口径可能与离线报表不一致。 |
| **标准报表、漏斗、留存、人群** | 多为 **T+1** | 离线数仓加工后稳定；用于产品决策与考核口径。 |
| **`spm` 成功后的迭代详情页** | 即时打开 URL | 用于 **注册结果、需求绑定、验收流转**；不等于业务指标已 T+0 可查。 |

**操作建议**：**开发联调阶段**在 **充分度简报** 之后，按 **「埋点需求平台侧验证」** 在埋点需求页使用 **PC + 实时验证**（可填 localhost）确认上报；发版后仍可用平台 **实时/调试通道** 做「有没有点、字段对不对」复核；用 **T+1 报表** 做趋势与漏斗结论。

## 交付自检清单

一次埋点交付完成前，按序自检：

1. **上下文**：已定位正确 **前端包根**（含 `package.json`；monorepo 时通常不是仓库最外层目录），并已阅读该仓库内项目说明（如 `README`、贡献指南，若存在）。
2. **初始化**：按 **`references/pc-web-spm-init.md`** 完成 `spm.config.json`、`index.html` Tracert 与 A/B 约定；必要时已补充 **`spm-dom.d.ts`**。
3. **开发**：新增/变更交互点已写 **`spm` 或合规 API**；B 位中文无已知重复。
4. **申请**：已运行 **`spm`** 或带 **`ab=`** 的 `path=`，并已 **审查 diff**（含分屏变量）。
5. **静态**：**`spm -c`**（带正确 `path=`）通过团队约定门槛。
6. **充分度（代码评审）**：已按上文「埋点充分度」清单结合 **实际代码路径与交互逻辑** 完成对照；在向用户给出平台验证步骤或验收模板 **之前**，已输出 **充分度简报**（简述覆盖/缺口/不适用及处置）。
7. **验证**：有迭代链接时，已完成或明确交接 **平台 PC 实时验证**；必要时已做 DOM 抽检。
8. **收尾**：MR/输出中含 **迭代链接** 与 **变更摘要**；若接入 CI，已说明或已改流水线命令路径。
