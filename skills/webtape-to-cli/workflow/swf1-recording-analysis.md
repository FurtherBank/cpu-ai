# 子工作流 1：录制分析与接口提炼 — 执行指导

## 输入信息

**必须入参**：
- `录制目录路径`：`webtape-workspace/recordings/<hostname>/`，可能含 1 个或多个会话子目录
- `目标功能描述`：用自然语言描述希望实现的 CLI 功能（如"查询最近编辑文档"）

## 目标要求

**任务**：从录制材料中提炼出核心业务接口规格，并评估信息完整性。

**目标**：输出「接口规格文档」，能让另一个 Agent 仅凭此文档完成 CLI 设计，无需再读录制原始文件。

**特性要求**：
- 核心接口清单中不包含埋点/静态资源/WS/SSE 等噪音接口
- 每个核心接口有明确的业务用途描述（不是 URL 的重复）
- 接口依赖关系已明确（A 的哪个输出字段 → B 的哪个输入参数）
- 每个接口有明确的实现策略（完整实现/推断实现/降级参数/待补录）
- **不做**：不输出完整的请求/响应 JSON，只输出字段摘要

## 工作依据

### 环节 1：录制材料摸底

1. 列出 `recordings/<hostname>/` 下所有会话目录（`ls`）
2. 对每个会话，检查以下文件的**存在性**与**文件大小**（`wc -c`）：
   - `_context.md`
   - `analysis_report.md`
   - `request.js`
3. 构建录制清单表（此步骤**不读任何文件内容**）：

| 会话 | analysis_report.md | _context.md 大小 | request.js |
|---|---|---|---|
| 0330-xxx | ✅ 存在 | 38KB | ✅ 存在 |
| 0320-xxx | ❌ 缺失 | 91KB | ❌ 缺失 |

**关键判断**：
- 有 2+ 个会话 → 多段录制模式，后续逐段处理
- 某会话 `_context.md` > 50KB → **标记为「大文件，须分段读」**
- 某会话缺少 `analysis_report.md` → **标记为「待分段读」**

---

### 环节 2：分层读取材料

**四层递进策略（严格按顺序，能解决问题就不进下一层）**：

#### 第 1 层（始终执行）
- 若存在 `analysis_report.md` → **全量读取**（5~20KB，安全）
- 若存在 `request.js` → **全量读取**（2~5KB，安全）
- 从这两个文件提取接口清单初稿

#### 第 2 层（第 1 层信息不足时触发）
```
_context.md ≤ 50KB → 全量读取
_context.md > 50KB → 禁止全量读取！
  执行：grep -n "^### " _context.md     ← 获取时间线节点列表
  执行：grep -n "#### \[req_" _context.md ← 获取接口 URL 概览
```

#### 第 3 层（需确认某接口请求/响应结构时触发）
- 定位 `requests/req_xxxx.json`，精确读取
- 对应 `responses/res_xxxx.json`，精确读取
- **不读 requests/ 或 responses/ 目录整体**

#### 第 4 层（接口参数不确定，且其他层无法解决时）
- 仅读取已知关键接口的完整 req/res 对
- 每次读取后立即评估是否还需要读更多文件

**多段录制时**：对每个会话独立执行以上四层，记录各段覆盖的功能区域。

---

### 环节 3：噪音识别与过滤

**三规则（按顺序应用）**：

| 规则 | 触发信号 | 处置 |
|---|---|---|
| ①域名过滤 | 主机名含 `collect.`/`log.`/`gw.alipayobjects.com`/`renderoffice.`/CDN 域名 | 直接排除，标注「埋点/静态资源」 |
| ②类型过滤 | `requests/req_xxx.json` 中 `"type": "websocket"` 或 `"type": "sse"` | 标记「已知限制」，寻找等价 REST 轮询接口 |
| ③频率过滤 | 同一 URL 模式在时间线中出现 3+ 次 | 标记「轮询/心跳」，排除（或标注等价 REST 接口）|

**陷阱**：目标站点自身的埋点接口可能用目标域名（如 `yuyan.antfin-inc.com/api/collect/...`），需结合路径中的 `collect`/`track`/`monitor` 关键词判断。

---

### 环节 4：数据依赖关系提炼

识别接口间的数据流：接口 A 的**响应字段 X** → 接口 B 的**请求参数 Y**

例：
```
POST /api/sprint/{id}/deploy → 响应 data.flowId
    ↓ flowId
GET /api/flow/{flowId} → 轮询进度
```

**孤立接口**（无上游依赖）= 入口接口，接受用户输入。

---

### 环节 5：信息完整性评估与盲区决策

对每个核心接口，评估录制覆盖状态并做出实现决策：

| 缺口类型 | 决策规则 |
|---|---|
| GET 接口有录制 | ✅ 完整实现 |
| GET 接口缺失，URL 规律明显（REST 惯例） | 推断实现，代码注释：「推断自 REST 规律，非实录」 |
| POST/PUT/DELETE 接口**缺失** | ❌ **绝对禁止猜测接口签名**；CLI 降级为用户直传参数；生成补录提示文案 |
| 分页终止字段不明确 | 使用三重终止保护，代码注释标注各字段来源 |
| 多段录制接口路径差异 | 以最新录制为准，差异处代码注释标注 |

**补录提示文案格式**：
```
⚠️ 需要补录以下操作才能完整实现 <功能名>：

缺口：<接口业务描述>
- 需要录制的功能：<自然语言描述>
- 操作页面：<具体 URL>
- 操作步骤：<步骤1，步骤2，...>
- 期望捕获的接口形态：<URL 模式> (<Method>)
```

**流程判定**：盲区中含写操作缺口 → 输出补录提示后**暂停整个流程**，告知用户补录完成后重新运行本 Skill。

- 站点：<hostname>
- 会话数：<N>
- 是否多段录制：<是/否>

## 核心接口清单

| 序号 | Method | URL | 业务用途 | 关键请求参数 | 关键响应字段 | 实现策略 |
|---|---|---|---|---|---|---|
| 1 | GET | /api/recent/list | 最近编辑文档列表 | offset, limit, action, type | data.list[], data.end | ✅ 完整实现 |
| 2 | POST | /api/books/{id}/docs | 创建文档 | title, format | data.id, data.slug | ⏸ 待补录 |

## 接口依赖关系

```
接口 N → 响应字段 X → 接口 M 请求参数 Y
```
