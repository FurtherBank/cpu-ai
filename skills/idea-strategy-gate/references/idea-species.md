# Idea 物种识别参考

执行前战略校验最常见的误判是：**用一种 Idea 的标准评估另一种 Idea**。
最典型的是用「软件产品」的标准（PMF、护城河、增长引擎）去评估「认知方法论」「研究计划」「内容作品」「生活方式」等本质不同的方向，从而得出错误结论。

本文档提供「维度推导 + 常见物种速查」两层方法，用来在阶段一识别 Idea 的物种，以便阶段三/四选择匹配的评估问题。

## 核心原则

- **不要硬套统一模板**：相同的「五问」（根问题强度、独特进入权、复利资产、上游杠杆、反方否决）在不同物种下的具体含义和证据标准不同。
- **不要默认 Idea = 待开发产品**：用户说「我想做 X」时，X 可能是工具、方法、研究、内容、习惯、资产、组织能力，不能默认归为产品。
- **不要让物种决定结论价值高低**：物种识别只用来选择「评估问题集」，不影响 Idea 是否值得做。

## 五维度推导（物种识别底座）

不要直接给 Idea 贴物种标签。先回答以下五个维度，再据此推导该 Idea 的真实物种和评估方式。

### 维度 1：交付物形态（Form）

这个 Idea 最终落地为什么？

- `software`：可执行的软件、工具链、Agent、Skill
- `service`：以人或团队为载体的服务、咨询、教学
- `methodology`：可复用的方法论、SOP、判断框架、思维清单
- `content`：文章、视频、课程、书籍、播客
- `asset`：可以反复调用的素材库、数据、模型、知识库
- `state`：用户自身的状态变化（健康、习惯、技能、关系）
- `organization`：组织能力、流程、团队、社群

允许多形态混合（例如「研究 + 文章」「方法论 + 工具」）。

### 维度 2：使用者（User）

这个 Idea 的最终消费者是谁？

- `self`：仅自用
- `others`：他人使用，自己不一定用
- `mixed`：自用并对外开放

判断它是否为「商业 / 工程项目」最关键的一刀就在这里。`self` 类 Idea 不需要回答 PMF、增长、商业模式问题。

### 维度 3：价值实现路径（Value Path）

它如何产生价值？

- `direct`：用一次就有一次价值（典型：内容、状态改善）
- `infrastructure`：建成后才能被反复调用（典型：工具、方法论、知识库）
- `compound`：每次使用都增厚自身（典型：方法论、案例库、模型）
- `option`：当前不直接产出价值，但建立未来选择权（典型：研究、关系网络）

### 维度 4：成熟度起点（Maturity）

进入战略校验时，它处于什么阶段？

- `concept`：只有方向描述，核心概念尚未形式化
- `hypothesis`：有清晰的假设，但未验证
- `prototype`：有可演示的原型或初版
- `validated`：已在真实场景跑通

`concept` 阶段的 Idea **不应直接进入战略压测**，应先触发「概念清晰度评估」（参见 [concept-clarity.md](./concept-clarity.md)）。

### 维度 5：成功定义（Success Criterion）

用户认为「这件事做成」的判定标准是什么？

- `used`：被使用（自己或他人持续调用）
- `understood`：被理解（用户自己或读者获得清晰认知）
- `replicated`：被复制（被他人采纳、传播）
- `monetized`：被变现（产生现金流）
- `compounded`：被沉淀（资产持续增厚）
- `experienced`：被体验（产生主观体验或状态变化）

注意：用户口头说的成功定义可能与他真正在意的不一致。需要在阶段一通过「如果只能保留一个判定标准，你会保留哪一个」来反推。

## 常见物种速查（推导后的快速参考）

以下不是穷举，仅作为推导后的对照。任何 Idea 都应先走五维度推导，再对照本表确认。

### 物种 A：产品/工具（Product/Tool）

- 典型组合：`software` + `others/mixed` + `infrastructure` + `prototype/validated` + `used/monetized`
- 评估重点：PMF、独特进入权（渠道/技术/场景洞察）、可复制性、可维护性
- 反方主问：是否有人愿意付费或反复使用？
- **不要**用此组合评估 `self + methodology` 或 `concept + research` 类 Idea

### 物种 B：方法论/思维框架（Methodology）

- 典型组合：`methodology` + `self/mixed` + `compound` + `concept/hypothesis` + `understood/compounded`
- 评估重点：可迁移性、可手动执行性、是否反复用得上、形式化清晰度
- 反方主问：去掉所有工具、纯手动版本是否仍然成立？
- **关键提示**：方法论的成功 ≠ 做成软件。用户回答「即使是 A4 纸版本也愿意用」就是该物种的强信号。

### 物种 C：研究/探索（Research/Exploration）

- 典型组合：`asset/methodology` + `self` + `option/compound` + `concept` + `understood/compounded`
- 评估重点：研究路径清晰度、领域基础是否扎实、证伪标准、阶段性可验证里程碑
- 反方主问：如果失败，是否仍能学到可迁移的洞察？
- **关键提示**：研究类 Idea 的「需要研究」是正常起始状态，不应被判定为「核心资产缺失」。但需要追问：「研究路径是否可设计」「是否有证伪标准」。

### 物种 D：内容/作品（Content/Work）

- 典型组合：`content` + `others/mixed` + `direct` + `hypothesis/prototype` + `replicated/used`
- 评估重点：受众清晰度、表达独特性、可持续产出节奏、是否有持续素材源
- 反方主问：在没有任何外部反馈的前提下，是否仍然愿意每周产出？

### 物种 E：商业/项目（Business/Project）

- 典型组合：`software/service` + `others` + `direct/infrastructure` + `prototype` + `monetized`
- 评估重点：商业模式、护城河、增长引擎、单位经济模型
- 反方主问：经济账是否算得过来？

### 物种 F：生活方式/习惯/技能（Lifestyle/Habit/Skill）

- 典型组合：`state` + `self` + `direct/compound` + `hypothesis/prototype` + `experienced/compounded`
- 评估重点：可持续性、收益/成本比、是否真正改善生活、依赖外部条件多寡
- 反方主问：3-6 个月后还会坚持吗？

### 物种 G：基础设施/资产（Infrastructure/Asset）

- 典型组合：`asset/software` + `self/mixed` + `infrastructure/compound` + `hypothesis/prototype` + `compounded/used`
- 评估重点：上游性、稳定性、迁移性、其他方向是否真的会调用它
- 反方主问：未来 6 个月里，下游真实任务会调用它几次？

### 物种 H：组织能力（Organization）

- 典型组合：`organization` + `mixed` + `infrastructure/compound` + `concept/hypothesis` + `replicated/compounded`
- 评估重点：流程清晰度、可培训性、激励机制、文化适配
- 反方主问：脱离创始人是否仍能运行？

## 多物种与混合形态

很多 Idea 是多物种叠加，必须**显式拆分**而非合并评估：

```markdown
示例：「写一个降低语言熵的工具」

拆分：
- 物种核心：方法论（Methodology）—— "降低语言熵"是认知方法论
- 物种外壳：产品/工具（Product）—— 想做成工具是表达形态
- 关键关系：外壳依赖核心成立，核心不依赖外壳

评估顺序：
1. 先评估核心物种（方法论）：手动版本是否成立？
2. 再评估外壳物种（工具）：核心成立后再考虑是否需要工具化
```

混合物种的判断规则：

- 若两个物种是「核心 + 外壳」：先评估核心，外壳暂缓，避免外壳的硬指标污染核心评估
- 若两个物种是「并列双形态」：分别评估，再判断关系
- 若两个物种是「上下游」：评估上游为主，下游作为复利信号

## 物种推导工作流

阶段一收集到 Idea 描述后，主控按以下顺序推导：

```markdown
## 物种推导：${idea_name}

### 五维度判断
- 交付物形态（Form）：${software/service/methodology/content/asset/state/organization}
- 使用者（User）：${self/others/mixed}
- 价值实现路径（Value Path）：${direct/infrastructure/compound/option}
- 成熟度起点（Maturity）：${concept/hypothesis/prototype/validated}
- 成功定义（Success Criterion）：${used/understood/replicated/monetized/compounded/experienced}

### 推导结论
- 主物种：${对照速查表选择}
- 是否多物种：${是/否，若是说明核心物种与外壳物种}
- 触发概念清晰度评估：${是/否，若 Maturity=concept 且核心概念依赖未定义术语，则必须触发}

### 评估问题集选择
- 将使用 [${物种}] 对应的评估问题集
- 不适用的标准维度：${列出该物种下不适用的常规问题}
```

## 常见误判模式（反例）

| 误判 | 错误后果 | 修正 |
|---|---|---|
| 把「方法论」当「产品」评估 | 强行追问 PMF、增长、商业模式，导致结论是「软件还做不成所以应该否决」 | 用方法论物种问题：手动版本是否成立、可迁移性、复利度 |
| 把「研究」当「产品」评估 | 把「需要研究」判定为「核心资产缺失」 | 用研究物种问题：研究路径、证伪标准、阶段性里程碑 |
| 把「内容」当「商业项目」评估 | 强行追问商业模式，忽略持续创作动力 | 用内容物种问题：受众、表达独特性、产出节奏 |
| 把「自用工具」当「市场产品」评估 | 强行追问目标用户群，但用户只想自用 | 用基础设施物种问题：自己未来调用频次、是否替代手动 |
| 把「混合物种」合并为单物种 | 用外壳指标污染核心评估 | 显式拆分核心 + 外壳，分别评估 |
