# 阶段一：边界锁定与最小输入

## 输入信息

**必须入参**：
- `用户原始需求`：用户关于 idea、项目、方向选择的原文。
- `候选范围`：多个 idea、单个 idea，或尚未列出的想法集合。
- `执行前约束`：用户是否明确不希望进入 MVP、实现方案或执行验证。

**上下文参考**：
- [战略五问门参考](../references/heuristic-gates.md)
- [Idea 物种识别参考](../references/idea-species.md)
- [概念清晰度评估参考](../references/concept-clarity.md)
- [语气校准参考](../references/tone-calibration.md)
- [Idea Card 模板](../templates/idea-card.md)

## 目标要求

**任务**：把对话从「发散聊想法」变成「可判断的方向校验任务」，并完成三件最关键的前置识别：物种、概念清晰度、目标函数。

**完成标准**：
- 已明确告诉用户本轮不做执行方案、MVP、功能设计、技术路线、增长打法。
- 已获得候选 idea 的基本描述。
- 已对**每个候选**完成物种识别（参见 [idea-species.md](../references/idea-species.md) 的五维度推导）。
- 已对**每个候选**完成核心概念清晰度评估（参见 [concept-clarity.md](../references/concept-clarity.md)）。
- 已确认是否触发「前置概念重构」路由——若触发，本工作流暂停并明确告知用户。
- 已获得或追问用户目标函数：长期主线、已有资产、资源约束、价值排序。
- 若信息不足，明确指出缺哪类信息，不能直接下结论。

## 工作依据

### 第一反应：防止任务滑坡

用户一谈 idea，很容易马上讨论「怎么做」。本步骤要把对话框定在「值不值得做」。

但**比框定更重要**的是：识别这个 idea 的**物种**和**概念清晰度**。
不识别物种就用统一标准评估 = 用产品 KPI 评估论文。
不评估概念清晰度就直接战略压测 = 在沙地上盖楼。

### 最小问题集

**多个 idea**：

```markdown
请每个 idea 只写一行：

1. idea 名称：
   一句话描述：
   它最终的形态是什么（软件 / 方法论 / 内容 / 研究 / 习惯 / 资产 / 其他）：
   为什么会想做它：
   已经有什么积累：

再补 4 个背景：
- 未来 1-3 年最想积累的主线是什么？
- 你现在最强的 3 个技能/资产是什么？
- 每周真实可投入多少小时？
- 你最看重：赚钱 / 能力复利 / 自用价值 / 影响力 / 长期自由度 / 组织能力，怎么排序？
```

**单个心动 idea**：

```markdown
本轮我会把它当成「需要校验的方向」来评估，不是默认应该做。
请先回答：
- 它最终的形态是什么？（软件 / 方法论 / 内容 / 研究 / 习惯 / 资产 / 状态 / 组织能力）
- 它的最终使用者是谁？（自己 / 他人 / 混合）
- 它解决的底层问题是什么？
- 过去 3-6 个月你自己遇到过几次类似问题？
- 现有工具/方法具体在哪一步失效？
- 如果推进它，会挤掉哪个你本来也想做的方向？
```

### 物种识别（关键步骤）

收到 idea 描述后，主控**必须**对每个候选执行五维度推导（参见 [idea-species.md](../references/idea-species.md)）。

物种识别的最低要求：
- 给出主物种判断（结合常见物种速查表）
- 若是混合物种，显式拆分核心物种 + 外壳物种
- 标注哪些「常规问题」对该物种**不适用**

**关键提示**：
- 用户说「我想做一个 X」时，X 可能是软件、也可能是方法论、研究、内容、状态。不要默认归为软件项目。
- 用户说「想做成一个工具」可能只是「外壳想象」，核心物种可能是方法论或研究。
- 当外壳物种与核心物种冲突时，先评估核心，外壳暂缓。

### 概念清晰度评估（关键步骤）

物种识别完成后，主控**必须**对每个候选的核心概念评级（参见 [concept-clarity.md](../references/concept-clarity.md)）。

最低要求：
- 列出 1-3 个决定可行性的核心概念
- 对每个概念评级（L0/L1/L2/L3）
- 按路由规则判断是否触发「前置概念重构」

**触发前置概念重构时的处理**：
- 暂停后续阶段
- 明确告知用户：「在评估这个 idea 是否值得做之前，需要先把概念 X 形式化」
- 给出建议（如调用 `t-concept-refactor` skill）
- 标记「概念形式化后回到本工作流」

**研究物种的特殊处理**：
- 研究类 Idea 允许从 L0/L1 概念起步
- 不应触发前置概念重构，而应调整后续问题集（追问研究路径、证伪标准）
- 用户回答「需要研究」是该物种的**正常起点**，不应被判定为「核心资产缺失」

### 方法论物种的特殊处理

- 方法论物种允许 L1 概念
- 关键判定：用户是否愿意「即使是 A4 纸版本也用」
- 用户给出肯定回答 = 该物种的强信号，不需要前置概念重构

### 处理用户的常见回答

**遇到用户说「都重要」**：
- 不继续让用户泛泛排序。
- 改问：「如果只能牺牲一个，你最不能接受牺牲哪一个：商业回报慢、影响力小、学习复利弱、自用价值弱、组织能力无法沉淀？」

**遇到用户开始问执行细节**：
- 回复：「这是下一阶段问题，我先记录，但当前只判断方向是否值得占用机会成本。」

**遇到用户对反方信号给出「这是执行问题」类回答**：
- 不要直接判定为「逃避战略问题」。
- 先区分：该回答确实属于执行层（接受），还是用执行层包装战略层（追问）。
- 追问方式：「这个『执行问题』的成立依赖哪个前提？这个前提你已经验证过了吗？」

## 产出格式

```markdown
## 边界确认
- 本轮判断范围：
- 明确不做：

## 候选物种识别

### ${idea_name}
- 五维度判断：
  - 交付物形态（Form）：
  - 使用者（User）：
  - 价值实现路径（Value Path）：
  - 成熟度起点（Maturity）：
  - 成功定义（Success Criterion）：
- 主物种判断：${物种 + 简要理由}
- 是否多物种：${是/否，若是说明核心 + 外壳}
- 不适用的常规问题：${列出该物种不适用的标准维度}

## 候选概念清晰度

### ${idea_name}
- 核心概念列表：
  - ${concept_1}：L0/L1/L2/L3，证据：
  - ${concept_2}：L0/L1/L2/L3，证据：
- 路由判定：
  - [ ] 直接进入战略压测（所有概念 ≥ L2，或研究/方法论物种允许低级别）
  - [ ] 调整问题集后进入战略压测（研究物种 + L1 概念）
  - [ ] 触发前置概念重构（产品物种 + L0/L1 概念）
  - [ ] 给出概念未定义预警（所有概念 L0 + 用户无意愿研究）

## 已收集信息
- 候选 idea：
- 用户长期主线：
- 已有资产：
- 资源约束：
- 价值排序：

## 信息缺口
- ${缺口及需要追问的问题}

## 路由决策
- [ ] 进入阶段二（归一化与候选分组）
- [ ] 触发前置概念重构（暂停本工作流）
- [ ] 信息不足，回到本阶段继续追问
```
