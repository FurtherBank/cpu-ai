# 用户问题代码根因分类框架

## 根因集合

### 一、逻辑错误（Logic Error）

代码实现了，但实现是错的。

- **算法/计算错误**：边界条件处理不当、运算符写错、浮点精度问题
- **状态管理错误**：竞争条件（Race Condition）、状态机转换遗漏
- **数据流错误**：空值未处理（NPE）、类型假设错误、数组越界
- **并发错误**：死锁、数据竞争、事务隔离级别不当

> **安全视角**：认证 Token 校验逻辑写错、权限判断条件写反、幂等性未保障，均属此类。

---

### 二、设计缺失或设计错误（Design Flaw / Gap）

逻辑实现是对的，但方案本身就是错的或不完整的。

- **需求理解偏差**：实现了开发理解的需求，而非用户真正的需求
- **架构设计缺陷**：模块耦合过重、无扩展性，导致改一处坏多处
- **接口契约不合理**：API 设计不符合调用方预期，或内部不同模块之间，职责和调用契约混乱，字段语义模糊
- **异常路径设计缺失**：只设计了 Happy Path，未考虑网络中断、超时、重试等场景

> **安全视角**：输入校验从未被设计（XSS/SQL 注入）、权限模型粒度过粗、敏感数据的存储/传输方案未做安全设计，均属此类。

---

### 三、配置与环境错误（Configuration / Infra Error）

代码本身没错，但运行环境/参数不对。

- **运维配置错误**：环境变量、Feature Flag、超时参数配置错误
- **资源不足**：连接池耗尽、内存溢出、磁盘写满
- **部署错误**：灰度比例配错、回滚不彻底、多版本不兼容共存
- **网络/防火墙策略**：端口封闭、DNS 解析错误、CDN 缓存未刷新

---

### 四、第三方依赖问题（External Dependency）

自身代码无错，但依赖方出了问题。

- **下游服务故障**：支付、短信、地图等 API 不可用或返回异常
- **SDK/库版本问题**：升级后行为变更、已知 Bug 引入
- **平台规则变化**：微信/iOS/Android 审核规则调整，导致功能被限制
- **数据源污染**：上游数据质量问题传导到自身产品

> **框架可递归性**：对依赖方自身而言，其故障同样可以用本框架归因——是它的逻辑错误、设计缺失、配置错误，还是它的上游依赖出了问题。故障链条可以沿依赖关系向上递归追溯，直到找到最原始的根因节点。

---

### 五、性能问题（Performance Degradation）

功能逻辑是对的，但效率不可接受。

- **算法复杂度过高**：O(n²) 的查询、循环内调数据库
- **缺少缓存**：热点数据每次重新计算/请求
- **数据库问题**：慢查询、索引缺失、锁竞争
- **资源泄漏**：内存泄漏、连接未释放，导致随时间劣化

> 性能问题通常是逻辑错误或设计缺失的子集，单独列出是因为其定位手段（Profiling、压测）和修复节奏（专项优化）与常规 Bug 有所不同。

---

### 六、兼容性问题（Compatibility Issue）

在特定环境下行为异常，其他环境正常。

- **客户端兼容**：低版本 iOS/Android、旧版浏览器的 API 缺失
- **分辨率/屏幕适配**：折叠屏、超小屏、横屏场景未测试
- **编解码/格式兼容**：文件格式、字符集、时区处理差异
- **多语言/本地化**：文本溢出、日期格式、RTL 布局未处理

> 兼容性问题同样可归入逻辑错误或设计缺失，单独列出是因为它的触发条件高度依赖环境，排查路径不同。

---

### 七、体验设计问题（User Experience Flaw）

功能逻辑正确，但使用过程不顺畅或不直观，用户体感差。

- **反馈缺失**：操作无响应、缺少 loading / skeleton 过渡态；长时操作无进度指示；状态变更后页面无感知提示
- **认知负担过重**：一屏信息量过大、主次不分；术语晦涩、非目标用户无法理解；完成任务需记忆过多上下文
- **流程路径摩擦**：达成目标需要多余步骤；操作路径违反用户心智模型或行业惯例；关键入口埋得过深
- **信息架构混乱**：导航层级不清，用户不知身处何处；功能入口分散难以发现；分类逻辑与用户认知不一致
- **视觉感知缺陷**：对比度不足影响可读性；重要信息不突出、视觉噪声干扰焦点；字体/间距/布局导致扫描困难
- **文案沟通失效**：错误提示无法指导下一步操作；按钮/标签文字模糊有歧义；空状态缺少引导
- **容错与可恢复性差**：破坏性操作无二次确认且不可撤销；表单报错后丢失已填内容；操作失败后恢复路径不清晰

> 体验设计问题本质上是设计缺失的子集，单独列出是因为其根因定位手段（可用性测试、用户访谈、启发式评估）与修复节奏（设计迭代、灰度验证）和代码层面的设计缺失有明显差异，且在用户产品前端以及作为三方依赖开发者的场景下高度集中。

---

## 结构总览

```
用户反馈问题
├── 逻辑错误         → 修代码；安全漏洞中的"写错了"属此类
├── 设计缺失/错误    → 重新设计；安全漏洞中的"从没考虑"属此类
├── 配置/环境错误    → 运维介入，通常可快速修复
├── 第三方依赖       → 降级/切换 + 向上递归归因
│     └── (递归) 依赖方自身的根因 → 同样适用本框架
├── 性能问题         → 逻辑/设计的子集，专项手段处理
├── 兼容性问题       → 逻辑/设计的子集，环境触发特征明显
└── 体验设计问题     → 设计缺失的子集，可用性测试定位、设计迭代修复
```

## 关键规律

**安全漏洞不是独立根因**，而是逻辑错误与设计缺失在安全维度的后果表现；将其单独列为根因会破坏分类的正交性，正确做法是在各根因下标注安全视角。

**修复成本随根因类型差异显著**：配置错误可分钟级回滚；逻辑错误需定位修复测试；设计缺失可能涉及周级重构；第三方依赖故障取决于对方，自身只能做降级兜底。这也是"越早发现越便宜"这一规律的根本来源。
