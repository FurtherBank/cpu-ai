# 步骤一：信息锚定 执行指导

## 输入信息

**必须入参**：
- `dimaLocalState`：含 `workItemId`、`status`
- `remoteDetail.description`：bug 描述文本
- `comments`：与问题相关的讨论
- `需求目录路径`：调用者通过提示词传入的当前需求所在目录（如 recordings-plan 对应的目录）

**上下文参考**：
- 项目 AGENTS.md（若存在，读取以了解项目规范）

## 目标要求

**任务**：把「模糊的 bug 描述」精确量化为「可操作的现象描述」
**目标**：创建 `{需求目录路径}/investigation.md` 文件，包含所有必要字段（见产出格式）

**特性要求**：
- 现象描述必须包含触发条件、环境、频率三要素，缺一不可
- `reproducible` 字段必须明确为 `true` 或 `false`（不允许 `unknown`，若真的不知道，默认为 `false`）
- 信息盲区至少 1 条，格式：「不知道 X，需要 Y 来确认」
- 本步骤禁止进行代码搜索或修改

## 工作依据

**三问锚定法**（必须在脑内对工单描述逐一回答）：

1. **表现是什么？** HTTP 5xx？数据返回错误？页面无响应？状态没更新？
   - 好的答案："接口返回 500，stacktrace 可见"
   - 差的答案："功能不工作了"

2. **是否必现？** 100% 每次都复现，还是偶尔出现（有频率估计更好）？
   - 这决定后续路径：必现 → 直接本地复现；偶现 → 需要特殊方法论

3. **何时开始？** 上线后立刻就有，还是运行一段时间后出现？
   - 上线即失败 → 配置/代码问题；运行后劣化 → 资源泄漏/数据积累类

**遇到描述极其模糊时**：不要猜，把"不知道"如实写进信息盲区，然后继续推进——后续步骤的搜索和证据采集会补充这些信息。不要因为信息不完整就卡在步骤一。

## 产出格式

在 `{需求目录路径}/investigation.md` 写入：

```markdown
# Bug Investigation: {工单 ID / 简短标题}

## 现象描述
- **触发条件**：{具体的触发步骤或条件}
- **环境**：{生产/预发/特定浏览器/特定数据量等}
- **频率**：{必现 / 偶现（约 X%）/ 特定条件下必现}
- **开始时间**：{大致何时开始，如"上线后立刻" / "T+2h 后出现"}

## 可重现性
- `reproducible`: {true / false}

## 已知线索
- {从 stacktrace / 日志 / 描述中直接可见的任何信息}

## 已排除项
- {（此时通常为空）}

## 信息盲区
- 不知道 {X}，需要 {Y} 来确认
- {可以有多条}

## 工单关联
- workItemId: {xxx}
```
