# 步骤三：根因归类与范围定界 执行指导

## 输入信息

**必须入参**：
- `investigation.md`（含第一现场位置）

**上下文参考**：
- [../references/bug-identify.md](../references/bug-identify.md)（根因七分类，本步骤必须对照）

## 目标要求

**任务**：选定根因类型标签；执行全局搜索建立影响范围清单
**目标**：根因类型有根据可查，影响范围清单基于真实代码搜索结果

## 工作依据

### 根因分类决策树（快速入口）

```
第一现场是什么类型的错误？
│
├── 有 stacktrace，指向空值/类型错误
│   → 根因类型：逻辑错误 > 数据流错误（空值未处理）
│
├── 有 stacktrace，指向 DB/SQL 错误
│   → 根因类型：逻辑错误 > 并发错误 or 设计缺失 > 接口契约
│
├── 本地正常，生产/预发不对
│   → 根因类型：配置/环境错误（先排查这个，最快）
│
├── 特定浏览器/操作系统出错，其他正常
│   → 根因类型：兼容性问题
│
├── 数据量少时正常，数据量大时慢/超时
│   → 根因类型：性能问题（N+1 / 索引缺失 / 内存排序）
│
├── 功能逻辑正确，但在边界条件下有隐患
│   → 根因类型：设计缺失 > 异常路径设计缺失
│
└── 依赖的外部服务/SDK 返回异常
    → 根因类型：第三方依赖问题（需确认是否可降级）
```

### 全局搜索（每次都必须执行）

**搜索的目标**：找到所有「可能受同一根因影响」的代码位置，建立修复范围的完整清单。

**常见搜索模式**：

```bash
# 搜索调用点（谁调用了出问题的函数）
rg "functionName|methodName" src/ --type ts -l

# 搜索同一危险模式（如所有未做 null 守卫的同类调用）
rg "getActiveDiscount" src/ --type ts -n

# 搜索所有「创建」操作的入口（幂等性修复场景）
rg "createOrder|INSERT.*orders" src/ --type ts -l

# 搜索特定 API 的使用（兼容性场景）
rg "structuredClone|\.at\(|Object\.hasOwn" src/ --type ts -n
```

**不能遗漏的入口类型**：
- HTTP Controller
- **定时任务（jobs/cron）** ← 最容易遗漏的入口
- **Webhook/回调处理** ← 第二容易遗漏的
- 消息队列消费者
- 工具脚本/数据迁移脚本

**"我记得只有一个入口"是最危险的信念**。搜索结果要比记忆可靠。

### 搜索无结果时

若 `rg` 确实没有找到其他入口，**必须显式记录**："已搜索 `{关键词}`，在 `src/` 下仅有 1 处引用，位于 `{文件路径}`"。这保证了"没有其他入口"是结论，而不是遗漏。

## 产出格式

在 `investigation.md` 中追加：

```markdown
## 根因归类

- **类型**：{逻辑错误 > 数据流错误（空值未处理）} [对照 bug-identify.md 的具体分类名]
- **依据**：{为什么选这个类型，1-2 句话}

## 影响范围清单

通过搜索命令 `rg "{keyword}" src/ -l` 找到以下相关位置：

- `src/services/order.service.ts:80` — 直接报错位置
- `src/services/cart.service.ts:45` — 同样调用了 getActiveDiscount，未做 null 守卫
- `src/jobs/daily-report.job.ts:120` — 调用了同一方法（定时任务入口）
- 已搜索 "{keyword2}"，在 src/ 下无其他引用

**修复必须覆盖上述全部位置。**
```
