# 阶段四：收尾报告 执行指导

## 核心动作

1. 确认 `tasks.md` 所有任务均已勾选，无遗漏（如有 CRITICAL 介入请求，停止流程并 yield）
2. 在 `{workspaceRoot}/.iteration/recordings-execute/[<reqId>] <summary>.md` 写出**最小化收尾报告**（格式见下；**只允许两节内容，禁止追加任何其他章节**）
3. 更新 Spec 目录状态（若目录名含 emoji 前缀）：

   ```bash
   mv ".../⏳ {spec-name}" ".../✅ {spec-name}"
   ```

   若目录名无 emoji 前缀，跳过此步骤

---

## 收尾报告格式

```markdown
# 执行报告 - <change-plan summary>

## 提交清单

| 仓库目录   | Commit Hash |
| ---------- | ----------- |
| {repoDir1} | {hash1}     |
| {repoDir2} | {hash2}     |
| ...        | ...         |

## 代码风险信号

> 来自编码过程中顺带识别（feature 风险自审环节产物）。按发现位置归组，**不**按 Cx 标注。

### 1. [风险类型] - [文件路径:行号]

- **问题描述**：{描述}
- **影响评估**：{影响范围}
- **优先级**：{高/中/低}

### 2. ...

> 若执行循环中未识别到任何风险信号，本节填写：「本次执行未识别到代码风险信号」。
```

---

## 强约束（写报告前自检）

- [ ] 报告**只包含**「提交清单」与「代码风险信号」两节
- [ ] **不出现**：测试结论 / 已知待处理项 / 功能描述列 / Cx 映射列 / 验证结果等任何额外字段
- [ ] 提交清单仅有「仓库目录、Commit Hash」二列，不附 Cx ID、不附描述

## 完成判定

- [ ] tasks.md 全 [x]
- [ ] 收尾报告已写入约定路径，符合上述强约束
- [ ] Spec 目录状态已更新（或确认无 emoji 前缀而跳过）
- [ ] 最终响应末尾输出 `[stage-yield] stage=EXECUTION artifact=<路径>`
