# 阶段一：规范对齐与规划生成 执行指导

> 本阶段在**单轮推理**中一次性产出最终的 `change-plan.md`，不切分中间产物。按「读 → 想 → 写 → 自检」四步组织，读阶段建立心智、想阶段做决策、写阶段直接产出文件、自检阶段守住质量门禁。

## 输入

- **规范文档**：`design.md`（RFC 路径）或需求工单 JSON（feature 直达路径）；**必须完整阅读**，不可只读摘要。
- **改造目标**：从规范文档或需求文件中提取的目标语句**原文**（用于直接引用，不改写）。
- **工作区仓库范围**：来自 `design.md` 的「工作区仓库范围」专节；feature 直达路径若无该节，须根据需求自行确定主仓 + 协同仓 + 只读参考仓，并在 `change-plan.md` 中明确标注「依据」。
- **目标仓 AGENTS.md**：用于了解架构约束（路由组织、entity/migration 约定、commit 规范等）。

---

## 第一步：读懂规范（条款原子化）

把规范文档每条语句原子化为「条款」（建议 ID：`S1`、`S2`…，或按章节 `A1`、`A2`、`B1`…）。每条标注：

- **类型**：领域概念 / 存储约束 / 行为规则 / 集成要求
- **优先级**：MUST / SHOULD / MAY（措辞模糊默认 MUST，并注「待确认」）

**关键陷阱**：

- **集合运算条款**（如「权限集 = A ∩ B ∩ C」）必须**逐元素拆**为独立条款，集合运算本身也是一条；这是最容易被静默漏掉的条款类型。
- **实现设计类条款**（如「触发某 agent 自动生成 X」「调用大模型完成 Y」）：在备注中标注「需在第二/第三步明确：prompt / 触发条件 / 输入输出格式」，避免转嫁给 EXECUTION。

> 该条款索引表**推荐保留**到 `change-plan.md` 的「规范条款索引」节，作为可追溯锚点。

---

## 第二步：检索系统（现状摘要）

围绕条款索引**定向**检索系统现状（不要全库扫描），覆盖以下维度：

- **数据模型**：以 migration 为结构真值（`**/migrations/**`、`**/entity/**`），列出与规范概念相关的表/字段及来源文件；migration 与 entity 冲突时以 migration 为准。
- **接口路由**：从主程序入口（`main.ts` / `bootstrap` / 控制器装饰器）出发，沿 import 图列出与规范相关的接口（method + path + handler）。
- **鉴权矩阵**：对每个相关接口确认鉴权方式（有 / 无 / 继承 / 未知）。
- **实现范式**（条件）：当条款要求「新建 agent / service / processor / workflow 节点」时，**必须**找到目标目录现有实现作为范式参照（基类、必须实现的方法、调用方式、workflow 插入点），以便为变更条目提供精确引用而非「参考现有实现」。
- **术语映射**（条件）：当系统中存在与规范概念**同名但语义不同**的词（如 `tenant_id` vs 规范「开放平台租户」），须建立术语映射表，避免落点错配。

**辅助停机信号**：连续多个文件不再发现新的相关注册模式或新表名时即停止；不必全库枚举。

---

## 第三步：决策落点（概念 → 落点 + 技术决策）

对每个领域概念，确定其在系统中的具体落点（表 / 字段 / 接口 / 服务），存在三种状态：

- **有对应物，语义一致**：直接复用，记录路径。
- **有对应物，语义不符**（术语映射告警）：触发「**冲突重写**」——必须新建字段/接口，**不**复用旧字段。
- **无对应物**：标记为「新增」并完成存在性技术决策（如：新表 vs 现有表加字段、DB vs Redis vs JWT、是否引入新服务边界）。

**决策边界（口诀）**：

- **存在性决策**（要不要这张表/接口/配置）→ 必须本阶段决定。
- **实现细节**（索引大小、TTL、变量命名风格）→ 留给 EXECUTION，无需在 plan 中固化。

> **判断口诀**：「如果选方案 A vs B 会导致 change-plan 中表名/字段名/接口名不同 → 必须本阶段决定」。

**实现设计类条款的硬约束**：对涉及「agent / 触发器 / 大模型调用」的条款，在写入变更清单前**必须**先决定：

- agent 类：完整的 system prompt（实际文本，不是「应该包含 X」）、输入数据来源、输出格式、错误处理策略
- 触发条件类：完整的布尔表达式（具体到变量路径，如 `context.chatInfo?.app_setting?.enableOpenAPI === true`）

如确实复杂到无法在本阶段决定，应回到 i-spec-design 补设计；不允许把 agent prompt 内容、触发条件留白给 EXECUTION。

---

**四态差距类型**：

- **新增**：当前完全没有 → 创建表/字段/接口/服务。
- **修改**：当前有但缺字段或行为不符 → 在原物上扩展。
- **保留+扩展**：当前语义一致，仅需新增关联 → 不改原物，外加新增。
- **冲突重写**：当前语义冲突 → 必须新建并保留旧物（旧物等迁移完成后再下线）。

**置信度规则**：

- **高**：双重证据（如 migration + handler 引用）无矛盾。
- **中**：单一证据，或存在继承 / 全局 filter 未完全展开。
- **低**：动态注册 / 无 migration / 纯推断 — 必须附**可执行的验证命令**（如 `rg "tenant_id" --type ts app/...`），不能写「需要进一步确认」。

**分批原则**：

- 新增字段默认 NULLABLE；新路由默认特性开关关闭；旧 token 与新 token 并存不互斥。
- 存量数据迁移批次（写新格式）**必须在**「读逻辑已支持新格式」批次**之后**。
- 删除旧字段须独立成批（不可逆操作）。

## 第五步：自检（写完 plan 后必跑）

逐项确认，全部通过方可结束 PLANNING。

### 充分性

- [ ] 每条 MUST 类条款在覆盖度矩阵中有 ✅ 或显式「不适用：原因」。
- [ ] 反向：每条 Cx 都有对应规范条款 ID（无规范来源的「顺便发现」条目须移入「可选/附加变更」或删除）。

### 遗漏源专项扫描（最常被静默遗漏的四类依赖）

- [ ] **缓存层**：搜索 Redis / cache key 拼接是否含受影响字段；如有，补充配置层或服务层条目。
- [ ] **消息队列**：搜索 emit / publish / consumer 中是否传递受影响字段；如有，补充服务层条目。
- [ ] **内部服务调用**：搜索 chromeFetch / axios / fetch 等内部 RPC 是否涉及受影响接口/字段；如有，补充 API 层或服务层条目。
- [ ] **前端 / SDK 依赖**：若前端同仓或对外 SDK 使用了受影响字段/接口，须在 plan 中显式说明契约变更。

### 粒度收敛（杜绝把决策转嫁给 EXECUTION）

- [ ] 描述中**无**「参考现有实现」「类似 X」「相关逻辑」「鉴权模块」等模糊词；必须落到表名 / 字段名 / 接口路径 / 文件路径级别。
- [ ] 描述中**无**「待与产品确认」「根据实际情况调整」「暂定 X」类悬空决策；如有意授权 EXECUTION 决策，必须改写为「粒度留白（已授权）」格式（见末尾）。
- [ ] 描述中调用的函数/类必须**在本清单中有定义条目**，或**在现有代码库中已存在**。
- [ ] 「冲突重写」类条目附「调用方影响评估」表（调用路径 + 影响类型 + 兼容策略）。
- [ ] **粒度独立检查**：对每条 Cx（包含高置信度条目）问「如果两个不沟通的工程师各自实现这条变更，他们的实现会有哪些实质差异？」若答案非空，粒度仍不足，须修复后重跑本检查。

### 分批可上线

- [ ] 每批结束后系统处于完整可用状态（旧功能不降级）。
- [ ] 存量数据迁移批次在「读逻辑已支持新格式」批次之后。
- [ ] 每批有明确的回滚方案（部署回滚 / 特性开关 / 数据回滚脚本三选一）。

### 自测用例

- [ ] 每条 Cx 至少 1 条主流程 + 1 条异常/边界用例；用例可被 i-verify 直接传入测试 skill。

### 工作区仓库范围

- [ ] 「工作区仓库范围（基础声明）」与 `design.md` 专节（或 feature 路径下已确认声明）一致；扩仓 / 缩仓已在表或备注中说明理由。
