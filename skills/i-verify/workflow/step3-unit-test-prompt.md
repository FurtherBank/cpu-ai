# Step 3 - 单测补全 Subagent 派发 Prompt

使用以下 Prompt 模板派发给 Subagent（将 `{...}` 替换为实际值后发送）：

---

```markdown
请为本次迭代的代码变更范围补充单元测试。

## 仓库信息

- 仓库路径：{仓库绝对路径}

## 单测规范

以下是仓库 AGENTS.md 中的 📐 仓库单测指引（完整内容）：

{📐单测指引全文}

## 本次变更范围

以下是本次迭代涉及的变更文件列表（来自 `git diff <基线>...HEAD --name-only`）：

```
{diff 文件列表，每行一个文件路径}
```

## 执行步骤

1. 阅读上方📐单测指引，明确：测试框架、运行命令、文件命名规范、覆盖率要求
2. 基于变更文件列表，识别需要补充单测的模块（**仅覆盖上方列表中的文件，不扩散到无关模块**）
3. 为每个模块编写单元测试，覆盖：
   - 正常路径（happy path）
   - 关键边界条件（空值、边界值、异常输入）
   - 如有外部依赖（数据库、HTTP、时间），使用 mock/stub
4. 在仓库目录下运行完整的单测命令，确认**全部通过**
5. 提交单测代码（commit message 格式按仓库规范，包含 `test:` 前缀）

## 执行要求

你是本次迭代单元测试的唯一责任方。

- 每个测试用例必须实际执行（运行测试命令并查看输出），不允许"推断"通过
- 不允许写空断言（如 `expect(true).toBe(true)`）或永真断言，这被视为未覆盖
- 遇到业务代码有明显 bug 时（如空指针、边界未处理），直接修复并报告，而非仅写"此处不可测"
- 覆盖率不满足仓库指引要求时，继续补充直到达标

记住：走过场的单测等于没有单测。单测阶段漏掉的 bug 会在预发或生产爆发，那时修复成本是现在的 10 倍。你就是这道防线。

## 返回格式

完成后严格按以下格式返回：

```
## 单测执行报告

### 覆盖模块
- 模块 1：{文件路径}，测试文件：{测试文件路径}，用例数：N
- 模块 2：...

### 测试运行结果
- 测试命令：{实际运行的命令}
- 通过：N / 失败：0 / 跳过：N（跳过原因说明）
- 覆盖率：{按仓库指引的覆盖率指标}

### 完整命令输出
（贴出测试命令的完整终端输出，不要省略）

### 提交信息
- commit hash：{hash}
- commit message：{message}

### 额外发现（如有）
{测试过程中发现的业务代码问题及修复说明}
