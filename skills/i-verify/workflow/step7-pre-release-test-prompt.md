# Step 7 - 预发验证 Subagent 派发 Prompt

使用以下 Prompt 模板派发给 Subagent（将 `{...}` 替换为实际值后发送）：

---

```markdown
请在预发环境中完成本次迭代的验证测试。

## 仓库预发部署 & 自测指引（自测部分）

{🏗️预发部署&自测指引全文，尤其是预发自测流程相关段落}

## 本次迭代自测需求

以下是变更计划表中的自测需求列表：

{自测需求列表，编号+描述，每行一条}

## 预发环境

{预发环境访问方式，如 URL、账号信息、访问前提等，从仓库预发部署 & 自测指引获取}

## 执行要求

你是本次发布的最后一道质量关卡。你的测试结论直接决定是否进入生产发布。

**强制要求**：
- 每个测试用例必须在预发环境**实际执行**，不允许基于代码阅读「推断」结果
- 发现问题时，必须精确记录：复现步骤、预期结果、实际结果（附命令输出或截图描述）
- 若所有用例均通过，请额外检查一个你认为最容易出问题的边界场景
- 跳过的测试用例必须说明跳过原因，「无法测试」不是理由

**你的测试报告会被人工审查，敷衍的报告会被打回重做。**

记住：漏掉一个 bug 到预发 = 给生产事故留了入口。你是防线，不是橡皮图章。

## 返回格式

无论结论如何，必须按以下格式返回完整报告：

```
# 预发验证报告

## 测试环境
- 执行时间：{timestamp}
- 预发环境：{URL 或环境标识}
- 测试结论：✅ 全部通过 / ❌ 发现 N 个问题

## 结果汇总
- 总用例数：N
- 通过：N
- 失败：N
- 跳过：N（说明跳过原因）

## 详细结果

### ✅/❌ 用例 1：{用例描述}
- 执行操作：{具体操作步骤}
- 预期结果：{expected}
- 实际结果：{actual}
- 证据：{命令输出/截图描述/请求响应}

（每个用例均需按此格式填写）

## 额外边界场景检查（所有用例通过时必填）
- 场景：{你认为最容易出问题的边界场景描述}
- 测试结果：✅ 通过 / ❌ 发现问题

## 发现的问题（如有）

### 问题 1：{问题标题}
- 严重程度：高 / 中 / 低
- 出现位置：{功能模块/页面/接口}
- 复现步骤：
  1. {步骤 1}
  2. {步骤 2}
- 预期结果：{expected}
- 实际结果：{actual}
- 证据：{具体输出}
- 初步判断（可选）：{后端/前端/配置/环境问题}
