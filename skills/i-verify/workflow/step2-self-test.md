# Step 2 - 自测验收执行指导

> **路径根**：自测报告等写入 **工作区根目录** `{workspaceRoot}/.iteration/`，与 `i-verify/SKILL.md` 约定一致。

## 输入信息

**必须入参**：
- `🧪 仓库本地自测指引`：从 AGENTS.md 提取的完整章节内容（硬依赖）
- `自测需求列表`：从变更计划表提取的所有自测需求条目（硬依赖）

## 目标要求

**任务**：逐条执行自测需求，发现失败则通过 `i-bug` 修复，循环至全部通过

**目标**：所有自测需求状态=PASS，自测报告存在，无遗留失败项

**特性要求**：
- 自测报告必须记录每条需求的执行结果和关键证据（不允许只写「通过」）
- `i-bug` 调用必须传入标准5字段格式，不允许仅传「有个测试失败」
- Bug 修复 commit 必须与功能 commit 分开（message 格式：`fix: [i-verify] <描述>`）
- 同一测试项累计失败 ≥3 次：门控阻断，升级人工
- 每次 i-bug 修复后：对受影响测试项重验 + 对已PASS项快速回归扫描

## 工作依据

**执行流程**：

```
初始化自测报告文件：{workspaceRoot}/.iteration/recordings-test/self-test-{YYYYMMDD-HHmmss}.md

FOR EACH 自测需求 req IN 列表:
  fail_count[req] = 0

  LOOP:
    按 🧪 指引执行 req
    观察执行结果

    IF 通过:
      标记 req.status = PASS
      在报告中记录：执行操作、预期结果、实际结果、关键证据
      BREAK

    ELSE 失败:
      fail_count[req] += 1

      IF fail_count[req] >= 3:
        输出阻断提示（见下方格式）
        进入 BLOCKED 状态
        等待人工介入

      在报告中记录失败现象

      构造 i-bug 调用（必须包含5字段）：
        症状描述: <具体报错/断言失败信息>
        期望行为: <req 的预期结果>
        受影响测试项: [req.id]
        相关文件: <从报错堆栈推断>
        已尝试方案: <无 or 上次修复尝试>

      调用 i-bug skill

      i-bug 修复完成后：
        对本次 req 重新执行（LOOP 继续）
        对所有已 PASS 的 req 进行快速回归扫描
          如有回归问题：当作新失败项处理

最终写入报告文件
```

**i-bug 调用上下文格式（不可简化）**：

```
症状描述：{执行步骤 N 时，<具体工具输出/错误信息/断言失败详情>}
期望行为：{自测需求原文中的预期结果}
受影响测试项：[{需求 ID 或描述}]
相关文件：{从堆栈信息中推断的文件列表，若不确定写"未知"}
已尝试方案：{无 / 上次修复内容简述（若有）}
```

**阻断提示格式**：

```
⛔ 自测阻断 — 需要人工介入

自测需求「{需求描述}」已连续失败 3 次，无法自动继续。

失败历史：
1. 第 1 次：{失败现象}，i-bug 修复：{修复内容}
2. 第 2 次：{失败现象}，i-bug 修复：{修复内容}
3. 第 3 次：{失败现象}

请人工分析后，告知处理方向（修复方案 / 跳过此项 / 调整测试预期）。
```

**方法论**：
- 先通读所有自测需求，评估哪些需求之间存在依赖（如 #3 依赖 #1 的数据），优先执行前置需求
- 快速回归扫描不等于完整重测：对已PASS项只验证其核心断言，不重走全部步骤
- 如多个 bug 共享相同的根因（如同一个接口返回字段错误），可合并为一次 i-bug 调用，但需在 `受影响测试项` 中列出全部
