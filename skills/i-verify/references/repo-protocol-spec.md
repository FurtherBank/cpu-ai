# 仓库协议规范：AGENTS.md 必须提供的指引

## 背景

`i-verify` 技能的所有关键步骤均依赖仓库在 `AGENTS.md` 中提供的指引。这是「机制优先于意志」原则的体现：AI 不猜测仓库的发布方式，而是严格按仓库提供的规程执行。

**指引完备性扫描（Step 0）仅产出警告并落盘，不因指引缺失而单独终止整次 i-verify。进入依赖某指引的阶段且确认该指引仍缺失时，按 `SKILL.md` 约定输出 `缺少{规范标题}指引，需要人工介入操作`（`{规范标题}` 见该文档表格），**当次 verify 停止**；补全后再次运行并从状态文件断点继续。**

---

## 仓库视角写作原则

仓库 `AGENTS.md` 中的流程指引应以**仓库维护者和实际操作者**为读者，描述本仓库的基线同步、自测、单测、非代码资产变更、预发部署、自测与生产发布应如何完成。它们是仓库自身的工程规程，不是为 `i-verify` 或某个迭代框架定制的适配层。

因此，编写或补全这些章节时：

- 只写仓库内真实可执行的操作、命令、判定标准、失败处理和人工确认点。
- 不在仓库文档中声明「与 i-verify 对齐」「供 i-verify 使用 / 解析」「某阶段应如何记录状态」「`.iteration/verify-state.json` 如何更新」等框架关联内容。
- 若某项要求同时会被人、其它自动化工具和 `i-verify` 使用，也应按通用仓库流程表达，例如写「预发未通过不得升生产」，不要写「i-verify 阶段七未完成不得进入阶段八」。
- 标题可沿用本规范要求的六类指引名称，以便自动化发现；正文语义仍保持仓库流程中立。

## 六类必须指引

### 🔀 仓库迭代合并指引

**对应步骤**：阶段一（基线同步）、阶段四（PR 合并）

**必须包含**：

- 迭代分支命名规范（如 `feature/*`、`fix/*`）
- 基线分支名（如 `main`、`develop`）
- 同步方式（rebase / merge）
- 推送命令（如 `gmr` 或 `git push origin HEAD`）
- PR 创建方式（通过 `gmr` 自动创建 or 手动创建 + PR 模板路径）
- PR 合并方式（squash / merge commit / rebase）
- CI 系统及检查项列表
- 代码评审规则（最少几个 approve，评审超时处理方式）

**示例结构**：

```markdown
## 🔀 仓库迭代合并指引

**迭代分支规范**：`feature/{工单ID}-{简短描述}` 从 `main` 创建
**基线同步**：`git fetch origin && git rebase origin/main`
**推送合并**：`gmr`（创建 PR 并推送），PR 标题格式：`feat: {简短描述} [工单ID]`
**合并方式**：Squash merge
**CI 系统**：GitHub Actions，检查项：lint / typecheck / unit-test / build
**评审要求**：至少 1 个 approve，owner review 可选
```

---

### 🧪 仓库本地自测指引

**对应步骤**：阶段二（自测验收）

**必须包含**：

- 本地启动服务的命令（若需要）
- 各类型自测的执行模式（API 测试、页面功能测试、脚本验证等）
- 自测结果判定标准

**示例结构**：

```markdown
## 🧪 仓库本地自测指引

服务启动：`docker-compose up -d && npm run dev`
API 测试：使用 `./scripts/test-api.sh` 或手动调用接口
判定标准：所有变更计划表中的接口返回正确 status code 和 response body
```

---

### 📐 仓库单测指引

**对应步骤**：阶段三（单元测试）

**必须包含**：

- 测试框架（如 Jest、Vitest、pytest、Go testing）
- 测试文件命名规范和存放位置
- 运行单测的命令
- 覆盖率要求（行覆盖率 / 分支覆盖率等）
- Mock 规范（如何处理外部依赖）
- 单测 commit 规范（message 格式等）

**示例结构**：

```markdown
## 📐 仓库单测指引

**测试框架**：Jest + TypeScript
**文件规范**：`src/**/*.test.ts`，测试文件与源文件同目录
**运行命令**：`npm run test:unit`（运行所有单测）；`npm run test:coverage`（含覆盖率报告）
**覆盖率要求**：行覆盖率 ≥ 80%，新增代码 ≥ 90%
**Mock 规范**：外部 HTTP 使用 `msw`，数据库使用 in-memory SQLite
**Commit 格式**：`test(scope): add unit tests for {module}`
```

### 📦 其它类型研发资产变更指引

**对应步骤**：阶段五（非代码资产变更）

**必须包含**：

- 各类非代码资产的变更执行模式（DB 迁移、配置变更、基础设施变更等）
- 每类变更的**执行时机**（如：「DB 迁移在预发部署后、预发测试前执行」）
- 不可逆操作的特别说明（如生产 DB 迁移需要人工确认）
- 各类变更的验证方式

> **时机标注约定**：
>
> - `合并后`：PR 合并到主干后立即执行
> - `预发部署前`：在预发部署命令之前执行
> - `预发部署后`：在预发部署完成、测试开始前执行
> - `生产发布前`：在生产发布命令之前执行
> - `生产发布后`：在生产发布完成后执行

**示例结构**：

```markdown
## 📦 其它类型研发资产变更指引

### 数据库迁移

工具：Flyway / Liquibase / 自定义 migrate 脚本
时机：预发部署后（预发测试前）；生产发布后
执行命令：`npm run db:migrate -- --env={staging|production}`
验证方式：检查 `schema_migrations` 表最新版本号
生产迁移：需人工在 terminal 输入 `CONFIRM MIGRATE PRODUCTION` 后执行

### 配置变更

时机：预发部署前；生产发布前
执行模式：通过配置中心 UI 或 `config-cli set <key> <value>`
验证方式：通过 `config-cli get <key>` 确认值已生效
```

### 🏗️ 仓库预发部署 & 自测指引

**对应步骤**：阶段六（预发部署）、阶段七（预发验证）

**必须包含**：

**部署部分**：

- 预发环境说明（是否独立、与生产的关系）
- 触发预发部署的方式（命令 / CI pipeline / 手动操作）
- 判断预发部署成功的方式（健康检查接口、日志关键词等）
- 预发部署失败时的处理方式（回滚命令）

**预发自测部分**（必须有独立段落）：

- 预发环境地址或访问入口
- 预发环境账号/权限获取方式
- 预发测试执行模式（可描述与本地测试的差异，也可完全独立定义流程）
- 预发测试通过的判定标准

> **注意**：预发自测流程由各仓库自行定义，不默认与本地自测相同；若某项目两者流程确实一致，在此指引的预发自测部分中显式说明即可。

**示例结构**：

```markdown
## 🏗️ 仓库预发部署 & 自测指引

### 预发部署

**部署命令**：`./scripts/deploy.sh staging`（或 CI Pipeline `deploy-staging` stage 自动触发）
**部署超时**：等待最多 10 分钟
**成功验证**：`curl https://staging.example.com/health` 返回 `{"status":"ok","version":"<commit-hash>"}`
**失败回滚**：`./scripts/rollback.sh staging`，等待旧版本恢复
**注意**：预发与生产共用 Redis，但数据库隔离。迁移前确认不会影响生产 Redis 数据

### 预发自测

预发地址：https://staging.example.com
账号：从 1Password vault "Staging" 中获取测试账号
执行模式：直接访问预发地址，按本地自测相同步骤执行（流程与本地一致，环境不同）
判定标准：与本地测试相同，但需额外验证与其他预发服务的集成正常
```

---

### 🚀 仓库生产发布指引

**对应步骤**：阶段八（生产发布）

**必须包含**：

- 生产发布触发方式
- 灰度发布支持（若支持：配置方式、灰度比例含义、全量时机）
- Feature Flag 支持（若支持：创建/配置/清理方式）
- 发布成功验证方式
- 生产监控地址及关注指标
- 回滚方式（命令 + 回滚后验证）
- 发布后必须完成的操作（如通知、工单关闭）

**示例结构**：

```markdown
## 🚀 仓库生产发布指引

**发布命令**：`./scripts/deploy.sh production`
**灰度发布**：通过 Feature Flag 系统（LaunchDarkly）

- 新建 Flag：`launchdarkly-cli flag create <name> --env production`
- 设置比例：`launchdarkly-cli flag percentage <name> <0-100>`
  **观察期**：灰度后观察 30 分钟，关注：错误率（<0.1%）、P99 响应时间（<200ms）
  **监控面板**：https://grafana.example.com/d/production
  **回滚命令**：`./scripts/rollback.sh production`
  **发布后操作**：在钉钉发布群通知发布完成，关闭对应工单
```

---

## 给仓库维护者的建议

- 指引要求「机器可执行」：命令要精确（含参数），不要写「大概就是执行下 deploy 命令」
- 区分预发和生产的差异（尤其是 DB 迁移、配置、Feature Flag）
- 明确不可逆操作（DB 迁移、生产配置变更等）的确认机制
- 更新指引后，可运行本 Skill 的前置门控验证指引完备性
