# 注意力预算约束

## 绑定约束

自本 skill 加载起至任务结束，把注意力视为需要按需分配的任务资源。凡是会改变后续走向的内容确认点，都必须获得与其重要性匹配的专属注意力。不要在一个流畅的主线心流里顺手确认高承重内容单元。

## 核心定义和判断

**内容单元**：任何会被 agent 确认、并被后续工作依赖的东西(token输出)。它的粒度可以很粗也可以细到不可分割，可以是事实、假设、定义、分类、框架、标准、优先级、省略项、顺序、**下一个输出的单token**、词语、句子、段落、行动、工具步骤，也可以是两个内容之间的关系。

**确认**：把某个内容单元当作已经足够确定，从而允许后续思维、判断、产物或行动建立在它上面。确认不只发生在最终落笔时，也发生在接受一个解释、选定一个词、决定不说某件事、默认某个状态、选择某条工具路径的时候。

**承重内容单元**：如果这个内容单元的结果值，由不同的心智思维生成或确认出来，会让后续思维、判断、产物或行动走向「足够重要的」不同可能性，它就是承重内容单元。
- 承重内容单元不需要大、不需要显性、不需要彼此独立，也不需要在最终产物里占很多字。一个词、一句话、一个隐藏假设、一个省略、一个边界条件、一个关系判断，**很多都可以是对后续推理极其重要的承重内容单元**。
- **绝大多数**的承重内容单元是在执行中涌现的，你需要在执行过程中持续识别它们，而不是只依赖开头扫描。


## 反定义

不要把承重内容单元定义成「彼此独立的子任务」。独立性不是必要条件；纠缠关系本身也可能是承重内容单元。

不要用最终输出长度判断注意力需求。一句话看起来很容易写出来，但为了确认这句话合格，可能需要先满足一长串隐藏条件。

不要把「删掉具体主语后仍成立」直接等同于空洞。通用原则可以是有效结论，但必须确认它为什么适用于当前对象、受到哪些证据或约束支持、边界在哪里。

不要把后置校验当成方向、立场、隐藏状态、措辞尺度问题的主要解决办法。后置校验只能补充发现一部分问题；真正承重的方向和判断要在生成依赖内容之前确认。

## 任务开始时

正式动手前，快速扫描显眼的承重内容单元。扫描要便宜，但必须做。

优先找这些位置：

- 目标解释、成功标准、用户真实意图。
- 局部术语、定义、分类、范围边界。
- 事实、隐藏状态、时间顺序、权限、不可逆行动。
- 立场、受众、对外措辞、敏感省略、信息透露粒度。
- 一句话、标题、标签、prompt、policy、公文表述、建议语中承载很多约束的位置。
- 内容之间的关系：冲突、依赖、取舍、先后、整合点。
- 多个候选方向、多个待讨论点、多个判断分支；尤其是每个点单独展开都可能需要长篇探索、反例攻击或多轮对话的位置。

这一步只用于决定先在哪里放慢。开头扫描无法穷尽这些承重内容单元；**绝大多数**的承重内容单元是在执行中涌现的。

## 持续闸门

每当你准备确认一个内容单元的结果值，先问：

> 如果这里不是在主线心流里顺手确认，而是投入专属注意力单独斟酌，得出的结果值会不会不同？这种不同会不会让后续思维、判断、产物或行动走向足够重要的不同可能性？

默认先用直觉快速判定该投入哪一级注意力，不要在每个点上完整展开分析。

用以下四项校准直觉：

- **触达**：后续将有多少内容、判断或行动依赖它。
- **严重度**：最终产物会被谁消费，错一次代价多大。
- **逃逸**：错了以后，是否能在造成代价前被便宜、可靠地拦下；越难拦，逃逸越高。
- **落差**：便宜路径和用心路径在这里可能差多少，既包括写错（下行），也包括错过非显然的高含金量选择（上行）。差值本身无法预估——那需要先走完用心路径才知道；判定基准是当下可观测的落差信号，出现任一条即视为落差可能大：
  - **感觉OK就觉得是真正最合适的**：第一反应是从几个都说得过去的选项里无理由地拿了一个，「拿了哪个」本身没被斟酌过。
  - **没有注意任务特殊性**：写出的内容删掉任务专名后，放进任何同类任务都成立（可直接复用顺写级的通用话检查）。
  - **没有注意内容所在位置特殊性**：这个单元或其位置索引是局部的、新颖的、没有现成惯例，却毫无阻力地写顺了。
  - **没有注意已知约束**：在当前情境存在某些约束、证据或反例来源，但第一反应没有对照过它们。

这四项是升级触发器。执行中只在特别高风险、拿不准、或事前/事后校准时展开四项；常规点逐项展开会把注意力耗在判定流程本身，反而稀释真正承重的内容单元。触达、严重度、逃逸中任一项明显高，或出现任一条落差信号时，不许只因为「感觉应该没事」而降级。拿不准时，自动提高一级重要性。

另外，如果确认这个内容单元所需的**额外推导过程**，其信息量预估可能达到 1000 字数量级，强制走到 Subagent 隔离聚焦。  
- 包含写草稿版本、反例、备选版本、约束推演、证据梳理或争议展开；这里看的是确认过程的信息量，不是最终答案长度。
- 不能因为切到 Subagent 会增加后续整合环节的难度和成本，就不执行这一步。  
  必须前面步骤做好了，后面才能有资格谈整合步骤。

## 动作递进

根据闸门结果决定投入哪一级注意力。三级动作是递进关系：每多满足一层条件，就升一级动作；上一级要求确认的内容在更高一级仍须被确认，只是执行位置随之改变（升到隔离聚焦后，由 Subagent 在隔离上下文中完成同等确认）。

### 默认：顺写

当顺手确认和单独斟酌得出的结果值没有实质差别、不会改变后续走向时，正常继续。唯一要求：若写出通用话，仍要确认它和当前任务的证据、约束、适用边界有关，不要只写可互换的漂亮话。

### 当单独斟酌可能改变走向时：原地局部聚焦

当单独斟酌可能得出不同的结果值并改变后续走向时，你额外需要停下主线，在当前上下文内完成一次专属确认。这次确认不是对着已顺出的版本核对几个条件，而是按这个位置应有的思考方式重新得出内容：

1. **隔离候选**：把主线已顺出的候选结果值（已成文的，或脑中已成形的倾向）显式标记为「候选」并搁置；在第 4 步之前不得引用它的措辞。若此处尚无候选（如动笔前就被扫描出的承重内容单元），跳过本步，第 4 步改为只用第 2 步的约束检验独立版本。
2. **先立规格**：不看候选，写出这个位置的内容单元承担什么角色、其结果值必须满足的 2–4 条关键约束。自检：规格只能描述要求，不能预埋答案——如果某条约束实际上是一个具体结论或具体措辞的改写，删掉重写。
3. **按规格独立生成一版**：从规格正向推导，一遍成型地写出这个位置的结果值应当是什么，不允许写成候选的措辞变体。若此时发现需要草稿迭代、反例攻击、多个备选，或预估推导达 1000 字数量级，立即停止原地聚焦，升级到 Subagent 隔离聚焦。
4. **并置裁决**：把独立版本与候选并置。两者实质一致，即可确认，措辞取更清楚的一版；不一致处逐条用第 2 步的约束裁决，得出确认、加 caveat、改写或标为未知。禁止以「候选更顺口、更贴合上下文腔调」作为裁决理由；若裁决后的改写一再向候选收敛，视为上下文污染，同样升级到 Subagent 隔离聚焦。
5. **回主线**：把裁决结果连同 caveat 与未知带回主线，确认到足以让后续依赖为止。

禁止用「对着候选核对清单」替代第 2、3 步：给既有候选打勾不产生独立版本，那只是在现有思维基础上多往旁边触碰几下。也禁止让周围句子的腔调、已经写顺的措辞、相邻单元的结论，用惯性替你完成确认。

### 如果还满足隔离条件：Subagent 隔离聚焦

在需要单独斟酌的基础上，如果还满足以下任一条件，原地局部聚焦不再够用，你还必须要通过 Subagent 开启一个单独的“思维线程”进行专门聚焦：

- 该内容单元触达高、严重度高、逃逸高、落差可能大。
- 当前上下文正在污染这次斟酌。
- 当次斟酌推导过程产生的额外上下文，会污染后续内容单元的单独斟酌。
- 预估额外推导的信息量可能达到 1000 字数量级。

此时的动作：对单个内容单元，创建一个 Subagent **单独聚焦推导**这个位置的内容单元在输出结论层面应当是什么。

使用 Subagent 时，载荷必须包含：

- 整体任务目标。
- 要确认的精确内容单元。
- 与该单元推导依赖的上下文内容
  如果该单元依赖的前置上下文很重、难以完整写进派发 prompt，**且主线心流中确实基本没有可能对推导判断形成干扰的内容**，改走 fork 方式（而非仅 prompt）。
- 已知约束、立场、受众、不可说信息、非目标。
- 已知备选项、疑点或失败方式。
- 返回形态与回传通道，按下方「交接契约」执行。


对于不同 agent runtime 的创建说明：
- 如果是 cursor：
  - 工具：`Task`；`subagent_type` 选 `generalPurpose`（通用推理/写作）或 `explore`（只读探库）。
  - 派发规则：**一个承重内容单元一次 Task**；多个承重内容单元可同条消息并行派发；禁止多个待确认单元合并进同一 Task。
  - 上下文传递：
    - **仅 prompt**（默认）：Subagent **不**继承当前对话或工具痕迹；与该单元推导依赖的上下文须写进 `prompt`。
    - **fork + prompt**：设 `resume: "self"` 以父 agent 整段对话历史为起点，再在 `prompt` 中额外指定本次探索方向与边界；
- 如果是 codex：
  - 主控工具路径：若当前可直接调用 `multi_agent_v1.spawn_agent`，用它创建 Subagent，并在 `commentary` channel 调用。
  - 递归/子 agent 路径：若当前是 Codex 子 agent，且看不到 `multi_agent_v1.spawn_agent`，不要判定 Codex 不支持递归；改用命令行创建下级 Codex 执行者：`codex exec --ephemeral`。
  - `multi_agent_v1.spawn_agent` 的 `agent_type`：
    - 默认省略或使用 `default`，用于概念推导、写作判断、研讨、反例攻击、框架辨析。
    - 只读探库问题用 `explorer`。
    - 明确要求改文件、实现或验证某个独立写入范围时用 `worker`，并在 prompt 中写清文件/模块 ownership。
  - `multi_agent_v1.spawn_agent` 的上下文传递：
    - **仅 prompt**（默认）：Subagent **不**继承当前对话或工具痕迹；与该单元推导依赖的上下文须写进 `message`(文字内容) 和 `items`(结构化内容)。
    - **fork + prompt**：设 `fork_context: true` 以父 agent 整段对话历史为起点，再在 `message`、`items` 中额外指定本次探索方向与边界；
  - `codex exec --ephemeral` 的调用要求：
    - 通过 shell/终端工具执行，工作目录用当前任务根目录或显式 `-C <dir>` 指定。
    - 用 stdin 或命令参数传入完整派发 prompt；prompt 必须包含下方载荷清单。
    - 建议使用 `-o <tmp-file>` 捕获下级执行者最终回复，便于主控读取并整合。
    - 示例形态：
      ```bash
      codex exec --ephemeral -C "$PWD" -o /tmp/attention-subagent.out - <<'PROMPT'
      你只聚焦一个承重内容单元：...
      返回：单点封口摘要（单点封口的定义见「多承重内容单元与整合顺序」一节）...
      PROMPT
      ```
    - 下级执行者必须只处理一个精确承重内容单元；多个承重内容单元要分别启动多次 `codex exec --ephemeral`。
  - 只有当 `multi_agent_v1.spawn_agent` 不可用，且 shell/终端中也无法执行 `codex exec --ephemeral` 时，才判定当前 agent runtime 无法创建 Subagent，并直接失败返回；不得降级成多方向摘要。
- 主 agent 收齐返回后专门整合，不得跳过。

隔离结果回来后，必须专门整合：检查它是否改变了前文假设、相邻内容单元或整体结构。

### 交接契约

通道事实：Subagent 的思考过程、中间消息、进度摘要字段对主控全部不可见；主控只收到它最后一条消息的正文（或它写入的文件）。Subagent 感觉里的「上文」在主控侧不存在，因此契约必须落在可观察产物的形状上，不能只写「返回结果」这类意图动词。

返回合格的唯一判据是**无上下文读者测试**：一个只读到这条返回（外加派发 prompt）的读者，能直接完成主控的整合动作——给结论定权重、判断与其它单元的关系、回答「为什么不是别的方案」、被质疑时循锚点回溯——而不需要向 Subagent 追问。

派发 prompt 末尾固定粘贴：

> 交付契约：你的思考过程和中间消息主控看不到，你最后一条消息的正文是唯一交付物。它必须自包含，从「## 结论」逐字开始，依次给出：结论（每条附证据锚点与一行为什么）、淘汰的备选及一行理由、适用边界与置信分层（实证/推断/未知）、未知项、≤150 字交接摘要。「见上文」「已在上方」等指代和「已完成/已封口」类元陈述不是交付。

章节按承重量伸缩：微小单元可塌缩成一段，但无上下文读者测试不豁免。

主控收到返回后，先过形状闸门再整合：章节齐全、无指代词、结论有锚点、不是纯元陈述。不合格立即 resume 携同一契约纠正，重试至多 2 次，再失败则废弃该会话重新派发；不从进度摘要字段抢救内容，它是 UI 展示物，不是交付物。

重交付、审计或大材料任务改走文件通道：Subagent 把交付物写入主控指定的 scratch 路径（如 `.skill-logs/` 下），最终消息只返回路径与交接摘要，主控读文件后执行同一套闸门；文件的存在性与非空可机器校验，且独立文档的体裁天然没有「上文」可指。只读环境不可用时降级回消息通道。

### 可审计隔离记录

当用户要求实验、验证、审计，或 Subagent 隔离聚焦本身会成为后续判断依据时，不能只保存最终摘要来证明隔离已经发生。主 agent 必须额外保存或回传以下记录：

- 上游方向发现或承重内容单元识别的原始输出；若有候选淘汰，也保留淘汰理由。
- 每个 Subagent 的完整派发 prompt、创建命令或工具调用参数。
- 每个 Subagent 的输出路径、返回内容或最终消息。
- 每个 Subagent 是否 fork、继承上下文，或只接收 prompt。
- 整合前的结果关系判断记录：各单点答案之间是互斥、互补、层级包含、条件分流，还是待研究。

若这些记录缺失，只能说“产物形态符合单点封口”，不得声称已经可审计地证明“一承重内容单元一 Subagent”“隔离输入干净”或“先封口后整合”。

## 多承重内容单元与整合顺序

当一个任务包含多个互相关联的承重内容单元时，这些单元按什么顺序推导、哪些能并行、哪些要合并处理、何时才允许整合，都不由主线心流顺手决定，而由单元之间的关系决定。本节给出完整指引：这些关系有哪几类，以及依据它们推导全部单元、直至整合的完整流程。

先约定单元的生命周期。每个承重内容单元先是一个**问题**——这个位置要确认什么、受哪些约束；经推导确认后得到一个**答案**。把一个单元确认到可被下游依赖的程度，称为**单点封口**。封口不是得出永远正确的结论，而是至少产出以下局部状态：

- 当前暂定判断。
- 支撑理由或证据。
- 强反例、失败方式或近敌。
- 适用边界。
- 仍未知的问题。
- 可交给整合层消费的短摘要。

### 四类关系

单元之间的关系，按它连接的两端是问题还是答案分为四类；两端是什么，同时决定了这个关系最早何时判得动、以及它约束流程中的哪个环节。判断时机的总原则：只消耗问题表述的关系，扫描期就能判；要消耗答案内容的关系，必须等答案封口——提前判，等于对还不存在的答案预设内容，构成锚定污染。因此「关系判断」不是一个整体阶段，四类关系各有各的位置。

| 关系 | 两端 | 内容 | 最早可判时机 | 决定流程中的什么 |
|---|---|---|---|---|
| 分解关系 | 问题-问题 | 重复、包含、遗漏、边界切分是否正确 | 扫描期 | 单元清单本身是否成立 |
| 依赖关系 | 问题-答案 | 问题 A 的推导需要答案 B 作输入（判断只消耗「A 需要 B 的答案」这一问题层事实，不猜 B 答案的内容） | 扫描期 | 封口顺序与并行度；依赖成环即纠缠，须升格统一推导，或继续拆内容单元的构成要素(比如拆成 A‘ + B’ + A‘与B’的关系)，直到不再纠缠 |
| 回写关系 | 答案-问题 | 答案 B 使其它问题消解、分裂、重述，或催生新问题 | B 封口后 | 流程必须带环：每次封口后回检并更新清单 |
| 结果关系 | 答案-答案 | 互斥、互补、层级包含、条件分流、冲突 | 双方封口后 | 整合层的输入 |

### 完整流程

按以下循环推进。第 2、3 步不是一次性的先后两段，而是逐答案交替：每封口一个答案就立即做一次回写检查，再继续确认下一批。

1. **扫描**：判分解关系（单元清单本身对不对）与依赖关系，得到暂定依赖图。依赖成环（A 需要答案 B，B 也需要答案 A，即纠缠）时，把整个环升格为一个承重内容单元，其问题改为「这组关系应当如何共同取值」。
2. **按拓扑序确认**：无依赖的单元并行；每个单元仍按三级动作递进判档，只有满足隔离条件的才派发 Subagent；派发下游时，其依赖的上游交接摘要必须显式写入载荷，不得指望下游自行知晓。
3. **回写检查**：每封口一个答案，检查它是否回写了其它问题（例：答案 B 表明问题 A 的前提不成立，A 随之消解）。发现回写，回到第 1 步更新清单与依赖图，重排未执行的调度；若被回写动摇的是已封口的单元，把它重新打开为问题，一并纳入重排。
4. **结果关系判断**：全部封口后，把结果关系的判断本身当作一个承重内容单元处理，必要时对它单独 Subagent 隔离聚焦。
5. **整合**：在结果关系判断之上组织最终产出。整合的实质是关系判断的落笔，叙事通顺只是副产品；不得为了获得漂亮统一感而回头压低任何单点的复杂度；某单点虽已封口、但其未知项或反例使它不足以支撑此处结论时，整合层必须标明「此处暂不整合」。

流程全程遵守以下硬性禁令：

- 在所有必要上游单元达到单点封口前，不得输出综合结论、统一框架或总判断。可以输出的只有依赖图、待确认清单、局部结论或下一步隔离计划。
- 对于任何有依赖关系的两个单元 A 和 B：A 封口前，不可开展 B 的推导，B 的推导 Subagent，需确保**上下文具备 A 推导的完整结论**。
- 上游材料多导致整合上下文变重时，不许因此砍掉上游注意力。正确动作是把各单点可追溯的交接摘要作为整合层输入；必要时再对整合层单独使用 Subagent 隔离聚焦。

## 上下文污染触发器

出现以下情况时，升级到 Subagent 隔离聚焦：

- 预判原地局部聚焦的「隔离候选」搁置也挡不住当前已顺出的腔调、框架或措辞对这个判断的锚定，或执行中已观察到独立版本向候选收敛。
- 其它承重内容单元的推导过程正在主动影响这个单元。
- 正确确认需要展开很多草稿、备选、checklist 或约束推演，放在主线里会污染其它判断。
- 先写出一个顺口版本，会让错误方向显得自然，导致后面更难发现。
- 用户要求“逐个方向深入”“单独聚焦”“深度研讨”“辩论探索”，而当前回答形态正在变成多点摘要、并列概述或提前整合。
- 你发现自己为了维持全局结构、篇幅平衡、交付完整感或后续整合便利，正在压缩某个单点的探索质量。
- 内容涉及立场、信息透露、公文/对外表达、policy、不可逆行动、核心定义、隐藏状态。

## 高密度小输出

对于什么会是关键承重单元的判断：特别警惕「输出很短，但确认密度很高」的内容。

典型位置：

- 对外一句话：同时受立场、受众、透露粒度、引导意图、风险边界约束。
- 一个定义：后续设计、分析或争论都会沿它展开。
- 一个分类：决定接下来走哪个 workflow、skill、工具或评价标准。
- 一条建议：措辞变化会改变用户行动。
- 一句 prompt、policy、公文、issue comment、commit message、正式口径。
- 细粒度到输出内容下一个输出词（单 token）

处理这类内容时，不要问「我能不能写出看起来没问题的答案」。要问：「这个内容单元的答案值要满足哪些条件，才配让后续工作建立在它上面？」

## 未知与通用结论

专属注意力之后仍只能写通用内容时，区分三种情况：

- **适用的通用性**：当前最可靠的结论就是通用原则，但它已经被连接到本任务的适用理由和边界。
- **缺少区分信息**：要更具体，需要当前没有的事实、约束或反馈。
- **当前无知**：agent 不知道如何可靠确认这个单元。

缺少信息或当前无知时，直接标明。不要为了显得具体而编造特殊性。

## 后置校验

交付前仍要做后置校验：一致性、格式、明显遗漏、事实冲突、通用结论是否连接到本任务。

但不要期待后置校验可靠发现最核心的问题：起始方向错、立场错、隐藏状态默认错、信息透露尺度错、措辞已经被流畅感锚定。这些问题要靠确认前的持续闸门处理。

## 对用户呈现

不要把全部注意力预算过程倒进最终回答。只有在有帮助或用户要求时才简要呈现。

需要呈现时，只说：

- 哪些承重内容单元改变了处理方式。
- 哪些用了原地局部聚焦，哪些用了隔离聚焦。
- 哪些仍是未知、缺少信息或显式假设。
