# 契约化质量义务执行指导

## 输入信息

**必须入参**：sealed `candidate-obligations.md`、任务契约、候选引用的现实模型锚点。

**上下文参考**：[义务契约 Schema](../references/obligation-contract-schema.md)、适用测量/验证规范。

## 目标要求

生成 `quality-obligations.md`，将候选主题转换为可触发、可取证、可判定、可失效的统一义务。

完成标准：

- 每项含稳定 ID、来源、触发、义务、证据要求、期限、依赖、失效/重开、逐项 `OK / NOT_OK / INDETERMINATE` 判据、当前状态和阻断未知。
- 一条义务只包含共同变化的主张；必要时拆分或提升为联合义务。
- 合同、用途、渠道、地区、能力维度和控制动作等集合采用一实例一义务或可独立三态的义务族，不得用聚合行掩盖一真一假。
- `OK` 需要有效证据，动作发生不等于状态成立。
- “已签认/通知/批准/检查”只能证明动作；义务必须另写动作要兑现的现实状态。例外不得绕过硬护栏或不可接受损失。
- 评价性义务写明评价者、评价任务、依据和边界。
- 不存在“关注、保障、优化、体验好”等不可判定谓词或无来源阈值。

## 工作依据

- 先定义必须成立的状态，再选择证据，避免从现成指标反推目标。
- 尝试构造义务内部两个主张一真一假；能构造则拆分。
- 对每个高严重度义务以及至少三项普通义务实际写出一组 `OK / NOT_OK / INDETERMINATE` 实例；无法稳定裁决则不得 seal。
- 后续版本、环境、事实或责任变化能推翻义务时，写具体重开事件。
- 缺证据保留 `INDETERMINATE`，不能删除义务或降低标准。
- 高严重度单项达到注意力门槛时按义务 ID 隔离；主控统一术语后再合入。

## 产出格式

```markdown
# 质量义务集合

## 义务表
每行必须含状态判据；义务族须列出实例化规则与实例状态。
## 三态汇总
## 高严重度义务单点封口
## 拆分、合并与删除记录
## 阻断未知
```

完成后运行 V3 和 T2；失败回候选生成或契约化。
