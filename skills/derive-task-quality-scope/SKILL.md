# 推导具体任务的质量范围

面对“把这件事做好”，我不会先套一张通用质量清单。任务真正的质量范围取决于目标、现实机制、利益相关者、环境、生命周期、损失、证据和决策权；若这些没有先接入现实，形式完整的索引也会稳定地遗漏关键问题。

本工作流把质量索引写成可触发、可取证、可失效的义务契约，再证明它们如何组合支持整体 OK。最终结论是带证明域、有效期和重开条件的可撤销声明，不承诺开放世界中的绝对完备，也不直接完成用户的原任务。

## 行为日志执行协议

- **规范引用**：[behavior-log/v1 契约](../behavior-log-audit/references/behavior-log-v1-contract.md)；日志骨架：[behavior-log-skeleton.md](../behavior-log-audit/templates/behavior-log-skeleton.md)
- **路径**：`{workspaceRoot}/.skill-logs/derive-task-quality-scope/yyyy-mm-dd/hhmm-<任务标题slug>.md`
- **初始化**：首次读取决定性来源或写入质量范围产物之前。写入元数据、执行边界、状态 `执行中`。任务标题取自用户原始任务的首个非空目标句，规范化为 slug
- **增量更新触发点**：
  - 任务契约、任务形态或证明域被确认或重开
  - 读取决定性私域/外部来源、发生权限受阻或采用替代取证
  - 创建、拆分、合并或淘汰承重质量义务
  - 确认关系、不变量、权衡、证据关口或失效传播
  - 调用 Subagent/子 Skill 并使用其结果
  - 校验或测试失败、回退、降级、阻断、风险接受或最终封口
- **子 Skill / Subagent 契约**：显式 Skill 子调用返回 `{交付物, behaviorLogPath, 材料性影响摘要}`；纯 Task 返回 `{交付物路径, auditSummary, 证据锚点, 未知}`，主控在父日志「关键行为」登记一行摘要
- **封口**：V1–V5 校验、T1–T4 独立测试和所有受影响回归均结束后。更新顶层执行摘要、影响清单、结果与验证、资源索引；终态与完成时间一致。子 Skill 应有日志但未返回路径时，终态最高 `部分成功`，并在「偏离、异常与未决事项」登记缺口
- **与交付物边界**：质量范围包写入调用者指定目录或 `.quality-scope/<task-slug>/`；行为日志只记录过程审计，通过「资源索引」链接交付物，不复制正文

运行时审计标准见 [references/audit-quality-standards.md](references/audit-quality-standards.md)。

## 工作流程（主控 Agent 调度模式）

第一步必须使用 `TodoWrite` 创建以下八个待办项；每阶段一个待办，名称和顺序不得改变：

- 冻结任务契约
- 建立来源与权利地图
- 恢复任务现实模型
- 生成多源候选义务
- 契约化质量义务
- 建立组合闭包
- 设计证据与裁决关口
- 独立挑战并封口质量范围

若运行时没有 `TodoWrite`，使用可用的计划/状态工具等价记录，但名称和顺序仍须一致。

### 全局约束

- 默认输出到 `<workspaceRoot>/.quality-scope/<task-slug>/quality-scope-package/`；用户指定 `outputRoot` 时改用 `<outputRoot>/quality-scope-package/`。
- 全流程只使用 `OK / NOT_OK / INDETERMINATE`；证据不足、来源冲突、无权裁决或超出适用包络时必须保持 `INDETERMINATE`。
- 每个节点只消费上游精炼文件和按需软上下文，不依赖完整对话历史。
- 加载 [references/attention-integration.md](references/attention-integration.md)：千字级承重内容单元必须一单元一 Subagent，先单点封口再整合。
- 使用 [templates/quality-scope-package.md](templates/quality-scope-package.md) 建立产物目录；每个文件经历 `draft → sealed`，重开时改为 `reopened` 并记录原因。
- 在关键产物后按 [workflow/validation-and-testing.md](workflow/validation-and-testing.md) 执行 V1–V5 和 T1–T4；校验失败或测试失败必须回退修订并重跑受影响关口。

### 冻结任务契约

**执行模式**：主控执行。

**我为什么做这一步**：先固定要证明的整体结果、边界和风险口径，避免后续索引追随第一个顺手方案。

**输入**：用户原始任务、工作区/情境、已知目标、约束、受众、时限、不可逆条件和非目标。

**目标与产出**：生成 `task-contract.md`，明确任务对象、整体 OK、受影响者、责任边界、环境、时间窗、不可接受损失、风险级别、假设、非目标、决策权和允许裁决状态。

**完成判定**：整体 OK 可被证据支持；原任务完成与范围发现完成分离；缺失项为 `UNKNOWN` 并有阻断影响。

**Audit Hook**：初始化日志，更新「执行边界」「依据与关键输入」「关键行为」「资源索引」。

执行时必须完整阅读 [workflow/task-contract.md](workflow/task-contract.md)。完成后运行 V1。

### 建立来源与权利地图

**执行模式**：主控维护总图；大型且独立的来源域可委派只读 Subagent。

**我为什么做这一步**：把“应从哪里知道”和“谁有资格确认”分开，防止用通用先验、会议沉默或无权意见填补现实缺口。

**输入**：sealed `task-contract.md`、可用工作区/私域/外部访问条件。

**目标与产出**：生成 `source-rights-map.md`；每个关键主张有来源、版本/时间、访问方式、权威范围、替代路径和权利角色。

**完成判定**：事实确认、语义裁决、执行、停止、披露和残余风险接受权没有混淆；敏感材料采用最小权限取证。

**Audit Hook**：更新来源、检索、权限受阻、替代取证、权利判断和外部事实日期。

执行时必须完整阅读 [workflow/source-rights-map.md](workflow/source-rights-map.md)。

### 恢复任务现实模型

**执行模式**：主控选择任务形态并整合；多个千字级系统域一域一 Subagent。

**我为什么做这一步**：质量义务必须附着在真实对象、状态、关系和反馈上，不能把理想流程当现实。

**输入**：sealed `task-contract.md`、`source-rights-map.md`、已取得事实与冲突登记。

**目标与产出**：生成 `task-reality-model.md`，恢复利益相关者/对象边界、成功机制、状态/时间、接口/资源、控制/责任、生命周期和反馈。

**完成判定**：每个关键事实可追溯；任务形态路由、不可逆点、低频/不可观测路径、外部依赖和反馈时延可定位。

**Audit Hook**：更新任务形态、模型来源、冲突、淘汰解释、分域交接和模型版本。

执行时必须完整阅读 [workflow/task-reality-model.md](workflow/task-reality-model.md)。完成后运行 V2 和 T1。

### 生成多源候选义务

**执行模式**：主控维护总账；达到注意力门槛时，一生成源一 Subagent，禁止读取其它源结论。

**我为什么做这一步**：让不同盲区方向的生成源独立提出候选，避免单一 MECE 轴或通用属性目录制造虚假完备。

**输入**：sealed 任务契约、来源权利地图和现实模型。

**目标与产出**：生成 `candidate-obligations.md`；逐项使用价值/受众、成功机制、状态/生命周期、约束/损失、标准/历史事实五类生成源。

**完成判定**：每个候选有来源坐标、对象、进入理由和缺失后的整体失败；同义项保留多重来源；弱相关词汇与虚假交叉被淘汰。

**Audit Hook**：保存生成源输入输出、Subagent 派发/返回、候选淘汰、合并和保留理由。

执行时必须完整阅读 [workflow/candidate-obligations.md](workflow/candidate-obligations.md)。

### 契约化质量义务

**执行模式**：主控统一契约；高严重度、不可可靠后置校验的单项可隔离 Subagent。

**我为什么做这一步**：把提醒性主题转换成能够触发、取证、判定和重新失效的义务，才有资格进入整体完成合取。

**输入**：sealed `candidate-obligations.md`、任务契约和候选引用的现实模型锚点。

**目标与产出**：生成 `quality-obligations.md`；每项具有稳定 ID、触发、义务、证据要求、期限、依赖、失效/重开、来源、逐项三态判据和当前状态。

**完成判定**：义务原子且可判定；可独立真假的集合已实例化；动作发生不等于完成；例外不能绕过硬护栏；评价性义务写明评价者、依据和边界；不得发明阈值。

**Audit Hook**：更新义务来源、备选定义、拆分/合并、删改理由、单点封口和状态判据。

执行时必须完整阅读 [workflow/obligation-contracts.md](workflow/obligation-contracts.md)。完成后运行 V3 和 T2。

### 建立组合闭包

**执行模式**：主控执行；整合被既有框架锚定或达到千字级时另起整合 Subagent。

**我为什么做这一步**：专门偿还分解产生的合成债，防止局部全绿、整体失败或不同版本绿灯拼贴。

**输入**：sealed 质量义务、现实模型和任务契约。

**目标与产出**：生成 `closure-model.md`，记录依赖、冲突、顺序、共享状态、权衡、失效传播、全局不变量、硬护栏、联合义务和完成层级。

**完成判定**：已知局部全绿/整体失败路径被命中；义务共同状态可达；价值冲突和残余风险有有权裁决者。

**Audit Hook**：更新关系判断、全局不变量、反例、权衡点、完成层级、决策权和未闭合关系。

执行时必须完整阅读 [workflow/closure-model.md](workflow/closure-model.md)。

### 设计证据与裁决关口

**执行模式**：主控设计总结构；专业或高污染证据主张可单点委派，义务定义者不得被当作独立评价者。

**我为什么做这一步**：区分分析、检查、演示、测试、真实使用反馈和独立评价，避免用证据数量或代理指标制造虚假确定性。

**输入**：sealed 质量义务、闭包模型和来源权利地图。

**目标与产出**：生成 `evidence-gates.md`；每项义务/关系具证据对象、方法、来源、版本、环境、责任人、有效性、独立性、三态和失效条件。

**完成判定**：证据不超出证明域；代理关系、样本、分母、阈值依据和泛化边界可解释；阻断未知连接禁止动作和重开动作。

**Audit Hook**：更新证据选择、测量有效性、替代证据、版本/环境、独立性和关口状态。

执行时必须完整阅读 [workflow/evidence-gates.md](workflow/evidence-gates.md)。完成后运行 V4 和 T3。

### 独立挑战并封口质量范围

**执行模式**：独立测试必须委派 Subagent；主控负责形状闸门、失败分类、回退和封口。

**我为什么做这一步**：产生范围的同一心流最难发现自己的遗漏，因此先寻找“全部义务显示完成但整体仍失败”的逃逸路径，再形成可撤销声明。

**输入**：前七阶段 sealed 产物、重大未知、未消除反驳和风险接受记录。

**目标与产出**：生成 `quality-scope.md`、`validation-report.md` 和 `tests/`；封口完整质量范围目录。

**完成判定**：重大逃逸被分类和回写；正文显示证明域、未知、残余风险、允许/禁止动作、有效期与重开条件；新鲜下游读者可正确消费。

**Audit Hook**：记录测试派发/返回、失败分类、修订路径、最终裁决、残余未知、重开条件和封口摘要；完成 behavior-log/v1 封口。

执行时必须完整阅读 [workflow/challenge-and-seal.md](workflow/challenge-and-seal.md) 与 [workflow/validation-and-testing.md](workflow/validation-and-testing.md)。运行 V5 和 T4；未全部通过不得 seal。

## 注意事项

- 不因用户要求“完整”而声称绝对完备；始终声明证明域和残余未知。
- 不把流程清单、质量属性表、风险表、知识目录或测试通过率单独当作质量范围。
- 不让后置检查代替承重定义、立场、权利、关系和隐藏状态的事前确认。
- 任何新失败先分类为执行未遵守、义务定义错误、关系缺失、证据无效或任务模型缺项；只有后三类及模型缺项需要修改范围。
- 最终文件应服务下游行动；过程草稿、搜索噪音和完整日志不得复制进 `quality-scope.md`。
