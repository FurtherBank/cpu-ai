# 组合闭包方法

## 目标

证明局部义务如何在同一相关版本、时间和环境中支持整体 OK，并捕获局部全绿但整体失败的路径。

## 关系类型

| 关系 | 检查问题 |
| --- | --- |
| 依赖 | A 成立是否要求 B 先成立 |
| 冲突 | A 与 B 是否不能同时达到各自目标状态 |
| 顺序 | 同样动作换序是否改变结果或可恢复性 |
| 共享状态/资源 | 两项是否读写同一对象、预算、权限或队列 |
| 权衡 | 同一决定是否改善一项、损害另一项 |
| 失效传播 | A 变化后哪些已完成义务必须重开 |
| 证据耦合 | 多个通过项是否共享同一错误规格、样本或评分器 |

## 闭包产物

- 关系图：义务 ID 与关系类型。
- 全局不变量：跨步骤或跨对象始终必须成立的状态。
- 硬护栏：不能被收益或其它通过项抵消。
- 联合义务：只能对组合状态判真的主张。
- 完成层级：制品、集成、真实用途、转交/运行、长期适用中哪些适用。
- 权衡裁决点：选项、多维证据、敏感性和有权角色。
- 失效传播图：变化事件到重开义务及下游关口。

## 反例驱动检查

对承重义务及现实接口逐项问：

- 两项局部都 `OK` 时，整体是否仍可失败？
- 这些 `OK` 是否来自从未同时存在的版本或时间切片？
- 某项后续变化是否会推翻其它历史完成状态？
- 接口、共享资源、组织责任或反馈是否无主？
- 义务集合是否根本无法同时成立，从而制造真空充分？

若发现逃逸，新增或修改关系义务/不变量；不要新增一个空泛“整体检查”主题。
