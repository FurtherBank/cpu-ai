你在一家互联网公司工作，主管口头跟你说，交给你一个技术改造的任务，让你全盘执行。

执行会基于这个 sop

# 技术改造全流程执行 SOP

**适用场景**：存量系统重构、核心链路升级、技术栈迁移、架构优化等任务。
**核心原则**：谋定而后动（RFC First）、小步快跑、灰度落地、随时可回滚。

---

## 阶段一：需求去模糊化与现状调研 (Discovery Phase)
**目标**：将主管的“口头指令”转化为“书面共识”，明确改造边界。

### 1.1 核心目标对齐 (Stakeholder Alignment)
*   **沟通对象**：主管、任务发起人。
*   **动作**：确认本次改造的“北极星指标”。
    *   **核心痛点确认**：具体要解决哪些阻碍业务或技术发展的核心问题？（如：系统稳定性、开发效率、性能瓶颈、扩展性限制等）。
    *   **预期收益 (ROI)**：改造完成后，预期达成什么可量化的效果？（如：P99 延迟降低 x%、人力维护成本降低 x%、支持 x 倍并发等）。
    *   **资源与排期**：预期的时间节点（DDL）及可用的人力资源。

### 1.2 现状资产盘点 (As-Is Analysis)
*   **代码审计**：梳理旧系统的核心逻辑、数据流向、关键类/函数。
*   **依赖梳理**：
    *   **上游**：谁调用了我们？流量特征是什么？
    *   **下游**：我们依赖了谁？是否存在强耦合？
    *   **数据**：涉及哪些数据库、缓存、消息队列？是否存在数据一致性隐患？
*   **历史债务识别**：通过访谈前任开发者或查看 Commit 记录，识别代码中的“黑魔法”或临时规避方案。

### 1.3 产出物
*   ✅ 《现状调研报告》（包含核心链路图、现有问题列表）。
*   ✅ 邮件/IM 确认记录（复述沟通结论并获得主管确认）。

---

## 阶段二：RFC 方案设计 (Design Phase)
**目标**：通过结构化文档暴露设计盲点，产出可执行的工程蓝图。

### 2.1 方案撰写 (Drafting)
遵循标准 RFC 模板，重点覆盖以下通用维度：
*   **范围定义 (Scoping)**：
    *   **In-Scope**：本次必须完成的功能和模块。
    *   **Non-Goals**：明确本次不做的事情（防止范围蔓延）。
*   **架构设计 (Architecture)**：
    *   总体架构图、时序图、ER 图。
    *   关键技术选型依据（对比 A/B 方案的优劣）。
*   **兼容性设计 (Compatibility)**：
    *   接口兼容性（是否破坏性变更）。
    *   数据兼容性（新旧数据格式如何共存）。
*   **迁移策略 (Migration Strategy)**：
    *   **关键策略**：双写（Dual Write）、双读（Dual Read）、影子流量（Shadow Traffic）等策略的选择。
    *   **数据迁移**：存量数据清洗与迁移方案。

### 2.2 风险控制设计 (Risk Management)
*   **开关机制 (Feature Flag)**：设计细粒度的配置开关，确保新逻辑可随时关闭。
*   **回滚方案 (Rollback Plan)**：
    *   代码回滚：镜像版本回退。
    *   数据回滚：如果新方案污染了数据，如何修复？

### 2.3 产出物
*   ✅ 《技术改造 RFC 初稿》。

---

## 阶段三：评审与定稿 (Review Phase)
**目标**：利用团队智慧“找茬”，确保 RFC 逻辑闭环，无设计硬伤。

### 3.1 预沟通 (Nemawashi)
*   **动作**：在正式评审前，私下找核心干系人（资深架构师、主管、SRE）过一遍方案。
*   **目的**：解决明显的逻辑漏洞，达成初步共识，避免评审会陷入无休止的争论。

### 3.2 正式评审 (Formal Review)
*   **参与方**：开发团队、QA（测试）、SRE（运维）、上下游业务方。
*   **评审重点**：
    *   方案的可行性与复杂度。
    *   对上下游的影响（SLA 变更）。
    *   监控与可观测性设计（是否缺监控？）。
    *   **灾难恢复预案**。

### 3.3 产出物
*   ✅ 《技术改造 RFC 定稿》（所有 Reviewer 标记 LGTM）。
*   ✅ 《测试用例初稿》（QA 介入）。

---

## 阶段四：代码实施 (Implementation Phase)
**目标**：将 RFC 翻译为代码，严格执行防御性编程。

### 4.1 环境准备
*   建立独立开发分支。
*   引入 Feature Flag 基础库或配置中心项。

### 4.2 编码与自测
*   **适配层开发 (Adapter Pattern)**：优先开发新旧逻辑的适配层，隔离变更。
*   **单元测试 (Unit Test)**：核心改造逻辑覆盖率需达到规定标准（如 80%+）。
*   **Code Review**：建议采用“小步提交”策略，单次 PR 不宜过大，确保逻辑可审查。

### 4.3 联调与验证
*   **功能验证**：QA 介入进行全回归测试。
*   **性能验证**：在 Staging 环境进行基准测试（Benchmark），对比新旧代码的 CPU/内存/延迟指标。

### 4.4 产出物
*   ✅ 完成 Code Review 的代码分支。
*   ✅ 性能压测报告。

---

## 阶段五：灰度上线与切换 (Release & Migration)
**目标**：零停机、无感知的平滑上线。

### 5.1 上线准备
*   **监控埋点**：确认核心业务指标（Business Metrics）和系统指标（System Metrics）的大盘已就绪。
*   **告警配置**：配置高敏度的告警规则。

### 5.2 分阶段发布 (Phased Rollout)
1.  **Stage 1 - 影子模式 (Shadow Mode)**：新代码运行但不影响业务（如：异步执行新逻辑并对比结果，仅记录日志）。
2.  **Stage 2 - 灰度引流 (Canary Release)**：
    *   按用户 ID、地区或流量比例（1% -> 5% -> 20%）。
    *   **观察期**：每个阶段停留足够时间（如 4-24 小时），观察错误日志和业务指标。
3.  **Stage 3 - 全量切换 (Full Release)**：流量 100% 打入新逻辑。

### 5.3 存量数据处理 (如果涉及)
*   启动后台任务清洗/迁移历史数据。
*   数据校验脚本持续运行，确保新老数据一致。

---

## 阶段六：收尾与复盘 (Closure)
**目标**：清理战场，沉淀价值。

### 6.1 清理工作
*   **代码清理**：待系统稳定运行（如 2 周）后，删除旧逻辑代码、下线 Feature Flag、回收旧资源。
*   **文档更新**：更新 API 文档、架构图、On-call 手册。

### 6.2 项目复盘
*   **价值回收**：对比阶段一设定的 ROI 指标，计算实际收益。
*   **经验沉淀**：总结遇到的坑和解决思路，分享给团队。

### 6.3 产出物
*   ✅ 《项目结项报告》（包含 ROI 分析）。
*   ✅ 清爽的代码库（无 Dead Code）。



### 执行建议：

1.  **SOP 的裁剪**：并不是所有改造都要走完每一个细分步骤。如果是小型重构，Review 和灰度阶段可以简化，但 **RFC 文档化** 和 **Feature Flag** 是底线，不可省略。
2.  **关于“通用描述”**：
    *   在阶段 1.1 中，我将具体的“性能瓶颈”泛化为“核心痛点确认”。
    *   在阶段 2.1 中，我将具体的“数据库字段变更”泛化为“兼容性设计”和“迁移策略”。
    *   这使得该 SOP 既适用于代码层面的重构（Refactoring），也适用于架构层面的迁移（Migration）。
