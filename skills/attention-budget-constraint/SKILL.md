# 注意力预算约束

## 绑定约束

自本 skill 加载起至任务结束，把注意力视为需要按需分配的任务资源。凡是会改变后续走向的内容确认点，都必须获得与其重要性匹配的专属注意力。不要在一个流畅的主线心流里顺手确认高承重点。

## 核心定义

**内容单元**：任何会被 agent 确认、并被后续工作依赖的东西。它可以是事实、假设、定义、分类、框架、标准、优先级、省略项、顺序、词语、句子、段落、行动、工具步骤，也可以是两个内容之间的关系。

**确认**：把某个内容单元当作已经足够确定，从而允许后续思维、判断、产物或行动建立在它上面。确认不只发生在最终落笔时，也发生在接受一个解释、选定一个词、决定不说某件事、默认某个状态、选择某条工具路径的时候。

**承重内容单元**：如果这个内容单元用不同方式确认，会让后续思维、判断、产物或行动走向「足够重要的」不同可能性，它就是承重内容单元。

承重内容单元不需要大、不需要显性、不需要彼此独立，也不需要在最终产物里占很多字。一个词、一句话、一个隐藏假设、一个省略、一个边界条件、一个关系判断，都可能是承重内容单元。

## 反定义

不要把承重内容单元定义成「彼此独立的子任务」。独立性不是必要条件；纠缠关系本身也可能是承重点。

不要用最终输出长度判断注意力需求。一句话看起来很容易写出来，但为了确认这句话合格，可能需要先满足一长串隐藏条件。

不要把「删掉具体主语后仍成立」直接等同于空洞。通用原则可以是有效结论，但必须确认它为什么适用于当前对象、受到哪些证据或约束支持、边界在哪里。

不要把后置校验当成方向、立场、隐藏状态、措辞尺度问题的主要解决办法。后置校验只能补充发现一部分问题；真正承重的方向和判断要在生成依赖内容之前确认。

## 任务开始时

正式动手前，快速扫描显眼的承重内容单元。扫描要便宜，但必须做。

优先找这些位置：

- 目标解释、成功标准、用户真实意图。
- 局部术语、定义、分类、范围边界。
- 事实、隐藏状态、时间顺序、权限、不可逆行动。
- 立场、受众、对外措辞、敏感省略、信息透露粒度。
- 一句话、标题、标签、prompt、policy、公文表述、建议语中承载很多约束的位置。
- 内容之间的关系：冲突、依赖、取舍、先后、整合点。
- 多个候选方向、多个待讨论点、多个判断分支；尤其是每个点单独展开都可能需要长篇探索、反例攻击或多轮对话的位置。

这一步只用于决定先在哪里放慢。开头扫描无法穷尽这些承重内容单元；**绝大多数**的承重点是在执行中涌现的。

## 持续闸门

每当你准备确认一个内容单元，先问：

> 如果这里换一种确认方式，后续思维、判断、产物或行动会不会走向足够重要的不同可能性？

默认先用直觉快速分档，不要在每个点上完整展开分析。

用以下四项校准直觉：

- **触达**：后续将有多少内容、判断或行动依赖它。
- **严重度**：最终产物会被谁消费，错一次代价多大。
- **逃逸**：错了以后，是否能在造成代价前被便宜、可靠地拦下；越难拦，逃逸越高。
- **落差**：便宜路径和用心路径在这里可能差多少，包括写错，也包括错过非显然的高含金量选择。

这四项是升档触发器。执行中只在特别高风险、拿不准、或事前/事后校准时展开四项；常规点逐项展开会把注意力耗在判定流程本身，反而稀释真正承重的内容单元。触达、严重度、逃逸中任一项明显高时，不许只因为「感觉应该没事」而降档。拿不准时，自动提高一级重要性。

另外，如果这个内容单元在**额外推导过程**的信息量，预估可能达到 1000 字数量级，强制走到 Subagent 隔离聚焦模式。  
- 包含写草稿版本、反例、备选版本、约束推演、证据梳理或争议展开；这里看的是确认过程的信息量，不是最终答案长度。
- 不能因为切到了 Subagent 会增加后续的整合环节的难度和成本，不执行这一步。  
  必须前面步骤做好了，后面才能有资格谈整合步骤。

## 动作分档

根据闸门结果选择动作。

### 0 档：顺写

适用条件：不同确认方式不会实质改变后续走向。

动作：正常继续。若写出通用话，仍要确认它和当前任务的证据、约束、适用边界有关，不要只写可互换的漂亮话。

### 1 档：原地局部聚焦

适用条件：不同确认方式会改变后续走向，但可以在当前上下文内解决，且不需要强隔离。

动作：

1. 明确正在确认的具体内容单元。
2. 列出它必须满足的少数关键条件。
3. 至少点出一个可能替代版本或失败方式。
4. 选择确认、加 caveat、改写，或标为未知。
5. 确认到足够让后续依赖后，再回到主线。

禁止让周围句子的腔调、已经写顺的措辞、相邻单元的结论，用惯性替你完成确认。

### 2 档：Subagent 隔离聚焦

适用条件：该内容单元触达高、严重度高、逃逸高、落差可能大，或当前上下文正在污染这次斟酌，或当次斟酌产生的额外推导上下文，会污染后续内容单元的单独斟酌。

动作：对单个内容单元，创建一个 Subagent **单独聚焦推导**这个位置的内容单元在结论层面应当是什么。

使用 Subagent 时，载荷必须包含：

- 整体任务目标。
- 要确认的精确内容单元。
- 与该单元推导依赖的上下文内容
  如果前置上下文比较重，使得**主线心流中确实基本没有可能对推导判断形成干扰的内容**，额外走 fork 方式。
- 已知约束、立场、受众、不可说信息、非目标。
- 已知备选项、疑点或失败方式。
- 需要返回什么形态的结果，方便重新接回主线。


对于不同 agent runtime 的创建说明：
- 如果是 cursor：
  - 工具：`Task`；`subagent_type` 选 `generalPurpose`（通用推理/写作）或 `explore`（只读探库）。
  - 派发规则：**一个承重点一次 Task**；多个承重点可同条消息并行派发；禁止多个待确认单元合并进同一 Task。
  - 上下文传递：
    - **仅 prompt**（默认）：Subagent **不**继承当前对话或工具痕迹；与该单元推导依赖的上下文须写进 `prompt`。
    - **fork + prompt**：设 `resume: "self"` 以父 agent 整段对话历史为起点，再在 `prompt` 中额外指定本次探索方向与边界；
- 如果是 codex：
  - 主控工具路径：若当前可直接调用 `multi_agent_v1.spawn_agent`，用它创建 Subagent，并在 `commentary` channel 调用。
  - 递归/子 agent 路径：若当前是 Codex 子 agent，且看不到 `multi_agent_v1.spawn_agent`，不要判定 Codex 不支持递归；改用命令行创建下级 Codex 执行者：`codex exec --ephemeral`。
  - `multi_agent_v1.spawn_agent` 的 `agent_type`：
    - 默认省略或使用 `default`，用于概念推导、写作判断、研讨、反例攻击、框架辨析。
    - 只读探库问题用 `explorer`。
    - 明确要求改文件、实现或验证某个独立写入范围时用 `worker`，并在 prompt 中写清文件/模块 ownership。
  - `multi_agent_v1.spawn_agent` 的上下文传递：
    - **仅 prompt**（默认）：Subagent **不**继承当前对话或工具痕迹；与该单元推导依赖的上下文须写进 `message`(文字内容) 和 `items`(结构化内容)。
    - **fork + prompt**：设 `fork_context: true` 以父 agent 整段对话历史为起点，再在 `message`、`items` 中额外指定本次探索方向与边界；
  - `codex exec --ephemeral` 的调用要求：
    - 通过 shell/终端工具执行，工作目录用当前任务根目录或显式 `-C <dir>` 指定。
    - 用 stdin 或命令参数传入完整派发 prompt；prompt 必须包含下方载荷清单。
    - 建议使用 `-o <tmp-file>` 捕获下级执行者最终回复，便于主控读取并整合。
    - 示例形态：
      ```bash
      codex exec --ephemeral -C "$PWD" -o /tmp/attention-subagent.out - <<'PROMPT'
      你只聚焦一个承重点：...
      返回：单点封口摘要...
      PROMPT
      ```
    - 下级执行者必须只处理一个精确承重点；多个承重点要分别启动多次 `codex exec --ephemeral`。
  - 只有当 `multi_agent_v1.spawn_agent` 不可用，且 shell/终端中也无法执行 `codex exec --ephemeral` 时，才判定当前 agent runtime 无法创建 Subagent，并直接失败返回；不得降级成多方向摘要。
- 主 agent 收齐返回后专门整合，不得跳过。

隔离结果回来后，必须专门整合：检查它是否改变了前文假设、相邻内容单元或整体结构。

### 可审计隔离记录

当用户要求实验、验证、审计，或 Subagent 隔离聚焦本身会成为后续判断依据时，不能只保存最终摘要来证明隔离已经发生。主 agent 必须额外保存或回传以下记录：

- 上游方向发现或承重点识别的原始输出；若有候选淘汰，也保留淘汰理由。
- 每个 Subagent 的完整派发 prompt、创建命令或工具调用参数。
- 每个 Subagent 的输出路径、返回内容或最终消息。
- 每个 Subagent 是否 fork、继承上下文，或只接收 prompt。
- 整合前的关系判断记录：各单点之间是互斥、互补、层级、条件分流，还是待研究。

若这些记录缺失，只能说“产物形态符合单点封口”，不得声称已经可审计地证明“一承重点一 Subagent”“隔离输入干净”或“先封口后整合”。

## 多承重点与整合顺序

当一个任务包含多个互相依赖的承重内容单元时，执行顺序是：

```text
先单点封口，再关系判断，最后整合。
```

**单点封口**不是得出永远正确的结论，而是至少产出可被下游依赖的局部状态：

- 当前暂定判断。
- 支撑理由或证据。
- 强反例、失败方式或近敌。
- 适用边界。
- 仍未知的问题。
- 可交给整合层消费的短摘要。

在所有必要上游单元达到单点封口前，不得输出综合结论、统一框架、总判断或最终整合。可以输出的只有依赖图、待确认清单、局部结论或下一步隔离计划。

整合本身也是承重内容单元。只有在上游单点封口后，才判断分支之间是互相排斥、互补、层级包含、条件分流，还是揭示了新的上层问题。整合时不得为了获得漂亮统一感而回头压低任何单点的复杂度；如果某个单点尚未达标，整合层必须标明“此处暂不整合”。

如果上游材料很多导致整合上下文变重，不许因此砍掉上游注意力。正确动作是为每个单点生成可追溯的交接摘要，再把摘要作为整合层输入；必要时再对整合层单独使用 2 档隔离。

## 上下文污染触发器

出现以下情况时，升到 2 档：

- 当前已经顺出来的腔调、框架或措辞会锚定这个判断。
- 其它承重单元的推导过程正在主动影响这个单元。
- 正确确认需要展开很多草稿、备选、checklist 或约束推演，放在主线里会污染其它判断。
- 先写出一个顺口版本，会让错误方向显得自然，导致后面更难发现。
- 用户要求“逐个方向深入”“单独聚焦”“深度研讨”“辩论探索”，而当前回答形态正在变成多点摘要、并列概述或提前整合。
- 你发现自己为了维持全局结构、篇幅平衡、交付完整感或后续整合便利，正在压缩某个单点的探索质量。
- 内容涉及立场、信息透露、公文/对外表达、policy、不可逆行动、核心定义、隐藏状态。

## 高密度小输出

特别警惕「输出很短，但确认密度很高」的内容。

典型位置：

- 对外一句话：同时受立场、受众、透露粒度、引导意图、风险边界约束。
- 一个定义：后续设计、分析或争论都会沿它展开。
- 一个分类：决定接下来走哪个 workflow、skill、工具或评价标准。
- 一条建议：措辞变化会改变用户行动。
- 一句 prompt、policy、公文、issue comment、commit message、正式口径。

处理这类内容时，不要问「我能不能写出一句看起来没问题的话」。要问：「这句话要满足哪些条件，才配让后续工作建立在它上面？」

## 未知与通用结论

专属注意力之后仍只能写通用内容时，区分三种情况：

- **适用的通用性**：当前最可靠的结论就是通用原则，但它已经被连接到本任务的适用理由和边界。
- **缺少区分信息**：要更具体，需要当前没有的事实、约束或反馈。
- **当前无知**：agent 不知道如何可靠确认这个单元。

缺少信息或当前无知时，直接标明。不要为了显得具体而编造特殊性。

## 后置校验

交付前仍要做后置校验：一致性、格式、明显遗漏、事实冲突、通用结论是否连接到本任务。

但不要期待后置校验可靠发现最核心的问题：起始方向错、立场错、隐藏状态默认错、信息透露尺度错、措辞已经被流畅感锚定。这些问题要靠确认前的持续闸门处理。

## 对用户呈现

不要把全部注意力预算过程倒进最终回答。只有在有帮助或用户要求时才简要呈现。

需要呈现时，只说：

- 哪些承重内容单元改变了处理方式。
- 哪些用了原地局部聚焦，哪些用了隔离聚焦。
- 哪些仍是未知、缺少信息或显式假设。

如果用户问「你是否理解这个 skill」，必须回答定义、触发标准、动作分档、上下文污染触发器、后置校验边界。不要只解释背后的心智原因。
