# Info Purify

本 Skill 用于对已有信息集合做通用信息净化。它的主目标是完成信息结构改造：先确认消费路径，再诊断原结构问题，形成目标结构判断，自上而下重建可消费结构；随后用来源追溯、事实概念权威核查、关系显式化和独立复核防止失真。

如果发现只有改变底层设计、流程、契约、权责或产品规则才能优化的问题，本 Skill 只记录为升级候选，不在净化阶段直接裁决。

本 Skill 的审查边界分为两块：通用净化问题由 Skill 内置处理；实际 case 下需要单独考虑的专项问题，由 Skill 产出专项审查交接清单，交给使用者结合具体信息体系继续判断。

## 主控原则

- 主要目标是结构降熵，不是信息盘点。来源保全、引用说明、台账追溯是防失真机制，不能抢占主线。
- 先形成结构判断，再补强来源追溯。判断必须来自消费路径、原结构问题和可观察材料，不能凭空裁决。
- 保全来源不等于保留原始结构。原始材料留在追溯视图里，净化正文必须允许重排、合并、拆分、上提、下沉和隔离。
- 最终交付必须能说明“原结构被怎样改了”。如果只输出内容清单、分类表或摘要，本 Skill 视为失败。
- 不能在净化阶段改变事实承诺、执行行为、上下游契约、责任归属或设计取舍；这类问题进入升级候选。

## 设计意图与非目标

`info-purify` 不是文案优化器、摘要器、普通重写工具或信息清单生成器，而是信息体系降熵器。它处理的核心问题是：一组材料里混有事实、推断、历史背景、候选方案、待确认问题、隐含关系和未裁决规则时，如何在不篡改来源、不替代决策的前提下，把它重构成可消费、可追溯、可继续判断的信息结构。

它不以“写出一篇顺滑正文”为唯一终点，也不以“列出全部信息”为终点。净化正文是面向目标读者的消费视图；结构诊断、目标结构、旧新映射、结构改动清单、追溯台账、事实承诺表、核心概念表、权威来源表、关系表、专项审查交接清单和升级候选清单共同构成交付物。

本 Skill 不负责替业务、产品、架构或组织机制做最终裁决。凡是需要改变规则、契约、权责、流程、模块边界或产品取舍才能解决的问题，都必须进入升级候选或专项审查交接，而不是被写成正文结论。

## 执行强度

七个阶段是状态机，不代表每次都要产出最大体量文档。主控仍需按阶段建待办，但可以根据风险选择执行强度：

| 模式 | 适用场景 | 产物要求 |
| --- | --- | --- |
| 轻量模式 | 单篇文章、小范围材料、低风险整理 | 保留最小来源追溯、待确认/低置信说明、升级候选和结构校验 |
| 标准模式 | 多材料、多版本、需要后续复用 | 产出目标结构、旧新映射、净化正文和模板中的主要校验表 |
| 审计模式 | 规则、规范、架构、长期维护知识库等高风险体系 | 完整保留台账、权威来源、关系、专项交接、升级候选和独立复核 |

轻量模式不是跳过阶段，而是在每个阶段记录“本阶段低风险、采用最小产物”的判断依据。

## 工作流程（主控 Agent 调度模式）

第一步必须使用 `TodoWrite` 按以下阶段创建待办项：每个阶段一个待办，名称和顺序与本文档一致。

运行时前提：本文按支持 `TodoWrite` 和 Subagent `Task` 的 Cursor/Agent 环境编写。若当前环境没有这些工具，主控应以普通待办文本替代 `TodoWrite`，并自行执行阶段七复核；复核结论仍必须按阶段七标准输出。

1. 阶段一：任务基本性质确认
2. 阶段二：原结构诊断与重构判断
3. 阶段三：目标结构设计与自上而下梳理
4. 阶段四：旧新映射与来源追溯校准
5. 阶段五：事实概念权威关系与领域风险核查
6. 阶段六：产物生成与主控校验
7. 阶段七：独立复核与交付收束

## 全局约束

- 不先润色，不先摘要，不先删减原始材料。
- 不把来源台账当成主要交付；台账必须服务结构重构和追溯校验。
- 不沿用原结构作为默认答案；必须判断哪些结构应保留、合并、拆分、移动、降级、隔离或升级。
- 不把推断、待确认、候选方案、开放问题写成既定事实。
- 不替代业务、产品、架构或组织决策。
- 升级候选是核心安全边界，不是附录；遇到需要升级才能优化的问题，必须写入升级候选清单。
- 未知类型材料必须先识别消费路径和专用约束，再生成临时检查项。
- 必须区分“Skill 已处理的通用问题”和“使用者需要继续判断的 case 专项问题”。
- 最终正文必须是目标读者可直接消费的新结构；只有清单、索引或分析报告时不得判定完成。

### 阶段一：任务基本性质确认

**执行模式**：主控执行

**输入**：
- 用户原始需求。
- 待净化信息集合。
- 用户给出的不可改变约束、目标读者、输出要求。
- 上下文参考：[信息净化方法](references/info-purify-method.md)

**目标**：把任务从“材料待处理”变为“净化目标、消费路径、不可改变约束、输入盲区已明确”。

**产出**：任务身份说明、主要目标、消费路径说明、不可改变约束清单、输入盲区清单。

实际执行到该步骤时，必须阅读 [workflow/01-scope-and-consumption.md](workflow/01-scope-and-consumption.md)，并按其中指导完成。

### 阶段二：原结构诊断与重构判断

**执行模式**：主控执行

**输入**：
- 阶段一产出的任务身份说明和消费路径说明。
- 原始信息集合。

**目标**：把任务从“材料待整理”变为“原结构问题、重构目标和结构改造策略已明确”。

**产出**：原结构诊断、结构问题清单、目标结构判断、改造策略、不可直接净化的问题初筛。

实际执行到该步骤时，必须阅读 [workflow/02-structure-diagnosis.md](workflow/02-structure-diagnosis.md)，并按其中指导完成。

### 阶段三：目标结构设计与自上而下梳理

**执行模式**：主控执行

**输入**：
- 阶段二产出的原结构诊断和目标结构判断。
- 原始信息集合。
- 消费路径说明。

**目标**：把信息状态从“知道哪里乱”变为“已有可替换原结构的目标结构草案，且信息已按新结构归位”。

**产出**：目标结构草案、结构层级说明、结构改动清单、无归属/待确认信息、升级候选初版。

实际执行到该步骤时，必须阅读 [workflow/03-target-structure.md](workflow/03-target-structure.md)，并按其中指导完成。

### 阶段四：旧新映射与来源追溯校准

**执行模式**：主控执行

**输入**：
- 目标结构草案。
- 结构改动清单。
- 原始信息集合。
- 消费路径说明。

**目标**：把信息状态从“目标结构草案”变为“每个强事实、强规则和关键结构改动都能回溯来源，旧信息去向已说明”。

**产出**：来源清单、信息块台账、旧新映射表、来源缺失记录、追溯风险、结构草案修正记录。

实际执行到该步骤时，必须阅读 [workflow/04-mapping-and-trace.md](workflow/04-mapping-and-trace.md)，并按其中指导完成。

### 阶段五：事实概念权威关系与领域风险核查

**执行模式**：主控执行

**输入**：
- 目标结构草案。
- 旧新映射表。
- 信息块台账。
- 信息类型识别结果。
- 消费路径说明。
- 领域正式规范（如有）。

**目标**：把信息状态从“结构已重建且有追溯”变为“事实、概念、权威、关系和领域风险已校验，不能由净化解决的问题已交接”。

**产出**：事实承诺表、状态分层表、核心概念表、权威来源表、关系表、事实冲突清单、概念问题清单、领域检查项、专项审查交接清单、升级候选增量。

实际执行到该步骤时，必须阅读 [workflow/05-validation-and-domain-checks.md](workflow/05-validation-and-domain-checks.md)，并按其中指导完成。

### 阶段六：产物生成与主控校验

**执行模式**：主控执行

**输入**：
- 阶段一至五全部定稿产物。
- [净化信息包模板](templates/purified-info-package.md)

**目标**：生成以最终正文为产物本体、以其他内容为净化日志记录的完整信息包，并完成主控可执行的结构校验。

**产出**：产物本体、净化日志记录、升级候选记录、结构校验记录、剩余风险。

实际执行到该步骤时，必须阅读 [workflow/06-output-and-validation.md](workflow/06-output-and-validation.md)，并按其中指导完成。

### 阶段七：独立复核与交付收束

**执行模式**：Subagent 委派

**输入**：
- 用户原始需求。
- 阶段一产出的任务基本性质。
- 阶段六产出的完整净化信息包。
- 复核标准：[workflow/07-review-and-close.md](workflow/07-review-and-close.md)

**目标**：把交付状态从“主控自检通过”变为“已由独立质量关卡复核，失败项已回退修正或风险已显式记录”。

**产出**：独立复核报告、回退修正记录、最终交付说明。

实际执行到该步骤时，必须阅读 [workflow/07-review-and-close.md](workflow/07-review-and-close.md)，并按其中指导完成。

**通过 Task 工具调用 Subagent 执行，配置**：
- `subagent_type`: `generalPurpose`
- `readonly`: true

**派发 Prompt**：

```markdown
请作为独立质量关卡，复核本次信息净化交付物。你只能根据下列输入判断，不要补写正文。

## 用户原始需求

${用户原始需求}

## 任务基本性质

${阶段一产物}

## 净化交付物

${阶段六产物}

## 复核标准

逐项检查：
1. 是否形成了原结构诊断和目标结构判断，而不是只列信息内容。
2. 净化正文是否是可直接消费的新结构，是否对原结构做了实质性重排、合并、拆分、上提、下沉或隔离。
3. 是否提供旧结构到新结构的映射，并说明每类原始信息的去向。
4. 原始信息是否已保全或说明无法保全的原因。
5. 净化正文中的强事实和强规则是否能回溯来源。
6. 事实、推断、假设、待确认、历史背景、候选方案是否混写。
7. 核心概念是否具备定义、边界、非范围、生产方、消费方、下一步判断。
8. 权威来源和责任边界是否明确。
9. 关系是否已显式化，而不是只有并列段落。
10. 领域特定检查项是否已应用，未知类型是否有临时检查项。
11. Skill 已处理的通用问题和使用者需要继续判断的 case 专项问题是否已分开记录。
12. 需要升级才能优化的问题是否进入升级候选清单。
13. 目标消费路径下，读者是否能直接使用结果。
14. 剩余风险和补充信息请求是否记录。

## 返回格式

先列失败项和风险，再列通过项。每个失败项必须说明对应交付物位置、问题原因、建议回退阶段。
```

主控等待 Subagent 返回后：
- 若存在失败项，回退到对应阶段修正。
- 若只存在剩余风险，更新风险说明。
- 若全部通过，输出最终交付说明。

## 最小交付要求

即使是轻量任务，也必须输出：

- 产物本体（最终正文或净化后结构）。
- 原结构诊断和目标结构判断。
- 旧结构到新结构的映射或结构改动说明。
- 来源/事实追溯方式。
- 待确认和低置信说明。
- 专项审查交接清单。
- 升级候选清单。
- 结构校验结论。
