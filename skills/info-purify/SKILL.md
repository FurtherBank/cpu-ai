# Info Purify

本 Skill 用于对已有信息集合做通用信息净化。它先保全来源，再拆块、判定事实与概念、归位权威来源、显式化关系、补齐领域检查项，最后输出净化正文、配套台账、升级候选和结构校验结论。

如果发现只有改变底层设计、流程、契约、权责或产品规则才能优化的问题，本 Skill 只记录为升级候选，不在净化阶段直接裁决。

本 Skill 的审查边界分为两块：通用净化问题由 Skill 内置处理；实际 case 下需要单独考虑的专项问题，由 Skill 产出专项审查交接清单，交给使用者结合具体信息体系继续判断。

## 设计意图与非目标

`info-purify` 不是文案优化器、摘要器或普通重写工具，而是信息体系降熵器。它处理的核心问题是：一组材料里混有事实、推断、历史背景、候选方案、待确认问题、隐含关系和未裁决规则时，如何在不篡改来源、不替代决策的前提下，把它转成可消费、可追溯、可继续判断的信息包。

它不以“写出一篇顺滑正文”为唯一终点。净化正文只是面向目标读者的消费视图；信息块台账、事实承诺表、核心概念表、权威来源表、关系表、专项审查交接清单和升级候选清单同样是交付物的一部分。

本 Skill 不负责替业务、产品、架构或组织机制做最终裁决。凡是需要改变规则、契约、权责、流程、模块边界或产品取舍才能解决的问题，都必须进入升级候选或专项审查交接，而不是被写成正文结论。

## 执行强度

七个阶段是状态机，不代表每次都要产出最大体量文档。主控仍需按阶段建待办，但可以根据风险选择执行强度：

| 模式 | 适用场景 | 产物要求 |
| --- | --- | --- |
| 轻量模式 | 单篇文章、小范围材料、低风险整理 | 保留最小来源追溯、待确认/低置信说明、升级候选和结构校验 |
| 标准模式 | 多材料、多版本、需要后续复用 | 产出净化信息包模板中的主要表格和正文 |
| 审计模式 | 规则、规范、架构、长期维护知识库等高风险体系 | 完整保留台账、权威来源、关系、专项交接、升级候选和独立复核 |

轻量模式不是跳过阶段，而是在每个阶段记录“本阶段低风险、采用最小产物”的判断依据。

## 工作流程（主控 Agent 调度模式）

第一步必须使用 `TodoWrite` 按以下阶段创建待办项：每个阶段一个待办，名称和顺序与本文档一致。

运行时前提：本文按支持 `TodoWrite` 和 Subagent `Task` 的 Cursor/Agent 环境编写。若当前环境没有这些工具，主控应以普通待办文本替代 `TodoWrite`，并自行执行阶段七复核；复核结论仍必须按阶段七标准输出。

1. 阶段一：任务定界与消费路径确认
2. 阶段二：来源保全与信息块台账
3. 阶段三：事实概念权威核查
4. 阶段四：结构净化与关系显式化
5. 阶段五：领域风险识别与专项审查交接
6. 阶段六：产物生成与主控校验
7. 阶段七：独立复核与交付收束

## 全局约束

- 不先润色，不先摘要，不先删减原始材料。
- 不把推断、待确认、候选方案、开放问题写成既定事实。
- 不替代业务、产品、架构或组织决策。
- 升级候选是核心安全边界，不是附录；遇到需要升级才能优化的问题，必须写入升级候选清单。
- 未知类型材料必须先识别消费路径和专用约束，再生成临时检查项。
- 必须区分“Skill 已处理的通用问题”和“使用者需要继续判断的 case 专项问题”。

### 阶段一：任务定界与消费路径确认

**执行模式**：主控执行

**输入**：
- 用户原始需求。
- 待净化信息集合。
- 用户给出的不可改变约束、目标读者、输出要求。
- 上下文参考：[信息净化方法](references/info-purify-method.md)

**目标**：把任务从“材料待处理”变为“净化目标、消费路径、不可改变约束、输入盲区已明确”。

**产出**：任务身份说明、消费路径说明、不可改变约束清单、输入盲区清单。

实际执行到该步骤时，必须阅读 [workflow/01-scope-and-consumption.md](workflow/01-scope-and-consumption.md)，并按其中指导完成。

### 阶段二：来源保全与信息块台账

**执行模式**：主控执行

**输入**：
- 阶段一产出的任务身份说明和消费路径说明。
- 原始信息集合。
- 输出模板：[净化信息包模板](templates/purified-info-package.md)

**目标**：把原始材料从“混合输入”变为“可追溯、可分类、可检查的信息块台账”。

**产出**：来源清单、信息块台账、来源缺失记录、初始类型标注。

实际执行到该步骤时，必须阅读 [workflow/02-source-and-blocks.md](workflow/02-source-and-blocks.md)，并按其中指导完成。

### 阶段三：事实概念权威核查

**执行模式**：主控执行

**输入**：
- 信息块台账。
- 任务不可改变约束。
- 已知权威来源。

**目标**：把信息状态从“已切块但未判定”变为“事实、推断、假设、历史背景、候选方案、待确认、核心概念和权威来源已分层”。

**产出**：事实承诺表、状态分层表、核心概念表、权威来源表、事实冲突清单、概念问题清单。

实际执行到该步骤时，必须阅读 [workflow/03-facts-concepts-authority.md](workflow/03-facts-concepts-authority.md)，并按其中指导完成。

### 阶段四：结构净化与关系显式化

**执行模式**：主控执行

**输入**：
- 阶段三产出的事实承诺表、核心概念表、权威来源表。
- 信息块台账。
- 消费路径说明。

**目标**：把信息状态从“事实和概念已分层”变为“可按消费路径阅读的净化结构，且生产、消费、依赖、冲突、前后置关系已显式化”。

**产出**：净化结构草案、关系表、无归属信息清单、待确认项清单、升级候选清单初版。

实际执行到该步骤时，必须阅读 [workflow/04-structure-and-relations.md](workflow/04-structure-and-relations.md)，并按其中指导完成。

### 阶段五：领域风险识别与专项审查交接

**执行模式**：主控执行

**输入**：
- 净化结构草案。
- 信息类型识别结果。
- 消费路径说明。
- 领域正式规范（如有）。

**目标**：把信息状态从“通用结构可用”变为“已覆盖具体信息类型的专用风险，未知类型已生成临时检查项，case 专项审查范围已交接”。

**产出**：领域检查项、类型识别说明、临时专用约束、专项审查交接清单、补充信息请求。

实际执行到该步骤时，必须阅读 [workflow/05-domain-checks.md](workflow/05-domain-checks.md)，并按其中指导完成。

### 阶段六：产物生成与主控校验

**执行模式**：主控执行

**输入**：
- 阶段一至五全部定稿产物。
- [净化信息包模板](templates/purified-info-package.md)

**目标**：生成完整净化信息包，并完成主控可执行的结构校验。

**产出**：净化后正文、配套台账、升级候选清单、结构校验清单、剩余风险。

实际执行到该步骤时，必须阅读 [workflow/06-output-and-validation.md](workflow/06-output-and-validation.md)，并按其中指导完成。

### 阶段七：独立复核与交付收束

**执行模式**：Subagent 委派

**输入**：
- 用户原始需求。
- 阶段一产出的任务身份与消费路径。
- 阶段六产出的完整净化信息包。
- 复核标准：[workflow/07-review-and-close.md](workflow/07-review-and-close.md)

**目标**：把交付状态从“主控自检通过”变为“已由独立质量关卡复核，失败项已回退修正或风险已显式记录”。

**产出**：独立复核报告、回退修正记录、最终交付说明。

实际执行到该步骤时，必须阅读 [workflow/07-review-and-close.md](workflow/07-review-and-close.md)，并按其中指导完成。

**通过 Task 工具调用 Subagent 执行，配置**：
- `subagent_type`: `generalPurpose`
- `readonly`: true

**派发 Prompt**：

```markdown
请作为独立质量关卡，复核本次信息净化交付物。你只能根据下列输入判断，不要补写正文。

## 用户原始需求

${用户原始需求}

## 任务身份与消费路径

${阶段一产物}

## 净化交付物

${阶段六产物}

## 复核标准

逐项检查：
1. 原始信息是否已保全或说明无法保全的原因。
2. 净化正文中的强事实和强规则是否能回溯来源。
3. 事实、推断、假设、待确认、历史背景、候选方案是否混写。
4. 核心概念是否具备定义、边界、非范围、生产方、消费方、下一步判断。
5. 权威来源和责任边界是否明确。
6. 关系是否已显式化，而不是只有并列段落。
7. 领域特定检查项是否已应用，未知类型是否有临时检查项。
8. Skill 已处理的通用问题和使用者需要继续判断的 case 专项问题是否已分开记录。
9. 需要升级才能优化的问题是否进入升级候选清单。
10. 目标消费路径下，读者是否能直接使用结果。
11. 剩余风险和补充信息请求是否记录。

## 返回格式

先列失败项和风险，再列通过项。每个失败项必须说明对应交付物位置、问题原因、建议回退阶段。
```

主控等待 Subagent 返回后：
- 若存在失败项，回退到对应阶段修正。
- 若只存在剩余风险，更新风险说明。
- 若全部通过，输出最终交付说明。

## 最小交付要求

即使是轻量任务，也必须输出：

- 净化后正文或净化后结构。
- 来源/事实追溯方式。
- 待确认和低置信说明。
- 专项审查交接清单。
- 升级候选清单。
- 结构校验结论。
