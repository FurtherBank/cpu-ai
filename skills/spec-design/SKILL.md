# 背景

该项目使用了一套 Spec-Driven 的迭代模式。其中该技能负责的环节是**讨论可行方案**。

整个迭代工作流的设计信息参考 `reference/spec-driven.md`。

# 第一轮沟通

## 上下文资源盘点

根据对齐后的任务改造目标，确认需要参考的上下文内容。

1. 我们通过对齐后的任务改造目标，先确定需要具体参考哪些方面的上下文内容。如：

- 当前版本方案设计
- 梳理相关依赖关系，如：
  - 上游：我们的方案强依赖了哪些模块/服务
  - 下游：我们的方案被哪些模块/服务依赖
  - 数据：我们的方案涉及哪些数据库、缓存、消息队列
- 代码中所涉及模块的技术债务，包括：
  - 扫描代码中的 TODO、FIXME
  - 复杂的“黑魔法”实现逻辑
  - 其它具有“坏味道”可能有坑的逻辑。具体可参考：[代码风险信号](/reference/代码风险信号.md)

2. 确定好获取这些上下文内容的方式，如：

- 直接阅读对应文件目录下的内容，或基于这些内容分析得到结论
- 通过特定 MCP 获取
- 访问或搜索对应网站获取
- 需要用户提供获取方式

其中，如果发现无法通过代码获取的外部依赖信息，必须暂停并询问用户，生成“待确认信息清单”，一次性问清。

## 方案构思

**动作**: 提出初步技术选型或改造路径。

**注意事项**:

- 对于关键技术点选型（如：新旧库替换兼容性、特定 SQL 语法支持、高并发下的锁机制）的可行性，如果你难以直接判断，不妨尝试通过 `workflow/poc.md` 的流程进行直接的实验验证。

## 详细方案设计

**动作**: 调用 `tech-design`技能，根据以上信息，构建完整可行的改造方案设计。
