## 广义接口的概念澄清

> **痛点现象 1：**我只靠基础模块直接给出的接口文档，跑不起来
>
> - 我拿到了这个模块的 API 文档，按照他的入参正确传进去了，发现跑不起来
> - 找人问半天才发现，需要在一个满足特定上下文的生命周期里面调用才行
>
> **痛点现象 2**：我使用基础模块内部接口实现功能，基础模块发了新版本，挂了
>
> - 业务方发现，我想用这个基础模块，直接通过最上层的接口，还没法完整实现需要的功能，这时我 hack 了内部的代码实现，用了内部的一个逻辑类，才成功的把这个功能做了出来
> - 然后，模块内部重构修改了一个方法，结果导致依赖该方法的业务方挂了——因为业务方引用了本该是私有的内部代码。
>
> **问题本质**：基础模块侧和业务侧对于基础模块「哪些内容是外部引用的接口」做不到自动+充分对齐，且基础模块侧可能都**没有这个意识**。

为了解决这个本质问题，我们在日常研发协作时，需要对接口是什么，有一个完备和准确的定义。

这里给出了他的“最终答案”：**“冰山模型”**

### 水面之上(Public)

水面之上的内容，其实就是基础模块侧和业务侧需要关注的**接口**。

不过这里要对接口的概念做澄清：模块之间**所有涉及到两边交互，必要关注的内容**都属于接口

- 「必要关注」说明：不关注、漏关注会引发 bug 等问题；  
  或者说我接这个模块，从开始接到上线修复到没 bug，**会关注到的属于基础模块的所有相关信息**
  - 不仅限于代码中显式写为 interface, api 等典型接口形式的内容
  - 也包括**原本需要关注但没有被显示声明并传达到的「隐式信息」**

基础模块开发者在设计模块时，一定要想清楚，哪些内容「应当」「不应当」在“水面之上”；  
并在实现中，通过必要的标记、说明，将接口的边界完整定义清楚。

- 标准：可以让业务方拿到模块，没有和你沟通的情况下，对模块中的任何内容(包括细节性内容)，问是否是在“水面之上”，能**毫无疑虑得出和你完全相同的答案**。

规范：**在“水面之上”的内容，需要对业务方承诺「向后兼容」，一定时间内不会有破坏性修改**；且破坏性修改需走严格的 RFC 流程。

### 水面之下(Internal)：

非水面之上的所有其它代码内容。  
这些内容，基础模块团队有权随时修改内部实现，并**不提前通知**业务方。  
业务层**绝对禁止直接引用“水面之下”的非 ts 类型代码**；

- 业务层如果发现必须要 import “水面之下”的代码才能实现功能解决问题，这是**基础模块接口设计的问题**，没划分好什么东西应该在“水面之上”。  
  需要让基础模块的同学支持修改。
- 此外的情况，业务层如果通过 import 了深层路径“水面之下”的代码，**后果自负**。  
  基础模块团队有权随时修改内部实现而不通知业务方，由此引发的线上故障由业务方承担。

### 代码接口的几种标准形式

后端不同系统之间，常见的接口形式都是 服务 API，不同 API 会通过 oneapi 等工具，规范其出入参。

但是前端不同模块之间，代码 export 的各种形式的内容，都有可能属于接口**。**

将代码内容作为接口导出，可以参考典型的几种接口形式：

1. 配置(Config)：静态配置入参(如 react props、api params)
2. 接口(Interface)：属性满足特定接口定义的对象/出入参满足特定类型的函数实现
3. 回调(Hook)：预定义生命周期钩子、回调（onCreate/onBefore/onSuccess）
4. 适配器实现(Adapter Impl)：相同功能，要在不同 target 环境下用不同的实际实现，要由**基础技术团队**负责定义**通用接口形式**，**上层业务团队**负责编写在实际环境中**满足接口定义的 adapter 专用实现**。
5. 插件(Plugin)：综合了以上机制的插件体系实现。
6. 业务上层封装：业务侧自己实现封装层，将基础技术模块的通用输出转为本业务形态。
