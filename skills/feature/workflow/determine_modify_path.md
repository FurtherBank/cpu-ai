# 确定代码修改路径 执行指导

> 本文件是主控 Agent 在阶段二中的详细执行指导。

## 步骤 1：三方库/外部依赖调研 (如涉及)

**重要**：当特性需要通过三方库实现时，必须先进行深度调研，确保方案的可行性和稳定性。

**深度调研过程**：

1. **识别需求信息**：
   - 明确实现该特性需要哪些三方库提供哪些核心能力（如：导出 Excel 需要支持多 Sheet、样式设置、大数据量处理等）。
   - 确定我需要从三方库中获取哪些具体信息。

2. **广义接口分析 (参考"冰山模型")**：
   - 参考 [广义接口定义.md](../references/广义接口定义.md)，识别三方库的"水面之上"内容：
     - **显式接口**：API 文档、配置项 (Config)、回调 (Hook)、类型定义 (Interface)。
     - **隐式约束**：生命周期要求（如必须在 DOM 挂载后调用）、上下文依赖、并发限制、环境要求（Node.js vs Browser）。

3. **风险与可行性评估**：
   - 搜索官方文档、社区 Issue 和最佳实践，验证库的功能是否能完全覆盖需求。
   - 评估潜在风险：包体积影响、性能瓶颈、维护活跃度、协议限制、破坏性更新历史。

4. **确定方案迭代**：
   - 基于调研结论修正初始想法，形成确定有效的集成方案。

## 步骤 2：确定修改路径

确定具体的代码修改路径。即：具体要改哪些代码，怎么改。

输出内容必须满足以下规范：

### 规范 1：对象指代需具体

**原则**：所有涉及"关系"的描述，都应指定能够直接指代到本体的具体目标事物。

**示例对比**：

❌：

- "在页面左上角加一个 logo"
- "修改用户信息接口"
- "优化列表渲染性能"

✅：

- "在 `TaskDetailPage` 组件的 `NavBar` 左侧添加产品 logo"
- "修改 `app/modules/user/controller/user.ts` 中的 `updateUserInfo` 方法"
- "优化 `UserListPage` 组件中 `Table` 组件的渲染，使用 `React.memo` 包装列表项组件"

### 规范 2：内容只是修改路径

**原则**：只描述修改路径，不描述最终编码结果。路径应包含：

- 具体文件路径
- 具体组件/函数/类名
- 具体修改位置（如：在哪个函数内、在哪个组件中）
- 修改类型（新增/修改/删除）

**示例对比**：

❌：

```
使用以下冒泡排序算法实现商品价格排序：
[完整的冒泡排序代码实现示例，而且这个示例数据结构是 int，项目实际数据结构，因为设计原因，是 string，造成误导]
```

✅：

```
通过冒泡排序算法实现商品价格排序：
- 注意：项目中商品价格为 `"¥XX.XX"` 格式的 string，需要先解析为数字再排序
- 排序后保持原 string 格式返回
```

## 代码修改路径

- `userId` 格式和定义参考：[具体类型定义文件路径]
