# Feature Task 编码与测试执行规范

> 本文件是 Coder Subagent 和 Test Subagent 的执行规范。请严格遵守以下要求。

## 编码核心规范

### 关键点注释 (Intent Annotation)

对于直接实现完成标准中关键点的代码，请在实现的位置增加一条以下格式的注释：
`// 👁️ ${关键点描述，例：导出过程加载态处理}`

### 可读性调试日志 (Readable Debug Logging)

**适用场景**：对于流程精密（如复杂状态流转）、复杂度高（如多层数据处理）、核心业务（如支付/权限）的逻辑实现。

优先使用当前项目中已有的日志工具（如 `logger.info`）。如果项目没有统一日志工具，且允许使用 `console`，在编写复杂逻辑时，使用以下 Emoji 前缀增强日志可读性：

- 🚀 `[Start]`: 核心流程/函数开始。必须包含关键入参。
- 🔑 `[Key]`: 关键节点或状态发生了重要变更。必须包含变更后的状态值。
- ⚡️ `[Step]`: 复杂计算或逻辑的中间步骤。
- 🔌 `[API]`: 外部接口调用或三方库交互前后。
- ✅ `[Success]`: 流程成功结束。包含最终返回值。
- ❌ `[Error]`: 捕获异常或进入错误分支。包含 Error 对象或具体原因。

## 单元测试 (可选)

在完成业务源代码编写后，请主动查阅项目中的 `AGENTS.md` 文件，确认是否有针对当前变更类型的单测规范。
- 如果有：请按规范编写并运行单测。
- 如果没有或明确不需要：请跳过。

## 交付自测闭环

Coder 在认为代码（及单测）就绪后，**必须使用 `Task` 工具（`subagent_type="generalPurpose"`）唤起一个 Test Subagent**。

- 派发任务：将主控传递给你的【自测用例】*完整内容*转发给 Test Subagent，要求其独立执行。
- 测试校验动作示例：运行 lint 检查代码、编写(服务端)单测并验证、直接 curl 确认接口返回值、直接打开浏览器进入网页实际操作测试、编写外部测试脚本进行实际调用测试。
- 修复闭环：如果 Test Subagent 返回失败反馈，Coder 必须自行修复代码，并重新唤起 Test Subagent 进行验证，直到自测通过。

## 代码风险自审

在自测通过后，Coder 必须按照 `./feature_review.md` 中的要求，对本次修改进行代码风险自审，并生成结构化的风险报告。
