# 企业级功能特性开发

在企业级项目中开发功能特性，需要先确定"怎样才算是任务真正且正确的完成"，确定代码修改路径，并在编码过程中通过高可读性的日志确保逻辑正确性，最后识别风险并输出记录。

## 工作流程 (主控 Agent 调度模式)

为了避免单次任务中上下文过载和遗忘，本技能采用**主控 Agent + 垂直领域 Subagent** 的协作模式。
作为主控 Agent，你的核心职责是**把控进度、维护核心状态（DoD、修改路径），并在合适的时机将精准裁剪后的上下文派发给专门的 Subagent 执行**。

> **任务阶段规划**：请使用待办列表工具（TodoWrite）按照本技能下方定义的工作流阶段来规划任务，而非自行拆分步骤。每个阶段对应一个待办项，阶段名称和顺序必须与本文档一致。

### 阶段一：确定完成标准 (主控 Agent 负责)


在开始编码前，主控 Agent 必须先仔细深度思考，明确真正实际的完成标准（即：怎样才算是任务真正且正确的完成）。

**执行方式**：主控执行
**指导文件**：[workflow/determine_dod.md](workflow/determine_dod.md)

**产出**：完成标准（关键实现点 + 自测用例），格式见指导文件中的产出格式模板

执行时读取上述指导文件，按其中的流程完成任务。

### 阶段二：确定代码修改路径 (主控 Agent 负责)

**执行方式**：主控执行
**指导文件**：[workflow/determine_modify_path.md](workflow/determine_modify_path.md)

在明确完成标准后，确定具体的代码修改路径。即：具体要改哪些代码，怎么改。

**输入**：
- `完成标准`：阶段一的产出物
- 上下文参考：如涉及三方库，参考 [references/广义接口定义.md](references/广义接口定义.md)

**产出**：结构化的代码修改路径列表，格式见指导文件中的产出格式模板

执行时读取上述指导文件，按其中的流程完成任务。

### 阶段三：编码实现、测试与风险自审 (委派给 Coder Subagent)

主控 Agent 明确了完成标准和修改路径后，**绝对不要自己写代码**。使用 `Task` 工具（`subagent_type="generalPurpose"`）唤起 Coder Subagent 执行。

**派发 Prompt**：
```markdown
请按照以下任务需求信息，严格遵循 `.cursor/skills/feature/workflow/feature_task.md` 中的要求，完成编码规范、测试流程以及代码风险自审。
完成后请返回修改的文件列表、测试结论以及代码风险报告。
## 任务原始需求
${完整的用户原始需求}

## 完成标准
【完成标准】: ${阶段一输出的【完成标准】**完整内容**}

## 自测用例
【自测用例】: ${阶段一输出的【自测用例】**完整内容**}

## 代码修改路径
【代码修改路径】: ${阶段二输出的【代码修改路径】**完整内容**}

```

*主控 Agent 等待 Coder Subagent 返回"修改完成的文件列表、测试结论及风险报告"后，直接进入最后阶段。*

### 最后：输出功能记录 (主控 Agent 负责)

在 Subagent 完成任务后，主控 Agent 负责汇总各阶段的产出，将修改时间戳、完成标准、自测用例、测试执行情况、代码风险信号输出到当前正在开发的【项目/仓库根目录】的 `docs/feature_record/${修改标题}.md` 文件中（请勿输出到整个工作区的根目录）。

**执行步骤**：

1. **确定修改标题**：
   - 根据用户需求或功能特性，确定一个简洁明确的标题
   - 标题应能清晰表达本次修改的核心内容

2. **收集记录信息**：
   - 修改时间戳：当前时间（格式：YYYY-MM-DD HH:mm:ss）
   - 完成标准：从阶段一输出的完成标准
   - 自测用例：从阶段一输出的自测用例
   - 测试执行情况：从阶段三 Coder Subagent 返回的综合测试结论
   - 代码风险信号：从阶段三 Coder Subagent 返回的风险报告

3. **创建记录文件**：
   - 文件路径：当前项目/仓库根目录下的 `docs/feature_record/${修改标题}.md`
   - 如果 `docs/feature_record` 目录不存在，需要先创建该目录
   - 文件内容应包含上述所有信息，格式清晰易读

**输出格式**：

## 注意事项

1. **严格遵守阶段顺序**：请严格遵守以下约定顺序，**禁止**跳过任何阶段：
   - 阶段一：确定完成标准（包括其中的自测用例）
   - 阶段二：确定代码修改路径
   - 阶段三：编码实现、测试与风险自审
   - 最后：输出功能记录
2. **完成标准要可验证**：每个标准都应有明确的验证方法。
3. **注释&日志即文档**：在实现复杂逻辑时，优秀的注释和日志不仅用于调试，也是逻辑流程的实时文档，请认真编写。
4. **参考项目规范**：修改路径应符合项目的代码组织规范和命名规范。
5. **风险信号识别要全面**：在阶段三中，应在完成编码和测试后，仔细回顾修改过程中接触到的所有代码，全面识别可能存在的风险信号。
6. **功能记录要及时**：在完成所有阶段后，应及时输出功能记录到当前项目/仓库根目录下的 `docs/feature_record` 目录中。
